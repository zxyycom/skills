# Agent Instructions

## 仓库边界

1. 本仓库是 Codex skills 主仓库, 通过 Git submodule 组织多个独立 skill 子仓库。
2. 主仓库维护跨 skill 的共享能力, 包括仓库说明、维护约定、决策记录、工具链、自动化、聚合打包发布和 submodule 指针。
3. 子仓库维护各自 skill 本体、随 skill 分发的文件和该子仓库自身的独立发布入口。
4. 子仓库不复制主仓库共享工具链、项目级维护文档或跨 skill 流程说明。
5. 长期文档只写当前主仓库和 submodule 内可维护的信息; 仓库外路径、临时来源、迁移来源和一次性调查上下文不进入长期文档。

## 内容 Owner

1. Skill 行为、触发条件、读取策略、执行流程、引用文件、边界和验收标准由对应子仓库的 skill 本体承接。
2. 跨 skill 共享的校验、打包、聚合发布、依赖入口、CI 和自动化由主仓库承接。
3. 子仓库独立发布只承接该子仓库 skill 内容的交付, 不扩展为共享工具链。
4. 脚本、安装、校验、打包、CI 和发布细节由 `docs/tooling.md` 承接。
5. 决策记录格式、门槛、状态关系和更新流程由 `docs/decisions/decision-record-rules.md` 承接; 决策清单由 `docs/decisions/decision-record-index.md` 承接。
6. 同一判断只在最稳定的 owner 位置完整解释; 非 owner 位置只保留摘要、触发条件或引用。

## 变更流程

1. 修改前先判断当前所在仓库。主仓库状态不能代表子仓库状态, 子仓库状态也不能代表主仓库指针是否已更新。
2. 修改子仓库内容前, 先在该子仓库检查状态并识别已有未提交改动。
3. 更新子仓库后, 先处理子仓库提交和推送, 再回到主仓库处理 submodule 指针。
4. 只修改主仓库文档、脚本、CI 或配置时, 不进入子仓库制造变更。
5. 需要确认整体状态时, 分别检查主仓库和所有 submodule。
6. 打包产物、依赖目录和 workflow 运行产物不作为长期源文件提交。

## 维护原则

1. 保持改动范围与用户请求对齐; 不为了整理文档、统一风格或顺手清理而重写无关内容。
2. 新增长期规则前先判断 owner; 只有当前文件适合承接项目级 agent 协作约定时, 才修改 `AGENTS.md`。
3. 新增或调整共享脚本时优先做成跨 skill 通用能力; skill 专属规则应留在对应 skill 或其子仓库。
4. 新增 skill 子仓库时, 同步确认主仓库聚合入口和子仓库独立发布入口仍覆盖该 skill。
5. 正文主要使用中文; 除用户要求、目标文件已有语言要求或代码/API 名称外, 新增维护文本保持中文。

## 决策记录

1. 决策记录只记录后续维护需要回放原因的判断, 不作为变更日志。
2. Skill 行为、触发条件、读取策略、引用 owner、规则边界或验收标准发生变化时, 按决策记录规则判断是否达到记录门槛。
3. 项目级决策只在改变长期维护契约、目录边界、自动化交付方式或跨文件 owner 时记录。
4. 单个 skill 专属决策要写明适用 skill 和当前子仓库路径, 避免写成全部 skill 的通用规则。
5. 普通文字修正、格式调整、链接修复和按既有规则执行的一次性细节不记录。

## 验证与交付

1. 修改主仓库维护文档、脚本、CI 或 submodule 指针后, 优先运行主仓库整体检查。
2. 只改某个子仓库 skill 本体时, 根据该子仓库内容选择最小验证; 回到主仓库后确认 submodule 指针状态。
3. 提交或汇报前说明实际运行过的验证; 未运行验证时直接说明原因。

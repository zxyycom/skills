# Skills 仓库编码规范

本文档是 `scripts/` 下实现代码通用质量规则的 owner。它帮助实现者和 reviewer 判断代码应放在哪里、如何处理边界、如何表达领域含义，以及需要什么验证证据。

项目级协作和内容归属由 [`AGENTS.md`](../AGENTS.md) 定义，各 skill 的行为契约由对应的 `skills/<skill-name>/SKILL.md` 及其引用文件定义，脚本目录、依赖、命令、生成、打包和发布规则由 [项目工具链](tooling.md) 定义。本文件不复制这些规则，只约束实现代码应保持的性质。

目标：每项代码变更都能定位规则 owner 和实现归属，显式处理受影响边界，并用与风险匹配的证据证明行为成立。

## 1. 使用方式

修改 `scripts/` 下的 TypeScript 源码或生成器前：

1. 从相关 `SKILL.md`、项目文档或工具链文档确认本次行为的规则 owner。
2. 确认承载逻辑的现有脚本、模块或生成器；没有明确归属时先澄清职责。
3. 列出改动跨过的输入、文件、进程和输出边界。
4. 选择能证明这些边界仍成立的验证。

随 skill 分发的 MJS、TypeScript 声明和 source map 生成文件不作为直接实现入口；需要改变其行为或公共类型时，修改主仓库源码、声明源或生成器，再通过 [项目工具链](tooling.md) 定义的入口同步。

## 2. 规则与实现各有 owner

代码和测试负责实现或验证规则，不成为新的规则来源。实现发现规则缺失、冲突或过期时，先确认稳定 owner，再按影响面同步 owner、代码、测试、fixture 和生成材料。

新增逻辑进入最接近其职责的现有实现归属。入口负责组合流程、映射结果和对外反馈；解析、校验、文件操作或领域判断进入负责该职责的模块。跨模块协作使用显式参数、类型和返回结果，不在调用方重复解释下游拥有的领域值。

共享层只承接多个调用方真正共享的不变量、数据结构或错误映射。只有调用位置相似、实现代码短或未来可能复用，不足以形成共享职责。

通过标准：reviewer 能回答“哪份规则定义行为、哪段代码负责实现、谁消费结果、谁负责失败反馈”。

## 3. 在边界完成解析和错误映射

CLI 参数、环境变量、配置、路径、文件系统、Git、子进程、Markdown、frontmatter、JSON、归档文件和网络响应都属于外部边界。边界数据按同一方向流动：

1. 以未经信任的外部值进入边界。
2. 在边界完成解析、校验和归一化。
3. 以领域对象进入内部逻辑。
4. 在入口把结果或失败映射为稳定输出、诊断和退出状态。

可预期失败使用明确返回结果或具有稳定语义的错误传递，由入口统一转换为调用方可行动的反馈。验证失败应指出文件、字段或规则及原因；机器输出与诊断输出保持各自通道职责。

默认值、格式推断、跳过和恢复策略只有在行为 owner 明确定义时才使用，并保留足够的定位证据。任何失败路径都应让调用方知道下一步该查看、修改或重试什么。

## 4. 用类型表达领域含义

1. JSON、YAML、frontmatter 等无法由 TypeScript 证明结构的外部数据先以 `unknown` 接收并完成运行时校验和收窄，不用 `any` 或未经证明的类型断言绕过边界。
2. 领域逻辑只接收已经校验的值，不重复解析原始输入。
3. 跨模块传递的配置、解析结果、执行结果和生成状态使用显式类型。
4. 有限状态和互斥分支优先使用判别联合、字符串字面量联合或等价结构，使非法组合不能轻易进入内部逻辑。
5. 稳定字段、错误类别和跨函数复用的固定值由职责内类型或常量表达，不建立跨功能的杂物常量池。

当多个字符串、通用 JSON 值或元组承载不同领域含义时，使用对象或专用类型区分；测试 fixture 和单一局部作用域可以保持简单。类型声明不能代替边界运行时校验，稳定规则和错误映射不能散落为魔法值。

通过标准：函数签名能区分原始输入、外部格式、领域对象和执行结果。

## 5. 按变化原因组织代码

目录归属、依赖选择和生成边界遵循 [项目工具链](tooling.md)。在同一实现归属内，文件、模块和函数围绕一个可独立说明的变化原因组织。

入口保持流程可读，具体职责进入可独立命名和验证的单元。只服务一个父模块的 helper、wrapper、常量和小类型留在父模块附近；共享职责形成后再移动到共享层。

文件大小和函数长度是阅读信号，不是单独的拆分依据。出现以下情况时重新判断边界：

1. 一个单元同时负责多个工具行为、多个外部边界或多个输出阶段。
2. 新增功能需要持续修改无关代码。
3. 解析、领域判断、文件操作、输出和错误映射互相遮蔽。
4. reviewer 无法用一个简短句子说明该单元的主职责。

通过标准：文件路径和命名能表达实现归属，局部改动不要求读者同时理解无关流程。

## 6. 验证覆盖风险，而不只覆盖代码路径

验证随影响面扩展：

1. 纯计算、解析或映射逻辑用单元测试、fixture 或等价局部验证证明不变量。
2. CLI 和验证入口同时覆盖成功、失败、退出状态以及 stdout/stderr 边界。
3. 文件系统、Git、归档和生成行为覆盖路径边界、确定性、失败恢复和产物漂移。
4. 共享工具链、项目配置或目录结构变化执行项目级检查。

具体命令和生成检查入口以 [项目工具链](tooling.md) 为准，最低验证范围遵循 [`AGENTS.md`](../AGENTS.md)。交付时记录实际运行的命令和结果；无法运行时说明原因、未覆盖边界和残余风险。

通过标准：验证证据能证明本次改动影响的边界和失败行为仍成立，而不只是通过类型检查。

## 7. 交付前检查

1. 规则 owner、实现归属、消费方和失败反馈责任是否清楚。
2. 外部输入是否只在边界解析、校验和归一化一次。
3. 失败、恢复和默认行为是否可追溯到 owner，并向调用方提供行动信息。
4. 类型是否区分原始输入、外部格式、领域对象和执行结果。
5. 文件和函数是否围绕独立变化原因组织，共享代码是否承接真实共享职责。
6. 生成行为是否从源码或生成器修改，并同步受影响产物。
7. 验证是否覆盖受影响边界、错误路径和输出通道。
8. 改动是否只同步受影响材料，没有把规则复制到新的 owner。

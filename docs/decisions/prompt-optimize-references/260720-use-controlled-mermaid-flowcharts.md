## 目的
- 保持 prompt-optimize 的格式选择可预测, 避免无边界的自由排版。
- 让适合图示的层级、分支和跨节点关系获得比扁平文本更清楚的表达。
- 确保图示仍由可解析的代码关系承接, 不依赖纯空间位置传递含义。

## 背景
- 有序列表适合步骤、规则和优先级, JSON 适合复杂结构化输入输出, 简单表格适合独立行对照; 这些固定默认值能降低临场格式选择的复杂度。
- 深层层级、多个分支或跨组关系强行改成列表、路径或字段对象时, 人类读者需要额外重建整体结构。
- Mermaid flowchart 通过方向、节点、边和 `subgraph` 显式编码关系, 既具有代码结构, 又能生成便于阅读的关系图。
- 图示依赖目标环境的渲染能力, 因此不能取代无条件可用的文本默认路径。

## 决策
- 采用: 目标已有明确格式契约时遵循契约; 没有契约时继续使用有序列表、JSON 和不超过 3 列的简单表格作为默认路径。
- 采用: 层级、分支或跨节点关系在图示能显著改善可读性且目标环境支持 Mermaid 时, 使用 Mermaid flowchart。
- 采用: 根据阅读方向选择 `flowchart LR` 或 `flowchart TD`, 并使用 `subgraph` 表达逻辑分组。
- 采用: 节点、边标签和方向必须显式表达关系, 不让含义只依赖渲染后的空间位置。
- 采用: 目标环境不支持 Mermaid 或图示收益不足时, 使用扁平路径、字段对象或 nodes/edges 表达。

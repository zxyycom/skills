# 将负向方案视为解空间锚点

## 索引摘要
- 目的: 让 agent 从目标、约束和结果建立解题路径, 避免替代方案被排除对象限定在局部框架中。
- 背景: 模型可以理解“不使用 X”, 但 X 的常见用法、修正路径和反例仍进入上下文, 生成替代方案时可能继续围绕 X 搜索。
- 决策: 先建立正向目标框架, 再分类负向方案; 只有容易回落、重要边界或明确契约需要时才点名 X, 并避免围绕 X 反复展开。

## 目的
- 让 prompt 优先激活目标状态、必要约束和推荐路径。
- 让负向边界继续保护明确契约和高风险场景, 同时减少它对方案搜索范围的锚定。

## 背景
- 较强模型通常能够准确识别用户不希望采用某个方案。
- 理解禁止不等于该方案退出上下文: 方案名称会同时带入其常见正确做法、修正路径和反例。
- 当 prompt 反复说明某方案不能做时, 替代方案容易继续围绕该方案的正反面生成, 而真正更优的做法可能需要跳出这层逻辑重新建立解空间。
- 安全、权限、合规、明确契约和默认容易回落的方案仍需要必要的负向边界。

## 决策
- 采用: 处理负向描述前, 先用目标状态、必要约束和预期结果建立正向解题框架。
- 采用: 将负向方案分类为迁移残留、低频极端方案、默认倾向冲突或硬性边界, 再决定删除、降级或保留。
- 采用: 推荐路径明确时直接写推荐路径, 不围绕被排除方案继续推导其替代做法。
- 采用: 默认容易回落、重要边界或明确契约需要点名时, 保留一次必要边界并配套正向做法。
- 采用: 推荐路径、接口契约或合法值集合已经自然排除旧方案时, 不再重复点名。

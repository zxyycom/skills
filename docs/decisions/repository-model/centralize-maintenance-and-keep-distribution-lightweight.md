## 目的
- 通过一个仓库复用项目约定、校验、生成、打包、CI、发布和决策记录，降低重复维护成本。
- 保持 skill 的选择和行为边界清楚，使仓库组织、安装选择和分发能力各自演进。

## 背景
- 每个 skill 单独建仓会重复维护仓库规则、自动化、发布入口、updater 和公共改进，当前维护规模下成本高于隔离收益。
- 使用者会按项目需要选择自己大致了解的 skill，而不是把仓库中的全部 skill 视为固定安装组合。
- 当前聚合 release 为每个 skill 提供独立 zip，随包 updater 只检查并可选替换当前 skill；这些机制的主要价值是集中获取和简化更新。
- 这些边界需要稳定 owner，避免仓库组织和简单分发被误解为统一安装产品或完整更新系统。

## 决策
- 采用: 使用 `docs/repository-model.md` 作为仓库目标、使用者假设、skill 选择安装、集中维护和轻量分发边界的当前 owner。
- 采用: 在 `skills/` 集中维护本仓库的 skill，同时保留各 skill 的行为边界和独立制品。
- 采用: 使用者在大致了解用途、主要行为和产物后按需选择 skill；具体写入和外部状态变化仍服从当前任务授权。
- 采用: 聚合 release 和显式运行的 updater 只承接获取与更新便利；更复杂的分发能力需由已确认的现实需求和独立 owner 驱动。
- 采用: skill 间的协作或分发关系必须由对应 owner 显式声明；主仓库与单个 skill 分别验证共享机制和自身行为。

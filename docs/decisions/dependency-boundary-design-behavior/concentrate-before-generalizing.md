## 目的
- 为工具、库、SDK、存储和基础设施依赖提供一套可执行的调用收口方法。
- 让同一责任的底层调用、协议、错误和配置进入明确 owner，使后续变化能够集中处理。
- 让单一具体实现也能建立有效边界，同时只向消费者承诺现实义务能够证明的契约。

## 背景
- 底层调用散落后，协议解析、错误、配置、认证和性能处理容易重复或漂移，修改与审查也需要跨多个位置追踪。
- 中间层经常被误解为多实现适配层。若把 provider、接口、工厂或运行时选择器当作前置条件，会在缺少现实替换义务时过早泛化；若因此拒绝建立任何边界，又会让已经存在的治理成本继续扩散。
- 同一工具可以服务数据访问、仓库配置、安装流程或诊断等不同责任。按工具名称集中全部调用会制造巨型接口，无法形成稳定 owner。
- `product-architecture-judgment` 判断工程事项是否值得做、应做到什么程度以及责任应放在哪一层；`common-denominator-design` 判断多个现实场景能够共同依赖什么契约。两者都不以单一底层实现的依赖收口、消费者边界和替换承诺为完整交付出口。

## 决策
- 采用: 新增独立分发的流程与决策型 skill `dependency-boundary-design`，面向工具、库、SDK、存储和基础设施依赖的接入、重构与审查。
- 采用: 以“把散落的底层调用收口到明确边界，让后续变化集中处理”为主承诺；先判断是否存在现实收口义务，不需要边界时以保持局部作为完整终止结果。
- 采用: 以消费者结果、责任和变化原因划分 owner；不因多个调用使用同一个可执行文件、包或 SDK 就强制归入同一边界。
- 采用: 当调用需要收口但稳定消费者语义尚未形成时，先建立诚实命名的具体依赖边界，集中调用、协议、错误与配置；消费者语义稳定后可以在其上增加局部门面。
- 采用: 边界契约从消费者必须保持的结果和明确约束形成；当前实现只作为迁移证据，并区分必须保持、明确修正和尚未确认的行为，避免把偶然现状固化成长期规则。
- 采用: 实施时先完成责任收口并验证没有计划外语义变化，再按现实证据继续演进消费者语义或替换能力；范围内直接调用、剩余例外和契约内变化的影响范围必须能够核对。
- 采用: provider 由多个实现共同履行同一消费者契约或实际选择切换义务定义，不由接口、构造函数或工厂等代码形态定义；只有未来设想时保留具体边界和重新评估信号。
- 采用: 新 skill 自包含运行，不依赖产品与架构判断、公约数设计或其他分发单元；一般领域抽象、跨场景共享契约和已明确契约的机械抽取继续由对应任务 owner 承接。

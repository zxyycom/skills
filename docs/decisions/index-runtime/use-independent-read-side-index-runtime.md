# 采用领域状态驱动的极简通用索引

## 索引摘要
- 目的: 让多个领域以低接入和维护成本获得稳定的状态查询与索引同步能力。
- 背景: 决策、调查和测试证据都需要索引，但领域事实、写入和动态状态语义并不相同。
- 决策: 领域提供 state、唯一 id、revision 和多 key 策略，通用层只管理派生副本、查询与确定性文件同步。

## 目的
- 让需要索引的领域只描述自身 state、稳定身份和可查询 key，不再重复实现索引文件格式、过滤、排序和同步。
- 让使用者继续通过领域命令工作，不承担通用索引概念、通用 CLI 或跨 skill 安装依赖。
- 让共享层的维护成本由三个现实领域共同使用的稳定契约约束，单个领域变化不扩大公共接口。

## 背景
- 决策记录、调查报告和测试证据都需要低成本定位长期增长的内容，并需要从权威领域源重建查询投影。
- 三个领域共同拥有 state、稳定身份、查询 key 和可重建索引，适合共享确定性文件管理与查询算法。
- 三者的源文件格式、合法状态、写入事务、Git 判断和动态采集方式不同，把这些义务放入共享层会增加接入、调试和长期维护成本。
- 索引查询必须适应长期项目规模；在 Node 下完整解析一千条模拟决策约需两秒，因此常规查询应直接读取已经同步的索引 state。

## 决策
- 采用: 使用 `tools/index-runtime/` 承接领域状态的派生索引。领域定义提供 `namespace`、`definitionVersion`、同步且确定性的 state parser、state 读取、revision 读取、唯一 id 和一个或多个 key 生成策略；parser 继续由领域拥有并校验领域字段与元数据，id 和 key 只对解析后的 state 执行确定性纯计算。
- 采用: 每条 state 只有一个稳定且唯一的 id；key 是可重复的派生查询值。key 策略显式选择 `exact`、`range` 或 `text`，通用查询只按这些固定语义过滤、排序和分页；时间等具有领域顺序的值由领域先转换为 epoch 数值或其他保持真实顺序的标量。
- 采用: 领域源继续拥有事实和写入规则。通用层保存可删除重建的 state 副本与 key，负责确定性 JSON、原子写入、完整同步和漂移检查；领域 writer 完成事实写入后调用完整同步。
- 采用: 索引保存不透明 `sourceRevision`。领域的完整 state 快照与 revision 必须来自同一时点；任何会改变索引成员、state、id 或 key 的源变化都产生新 revision。parser、id 或 key 契约变化时提升 `definitionVersion`。常规运行时在查询前核对当前 revision，并重新解析持久化 state 与核对派生结果；不一致时明确失败，不静默使用旧 state。
- 采用: 动态 state 以完整 state 形式在查询时按 id 临时替换或追加，使用同一组 key 策略，但不写入索引文件。pending、inventory 和 review trigger 的产生方式仍由领域负责。
- 采用: 索引文件只保存格式版本、领域定义版本、revision、key 定义和 `id/state/keys` 条目，不保存生成时间、通用领域元数据、搜索权重或隐藏缓存。
- 采用: 通用层不提供跨领域 federation、通用 CLI、领域详情读取、图语义、可插拔存储或领域源写入事务。关系使用普通 key 表达，领域命令按需执行反向查询或遍历。
- 采用: 领域接入是后续独立变更。接入必须复用既有领域解析器、删除旧索引路径并把共享源码内联进自包含分发产物；建立中间层本身不改变任何 skill 的运行时依赖。
- 采用: 使用三个隔离领域 fixture 验证同一契约，并将 Node 与 Bun 的一千条和五千条 state 测量作为中间层规模证据，不建立通用层持续性能 SLO；领域 reader 的源解析和端到端成本在各自接入时验证。
- 不采用: 统一领域文件格式、强制通用时间字段，或让通用层解释领域生命周期和动态状态来源。

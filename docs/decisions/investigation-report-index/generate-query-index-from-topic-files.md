## 目的
- 让调查主题随着数量增长仍可按身份、分类、状态、最新报告时间和文本稳定发现，不要求 agent 逐份解析正文。
- 让索引首先支持找到同一核心问题并继续调查，同时保留主题内报告数量、顺序和最近记录信息。
- 让主题 Markdown 自包含全部权威调查信息，索引损坏、缺失或被删除时可以无损重建。
- 让 `investigation-report` 复用已经建立的通用索引协议，同时保持独立分发和领域命令边界。

## 背景
- 原 `investigation-index.md` 手工复制主题路径、标题、核心问题、状态和最新报告时间，主题文件与索引需要双写，容易形成遗漏和投影漂移。
- 每个主题文件承接一个稳定核心问题、调查状态和追加报告序列；每份 H3 报告只表达特定形成时点的认识，不独立拥有生命周期。
- 路径首段只是分类，不能作为主题身份；同一分类可以包含多个核心问题不同的主题文件。
- 通用索引已经提供唯一 id、多 key、source revision、确定性 JSON、查询和原子完整同步，调查领域只需定义自身 state、源发现和派生策略。
- 调查报告正文与状态语义仍由调查领域拥有；把 Markdown 解析、报告合法性或写入事务放入通用层会破坏现有 owner 边界。

## 决策
- 采用: 调查根目录内全部合法主题 Markdown 是成员和领域事实的唯一来源；每个主题只产生一个 `investigation-index.json` entry，同一主题内的报告不产生独立 entry。
- 采用: 相对调查根目录的 `<category-id>/<semantic-slug>.md` 路径同时作为当前集合中的 id 和 state path；移动主题会显式产生新 id，不从内容猜测重命名。
- 采用: state 对主题标题、核心问题和全部报告标题保存 Markdown AST 提取的语义纯文本，并投影状态、最新报告时间、报告数量和按形成顺序排列的报告标题；不复制展示标记、报告正文或结果摘要。
- 采用: 派生 `category`、`status`、`latest-report-at` 和 `text` keys；最新报告时间先转换为 epoch 毫秒，text 聚合主题标题、核心问题和全部报告标题，路径查询使用通用保留 id。
- 采用: 最新报告只表示主题内最近形成的记录，不自动充当累积当前口径。未来出现跨主题的正文级历史知识检索需求时，可以增加独立报告读取侧，但不替换主题主索引。
- 采用: source revision 对排序后的 POSIX 路径和完整 Markdown 文本计算 SHA-256，只规范化 CRLF。任何成员或内容变化都使旧索引失效，即使 state 投影没有变化。
- 采用: `sync-index` 在全部主题有效后执行完整同步和写前 revision 复核；默认 `check` 验证完整集合与索引新鲜度。带 topic 或 report 筛选的检查只证明局部报告结构，不证明全局索引可查询。
- 采用: 通过领域 `list` 命令暴露主题路径、category、状态、最新报告时间和文本查询；命令先核对当前 source revision，再查询持久化主题 state 与 keys，不要求使用者理解通用索引 API 或逐份解析报告正文。
- 采用: 删除 `investigation-index.md` 和对应解析、兼容及手工维护路径，不提供双轨读取或迁移模式。首次同步直接从当前主题 Markdown 生成当前 JSON 格式。
- 采用: 构建器把通用索引源码内联进 `investigation-report` 自包含分发模块，并随包提供领域索引 JSON Schema；使用者不需要安装另一项 skill 或理解通用 CLI。

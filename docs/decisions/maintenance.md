# 决策记录维护说明

本文件说明 `docs/decisions/` 的维护方式。索引负责导航, 本文件负责规则, 每个决策文件负责一条需要回放的具体判断。

## 目录结构

```text
docs/decisions/
README.md
maintenance.md
<impact-area-id>/
YYMMDD-<status>-short-title.md
```

## 影响面 ID

影响面 ID 表示决策影响的稳定责任边界。

命名规则:

1. 使用 kebab-case。
2. 使用名词或名词短语。
3. 表达长期影响面, 例如 `decision-records`、`skill-behavior`、`skill-references`、`project-tooling`。
4. 不使用动词开头的一次性任务名。
5. 不包含日期、阶段、版本号或临时问题编号。

选择规则:

1. 决策主要影响哪个长期边界, 就放入哪个影响面目录。
2. 同时影响多个边界时, 选择后续最可能查找该决策的主要影响面。
3. 找不到合适影响面时再新增目录; 新增后同步更新 `README.md` 的影响面说明。

## 决策状态

决策状态写入文件名和正文。文件名提供目录浏览时的快速判断, 正文负责说明状态关系和导致状态变化的后续决策。

状态值:

1. `active`: 当前仍完整生效的决策。
2. `amended`: 决策仍有回放价值, 但部分规则、命名或适用方式已被后续决策修订。
3. `superseded`: 决策已被后续决策替代, 后续维护不再按该记录作为当前规则执行。
4. `invalidated`: 决策因后续发现冲突、前提错误或无效结论而不再作为依据。

使用规则:

1. 新决策替代、修订或判定旧决策无效时, 新决策保持 `active`, 旧决策改为 `amended`、`superseded` 或 `invalidated`。
2. 非 `active` 状态必须在 `## 状态` 中链接导致状态变化的后续决策。
3. 状态变化由造成变化的后续决策承接, 不为单纯改状态再新增一条决策。
4. 旧记录仍能解释当前设计为什么曾经存在时保留并标状态; 低于当前门槛且没有独立回放价值时可以删除。
5. `superseded` 用于正常演进后的替代; `invalidated` 只用于后续明确判断旧结论与当前契约冲突、前提错误或不应再沿用。

## 记录门槛

决策记录只在后续维护需要回放原因时新增。判断时先问三个问题:

1. 如果没有这条记录, 后续是否难以解释为什么采用当前方案, 而不是另一个看起来也合理的方案?
2. 这次决定是否会改变后续如何维护 skill 或项目?
3. 这次决定是否沉淀了可复用的判断标准、owner 边界、验收标准或长期约束?

skill 相关决策可以稍细。以下情况通常需要记录:

1. 改变 skill 的触发条件、读取策略、执行流程、完成检查或输出边界。
2. 改变 reference 文件的 owner 分工、结构拆分、规则归属或跨文件引用关系。
3. 增加、删除或改写会影响后续输出的规则、判断标准、边界条件或验收标准。
4. 用户通过多轮反馈逐步收敛出新的维护偏好, 且该偏好以后会复用。

项目级决策门槛更高。只有改变长期维护契约、目录边界、自动化交付方式或跨文件 owner 的决定才记录。脚本内部实现步骤、CI 单步调整、普通文案修正、格式调整、链接修复和一次性执行细节不记录。

决策目录可以被整理。低于当前门槛、没有独立判断价值的旧记录可以合并或删除; 如果旧记录仍能解释当前设计为什么存在, 保留并在新记录中说明替代关系。

## 决策文件

文件名格式:

```text
YYMMDD-<status>-short-title.md
```

文件名开头使用 6 位日期, 例如 `260627` 表示 `2026-06-27`。标题部分使用英文 kebab-case。`<status>` 必须使用 `active`、`amended`、`superseded` 或 `invalidated`。正文标题保留完整日期和中文标题。

正文分为最小结构和完整结构。最小结构用于单点命名、模板、校验或维护偏好调整; 完整结构用于需要回放上下文、关键备选或多轮收敛过程的决策。

正文最小结构:

```text
# YYYY-MM-DD - <标题>

## 状态
- 当前状态: <active | amended | superseded | invalidated>
- 导致状态变化的决策: <active 写“无”; 其他状态链接到后续决策>
- 状态说明: <状态含义和旧记录还能怎样使用>

## 问题
- <触发问题和为什么值得记录>

## 决定
- 采用: <最终采用的规则、结构或做法>
- 不采用: <放弃的方案和原因, 没有明显备选时可省略>
- 触发条件: <以后什么情况下沿用这条判断>

## 影响
- <对 skill 行为、维护方式或后续工作的影响>

## 验证
- <如何确认该决定已经落地>
```

完整结构:

```text
# YYYY-MM-DD - <标题>

## 状态
- 当前状态: <active | amended | superseded | invalidated>
- 导致状态变化的决策: <active 写“无”; 其他状态链接到后续决策>
- 状态说明: <状态含义和旧记录还能怎样使用>

## 问题
- <触发问题和为什么值得记录>

## 背景与约束
- <当时已经成立的项目边界、用户补充或上下文限制>

## 决策过程
1. <先提出的问题、初始方案或不确定点>
2. <多轮对话中新增的约束、反例或用户确认>
3. <最终收敛出的判断>

## 决定
- 采用: <最终采用的规则、结构或做法>
- 不采用: <放弃的方案和原因, 没有明显备选时可省略>
- 触发条件: <以后什么情况下沿用这条判断>

## 影响
- <对 skill 行为、维护方式或后续工作的影响>

## 验证
- <如何确认该决定已经落地>
```

## 写法要求

1. `## 问题` 写触发原因, 不写普通任务标题。
2. `## 决定` 必须显式写出采用什么、为什么不采用关键备选项、以后何时沿用。
3. `## 影响` 写未来维护会因此怎样变化, 不复述已完成的文件改动。
4. `## 验证` 写如何确认决定已经落地, 可以是文档位置、脚本校验或人工检查点。
5. `## 背景与约束` 写当时成立的边界, 包括用户明确补充的约束; 短决策没有独立约束时省略。
6. `## 决策过程` 用来回放“先提出问题, 再通过多轮对话逐步确定细节”的过程; 没有关键备选或多轮收敛时省略。
7. 省略可选小节时不要留下空标题。
8. 校验脚本只强制最小结构、状态关系、文件命名和索引链接; 完整结构由记录价值决定。

## 更新流程

1. 新增决策文件, 放入对应影响面目录。
2. 文件名写入 6 位日期和当前状态, 正文 `## 状态` 写清状态关系。
3. 在 `README.md` 的决策清单中追加链接。
4. 如果新增影响面目录, 同步更新 `README.md` 的“影响面”。
5. 新决策替代、修订或判定旧决策无效时, 更新旧决策文件名、`## 状态` 和索引链接。
6. 新决策替代旧决策时, 先判断旧记录是否仍有回放价值; 有价值则保留并说明替代关系, 无独立判断价值则合并或删除。

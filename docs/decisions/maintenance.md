# 决策记录维护说明

本文件说明 `docs/decisions/` 的维护方式。索引负责导航, 本文件负责规则, 每个决策文件负责一条具体判断。

## 目录结构

```text
docs/decisions/
README.md
maintenance.md
<impact-area-id>/
YYYY-MM-DD-short-title.md
```

## 影响面 ID

影响面 ID 表示决策影响的稳定责任边界。

命名规则:

1. 使用 kebab-case。
2. 使用名词或名词短语。
3. 表达长期影响面, 例如 `project-documentation`、`agent-instructions`、`decision-records`。
4. 不使用动词开头的一次性任务名。
5. 不包含日期、阶段、版本号或临时问题编号。

选择规则:

1. 决策主要影响哪个长期边界, 就放入哪个影响面目录。
2. 同时影响多个边界时, 选择后续最可能查找该决策的主要影响面。
3. 找不到合适影响面时再新增目录; 新增后同步更新 `README.md` 的影响面说明。

## 决策文件

文件名格式:

```text
YYYY-MM-DD-short-title.md
```

文件名使用英文 kebab-case。正文标题可以使用中文。

正文结构:

```text
# YYYY-MM-DD - <标题>

问题:
- <发现的问题或维护风险>

决策过程:
- <比较过的方案、取舍依据或用户确认的信息>

决定:
- <最终采用的规则、结构或做法>

影响:
- <对 skill 行为、维护方式或后续工作的影响>

验证:
- <如何确认该决定已经落地>
```

## 记录边界

需要记录:

1. 改变 skill 行为、触发条件、读取策略、内容 owner、引用结构、项目约定或验收标准的决定。
2. 项目文档结构、维护方式或长期边界的调整。
3. 用户明确指出的问题, 且解决过程体现了后续维护需要复用的判断。
4. 新判断替代旧判断的情况。

不需要记录:

1. 拼写、格式、单个链接修复。
2. 一次性执行细节。
3. 无长期判断价值的临时调查路径、外部来源或工具位置。

## 更新流程

1. 新增决策文件, 放入对应影响面目录。
2. 在 `README.md` 的“当前决策”中追加链接。
3. 如果新增影响面目录, 同步更新 `README.md` 的“影响面”。
4. 新决策替代旧决策时, 不删除旧文件; 在新文件中说明替代关系和影响。

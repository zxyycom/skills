# 2026-07-12 - 区分并行写入所有权与只读调查范围

## 状态

- 当前状态: active
- 导致状态变化的决策: 无
- 状态说明: 作为 `skills/subagent-orchestration/` 划分并行子代理范围和权限的当前规则使用。

## 问题

- `skills/subagent-orchestration/SKILL.md` 原先要求子代理工作范围、文件范围和决策范围保持互斥, 没有区分写入所有权与只读证据范围。
- 多个实现代理并行写入同一状态会产生冲突, 但独立审查、对照调查和跨范围取证需要读取相同文件、模块或决策面。
- 把所有重叠都排除会限制有效调查; 只写“只读可重叠”又可能让重复任务和隐含写入失去边界。

## 决定

- 采用: 只要求并行子代理的写入范围互斥, 并按实际可能修改的文件、配置、生成产物或外部状态划分所有权; 原因是冲突来自共享状态变更, 不是共享证据读取。
- 采用: 只读子代理可以为获取独立结论或互补证据而重叠读取同一范围, 任务必须明确只读权限、重叠目的和各自输出, 并基于稳定输入产出结果, 且不得修改共享工作区或外部状态。
- 不采用: 允许没有独立验证或证据互补价值的只读任务无条件重叠。原因是这种重叠只会增加重复理解成本。
- 触发条件: 并行派发实现、调查、审查或审计子代理时应用本规则; 若执行环境提供隔离工作区或事务化外部状态, 重新评估写入所有权的冲突边界, 但仍需显式声明每个任务的权限。

## 影响

- `skills/subagent-orchestration/SKILL.md` 的元数据、核心原则、主代理职责、任务模板、权限与并行边界和完成标准统一按读写权限表达范围。
- 主代理可以让只读 reviewer 覆盖实现代理的文件范围, 也可以让多个只读调查代理从不同目标检查同一决策面。
- 可写任务继续保持互斥所有权; 只读重叠不能作为隐含修改共享状态或派发重复任务的理由。

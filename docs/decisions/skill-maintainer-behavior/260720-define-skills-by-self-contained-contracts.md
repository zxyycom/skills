# 2026-07-20 - 让分发 skill 以自身契约定义边界

## 索引摘要
- 目的: 让每个分发后的 skill 在没有其他 skill 配合时也能被发现、理解和执行。
- 背景: 仓库维护时可以比较多个能力, 但安装环境中的 skill 组合并不固定; 用相邻能力的差异解释触发或边界会把其他 skill 变成隐含前提。
- 决策: 维护时可以比较可访问能力以判断 owner, 但分发入口和行为正文只用自身输入、目标结果、判断、步骤、边界和验收定义能力, 不假定其他 skill 存在。

## 目的
- 让每个 skill 的最小分发包能够独立兑现发现入口和行为契约。
- 让能力边界由自身负责的输入、行为和完成结果表达, 不随安装环境中其他 skill 的增减而失效。

## 背景
- 仓库维护者需要比较多个能力, 才能识别重复、冲突和 owner 归属。
- 分发后的运行环境可能只安装其中一个 skill, 也可能使用不同版本或完全不同的能力组合。
- 如果 `description` 或正文依赖“与另一个 skill 的区别”才能解释自身用途, 未安装的相邻能力会成为隐含运行前提, 并让触发边界随外部组合漂移。
- 维护分析与分发契约承担不同作用: 前者可以比较, 后者必须自包含。

## 决策
- 采用: 维护 skill 时可以检查当前环境中可访问的同类能力, 以判断 owner、重复和冲突。
- 采用: 分发后的 `description`、行为正文和主动引用只用该 skill 自身的输入、目标结果、前置条件、判断、步骤、输出、边界和验收表达能力。
- 采用: 相邻能力的比较只作为维护判断和决策回放材料, 不作为分发后 skill 的触发说明、执行前提或完成条件。
- 采用: 使用现实请求和边界请求验证触发范围时, 只要求该 skill 在自身契约内正确承接或不误触发, 不假定其他 skill 会接手。

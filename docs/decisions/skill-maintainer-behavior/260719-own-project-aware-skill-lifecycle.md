# 让 skill-maintainer 承接环境自适应的 skill 生命周期

## 索引摘要
- 目的: 让可复用的 agent 能力由新 skill、既有 skill 或其他 owner 清楚承接，并让选定的 skill 在简约或完整的维护环境中都能交付。
- 背景: Skill 可能位于成熟仓库、独立目录或安装位置；把项目级 owner、校验器和发布工具链作为默认前提会限制其可移植性。
- 决策: 使用 `skill-maintainer` 承接 skill 模型、能力归属和行为契约，采用“自包含最小基线加环境适配”的交付方式，并随包提供独立结构验证器。

## 目的
- 让每项可复用的 agent 能力先确定应由新 skill、既有 skill 还是其他 owner 承接，再让选定的 skill 形成清楚、可验证和可交付的行为单元。
- 让分发后的 skill 能在只有目标路径的简约环境中独立工作，也能接入拥有完整规则和工具链的项目。
- 让维护位置负责源码和发布，让行为契约决定 skill 的实际适用范围。
- 让可机械判断的结构契约由随包脚本验证，让语义质量继续由审阅和环境检查证明。

## 背景
- Skill 的维护环境并不总是一个具备 agent 指令、决策记录、共享工具链、校验器和打包流程的仓库。
- 只依赖固定模板会忽略成熟项目的本地契约，只依赖项目契约又会让简约环境缺少可执行的默认路径。
- 缺少对 skill 作用、组成和主要类型的共同模型时，owner、资源和验证方式只能依赖临场猜测。
- 只有文字检查项时，缺失入口、无效 frontmatter、目录命名和断链等结构问题无法稳定复现。
- 一个可移植的维护 skill 需要同时提供自包含基线和环境适配入口。

## 决策
- 采用: 在 `skills/skill-maintainer/` 新增独立 skill，使用不同于全局 `skill-creator` 的名称和触发入口。
- 采用: 在 `SKILL.md` 默认上下文中说明 skill 的作用、发现入口、行为正文、按需资源和环境集成层，并按主要作用区分知识与规则、流程与决策、工具与自动化、产出与模板、协调与入口五类。
- 采用: 一个 skill 可以兼具多类特征，以主承诺占比最高的类型决定正文重心、资源选择和验证方式。
- 采用: `skill-maintainer` 承接能力应由新 skill、既有 skill 还是其他 owner 承接的判断，并维护选定 skill 的行为契约、资源结构和完成验收。
- 采用: 以 skill 目录、包含 `name` 和 `description` 的 `SKILL.md` 及完整行为契约作为最小交付；按需增加 references、scripts 和 assets。
- 采用: 随 `skill-maintainer` 分发 Node 单文件脚本 `scripts/validate-skill.mjs`，机械检查可移植结构基线；当前环境的额外契约继续由其验证入口补充。
- 采用: 验证器的 TypeScript 源码、构建入口和测试由主仓库 `scripts/skill-validator/` 承接，skill 目录只保存生成后的 JavaScript 与 source map。
- 采用: 当前环境存在指令、运行时约定、metadata、文档、校验或发布入口时接入这些契约，其余部分由最小基线补足。
- 采用: `description` 只以设计、创建、更新、重构、审阅和相邻 skill 边界冲突作为触发场景；安装、启停和发布作为用户明确要求时才执行的独立外部状态变更。
- 采用: 本仓库承接该 skill 的源码、人类介绍、决策记录、打包和发布；行为契约保持对其他维护环境的独立适用性。

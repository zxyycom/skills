# 将 Skill 深度设计发现设为独立行为

## 索引摘要
- 目的: 让隐性工作流、潜藏决策和约束在 Skill 实现前成为可验证设计，同时保持生命周期维护职责聚焦。
- 背景: 现有维护能力覆盖归属、结构、分发和交付，但创建只是其中一条路径，无法稳定承接复杂领域能力的深度发现。
- 决策: 新增独立 `skill-design-discovery`，以轻量或深度发现形成证据化设计契约，再交由当前环境的实现入口落地。

## 目的

- 在创建、显著扩展或大幅重构 skill 前，恢复真实结果、工作流、潜藏决策、约束来源、人机权限和验证义务。
- 让复杂领域经验成为实现者可以直接使用和检查的设计契约，而不是只把表面步骤改写成 `SKILL.md`。
- 保持设计发现与 skill 生命周期维护各自拥有稳定触发、行为责任和完成验收。

## 背景

- `skill-maintainer` 需要处理能力归属、skill 组成、分发单元、环境集成、结构验证和交付，创建只是其生命周期流程中的一种任务。
- 隐性领域能力的主要困难发生在实现前：用户经验难以直接表达，现实案例与现有材料可能冲突，启发式判断容易被误写成硬规则，人类权限与验证义务也经常缺失。
- 如果把完整深度发现强制加入所有维护任务，局部修订会产生过度访谈和一次性规划文件；如果只保留普通 workflow map，又无法恢复决定行为质量的判断依据和约束强度。
- 设计发现可以独立产出可实现契约，不需要把任何特定维护 skill 作为运行前提。

## 决策

- 采用: 在 `skills/skill-design-discovery/` 创建独立的流程与决策型 skill，并作为独立分发单元维护。
- 采用: 该 skill 只在新建、显著扩展或重大重构前、关键行为仍隐含或冲突时展开；目标、边界和验收已经清楚的局部维护不进入深度发现。
- 采用: 发现同时使用用户说明、规范性材料、现实案例、现有实现和可用运行证据，并区分已确认目标、观察事实、经验、推断和未知项。
- 采用: 设计结果同时包含工作流、决策点、约束类型与来源、人类权限、资源 owner 候选和验证义务，不以步骤列表代替判断模型。
- 采用: 深度契约显式包含端到端 Workflow Map，并为关键步骤说明控制方式、必需资源、完成证据、交接条件和失败处理；不恢复固定管控等级或强制整图确认。
- 采用: 使用跨触发、流程、判断、依赖、权限、owner 和行为差异的评估提示即时推导验证义务，不在当前 skill 中固化目标领域的验证案例。
- 采用: 根据复杂度选择轻量或深度路径；设计契约默认可以在对话中交付，只有复杂交接或当前环境要求时才落盘。
- 采用: 该 skill 不拥有目录结构校验、metadata、分发、打包、安装、更新和发布；当前请求同时包含实现时，由当前环境的实现或维护入口接续。
- 采用: 发现到的判断不自动进入长期决策记录；只有已经确认并达到当前项目记录门槛的取舍才由对应 owner 持久化。
- 采用: 分发后的入口与正文只使用自身目标、输入、流程、边界和验收定义能力，不把仓库内相邻 skill 作为隐含依赖。
- 采用: 在当前设计尚未稳定期间，将固定版本的上游原始 skill 作为随包参考保留；它不进入普通运行路径，其 evals 只作为上游证据，许可、NOTICE、署名和商标说明一并保留。

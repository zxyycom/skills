## 目的
- 为项目中的版本状态查询建立唯一、可维护的责任边界。
- 让业务调用表达“某修订中是否存在文件”“两个修订间哪些路径变化”等意图，而不是依赖 Git 命令、index 结构或第三方库对象。
- 将未来替换 Git 库、修正语义或尝试接入其他版本管理系统的主要改动集中在同一 owner。

## 背景
- 项目需要读取已经提交的修订、准备进入下一版本的内容以及当前工作区变化，这些场景具有共同的版本管理责任，但不是同一种状态。
- 如果各调用方分别执行 Git 命令、解释输出和映射错误，正确性、性能与替换成本会随调用点增长。
- SVN 等其他版本管理系统在游戏开发等场景中具有现实价值，但当前项目没有需要立即支持的明确消费者；以完整跨系统兼容为前提会把假想差异提前固化进公共契约。

## 决策
- 采用: 由项目自有版本管理中间层集中承接仓库发现、修订解析、提交快照、待提交快照、版本文件与路径变化查询；调用方只依赖项目类型、结果和错误语义。
- 采用: 默认实现和当前返回语义可以面向 Git；第三方 Git 库对象、原始命令输出和解析细节不得越过中间层边界。
- 采用: 已提交的 `revision` 快照与准备提交的 `pending` 快照必须显式区分；Git 可以用 index 实现 `pending`，其他系统若以后接入则在中间层内模拟、拒绝或修订契约，不由调用方猜测。
- 采用: 公共操作只从项目内已经存在的查询责任提炼，不为尚未确认的后端预建 provider 注册、能力协商、多仓库发现或通用降级框架。
- 不采用: 当前不把中间层描述为 Git、SVN、Perforce 等系统的共同抽象，也不以一次实现完成跨版本管理系统兼容为验收目标。

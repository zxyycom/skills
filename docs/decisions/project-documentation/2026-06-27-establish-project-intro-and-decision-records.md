# 2026-06-27 - 建立项目介绍和决策记录

问题:
- 本仓库用于维护 `prompt-optimize` skill, 但 README 只说明目录和目标, 缺少项目定位、skill 内容结构和维护边界。
- skill 修改过程中的问题发现、方案取舍和用户思考没有持久 owner, 容易只留在对话或 git diff 中。
- 如果把维护历史直接写入 `SKILL.md`, 会增加 skill 运行时上下文负担, 让入口文件承担不属于它的解释责任。

决策过程:
- README 适合承接项目基础介绍和目录导航, 但不适合展开具体执行约定。
- `AGENTS.md` 适合承接 agent 在本仓库工作时的读取顺序、内容 owner、修改原则和验证检查。
- `docs/decisions/` 适合承接修改过程中的长期判断, 避免把决策历史写入 skill 入口或引用规则。

决定:
- README 承接项目定位、目录结构、skill 内容和维护约定。
- `AGENTS.md` 承接 agent 维护本仓库时的项目级指令。
- `docs/decisions/` 承接会影响 skill 行为、结构、维护约定或后续判断的决策记录。

影响:
- 后续修改 `prompt-optimize` skill 时, 需要先判断本次改动是否改变长期行为或维护约定；如果改变, 同步追加决策记录。
- skill 入口继续专注运行时导航和行为约束, 决策历史留在 docs 中, 减少运行时上下文噪音。

验证:
- 根目录存在 `AGENTS.md`。
- 决策记录存在于 `docs/decisions/`, 并包含索引、维护说明和首批决策。
- README 的目录结构和维护约定同步包含 agent 指令与决策记录。

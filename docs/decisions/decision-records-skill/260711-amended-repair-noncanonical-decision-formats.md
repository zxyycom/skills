# 2026-07-11 - 将非标准决策材料修复为唯一契约

## 状态
- 当前状态: amended
- 导致状态变化的决策: [2026-07-11 - 使用显式确认控制决策写入](260711-active-require-confirmed-decision-writes.md)
- 状态说明: 唯一目标格式、不维护并行格式和保留可确认语义仍有效；发现非标准材料后自主修复的授权边界已改为先报告并等待显式确认。

## 问题
- 如果 skill 为任意已有格式长期维护兼容分支或独立迁移模式, agent 需要同时理解多套目录、模板、状态和索引语义, 容易让同一项目继续产生并行记录。
- 已有材料仍可能包含需要保留的长期判断, 不能因为格式不符合当前契约就忽略内容或直接重新开始。

## 背景与约束
- `decision-records` 的出发点是在不同项目中使用同一套显式、可审核的 Markdown 契约。
- 用户明确要求让 agent 参考当前创建格式自行完成修复, 而不是把兼容任意格式作为 skill 的长期能力。
- 修复必须保留能够从项目 owner、版本历史和相互引用确认的原始语义, 不能为了满足模板虚构状态或取舍。

## 决定
- 采用: 将 bundled reference 定义的目录、文件名、状态、正文结构和索引关系作为唯一目标格式。
- 采用: 用户要求建立、记录或维护时, 把不符合契约的已有决策材料视为待修复输入, 由 agent 完成内容筛选、语义映射、重命名、重组、关系恢复和索引同步。
- 采用: 只有证据不足且继续推断会改变决策含义时才请求用户判断; 其余能够从现有材料确定的迁移由 agent 自主完成。
- 采用: 当前格式完整承接内容后, 不长期保留会继续演进的旧目录、旧索引或重复记录; 用户只要求解释、检查或审阅时保持只读并报告修复范围。
- 不采用: 为任意既有 ADR 或决策格式维护兼容模式。原因是多套格式会扩大 skill 的长期语义边界, 并增加同一项目出现平行 owner 的风险。
- 触发条件: 目标工作区已有决策材料, 但目录、命名、状态、正文结构、关系或索引不符合 `skills/decision-records/references/decision-record-rules.md` 时, 沿用本决策修复。

## 影响
- `decision-records` skill 从只会新建固定结构, 扩展为能够把已有材料收敛到同一固定结构。
- agent 不需要长期维护格式适配矩阵, 但需要在迁移时审慎区分决策、现行规则、任务和日志。
- 项目最终只保留一个决策记录 owner 和一种继续演进的格式, 迁移前形态由 Git 历史承接。

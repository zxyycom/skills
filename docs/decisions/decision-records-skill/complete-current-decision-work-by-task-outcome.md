# 按当前任务出口验收决策记录工作

## 索引摘要
- 目的: 让恢复或审阅、候选和维护分别按实际结果验收，并安全使用带 warning 返回的查询结果。
- 背景: 只读查询可能在局部材料有误时返回可恢复结果，统一要求严格检查会扩大任务范围，而维护写入仍需要完整一致性门禁。
- 决策: 按恢复或审阅、候选和维护三类出口验收；warning 限定查询结论，严格 check 作为维护写入的最终门禁。

## 目的
- 让每类决策记录任务都有与实际目标一致的完成条件。
- 在局部材料损坏时继续恢复可用判断，同时避免把不完整结果解释成完整事实。
- 让维护写入仍以当前格式、完整索引和严格集合检查作为交付边界。

## 背景
- `list`、`show` 和 `trace` 可以在当前索引可解析时返回仍可恢复的结果，并用 warning 暴露局部正文、投影或关系问题。
- 统一要求所有查询先通过严格 `check`，会把恢复或审阅扩大成修复无关结构问题。
- warning 只说明查询命令能够完成，不能证明结果完整；写入或状态变化则会改变长期依据，需要更严格的最终检查。
- 候选判断尚未确认时不应进入记录或索引，也不需要假装完成维护验收。

## 决策
- 采用: `decision-records` 分别以恢复或审阅、候选决策和维护作为任务出口，不要求使用者先选择独立的只读或写入模式。
- 采用: 查询成功只证明返回结果可用，不证明结果完整；warning 可能影响相关主题时，不根据缺失记录、关系或节点断言它们不存在。
- 采用: 索引缺失或无法按当前 schema 解析、目标未登记或 Git `HEAD` 查询失败时阻断查询，因为请求无法可靠完成。
- 采用: 恢复或审阅以相关判断、生命周期和对齐状态已经恢复，偏离已经归类，warning 已限定结论范围为完成条件，不要求修复无关结构问题。
- 采用: 未确认候选以目的、背景、采用方向、必要关系和确认状态清楚为完成条件，不写入记录或索引。
- 采用: 维护写入要求完整读取固定契约、保持自包含 Markdown 与真实历史关系、重建派生索引，并以严格 `check` 返回 `0` 作为完成门禁。

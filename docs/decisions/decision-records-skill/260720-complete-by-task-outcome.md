# 按任务出口验收决策记录工作

## 索引摘要
- 目的: 让查询、候选、维护和迁移按各自结果验收，并安全使用带 warning 返回的降级查询。
- 背景: 非阻断查询已经能够返回可恢复结果，但统一要求严格检查通过会扩大查询范围，而仅报告 warning 又不足以约束对缺失记录和关系的推断。
- 决策: 分别按恢复或审阅、候选、维护或迁移验收；warning 限定查询结论，严格 `check` 只作为写入和迁移门禁。

## 目的
- 让每类决策记录任务都有与其实际目标一致的完成条件。
- 在局部材料损坏时继续恢复可用判断，同时避免把不完整结果解释为完整事实。
- 不通过用户可见的“只读模式”增加额外交互，而由当前指令和任务出口限定改动范围。

## 背景
- `list` 和 `trace` 已经在中央索引可解析时带 warning 返回可用结果，`check` 与写事务继续保持严格失败。
- 原入口的统一完成标准要求 CLI 检查通过，容易让单纯查询或审阅被扩大为修复全部结构问题。
- warning 能说明某些记录、关系或派生字段无法可靠解析；查询退出码为 `0` 只表示命令完成，不能证明结果完整。
- 候选决策、已确认维护和非标准格式迁移的确认状态与验收条件也不同，不适合共用一组累积检查项。

## 决策
- 采用: `decision-records` 分别以恢复或审阅、候选决策、维护或迁移作为任务出口，不要求使用者先选择独立的只读或写入模式。
- 采用: `list` 和 `trace` 在中央索引可解析且查询目标存在时继续返回当前可恢复结果，并通过 warning 暴露结构或关系问题。
- 采用: 中央索引缺失或无法解析、`trace` 起点不存在时仍阻断查询，因为请求无法可靠完成。
- 采用: 查询成功只证明结果可用，不证明结果完整；warning 可能影响相关主题时，不根据缺失记录、关系或节点断言它们不存在，并在交付中说明不完整性。
- 采用: 恢复或审阅以相关判断已经恢复、偏离已经归类、warning 已经限定结论范围为完成条件，不要求无关结构问题全部修复或 `check` 返回 `0`。
- 采用: 未确认候选以概览和确认状态清楚为完成条件，不写入当前索引。
- 采用: 写入和迁移仍要求确认范围、读取固定契约、保持历史语义，并以严格 `check` 返回 `0` 作为完成门禁。

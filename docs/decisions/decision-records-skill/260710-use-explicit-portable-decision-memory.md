## 目的
- 建立公开、可审阅、可版本化且能跨项目复用的长期决策记忆。

## 背景
- 用户希望把重要决定沉淀为长期记忆，但这种记忆需要能够被查看、审核和持续优化，不能隐藏在模型状态或工具内部。
- 方案需要简单直接，并能在不同项目中复用，而不是为每个项目重新发明目录、状态和正文格式。
- skill 可以说明怎样判断、编写和维护决策，但目录、文件、索引和关系的一致性需要确定性脚本检查。

- 本仓库已经有按影响面组织、带状态关系的 Markdown 决策记录，可以作为真实用例。
- skill 包会独立分发，实际用户决策不能写回 skill 包；目标项目需要拥有自己的可版本化记录。
- 校验器应帮助发现结构错误，不应保存另一份状态、自动改写决策或替代用户和 agent 的语义判断。

1. 初始设想把长期记忆拆成记录、回忆和初始化等多个 skill，并讨论项目级与跨项目存储。
2. 用户随后明确核心是一个较为通用的 skill 加配套脚本：skill 说明决策怎样写和维护，脚本检查决策结构。
3. 用户进一步确认采用一套方案供不同地方复用，并要求长期记忆公开可见、可审核、可优化且简单直接。
4. 因此将主承诺收敛为固定的显式决策协议，不再把隐藏记忆、多个入口或任意格式兼容作为核心。

## 决策
- 采用: 使用固定的 `docs/decisions/` Markdown 方案在不同项目中保存决策，包括规则 owner、索引 owner、稳定影响面、文件名状态和正文关系。
- 采用: 目标项目的决策目录是长期记忆的数据 owner；skill 包只承接通用流程、格式 reference 和校验器。
- 采用: skill 负责记录门槛、取舍表达、状态选择、owner 同步和维护事务等语义判断。
- 采用: 配套脚本只读检查目录、文件名、日期、必需小节、固定字段、索引覆盖和状态来源链接，不评价决策质量或用户意图。
- 采用: 所有必要状态都通过可直接阅读的 Markdown、文件名和相对链接表达；校验器不生成配置、缓存、数据库或修复结果。
- 采用: 项目可以增加更严格的领域门槛，但继续使用同一结构契约；确需其他格式时显式分叉并同步维护 skill、规则和脚本。
- 不采用: 兼容任意决策格式。原因是可配置标题、状态和关系会削弱统一契约，使 skill 与校验器难以稳定协作。
- 不采用: 把完整对话、操作日志、未确认推断或隐藏工具状态作为决策记忆。原因是这些内容难以审核当前有效性和后续演进。

# 2026-07-18 - 在决策写入前明确通知用户

## 索引摘要
- 背景: 为每项决策维护操作重复询问授权会增加交互成本，但静默新增或调整长期决策又会让用户失去对长期记忆变化的感知。
- 决策: 当前指令足以表达决策维护意图时不重复询问操作授权；新增、调整或退出当前集合前明确告知用户将写入或改变的判断及集合变化，长期含义或范围不清时再请求判断。

## 背景
- 用户指令可能已经清楚表达记录、调整或停止沿用某项决策的意图，此时再次请求操作授权不会增加有效信息。
- 新增或调整决策会改变后续工作的默认依据，即使维护意图已经明确，也不应在用户无感知的情况下写入。
- 操作授权、长期判断是否已经形成以及写入前通知是不同问题，不应通过逐项枚举维护操作混在一起。
- 普通工作中由 agent 发现的候选仍可能只是建议或一次性选择，需要在长期含义和范围明确后才能写入。

## 决定
- 采用: 当前指令足以表明用户要维护某项决策时，将该指令作为操作授权，不重复询问是否执行。
- 采用: 新增、调整或退出当前集合前，明确告知用户将写入或改变的判断以及当前集合变化。
- 采用: 当前指令只改变本次任务的执行方式时，归为一次性例外；只有用户已经决定改变后续工作的默认依据时，才归为长期修订。
- 采用: 长期含义、调整对象或影响范围仍不清楚时，先给出最小候选概览并请求用户判断。
- 采用: 非标准格式迁移继续要求用户确认已说明的修复范围，普通维护意图不自动扩大为全目录迁移授权。

# 2026-07-10 - 使用 CLI 维护活动索引并归档失效决策

## 索引摘要
- 背景: 只做结构校验不能帮助维护者在文件重命名或状态变化后同步索引，重复手工编辑容易产生旧路径和状态漂移。
- 决策: `decision-records.mjs check` 作为默认只读命令，检查目录、文件名、正文状态、链接、归档位置和活动索引。

## 背景
- 只做结构校验不能帮助维护者在文件重命名或状态变化后同步索引，重复手工编辑容易产生旧路径和状态漂移。
- 决策会保留多个状态，但日常读取首先需要当前仍有效的判断；默认展示全部历史会降低活动决策的可见性。
- `invalidated` 表示旧判断不应再作为依据，继续与仍有正常演进价值的记录混放，会增加误读风险。

- 2026-07-10 的前一条决策已确认使用显式 Markdown 形成可审核、可优化的长期记忆，并把语义判断留给 skill。
- CLI 的写入行为必须可预期、可审核且由用户显式触发，不能恢复成隐藏状态或自动修改全部决策。
- 移动失效记录会改变相对链接；在关系重写规则稳定前，不应让 CLI 静默自动归档。

1. 初版脚本只提供只读结构检查，并要求索引覆盖全部保留记录。
2. 用户提出把脚本做成 CLI，至少支持同步文件名到索引、检查文件名和正文状态一致性。
3. 用户进一步明确所有状态都保留，但默认只显示 `active`，完全失效的记录应考虑进入专用归档目录。
4. 因此把 CLI 拆成只读检查、状态列表和显式索引同步，并先固定归档目录契约，自动移动留到链接重写可以安全预览后再实现。

## 决定
- 采用: `decision-records.mjs check` 作为默认只读命令，检查目录、文件名、正文状态、链接、归档位置和活动索引。
- 采用: `decision-records.mjs list` 默认只显示 `active`；使用 `--all` 或 `--status` 显式查看其他保留状态。
- 采用: `decision-records.mjs sync-index` 默认只检查漂移；只有 `--write` 才更新 `decision-record-index.md` 最后的 `## 活动决策`。
- 采用: 活动索引只列出 `active`，并根据文件名和正文标题按影响面生成；其他状态通过 CLI、文件树和状态来源关系访问。
- 采用: 所有已经形成的决策默认保留；后续变化通过状态、来源关系和归档位置表达，不通过删除抹去历史。
- 采用: `amended` 和 `superseded` 保留在原影响面；`invalidated` 移入 `archive/<impact-area-id>/`，正文仍保留原状态和原因。
- 采用: CLI 不保存缓存或扫描状态；当前唯一写操作是显式同步公开索引。
- 不采用: 在 `check` 或 `list` 中自动修复索引。原因是只读命令需要保持可安全重复执行。
- 不采用: 因决策不再活跃或回放价值降低而删除记录。原因是删除会破坏长期记忆的审核和优化链路。
- 不采用: 当前版本自动移动 invalidated 文件。原因是移动会改变来源和反向引用，应该先形成可预览、原子更新链接的归档命令。

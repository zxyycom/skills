# 2026-07-11 - 使用显式确认控制决策写入

## 状态
- 当前状态: active
- 导致状态变化的决策: 无
- 状态说明: 作为 `skills/decision-records/` 提出候选、记录决策和修复非标准格式的写入边界使用。

### 关系
- 修订: [2026-07-11 - 将非标准决策材料修复为唯一契约](260711-amended-repair-noncanonical-decision-formats.md)

## 问题
- 如果 agent 对工作中的每个选择都询问是否记录，长期记忆判断会频繁打断原任务。
- 如果明确选择未经长期确认就直接写入，一次性例外可能被误当成后续规则。
- 如果普通记录或维护请求自动触发非标准格式迁移，实际写入范围会超出用户预期。

## 背景与约束
- 用户明确要求 agent 在发现重要长期决策时主动提出记录，先给出当前决策概览，确认后再写入。
- 用户明确要求非标准格式只触发提醒，不静默修改；修复后仍只维护随包 reference 定义的一种格式。
- 候选判断应尽量放在自然更新或交付阶段提出，避免为了长期记忆反复暂停当前工作。

## 决定
- 采用: agent 只在自然阶段发现会持续改变行为、owner、边界、兼容性、风险处理或验收方式的重要取舍时主动提出记录，从而保留高价值决策而不逐项打断工作。
- 采用: 候选概览固定说明当前决定、核心理由、适用与重评条件、影响范围和既有决策关系，让用户可以一次确认、修改或放弃。
- 采用: 用户已经明确要求记录一项内容完整、长期适用的决定时，本次要求直接作为确认，不再重复询问。
- 采用: 当前任务的局部选择和一次性例外不自动改变决策历史；只有确认将长期影响后续工作时才写入或修订。
- 采用: 发现非标准格式时先保持只读，报告不符合项、可保留语义和预计修复范围；只有用户显式确认修复范围后才迁移，并在完成后收敛旧入口。
- 不采用: 为每个候选选择询问是否记录。原因是判断成本和交互等待会压过长期记忆带来的收益。
- 不采用: 把普通记录请求视为全目录修复授权。原因是格式迁移会重命名、移动或改变状态关系，需要独立且明确的写入边界。
- 触发条件: 普通工作中出现重要长期取舍、用户要求记录或修订决策，或目标目录存在非标准决策材料时，沿用本确认流程。

## 影响
- `skills/decision-records/SKILL.md` 先区分只读恢复、明确记录、候选提议和显式修复，再决定是否写入。
- agent 可以主动沉淀重要决定，但不会因为每个局部选择反复阻塞原任务。
- 非标准格式继续以唯一契约为修复目标，但迁移授权由用户显式提供。
- 决策概览和最小记录都会明确核心理由以及适用与重新评估条件。

## 目的
- 在长期行为改变前恢复既有方向，防止后续维护静默偏离。

## 背景
- 只在用户显式要求回放或记录时读取决策，不能保证普通维护任务在改变长期行为前看见已有方向。
- 把“当前任务指令优先”作为唯一冲突处理规则，会混淆一次性例外和长期修订，后续可能在没有说明的情况下偏离已确认判断。
- 当前索引表示后续工作的默认依据，但现实会变化；记录当时背景比预估未来所有适用场景和重评条件更可靠。

- 决策记忆继续使用公开、可版本化的 Markdown，不引入数据库、模型隐藏状态、全局知识库或强制扫描全部历史。
- 现有“重要长期取舍确认后才写入”的边界继续有效，主动恢复不等于自动修改决策历史。
- 代码、配置、规范和项目文档继续承接当前行为；决策记录只解释形成背景、采用的决策和演进关系。

## 决策
- 采用: 任务可能改变长期行为、owner、边界、兼容性、风险处理或验收方式时，在实质改动前从 JSON 当前索引读取相关记录，因为只有在决策点前恢复上下文才能实际防止方向静默漂移。
- 采用: 恢复时读取简单背景和采用的决策，再对照当前事实和行为 owner 判断是否沿用，避免把当前成员误解为脱离背景的永久规则。
- 采用: 将当前任务与既有判断归为对齐、明确的一次性例外、长期修订或一致性问题；一次性例外按当前指令执行并说明不改变历史，长期修订沿用确认后写入流程，一致性问题在没有依据时不由 agent 自行选边。
- 采用: 目标项目需要冷启动发现能力时，只在自己的 agent 指令中增加指向 JSON 当前索引和本 skill 的短路由，不复制固定契约或建立第二套记忆入口。
- 不采用: 每个任务都读取全部决策历史。原因是 JSON 当前索引的路径、标题、精简背景和决策已经能够限定读取范围，完整历史只在审计原因时需要。
- 不采用: 发现当前实现或指令不同就自动改写决策。原因是差异可能只是一次性例外、实现缺陷或尚未确认的新方向。

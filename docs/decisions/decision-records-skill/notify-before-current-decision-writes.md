# 在当前决策写入前明确通知用户

## 索引摘要
- 目的: 在不重复索取操作授权的前提下，让用户感知长期决策集合和状态的变化。
- 背景: 当前指令可能已经明确授权维护，但静默新增记录或改变状态会让后续默认依据在用户无感知时变化。
- 决策: 当前指令足以表达维护意图时直接执行；新增记录或改变状态前告知判断与集合变化，长期含义或范围不清时再请求判断。

## 目的
- 在不增加重复授权成本的前提下，让用户感知长期决策集合和状态的变化。
- 区分操作授权、长期判断是否已经形成以及写入前通知，避免把三者混成逐项确认。

## 背景
- 用户指令可能已经清楚表达新增、修订、归档或调整对齐状态的意图，此时再次询问是否执行不会增加有效信息。
- 决策维护会改变后续工作的默认依据；即使维护意图已经明确，也应在实际写入前让用户知道将改变什么。
- 普通工作中由 agent 提炼的候选可能仍是建议或一次性选择，只有长期含义、维护对象和影响范围清楚时才能写入。

## 决策
- 采用: 当前指令已经明确长期判断、维护对象和影响范围时，将该指令作为维护授权，不重复询问是否执行。
- 采用: 新增记录或改变生命周期、对齐状态前，明确告知用户将写入或改变的判断及集合变化。
- 采用: 当前指令只改变本次任务的执行方式时，归为一次性例外；只有后续工作的默认依据发生变化时才进入长期修订。
- 采用: 长期含义、维护对象或影响范围仍不清楚时，给出最小候选概览并只请求这一项判断。

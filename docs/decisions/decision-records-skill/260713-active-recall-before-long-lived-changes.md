# 2026-07-13 - 在长期行为变更前恢复决策并显式处理偏离

## 状态
- 当前状态: active
- 导致状态变化的决策: 无
- 状态说明: 作为 `skills/decision-records/` 主动恢复长期决策、检查重评条件和处理方向偏离的当前行为依据使用。

## 问题
- 只在用户显式要求回放或记录时读取决策，不能保证普通维护任务在改变长期行为前看见已有方向。
- 把“当前任务指令优先”作为唯一冲突处理规则，会混淆一次性例外和长期修订，后续可能在没有说明的情况下偏离已确认判断。
- `active` 决策仍可能因前提变化而需要重评；如果不检查记录中的触发条件，长期记忆会退化成机械执行的永久规则。

## 背景与约束
- 决策记忆继续使用公开、可版本化的 Markdown，不引入数据库、模型隐藏状态、全局知识库或强制扫描全部历史。
- 现有“重要长期取舍确认后才写入”的边界继续有效，主动恢复不等于自动修改决策历史。
- 代码、配置、规范和项目文档继续承接当前行为；决策记录只解释形成原因、适用条件和演进关系。

## 决定
- 采用: 任务可能改变长期行为、owner、边界、兼容性、风险处理或验收方式时，在实质改动前从活动索引读取相关 `active` 记录，因为只有在决策点前恢复上下文才能实际防止方向静默漂移。
- 采用: 恢复时同时对照当前行为 owner，并检查 `触发条件` 是否因上下文变化而需要重新评估，避免把 `active` 误解为不受前提约束的永久规则。
- 采用: 将当前任务与既有判断归为对齐、明确的一次性例外、长期修订或一致性问题；一次性例外按当前指令执行并说明不改变历史，长期修订沿用确认后写入流程，一致性问题在没有依据时不由 agent 自行选边。
- 采用: 目标项目需要冷启动发现能力时，只在自己的 agent 指令中增加指向活动索引和本 skill 的短路由，不复制固定契约或建立第二套记忆入口。
- 不采用: 每个任务都读取全部决策历史。原因是活动索引和影响面已经能够限定读取范围，完整历史只在审计原因时需要。
- 不采用: 发现当前实现或指令不同就自动改写决策。原因是差异可能只是一次性例外、实现缺陷或尚未确认的新方向。
- 触发条件: 任务可能造成长期行为变化、当前指令与既有判断不同、决策前提发生变化，或目标项目需要增加决策冷启动路由时，沿用本恢复与偏离分类流程。

## 影响
- `skills/decision-records/SKILL.md` 将长期变更前恢复、重评条件检查和偏离分类纳入读取流程与完成标准。
- 明确的一次性例外可以继续执行当前任务，但交付需要说明范围以及决策历史未改变。
- 新的长期方向仍需确认后写入，不因主动恢复而增加自动写入或频繁打断。
- `docs/skills/decision-records.md` 和 agent 接口以“长期决策记忆”而不只是“记录文件维护”描述该能力。

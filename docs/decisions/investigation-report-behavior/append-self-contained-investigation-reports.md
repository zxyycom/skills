# 以完整独立报告演进调查主题

## 索引摘要
- 目的: 让每次调查都具备正确背景并能独立汇报，同时保留同一问题的认识演进。
- 背景: 顶部概述加增量调查段会让后续结论依赖旧上下文，缺失当前背景时容易产生错误推断。
- 决策: 同一核心问题使用稳定主题文件，每次实质更新追加包含背景、起因和调查结果的完整报告。

## 目的
- 让 `skills/investigation-report/` 形成的每份报告都能脱离原对话和历史报告独立解释本次调查，避免项目处境、阶段或对象状态缺失时产生看似合理但实际错误的结论。
- 在同一核心问题上保留可回放的认识演进，同时让最新报告直接承担当前汇报口径，不再维护可能与正文漂移的顶部认识摘要。
- 让固定结构只约束不可缺少的语义，允许基线、方法、证据、推断、结论和未知按具体问题出现。
- 保持用户明确要求沉淀时才使用报告的非主动式入口，以及稳定主题路径、轻量索引和机械检查边界。

## 背景
- 原行为把顶部 `调查概述` 作为当前口径，并把后续工作追加为调查段。增量段即使保留局部材料，也容易省略理解新结论所需的项目现状、阶段和约束，迫使读者从旧段或原对话拼接背景。
- 背景不是可选装饰。调查结果与推断都依赖对象当时所处的条件；背景缺失或只写差异时，后续 agent 会把过期前提带入当前判断。
- 起因与背景承担不同责任：背景解释结果所处的上下文，起因解释为什么此时开展调查。一次复查的起因可以只是距上次已久、需要了解现状，但它仍需要完整当前背景。
- Git commit hash 只能定位已提交仓库快照，不能统一代表项目阶段、运行环境、外部状态或未提交改动，因此不适合作为所有报告固定背景字段。
- 范围、方法、证据、推断、结论和未知很重要，但并非每份报告都应以相同固定字段填写；强制模板会把报告推向填表，并掩盖背景、起因和结果这三个真正不可缺少的语义。

## 决策
- 采用: 一个稳定主题文件持续承接同一核心问题；能够独立调查、交接和结束的新问题创建兄弟文件。
- 采用: 每次新证据、现状复查、背景变化或解释变化产生实质认识时，在原主题文件末尾追加一份完整报告。每份报告依次包含非空的 `背景`、`起因` 和 `调查结果`，并能脱离其他报告独立阅读。
- 采用: 最新一份完整报告是当前认识的报告正文 owner，较早报告保留其形成时的完整快照。主题文件顶部只保留核心问题、状态和逐字等于最后一份报告形成时间的最新报告时间。
- 采用: 背景写本次结果所依赖的当前项目处境、阶段、对象状态和关键条件；起因写为什么此时调查；调查结果先写实际发现、适用边界或明确未知。
- 采用: 范围与方法、对象基线、证据、推断、结论、冲突、未知和下一步作为可选支撑内容放在三个核心章节之后。推断和结论必须回指调查结果与证据，不能替代结果。
- 采用: Git hash 只在结果依赖已提交快照时作为条件性定位；未提交改动影响结果时同时记录相关 diff、内容或工作区状态。与 Git 快照无关的报告不要求 hash。
- 采用: 纯错别字、格式、链接和不改变语义的定位修复可以原地编辑；会改变背景事实、调查结果、推断、结论或适用边界的维护通过追加新完整报告表达。
- 采用: 检查器机械验证调查信息、完整报告外壳、三个核心章节、形成时间顺序、最新报告时间和索引投影；背景充分性、证据质量、推断成立性与是否应拆分继续由人工审阅。
- 采用: `investigation-report` 继续只在用户明确要求记录、沉淀、创建、更新或审阅调查报告时触发，不扩张为普通调查的主动记录建议。

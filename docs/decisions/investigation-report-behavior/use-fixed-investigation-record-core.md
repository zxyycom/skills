# 使用固定调查记录核心并按场景补充正文

## 索引摘要
- 目的: 让每轮调查完整保存可比较的形成时输入、证据基础和结果边界，同时避免按报告规模或既有形态分配格式。
- 背景: 真实报告标题各异，但共同保留形成时背景、调查目的、调查范围与依据以及结果边界；互斥格式不能承接混合场景。
- 决策: 每份报告固定使用形成时背景、调查目的、调查范围与依据、调查结果与边界四段；场景支撑先写入对应核心，需要独立组织时再在其后展开。

## 目的
- 让 `skills/investigation-report/` 形成的每份报告保存结果产生时实际使用的背景、目的、证据条件和结论边界，使未来读者能够重放判断，不必补入默认前提或依赖原对话。
- 让同一核心问题在不同时间、对象状态、假设、调查目的或证据范围下形成的认识可以独立比较，即使结果相同也能判断适用条件是否一致。
- 让固定结构只承接所有调查都不可缺少的记录核心；计量、诊断、机制、方案和变更等内容继续由结论依赖的真实场景触发，不形成按篇幅或主题互斥的模板。

## 背景
- 前序决策为避免调查目标被主题名称、动因或默认前提隐含，固定了背景、调查动因、调查目标和调查结果。随后尝试把动因与目标按场景合并或分列，但这种设计仍要求 agent 在两套格式间选择，并把调查场景错误地建模成互斥结构。
- 对 13 份没有统一模板的工程调查档案进行审阅后，发现它们虽然章节名称不同，却都保存了带日期的调查身份、调查目的、结果，以及证据来源、方法、测量口径、适用范围或不能主张的内容。只固定背景、目的和结果会让没有证据基础的普通总结也通过。
- `形成时背景` 与 `调查范围与依据` 承担不同责任：前者说明结果是在怎样的处境和已知条件下形成，后者说明本轮实际检查了什么、凭什么形成结果以及没有覆盖什么。把两者合并会让对象条件与证据条件互相遮蔽。
- 计量比较、故障诊断、状态机制、候选方案和实际变更可以同时出现在一份报告中。将它们做成多套互斥格式会遗漏混合场景；全部做成固定字段又会使 skill 膨胀并产生填表文本。
- 固定核心各项的篇幅由实际信息决定；是否另设支撑章节由结论的场景依赖和复核需要决定，不能用简单或复杂作为格式分流条件。

## 决策
- 采用: 一个稳定主题文件持续承接同一核心问题；能够独立调查、交接和结束的新问题创建兄弟文件。每份报告是一份形成于特定时点、可脱离原对话和同主题其他报告独立阅读的认识快照。
- 采用: 每份报告依次使用非空的 `形成时背景`、`调查目的`、`调查范围与依据` 和 `调查结果与边界` 四个固定章节；固定核心只使用这些精确标题和顺序，不通过旧标题、章节别名或格式推断建立兼容分支。
- 采用: `形成时背景` 保存本轮结果所依赖的当时处境、触发认识、已知事实、假设、未知、关键约束和必要基线；即使与上一轮相同也完整重述，后来的认识不得倒写进旧报告。
- 采用: `调查目的` 独立写明本轮要回答的问题、支持的判断和预定边界，不能只由主题名称、背景或原对话隐含。
- 采用: `调查范围与依据` 写明实际检查的对象、来源、时点或版本、方法、实际覆盖范围以及未覆盖内容。实际依据仅为一次直接观察或单一材料时可以用一句话如实说明，但不能省略依据或虚构方法。
- 采用: `调查结果与边界` 区分事实、推断、建议、实际动作和未知，说明结果状态、适用条件、不能外推的内容以及复核条件；没有形成结论时记录当前未知和缺少的证据。
- 采用: 计量口径、假设与反证、时序与状态、方案对照、授权、恢复和验证等场景支撑先写入对应核心；需要独立组织才能复核结论时，再在四项核心之后展开。多种场景义务可以叠加；当前不把它们固化为替代核心的 profile 或互斥模板，反复出现且需要机械验证的新义务另行判断。
- 采用: 当背景、目的、证据范围、结果或解释发生实质变化并形成可独立汇报的认识时，在原主题文件末尾追加完整报告。最新报告承担当前认识，较早报告保留历史回放和跨背景比较。
- 采用: 检查器机械验证四个固定章节的存在、唯一、非空和顺序，以及形成时间和索引投影；内容是否充分、场景义务是否触发、证据是否支持结果和是否应该拆分继续由人工审阅。
- 采用: `investigation-report` 继续只在用户明确要求记录、沉淀、创建、更新或审阅调查报告时触发，不扩张为普通调查的主动记录建议。

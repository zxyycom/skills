## 目的
- 为格式、需求、数据模型、接口、流程和代码复用提供一套跨领域的统一判断方法。
- 让 agent 不只提出“做抽象”, 而能说明哪些场景可以共同依赖什么契约、统一为何成立, 以及为何需要一个或多个稳定公约数。
- 同时考虑存储、传输、领域语义和消费使用的不同义务, 避免为了单一表示把复杂度转移给其他层。

## 背景
- 多个对象可以拥有相似名称、字段或实现形状, 但它们的消费者、约束、生命周期和失败语义并不一定相同。直接统一容易产生大量可选项、类型标志、条件分支和调用方约定。
- 反过来, 沿用每个现有格式或实现也不能证明差异具有长期意义; 表面差异可能只是命名、展示或历史偶然。
- “抽象”能够提示寻找共性, 但不能单独约束共性的适用场景、充分性和消费者成本。“公约数”会要求设计者明确适用集与共同依赖边界; “最小公约数”则容易把 agent 引向字段更少或结构更空, 数学隐喻也会退化为没有设计价值的 `1`。
- 缺少证明真实共性的证据时, 暂时保持局部是安全的可逆动作, 但它不能证明场景之间不存在公约数。若不区分否定结论与未知状态, 后续维护会把尚待确认的假设固化为拆分依据。
- `product-architecture-judgment` 负责判断工程事项的产品价值、合理程度、责任 owner 和实现层级, 没有以跨场景比较为主流程形成共享契约、变体及表示映射。两项能力具有不同触发和验收出口。
- `skill-design-discovery`、`skill-maintainer` 和 `prompt-optimize` 分别拥有 skill 设计发现、能力与分发边界、结构化文本优化, 都不以跨领域的场景公约数为交付出口。

## 决策
- 采用: 新增独立分发的流程与决策型 skill `common-denominator-design`, 面向已经进入跨场景统一、合并或复用判断的任务。
- 采用: 将“公约数”定义为明确场景集合和责任层内, 各场景可以按同一含义共同依赖的契约; 消费者只依赖候选契约时应能处理适用集合中的任一场景, 除公开变体外不需要额外场景知识。
- 采用: 先检验候选能否完整承接共同目标、保持语义和约束一致并让消费者直接使用, 再通过边界收敛把局部义务和纯便利内容移出共享层; 不把“最小”放入能力名称或主承诺。
- 采用: 从现实场景的消费者、目标、语义、合法状态、操作、存储、失败和演进义务恢复比较矩阵, 再检验候选的语义保真、使用成本、存储往返、变化隔离、分组信号和边界收敛。
- 采用: 将“候选不成立”和“证据不足”作为不同结论; 只有现有证据支持时才否定公约数, 关键缺失信息按影响确认, 其余未知记录为假设或验证义务并暂时保持局部。
- 采用: 公约数数量由证据推导, 允许选择一个公约数、共享核心加显式变体、多个场景公约数或保持局部; 现有格式数量和追求单一抽象都不作为预设答案。
- 采用: 存储、传输和消费层义务不同时可以分别形成适合本层共同目标的公约数, 但必须明确事实源、映射方向、丢失规则和兼容责任。
- 采用: 新 skill 自包含运行, 不依赖产品与架构判断或其他分发单元; 上游价值与 owner 判断以及下游具体实现继续由当前任务和对应行为 owner 承接。

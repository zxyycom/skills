# 按共享测试基座组织多分支证明点

## 索引摘要
- 目的: 在保持证明点简短可审计的同时，避免多分支测试因机械拆分而重复 fixture、增加执行成本或造成共享数据底座漂移。
- 背景: 一条测试链路可以复用初始状态、基础数据和执行上下文，并在条件节点后形成多个可观察分支；按叶子结果逐项拆分会增加准备、同步和审计成本。
- 决策: automated case 可按共享基座或连续链路聚合分支；`Proves:` 使用原子结果列表，复杂分支用 Mermaid，只有独立边界才拆分。

## 目的
- 让账本按可共同维护的测试基座和行为链路组织，而不是把每个测试函数或分支叶子机械拆成独立 case。
- 让每个证明点保持简短、直接并可审计，同时保留多分支测试的共享数据、执行成本和漂移约束。
- 让维护者能从 Mermaid 恢复共享基座、分支条件、处理阶段和叶子断言之间的关系。

## 背景
- 同一初始状态、fixture、外部进程或连续行为链路经常需要覆盖多个 owner 已承诺的分支结果。
- 把每个分支拆成独立 case 可能复制昂贵准备工作，并让相同数据底座、状态同步或断言上下文分散演进。
- `Proves:` 承接的是当前语境下的具体证明点；case 标题、行为 owner、`Code:` 和 Mermaid 已经能够提供背景，不需要把证明点扩写成自由段落。
- 单纯按功能主题聚合仍会掩盖独立行为；是否保留多分支 case 需要同时考虑责任边界、共享链路、执行成本、审计范围和底座漂移风险。

## 决策
- 采用: automated case 是共享测试基座或连续行为链路的审计单元，可以包含多个测试函数和多个分支叶子，不按测试函数逐项登记。
- 采用: 每个 automated case 声明一个非空 `Proves:` 列表；每个列表项只表达一个简短、直接的分支条件与可观察结果或内部不变量，背景和理由由其他 owner 承接。
- 采用: 多个分支共享初始状态、基础数据、fixture、执行上下文或行为主干，且拆分会增加准备、运行、状态同步或审计成本，或造成共享底座漂移时，优先保留在同一 case。
- 采用: 多分支 case 使用 fenced Mermaid `flowchart LR` 或 `flowchart TD`，通过 `subgraph` 分组共享阶段或分支族，并按共享基座、分支条件、处理阶段和叶子结果组织；每个叶子对应一个 `Proves:` 项。
- 采用: 分支条件或结果不同本身不触发拆分。分支形成独立 owner、观察入口或行为链路，使用独立 fixture、运行环境或维护周期，或者共享底座已经稳定集中且拆分能降低总维护与审计成本时，再建立独立 case。
- 采用: CLI 机械要求 `Proves:` 至少包含一个列表项；Mermaid 只表达分支结构，不代替证明点列表，其语义质量继续由 `test-evidence-review` 审查。

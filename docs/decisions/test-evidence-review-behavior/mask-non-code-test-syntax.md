## 目的
- 让常见语言中的注释、字符串、模板、原始字符串和正则样本不再形成测试入口。
- 保持入口位置、marker 邻接和现有跨语言正则发现契约不变，避免降噪过程破坏映射诊断。
- 让 exempt case 聚焦无法由通用发现器可靠区分、且仍需要长期审计的真实误报。

## 背景
- 测试发现器直接对完整源文件运行入口正则，无法区分可执行语法与字符串、注释或正则字面量中的样本。
- CLI 扫描自身仓库时会把发现器模式、fixture 字符串和生成后的 bundle 识别为未登记测试入口；这些结果不代表项目测试，也不值得逐项建立长期豁免。
- 为每种语言引入完整 AST 解析会显著扩大单文件分发依赖和兼容边界，而当前问题主要来自可确定识别的非代码词法上下文。

## 决策
- 采用: 在测试入口正则之前屏蔽 Rust、TypeScript、JavaScript、Python、Go、Java 和 C# 的常见注释与字符串形式，并按语言处理模板、原始字符串、文本块和 JavaScript/TypeScript 正则字面量。
- 采用: 屏蔽以等长空白替换非换行字符，保留 UTF-16 offset、行和列，使 marker 邻接、错误位置和源码映射继续使用原始坐标。
- 采用: 为全部声明语言维护表驱动发现测试，同时证明真实入口保留、常见非代码样本被忽略。
- 不采用: 在当前分发单元内引入七种语言的完整解析器；宏、别名、自定义框架和少见语法继续作为已披露的剩余边界。

# 2026-07-21 - 分离测试入口采集与账本维护

## 索引摘要
- 目的: 让复杂项目能够替换入口收集方式，同时让账本维护只依赖稳定、可校验的数据契约。
- 背景: 账本 CLI 同时拥有文件选择、跨语言正则和账本校验，按语言屏蔽非代码上下文仍会漏掉有效入口，也让自定义发现规则必须进入账本实现。
- 决策: 以 Schema 为结构真源，把工具拆成输出标准清单的可替换采集层和只消费清单的账本层，并保留组合兼容入口。

## 目的
- 让账本维护模块不需要知道测试入口来自源码正则、AST、框架注册表还是外部清单。
- 为常见项目保留零配置入口检查，同时允许复杂项目独立定义文件范围、排除规则和正则，或完全替换采集实现。
- 让配置、标准清单、诊断、报告和查询结果只有一个结构 owner，避免运行时校验、JSON Schema 与 TypeScript 类型相互漂移。
- 保留日常单命令体验，并让严格校验与可恢复查询使用一致、可机器判断的完成语义。

## 背景
- 既有 `test-evidence.mjs` 同时读取源码、决定语言范围、发现入口、解析账本并检查 Git Scope，任何发现策略变化都会进入账本维护模块。
- 对注释、字符串和原始字符串做轻量语言屏蔽不能可靠等价于语法解析；例如 Go 原始字符串中的入口文本可能被隐藏，而其他宏、别名和框架注册形式仍可能漏报。
- 人工维护“哪些语言位置算代码”会不断扩大近似解析器，仍不能满足项目自定义框架或语法级精度需求。
- 查询把严格 error 以人类 warning 文案输出，但机器 diagnostic 仍保留 error severity，调用方无法直接判断该诊断是否阻断当前命令。
- 手写 TypeScript 数据类型再反向生成或同步 Schema，会让结构 owner 分散到多个文件。

## 决策
- 采用: `test-entry-regex` 是独立采集层。它提供七种语言的常见全文正则基线，并允许项目通过独立配置选择候选文件、排除文件、启用内置 detector 和声明自定义正则；没有定制需求时不创建配置。
- 采用: 正则采集不猜测匹配位于注释、字符串、fixture 或可执行语法。要求语法级精度的项目使用 AST、框架清单或其他自定义采集器，不在通用工具中继续维护近似语言解析器。
- 采用: 所有采集器输出版本化 `TestEntryInventory`，包含入口、marker 绑定和上游 diagnostics。`test-evidence-ledger` 只在 Schema 边界接收该清单，再维护 catalog、入口角色、Git Scope、review trigger 和查询；它不导入文件发现或正则实现。
- 采用: `test-evidence.mjs` 保留为组合兼容入口，在进程内连接默认正则采集器与账本层，并迁移旧 v2 组合配置；新配置分别使用账本 v3 和可选采集器 v1。
- 采用: Valibot Schema 是配置、清单、诊断、报告、inspection 和 query 的源码真源；源码类型使用 `InferOutput`，构建时从同一 Schema 生成 JSON Schema，再生成分发 TypeScript 数据声明。不得从手写类型反向生成 Schema。
- 采用: diagnostic 同时包含固有 `severity` 和命令级 `blocking`。严格检查按 `blocking` 决定完成状态；恢复查询复制严格诊断并设为非阻断，不篡改其 error 或 warning 严重度。
- 不采用: 让账本配置继续承接语言、文件发现或正则字段，也不要求复杂项目把入口规则提交给通用账本维护模块。

# 2026-07-19 - 分离测试价值审查与账本机械校验

## 索引摘要
- 目的: 抑制无意义、重复和实现耦合的测试增长，同时让测试证明目标与实现保持可审计映射。
- 背景: 只维护测试账本会增加同步成本，单靠测试执行又无法阻止 AI 以 TDD、防回归或覆盖率为由持续新增低价值测试；调用 skill 后再判断是否启用账本还会造成流程分叉和漏登记。
- 决策: 使用 `test-evidence-review` 对自动化测试执行证据准入，并默认以账本保存通过准入的稳定证明目标，再用可配置的跨语言 CLI 校验主要入口、辅助归属、豁免和未登记测试文件。

## 目的
- 抑制无意义、重复、只覆盖实现细节或无法追溯到稳定 owner 的测试增长。
- 让新增和现有测试能够说明独立证明目标，并在账本与源码之间保持低成本、可机械检查的映射。

## 背景
- AI 在采用 TDD、补回归测试或追求覆盖率时，容易把不同字面量、相似非法输入和实现路径分别写成测试，却没有增加新的稳定证据。
- 单独维护账本只能记录测试存在，不能在编写前判断它是否值得存在；如果所有测试函数都成为独立 case，账本本身又会形成新的维护负担。
- 账本状态、源码路径、marker 和未登记测试文件适合确定性校验；证明目标、等价类、owner 承诺和可观察结果仍需要语义审查。
- skill 的触发已经表达了采用这套测试审查方式；再让 agent 判断项目是否启用账本，会让相同任务产生不同登记结果并增加使用成本。

## 决策
- 采用: 新增 `test-evidence-review` skill，作为测试证据准入、重复判断、语义巡检和账本登记流程的行为 owner。
- 采用: 每次调用 skill 都启用账本契约；账本不存在时，修改类任务按默认路径初始化并登记当前范围，只读任务输出拟登记映射而不越过写入授权。
- 采用: 让账本保存稳定证明目标，而不是枚举全部测试函数；每个 implemented case 使用唯一主要 `@case`，辅助测试使用可重复的 `@supports`，发现豁免使用带具体原因的 `@test-exempt`。
- 采用: 从 Docnav case catalog validator 提取 Markdown 解析和双向映射逻辑，并扩展为可配置的 Node CLI；以文件为首个强制归属单元，发现 Rust、TypeScript、JavaScript、Python、Go、Java 和 C# 常见测试入口，并允许项目在 ignore、warn 和 error 之间选择未登记策略。
- 采用: 让测试框架和 CI 继续负责测试执行，让项目测试策略继续拥有测试层级和覆盖要求；`product-architecture-thinking` 可以在相关工程判断中消费测试证据结论，但不成为测试账本或测试价值规则的新 owner。

# 让决策查询带警告返回可用结果

## 索引摘要
- 目的: 让局部记录或关系损坏不阻断其他长期判断的恢复，同时保留严格结构检查入口。
- 背景: `list` 和 `trace` 与 `check` 共用全库严格校验门禁，任一无关记录、关系或派生字段漂移都会让全部只读查询失败。
- 决策: 中央索引可解析且目标存在时，`list` 与 `trace` 返回可恢复结果并以 warning 暴露错误；检查和写事务仍严格失败。

## 目的
- 让 agent 在仓库存在局部编辑中状态或单条损坏记录时，仍能恢复其他可用的当前判断和关系。
- 让查询降级显式暴露不完整性，同时保持维护和交付的严格一致性门禁。

## 背景
- `check` 需要全库严格失败，才能证明目录、索引、正文和关系共同符合固定契约。
- `list` 和 `trace` 的主要用途是恢复已有信息；把同一严格门禁放在查询前，会让一个无关主题的索引漂移或正文错误阻断全部恢复。
- 扫描器即使发现局部错误，仍能返回已解析索引、记录元数据和有效关系边；这些结果可以在附带诊断时继续使用。
- 中央索引决定当前成员。索引缺失或无法解析时，CLI 无法可靠区分当前与归档记录，因此不能把这种情况作为普通局部 warning。

## 决策
- 采用: `check`、`sync-index`、`activate` 和 `archive` 继续要求完整校验通过，并在结构或索引错误时返回失败。
- 采用: `list` 在中央索引可解析时返回能够扫描到的目标生命周期记录；记录自身存在结构错误时继续显示并标记为 `invalid`。
- 采用: `trace` 在中央索引可解析且起点存在时，使用成功解析的关系边返回能够恢复的关系子图；损坏或被跳过的关系由 warning 说明。
- 采用: 非阻断查询把完整校验问题写入 stderr，查询本身成功时退出码为 `0`，让调用方同时获得可用结果和诊断。
- 采用: 中央索引缺失或无法解析、`trace` 起点不存在时仍返回失败，因为这时无法可靠完成请求。

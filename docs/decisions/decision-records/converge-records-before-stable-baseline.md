## 目的
- 让决策关系只表达已经作为后续工作依据的判断如何发生变化，不把同一维护事务中的方案试探和收敛保存成长期历史。
- 在保留既有决策稳定身份的同时，允许当前事务中新建的记录在进入稳定基线前修正语义。

## 背景
- 既有契约只允许错别字、格式、链接和不改变语义的澄清原地完成，任何目的、范围、背景前提、采用方向或核心理由变化都要求创建后序记录。
- 新决策在设计和实现尚未交付时可能连续调整。记录及索引仍只存在于工作区或暂存区、也没有被其他工作依赖时，这些版本没有分别成为长期依据。
- 把每个中间版本都激活、归档并建立关系，会把一次方案收敛表现成多次正式政策变化，增加索引噪声和后续追溯成本。
- 已经进入版本基线、明确对外发布或被其他工作依赖的判断仍需要稳定身份和演进关系，不能因为当前工作区尚有未提交改动而原地覆盖。

## 决策
- 采用: 区分已建立记录与当前维护事务中新建的未建立记录；精确定义和维护事务由 `skills/decision-records/references/decision-record-rules.md` 承接。
- 采用: 记录及索引版本已经存在于当前 Git `HEAD` 时，默认视为已建立；已经明确对外发布、交接或被其他工作作为依据时同样视为已建立。
- 采用: 已建立记录发生决策语义变化时，继续创建完整后序、归档直接前序并建立真实演进关系。
- 采用: 当前事务中新建、尚未进入稳定基线且未被其他工作依赖的记录，可以在原路径更新完整语义和索引投影，不为同一事务的中间版本创建归档记录或关系。
- 采用: 无法确认记录是否已建立、是否已被依赖或变化是否影响决策语义时，继续按决策演进处理。
- 不采用: 在生命周期索引中增加草稿状态。原因是索引应继续承接已经确认并作为默认依据的决策，草稿状态会让长期检索承担工作日志职责。

# 分离逻辑归档与决策关系

## 索引摘要
- 目的: 准确区分普通退出与真实决策演进，并保持当前集合和关系链可验证。
- 背景: 强制每条逻辑归档记录都被后续记录引用，会把没有后续判断的普通退役误建模为决策演进；现有校验只要求任意入链，也不能证明 `archive --by` 指定的后续记录真实承接全部前序。
- 决策: JSON 索引维护当前成员；记录可独立归档，真实演进才建直接关系；`archive --by` 校验后续承接全部前序，CLI 拒绝关系环。

## 目的
- 准确区分普通退出与真实决策演进，并保持当前集合和关系链可验证。

## 背景
- JSON 当前索引、显式 Markdown 摘要和稳定文件路径已经能够低成本恢复当前判断，并避免生命周期变化造成链接漂移。
- 逻辑归档表示一项判断不再作为后续工作依据，但退出原因不一定是另一项需要长期记录的新判断。
- 强制所有归档记录拥有后续关系，会为普通退役制造占位决定，也会让当前索引保留不必要的“墓碑”判断。
- 现有结构检查只验证归档记录是否成为任意关系目标；它不能排除关系环，也不能确认 `archive --by <new>` 中的 `<new>` 就是直接承接全部前序的记录。

## 决策
- 采用: `docs/decisions/decision-index.json` 继续作为当前成员关系的唯一 owner，每个当前条目保存由 Markdown 显式摘要派生的 `path`、`title`、精简 `background` 和 `decision`。
- 采用: 决策文件继续使用稳定路径；退出当前集合不移动或重命名 Markdown。
- 采用: 一项判断只需退出后续工作依据且没有后续判断时，允许使用 `archive <path...>` 独立逻辑归档，不建立关系或占位记录。
- 采用: 只有修订、替代、判定无效或归并真实发生时，才由新记录单向链接本次直接作用的前序。
- 采用: `archive <old...> --by <new>` 必须确认 `<new>` 直接引用每个 `<old>`，再归档前序并激活后续。
- 采用: CLI 拒绝关系环，并继续通过反向扫描和 `trace` 查询前后关系链。
- 采用: 当前记录独立表达完整判断；索引摘要必须只包含作者明确概括的一条背景和一条决策。

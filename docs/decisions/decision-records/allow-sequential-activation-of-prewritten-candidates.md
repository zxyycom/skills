# 允许逐条激活预写决策候选

## 索引摘要
- 目的: 允许一次准备多条已确认决策并逐条激活，同时保持遗漏候选能够被明确发现。
- 背景: 单条 activate 的完整集合校验会让其他预写候选阻断目标激活，迫使维护者交替写入和激活文件。
- 决策: 识别未进入 HEAD 的完整未激活候选；作用域维护可在逐条提醒后继续，索引排除候选，严格 check 继续阻断遗漏。

## 目的
- 支持先完整写好多条已经确认的决策，再按明确顺序逐条激活，而不要求在每次激活之间重新创建文件。
- 让每次激活仍只改变显式目标，其他预写文件保持未生效且可被清楚识别。
- 保持最终集合严格完整，避免候选被静默忽略或作为孤立文件长期遗留。

## 背景
- `activate <path>` 只表达一个目标记录及其对齐状态，单条激活有利于保持状态变化和失败恢复边界清楚。
- 当前完整集合校验把其他 `createdAt: null` 文件视为普通无效记录；同时预写多条文件时，任一目标激活都会被其余文件阻断。
- 直接忽略全部未登记 Markdown 会掩盖损坏记录、错误路径和已进入正式历史但缺失元数据的文件，破坏严格检查的证明价值。
- 预写候选在激活前不是有效决策，也不应为了支持维护流程而引入新的生命周期状态或隐藏登记表。

## 决策
- 采用: 未进入 Git `HEAD`、尚未进入索引、正文结构完整，并声明 `status: active`、显式 `alignment` 和 `createdAt: null` 的新 Markdown，识别为未激活决策候选。
- 采用: 未激活候选不是集合成员、不生效、不进入索引，也不标记为 `pending`；`pending` 继续只描述已经激活但路径尚未进入 `HEAD` 的记录。
- 采用: `activate <path> --alignment aligned|unaligned` 继续只激活显式目标；其他合格候选不阻断该事务，命令成功后必须逐条列出仍未激活的候选并说明严格检查仍会失败。
- 采用: 状态维护和索引同步只在明确目标与已建立记录上执行；存在其他合格候选时可以完成当前事务，但必须显示同样的未激活提醒，且生成索引始终排除这些候选。
- 采用: 严格 `check` 在任一未激活候选存在时返回失败；只有所有候选都已激活或丢弃后，集合才能通过最终验收。
- 采用: 容忍范围只覆盖上述可机械识别的候选。格式不完整、已进入 `HEAD` 或不满足候选状态组合的 Markdown 继续作为阻断错误；索引有效时，带非空 `createdAt` 却未登记的 Markdown 也不自动吸收。只有索引缺失或损坏时，才按固定契约从全部非候选且有效的当前格式 Markdown 恢复完整索引。
- 采用: 候选不能充当关系前序或关系目标；候选自身声明关系时，目标仍必须是已经归档且存在于 `HEAD` 的已建立决策。

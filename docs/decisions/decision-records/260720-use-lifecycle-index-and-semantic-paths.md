## 目的
- 让决策的稳定身份表达主题与语义，不把形成时间混入名称和路径。
- 让单一 JSON 索引同时承接活动与归档记录的生命周期元数据、低成本摘要和直接关系。
- 让 CLI 不可用时仍可直接读取索引和 Markdown，理解记录内容及正常命令会产生的大致效果。

## 背景
- 决策记录是按语义恢复长期判断的知识集合，不是以日期作为主要身份的时间日志。
- 日期仍有展示、排序和历史回放价值，但它属于记录元数据，不需要重复进入文件名和正文标题。
- 只在索引中保存活动成员会让归档记录的时间和摘要退出集中入口，查询全部生命周期时仍需重新扫描 Markdown。
- 标题、目的、背景和决策摘要虽然来源于 Markdown，但在索引中保留可读投影，可以在 CLI 或 Node 环境不可用时继续完成基本恢复。
- 正常维护仍以 CLI 的同步和校验为准；无 CLI 降级只需支持直接阅读和理解操作效果，不承诺人工复刻等价事务。

## 决策
- 采用: 决策 Markdown 使用 `<topic-id>/<semantic-slug>.md` 作为相对决策根目录的稳定身份，路径和正文标题不包含日期。
- 采用: Markdown 继续使用 `索引摘要`、`目的`、`背景` 和 `决策` 四个必需章节，`关系` 按需出现；完整章节承接完整语义。
- 采用: JSON 索引以 `records` 登记全部决策，每条记录保存 `path`、`status`、`createdAt`、`title`、`purpose`、`background`、`decision` 和 `relations`。
- 采用: `status` 使用 `active` 或 `archived`；`createdAt` 是索引拥有的不可变元数据，使用带时区的 RFC 3339 字符串并保存来源能够可靠提供的最高精度。
- 采用: `title`、`purpose`、`background`、`decision` 和直接 `relations` 从 Markdown 同步；关系项保存类型和目标路径，反向关系由索引推导。
- 采用: 索引中的 `records` 固定按 POSIX `path` 升序排列，不因状态或时间变化重排。
- 采用: CLI 继续作为正常查询、维护和严格校验入口；CLI 不可用时可以直接读取索引和正文，并依据命令说明理解大致操作效果。
- 采用: 既有记录时间的恢复来源和迁移方式作为独立问题处理，不在本判断中固定。

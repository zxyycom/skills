# 从自包含决策文件生成完整索引

## 索引摘要
- 目的: 让每条决策自包含全部权威状态，并使集中索引成为可以随时无损重建的查询投影。
- 背景: 由集中索引独占生命周期和创建时间会形成第二真相源，索引损坏后无法仅凭决策文件恢复完整集合。
- 决策: 决策文件保存 status、alignment 和 createdAt 等权威元数据；集中 JSON 索引完全由 Markdown 生成，不拥有独立事实。

## 目的
- 让单条决策文件脱离集中索引后仍能独立恢复自身身份、生命周期、对齐状态、创建时间和完整语义。
- 让集中 JSON 索引只承担快速筛选、摘要展示和关系查询，不再成为必须与正文分别维护的状态 owner。
- 让索引可以从全部决策文件确定性重建，并通过机械比较发现任何投影漂移。

## 背景
- 生命周期、对齐状态和创建时间都属于单条决策自身的权威信息，把其中一部分只保存在集中索引会造成两个需要协调的真相源。
- 对齐状态会随决策落实而改变；如果只修改索引，单独阅读 Markdown 无法判断该决策当前应被如何执行。
- 标题、摘要和关系已经由 Markdown 生成，继续让状态元数据由另一文件独占没有必要。
- 集中索引仍然适合承载全库查询投影，但它应当能够被删除并从决策文件完整重建，而不是保存无法从正文恢复的事实。

## 决策
- 采用: 每条决策 Markdown 在正文标题前使用严格 frontmatter，保存 `status`、`alignment` 和 `createdAt`；标题、索引摘要、完整章节和关系继续保存在正文。
- 采用: `status` 只能是 `active` 或 `archived`；活动记录的 `alignment` 只能是 `aligned` 或 `unaligned`，归档记录的 `alignment` 必须是 `null`。
- 采用: `createdAt` 是首次激活时写入决策文件的不可变秒级 RFC 3339 时间，带显式时区且不含小数秒。
- 采用: 决策继续使用 `<topic-id>/<semantic-slug>.md` 作为稳定身份；索引记录固定按 POSIX 路径升序排列。
- 采用: 集中 JSON 索引登记全部活动和归档记录，并投影 `path`、`status`、`alignment`、`createdAt`、标题、目的、背景、决策和直接关系。
- 采用: 索引中的每个字段都从 Markdown 生成，索引不拥有独立状态；删除索引后可以从全部有效决策文件无损重建。
- 采用: 正常维护不直接编辑索引；CLI 在状态命令或正文同步后生成完整候选索引、严格校验，再一次性写回。
- 采用: 实现只承接当前格式和当前 schema，不读取旧 schema、不提供兼容映射，也不为缺失对齐状态设置默认值。

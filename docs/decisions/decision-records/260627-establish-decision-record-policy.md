# 2026-06-27 - 建立决策记录策略

## 索引摘要
- 目的: 只沉淀值得长期回放的重要判断，并让索引与维护规则各有明确 owner。
- 背景: 决策记录需要帮助后续维护者回放重要设计原因, 但过多项目级记录会把真正重要的判断淹没。
- 决策: 按影响面目录组织决策，分别由清单索引和维护规则承接导航与契约；文件命名由后续状态规则确定。

## 目的
- 只沉淀值得长期回放的重要判断，并让索引与维护规则各有明确 owner。

## 背景
- 决策记录需要帮助后续维护者回放重要设计原因, 但过多项目级记录会把真正重要的判断淹没。
- 早期记录把项目文档、AGENTS、工具链和决策目录调整都写成独立决策, 粒度过细, 接近变更日志。

- 本项目的主要维护对象是 skill, skill 行为、读取策略、规则 owner、边界和验收标准比普通项目文档更值得记录。
- 项目级文档、AGENTS、脚本和 CI 都有各自 owner, 普通调整可以由 owner 文档、git diff 和提交说明承接。
- 用户希望决策目录按影响面组织, 同时明确决策常常来自“先提出问题, 再多轮对话逐步收敛”的过程。

1. 先确定决策记录要从单文件改为目录组织, 一级目录使用影响面 ID, 单条决策使用日期加标题命名。
2. 后续发现项目级记录太多: 很多内容只是把已经落地到 README、AGENTS 或工具链文档里的规则再记录一遍。
3. 用户进一步明确: 决策的作用是回放当时情况和设计原因, 项目级决策不应细到每个执行细节, 不重要的记录可以直接抛弃。
4. 因此把决策记录策略收敛为高门槛机制: 只保留后续维护需要回放原因的判断, 并允许删除低价值旧记录。

## 决策
- 采用: 决策记录按 `docs/decisions/<impact-area-id>/` 组织, 清单索引和维护规则分别由 `decision-record-index.md` 与 `decision-record-rules.md` 承接; 具体文件命名以后续状态规则为准。
- 采用: skill 相关决策可以稍细, 但必须影响后续行为、读取策略、规则 owner、边界或验收标准。
- 采用: 项目级决策只记录改变长期维护契约、目录边界、自动化交付方式或跨文件 owner 的判断。
- 采用: 决策正文使用显式小节, 在 `## 决策过程` 中回放多轮收敛过程。
- 采用: 低于当前门槛、没有独立回放价值的旧记录可以删除, 不需要为了保存历史而保留。
- 不采用: 为 README、AGENTS、工具链脚本、CI 步骤、格式调整或一次性执行细节分别保留决策记录。

## 目的
- 让决策身份稳定表达主题与语义，不把日期混入路径或正文标题。
- 让单一 JSON 索引恢复全部活动和归档记录的生命周期、参考时间、摘要和直接关系。
- 让同日多条记录保留可比较的先后参考，并让既有项目可以从版本历史恢复可用时间。
- 让首条已确认记录无需独立空索引步骤即可进入可维护集合。

## 背景
- 决策记录主要按语义恢复长期判断，日期是辅助参考而不是身份。
- 全生命周期索引让 CLI 或 Node 暂时不可用时仍能大致读取状态、摘要和关系，但正常维护仍需要确定性校验。
- `createdAt` 只用于提供秒级参考和可比较先后，不承担判断形成瞬间的取证语义。
- 本仓库提交频繁，沿文件历史找到的最早 Git 作者时间比统一迁移或导入时刻更接近记录真实建立时间。
- 首次激活已经明确目标和活动状态；额外创建空索引没有独立产品意义，但损坏索引不能被静默覆盖。

## 决策
- 采用: 新决策使用 `<topic-id>/<semantic-slug>.md` 和无日期正文标题作为稳定身份；既有日期路径作为兼容身份保留。
- 采用: Markdown 使用 `索引摘要`、`目的`、`背景` 和 `决策` 必需章节，`关系` 按需出现；完整章节承接完整语义。
- 采用: schema v3 索引以 `records` 登记全部决策，每条保存 `path`、`status`、`createdAt`、四个检索投影和直接 `relations`。
- 采用: `createdAt` 是不可变的秒级 RFC 3339 元数据，必须带显式时区且不含小数秒；它只提供参考时间和可比较先后。
- 采用: 新记录以首次 `activate` 的秒级时刻建立 `createdAt`。显式迁移既有记录时，优先沿重命名历史取最早 Git 作者时间，Git 无证据时再参考操作系统时间；来源冲突或只能恢复到日期时披露不确定性。
- 采用: 标题、目的、背景、决策和直接关系从 Markdown 同步，索引记录固定按 POSIX `path` 升序排列。
- 采用: 每个生命周期写命令从全部 Markdown 和现有索引生成完整候选索引，写入后执行完整校验；失败时恢复原索引。
- 采用: 索引不存在时，`activate` 只在目录中恰好有一条有效 Markdown、目标就是该记录且没有其他结构错误时创建 schema v3 索引；已存在但无效的索引不覆盖，首次校验失败则删除新索引。

# 2026-07-20 - 在首次激活时自动初始化当前索引

## 索引摘要
- 目的: 让首条已确认决策无需独立的空索引初始化步骤即可进入当前集合。
- 背景: `activate` 已经明确表达目标记录和当前成员意图，但索引缺失时仍会失败；单独初始化空索引会增加一个没有独立语义的维护步骤。
- 决策: 索引缺失且目录有效时，`activate` 创建 schema v2 索引并激活目标；无效索引不覆盖，首次校验失败则删除新索引。

## 目的
- 让决策记录目录从首条已确认判断自然进入可维护状态。
- 保留索引成员的显式意图和写入后的完整结构校验，不引入独立的空初始化状态。

## 背景
- 新项目可以先写出首条已经确认的决策 Markdown，再通过 `activate <path>` 明确它应进入当前集合。
- 原有 CLI 在 `decision-index.json` 缺失时拒绝所有成员操作，即使 `activate` 已经给出唯一目标。
- 新增独立 `init` 命令会先创建空当前集合，随后仍需运行 `activate`；这个中间状态没有额外决策语义。
- 已存在但损坏的索引可能承接无法解析的成员信息，不能把它当作从未初始化并静默覆盖。

## 决策
- 采用: 首次写入先创建符合固定契约的已确认决策 Markdown，再运行 `activate <path>`。
- 采用: 索引不存在、决策目录和已有 Markdown 除索引缺失外均通过扫描且目标记录存在时，`activate` 直接生成 schema v2 索引并把目标记录加入当前集合。
- 采用: 首次索引写入后运行完整 `check` 语义；校验失败时删除新建索引，不留下部分初始化状态。
- 采用: 已存在但为空、无法解析或不符合 schema 的索引保留原状并报告错误，不作为首次初始化处理。
- 采用: `sync-index` 不推断空当前集合，`archive` 也不在缺少成员依据时初始化；这些命令继续要求有效索引。

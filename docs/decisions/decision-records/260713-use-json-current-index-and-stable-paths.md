# 2026-07-13 - 使用 JSON 当前索引和稳定决策路径

## 索引摘要
- 目的: 让当前决策集合可机器维护、低成本检索，并为历史记录保留稳定身份。
- 背景: 当前索引既要明确成员，也要用经确认且与正文一致的精简背景和决策支持低成本判断；主题分类已经包含在路径中。
- 决策: 使用 `docs/decisions/decision-index.json` 作为当前成员关系的唯一 owner，并保存由 Markdown 显式摘要派生的 `path`、`title`、精简 `background` 和 `decision`；决策文件使用稳定路径，逻辑归档只退出当前集合，历史演进由新记录单向链接直接前序并由 CLI 反查。

## 目的
- 让当前决策集合可机器维护、低成本检索，并为历史记录保留稳定身份。

## 背景
- 当前索引既要明确成员，也要用精简背景和决策支持低成本判断；主题分类已经包含在路径中，不需要重复字段。
- 状态同时写入文件名、正文和 Markdown 索引，使一次生命周期变化需要重命名文件并修复链接，长期维护成本随关系数量增长。
- Markdown 索引混合阅读说明和机器维护区，CLI 很难在不处理排版的情况下可靠改变当前成员。
- 物理归档会改变完整路径，而长期决策需要稳定身份才能被行为 owner、历史关系和外部说明持续引用。

- 决策内容、原因和取舍继续使用可直接审阅的 Markdown，不把长期记忆迁入数据库或隐藏状态。
- 当前集合需要是公开、可版本化、可校验的明确状态，并允许 agent 在长期行为变化前低成本定位相关判断。
- 已有短日期正文、按需展开背景、直接前序关系和归并读取边界继续有效。
- 用户确认由 CLI 维护 JSON 当前索引，文件名不再包含状态，归档记录不进入当前索引但仍可通过参数查询。
- 用户进一步确认决策记录只保留简单背景和采用的决策，不预估适用场景或重评条件。
- 用户确认索引需要包含标题、精简背景和决策，不能只有定位信息；Markdown 必须显式提供摘要，不能由 CLI 默认取正文第一条；`area` 可由 `path` 第一段获得。
- 用户确认候选概览经确认后直接形成索引摘要，并要求修改完整背景或采用决定时同步复核摘要与正文的语义一致性。

## 决策
- 采用: 使用 `docs/decisions/decision-index.json` 作为当前成员关系的唯一 owner；每个当前条目只保存由 Markdown 派生的 `path`、`title`、精简 `background` 和 `decision`。
- 采用: 决策文件固定使用 `YYMMDD-short-title.md`，创建后不因生命周期、关系、标题润色或 owner 调整而重命名或移动。
- 采用: 逻辑归档只表现为记录退出 JSON 当前集合；Markdown 保留在原主题目录，不建立物理归档目录。
- 采用: 决策正文要求显式 `索引摘要`、`背景` 和包含 `采用` 的 `决定`；索引摘要中的背景与决策由作者明确概括，CLI 不从正文位置或顺序猜测。
- 采用: 经确认的候选概览中的背景和采用决策直接写入 `索引摘要`；正文沿用该方向展开原因、约束和细节。修改完整背景或采用决定时同步复核摘要，确保影响采用方向的限定仍能从索引恢复。
- 采用: 从正文和文件名移除状态；修订、替代、判定无效和归并只由新记录单向指向直接前序，CLI 负责反查后续记录和关系链。
- 采用: CLI 默认列出当前记录，通过 `--archived` 或 `--all` 查询历史，通过 `trace` 查询关系链；`activate` 和 `archive --by` 校验式更新成员关系，`sync-index --write` 只刷新派生字段。
- 采用: 当前记录继续独立表达完整结论，关系只链接直接前序；历史妨碍日常理解时使用归并记录建立新的读取起点。
- 不采用: 从全部 Markdown 自动推断当前集合。原因是“哪些判断仍作为后续依据”本身是需要显式确认和版本审阅的长期状态。
- 不采用: 在 JSON 中复制完整背景、备选方案和决策过程。原因是索引只承接快速判断所需的精简背景和决策，完整语义继续由 Markdown 唯一承接。

# 从 HEAD 路径临时识别待提交决策

## 索引摘要
- 目的: 让提交前反复收敛的决策保持可见，同时避免形成虚假的正式演进历史。
- 背景: 持久化草稿状态或查询历史基线都会为简单的提交前判断增加额外维护成本。
- 决策: 始终查询当前记录，并按 Markdown 路径是否存在于 HEAD 临时标记 pending。

## 目的
- 让 agent 默认看到当前工作区中准备采用的决策，同时清楚区分它是否已经进入正式版本历史。
- 让提交前的方案调整可以原地收敛或直接放弃，不把中间版本归档成长期演进记录。
- 用一次可预测的 Git 机械检查建立写操作边界，不引入额外 baseline、缓存或持久草稿状态。

## 背景
- 新决策即使已经写入 Markdown 和索引，也可能在首次提交前随设计与实现连续调整；这些版本尚未成为稳定的历史依据。
- 把 `pending` 写入索引会要求额外的显式状态迁移，而从 `HEAD` 读取旧索引、正文和关系又会为当前查询引入第二套数据视图。
- 当前索引已经给出需要核对的记录集合。批量取得决策根目录在 `HEAD` 中的 Markdown 路径后，按路径存在性即可区分首次提交前的记录。
- Git 对大幅改名只能提供启发式相似度。把重命名识别纳入生命周期判断会让同一工作区在不同内容变化下产生不稳定结果。

## 决策
- 采用: `decision-records` 只服务 Git 工作区；Git 命令、仓库或 `HEAD` 路径不可读取时直接报错退出，不提供非 Git 降级模式。只有 `HEAD` 仍为有效符号引用且尚不能解析到任何版本时，才把路径集合视为空；其他 Git 查询错误不得伪装成尚无首个提交。
- 采用: `list`、`show`、`trace` 和写命令始终读取当前工作区的 JSON 索引、Markdown 与关系，不从 `HEAD` 恢复另一份索引或正文视图。
- 采用: 每次 CLI 调用在正常路径中只执行一次批量 Git 树查询，取得决策根目录下全部 `HEAD` Markdown 路径；不逐文件启动 Git，也不读取历史内容。
- 采用: 当前已登记且正文存在的记录路径不在 `HEAD` 时，CLI 临时显示 `pending`。该标记从不写入索引或正文，`git add` 不会使其消失，首次提交后也不需要显式状态迁移。
- 采用: `active [pending]` 可以表达当前事务准备采用的判断，但尚不充当正式历史前序。它在首次提交前原地收敛；不再成立或语义身份变化时用 `discard` 删除正文和索引条目，再按需创建新记录。
- 采用: `archive` 拒绝 `pending` 记录；`discard` 只接受路径不在 `HEAD` 且没有被其他关系引用的记录，并且不修改 Git 暂存区。
- 采用: 历史关系目标必须在当前索引中已经归档，且其 Markdown 路径存在于 `HEAD`，避免把尚未正式建立的中间判断写成演进前序。
- 采用: CLI 不推断重命名。已存在于 `HEAD` 的路径从工作区消失是严格错误；另一路径上的当前文件仍独立按 `pending` 处理。
- 不采用: 持久化 `pending` 生命周期或额外 baseline 查询范围。它们会引入显式迁移、双重内容来源和当前任务不需要的历史读取成本。

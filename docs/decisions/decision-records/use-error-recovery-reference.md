## 目的
- 为旧项目升级、索引缺失或损坏、写入中断和 CLI/Node 故障提供按故障类型可直接执行的恢复入口。
- 让正常查询和维护不承担异常路径的上下文成本。
- 让恢复优先保留 Markdown、生命周期、时间和关系证据，并最终回到机械校验。

## 背景
- 固定契约适合定义正确状态，却不需要在每次正常使用时展开所有故障分类和修复顺序。
- CLI 不可用时仍可以直接读取索引和 Markdown，大致判断内容与正常命令效果；此时无法证明人工操作与校验式事务等价。
- 索引损坏或旧 schema 升级需要结合最后有效索引、Markdown 和 Git 历史，临时猜测容易覆盖仍可恢复的状态和时间。
- schema v2 到 v3 同时改变索引字段、CLI 调用和可导入模块的结构化结果；分发包维护当前 v3 结构，旧项目需要一次性升级说明。

## 决策
- 采用: 在 skill 分发单元内维护独立恢复手册，承接触发条件、故障分类、恢复顺序和完成检查；schema 与正常维护事务引用固定契约。
- 采用: 只有现有集合冷启动或升级、CLI/运行时不可用、索引缺失或损坏、写入中断，或严格检查失败且普通诊断不足时读取恢复手册。
- 采用: 恢复先保留现状和 Git 证据，再区分刷新投影、重建索引、恢复缺失成员和修复运行时；不以空索引覆盖已有材料，也不为消除错误删除无法解释的记录。
- 采用: 恢复手册用独立升级报告说明 schema v2 到 v3 的产品目的、生命周期与时间映射、CLI 和模块调用调整；skill 分发包只维护当前 v3 结构。
- 采用: 时间优先沿文件历史取最早 Git 作者时间，生命周期优先取最后有效索引或可确认的 Git 状态；不能确认的语义、状态、时间或关系请求判断。
- 采用: CLI 不可用时只描述直接读取结果和正常命令的大致效果，不把人工编辑称为等价事务。
- 采用: 恢复以严格 `check` 返回 `0`、全生命周期列表符合预期且 Git diff 没有证据丢失为出口。

# 2026-06-27 - 按影响面组织决策记录

问题:
- 单文件保存全部决策会随着项目演进不断膨胀, 后续查找和维护成本会变高。
- 不同决策影响的边界不同, 混在一个文件里会弱化 owner, 也难以看出某个影响面的历史脉络。
- 用户希望参考能力面命名思路, 用一级目录表达影响面或层级, 再用单个文件承接每条决策。

决策过程:
- 决策记录本身是长期维护材料, 需要比普通日志更稳定的组织方式。
- 一级目录使用稳定影响面 ID, 可以让后续维护者按责任边界查找历史判断。
- 单条决策独立成文件, 可以降低编辑冲突, 也避免索引文档承接过多细节。
- 索引文档负责导航, 维护文档负责规则, 决策文件负责具体判断, owner 更清晰。

决定:
- 将 `docs/decisions.md` 迁移为 `docs/decisions/` 目录。
- 新增 `docs/decisions/README.md` 作为主索引。
- 新增 `docs/decisions/maintenance.md` 作为决策记录维护文档。
- 一级目录使用影响面 ID, 当前包括 `project-documentation`、`agent-instructions` 和 `decision-records`。
- 每条决策使用 `YYYY-MM-DD-short-title.md` 文件承接。

影响:
- 后续新增决策时, 先选择影响面目录, 再新增单条决策文件。
- README 和 AGENTS 中引用决策记录时, 指向 `docs/decisions/` 而不是单个 Markdown 文件。
- 决策记录的索引和维护规则不再与历史决策混在同一个文件中。

验证:
- 决策记录已经迁移到 `docs/decisions/`。
- 索引、维护文档和各影响面目录已创建。
- README 和 AGENTS 已更新决策记录路径。

## 目的
- 让 CLI 已有的目录覆盖能力具备明确、可移植且不会分裂身份的契约。
- 保持默认项目零配置，同时允许已有项目在工作区内选择适合自己的决策目录。
- 区分尚未初始化与当前成员来源损坏，避免查询为了冷启动创建占位文件。

## 背景
- 默认布局长期使用 `docs/decisions/decision-index.json` 和主题目录，能够满足大多数项目。
- CLI 已经提供 `--decisions-dir <path>`，并把记录路径解释为所选目录下的相对路径。
- 固定契约仍直接把默认路径写成唯一 owner，导致非默认目录看似违反契约，也没有说明切换目录后稳定身份如何解释。
- 决策根目录整体不存在通常表示项目尚未建立决策集合；已有 Markdown 但索引缺失或无效时，则无法从正文推断哪些记录仍是当前成员。

## 决策
- 采用: 将保存 `decision-index.json` 和主题 Markdown 的工作区内目录定义为决策根目录。
- 采用: 决策根目录默认是 `docs/decisions`；目标项目需要其他位置时，对同一集合的所有 CLI 调用一致传入 `--decisions-dir <path>`。
- 采用: `decision-index.json` 是相对所选根目录的当前成员 owner，记录的稳定身份是相对根目录的路径，根目录本身不进入身份。
- 采用: 所选根目录内继续使用唯一索引、主题分类、稳定文件名、逻辑归档和关系契约，不因目录覆盖产生第二套格式。
- 采用: 根目录整体尚不存在时将集合解释为尚未初始化，查询不创建文件；已有材料但索引缺失或无效时说明当前成员无法可靠恢复。
- 采用: 当前维护意图已经明确初始化成员时，由首个 `activate` 创建索引；否则先恢复或确认成员判断。

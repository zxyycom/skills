# 使用 ai-ready-docs 作为文档 AI 阅读优化能力的稳定身份

## 索引摘要
- 目的: 让 skill 的稳定身份和入口承诺准确表达把文档优化到 AI-ready 状态的能力。
- 背景: prompt-optimize 缩窄了处理对象，而对称的人机共读定位又弱化了 AI 阅读优化这一核心。
- 决策: 将既有 skill 重命名为 ai-ready-docs，以 AI 的准确阅读、理解和使用为主目标，人类可维护性作为次级约束。

## 目的
- 让人类能够从名称直接识别这是一项把文档优化到适合 AI 阅读和使用状态的能力。
- 让 agent 通过入口描述继续识别 prompt、规则、任务、需求、模板、工作流、skill 说明和 agent 指令等现实处理对象。
- 让 AI 能从实际获得的文本恢复用途、对象、关系、范围、权威性和边界，并可靠用于预期任务。
- 保持文档仍可由人自然阅读和维护，不为 AI 另建内容不同的第二事实源。
- 保持 skill 身份、目录、分发制品和 updater 配置一致，避免名称只停留在展示文案。

## 背景
- `prompt-optimize` 的实际对象已经扩展到说明、规则、任务、需求、模板、工作流、skill 说明和 agent 指令等多类文档，`prompt` 不能概括完整输入范围。
- 该 skill 的核心动作仍是优化文档；整理、审阅、重构、改写和生成前规划是达到目标状态的不同路径。新名称用 `ready` 表达优化后的结果，不把其中任一路径写成输入类别。
- 使用者明确将核心能力定义为把文档优化到适合 AI 阅读、理解和使用的状态，并选择 `ai-ready-docs` 作为新名称。
- 清楚目标、稳定术语、显式关系和减少歧义通常同时有利于人类阅读；真正额外面向人的行为主要是保留阅读脉络，以及在复杂关系确有帮助时使用显式关系图或 flowchart。
- 将人类阅读和 AI 信息恢复写成对称主目标，会让触发、流程和验收偏离 `AI-ready`，并为有限的人类侧行为建立不必要的第二套契约。
- 人类通常从“让文档更适合 AI”识别这项能力，agent 则从正在处理的文档类型和目标不清、关系隐含、重复冲突等问题触发；名称不需要独自承担两种发现路径。
- 既有的任务出口、主承诺、文档骨架、内容 owner、负向描述和规则收敛仍是有效判断，但需要围绕 AI 的实际消费路径和信息恢复重新组织。

## 决策
- 采用: 将 `prompt-optimize` 重命名为 `ai-ready-docs`，并让 skill 目录、frontmatter `name`、标题、人类介绍、仓库入口、发布制品和 updater 配置统一使用新身份。
- 采用: 将核心定位表述为把文档优化到 AI 容易阅读、准确理解和可靠使用的状态；AI 应能从实际文本恢复用途、对象、关系、范围、权威性、约束和操作依据。
- 采用: `AI-ready` 是优化结果，不是输入文档的预设类别；触发由明确的 AI 阅读或使用优化目标，或者 AI 已实际依赖该文档且存在会改变结果的问题症状决定，“可能供 AI 阅读”本身不足以触发。
- 采用: `AI` 不只是面向人类发现能力的名称入口，也是行为契约中的主要消费者；主流程和验收先证明 AI 的阅读、信息恢复和使用结果。
- 采用: 人类可读性不与 AI-ready 构成对称主目标。文档继续保持可阅读、可维护和自然脉络，必要时补充显式关系图或 flowchart；纯粹面向人的润色、排版和视觉体验设计不由本 skill 主导。
- 采用: 入口 `description` 先表达要通过优化达到的结果，再列出现实处理动作、对象、问题症状和使用时机；名称不独自承担全部触发条件。
- 采用: 主执行流程建立 AI 消费契约，围绕实际可用文本、预期任务和可观察结果优化信息与关系；现有主承诺、骨架、归位、规则收敛和格式判断继续作为该流程的一部分。
- 采用: 不为 AI 维护内容不同的专用文档副本；通过同一内容源、分层结构、附录、参考或可追溯的结构化块提供需要的信息密度。
- 采用: 本 skill 承接文档内容、语义关系、阅读结构和表达方式；领域事实、视觉制品渲染以及 OCR、embedding、索引、召回和运行时上下文组装继续由对应 owner 承接。
- 采用: 已进入 Git 历史的决策与调查路径保持稳定；正文链接按当前行为 owner 更新，历史形成时认识不因改名重写。
- 采用: 仓库只发布 `ai-ready-docs` 制品，不保留会与新入口重复触发的 `prompt-optimize` 行为副本；既有旧名称安装需要按新身份重新安装。

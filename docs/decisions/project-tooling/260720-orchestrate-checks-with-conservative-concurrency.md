# 2026-07-20 - 用保守顶层并发编排完整检查

## 索引摘要
- 目的: 缩短完整检查的反馈时间，同时让不同开发机和 CI 的资源使用与打包前置条件保持可预期。
- 背景: 串行执行会浪费彼此独立的只读检查等待时间，而固定较高或无界的顶层并发会叠加各任务内部的 Git 与 Node 子进程负载。
- 决策: 由 `scripts/check.ts` 统一编排完整检查，默认最多并发两个顶层任务并允许显式覆盖；失败后停止领取新任务、等待已启动任务，且仅在全部前置检查通过后打包。

## 目的
- 缩短 `bun run check` 的等待时间，并保持资源使用和打包前置条件可预期。

## 背景
- 前置检查彼此独立，但部分任务会自行启动 Git 或 Node 子进程；无界顶层调度会放大资源占用，打包则依赖全部前置检查成功。

## 决策
- 采用: 由 `scripts/check.ts` 编排完整检查，默认最多并发两个顶层任务并允许 `CHECK_CONCURRENCY` 显式覆盖。
- 采用: 失败后停止领取新任务并等待已启动任务；只有全部前置任务成功才执行 `pack:skills`。
- 采用: 将并发计算与任务执行分离，在进程内验证调度失败语义。

# 让完整检查默认报告警告并显式启用阻断

## 索引摘要
- 目的: 让并行修改中的工作区完成全部检查，同时为 CI 和真实依赖保留明确门禁。
- 背景: 新增检查若必须逐项声明为非阻断，会增加维护负担，临时损坏也会过早终止其余诊断。
- 决策: 前置检查默认以 warning 继续；任务可显式声明 blocking，`--strict` 将全部失败升级为阻断。

## 目的
- 让日常 `bun run check` 在决策、测试、调查报告或其他工作区内容临时不完整时，仍能运行全部可执行检查并汇总诊断。
- 让需要阻止打包或发布的检查通过显式声明或调用参数进入门禁，不依赖隐含分类。
- 保持不同开发机和 CI 的顶层资源使用可预期。

## 背景
- 前置检查会逐步增加，维护者不应为了获得非阻断行为而给每个新任务额外登记 warning 策略。
- 并行修改可能让决策、测试、调查报告或其他检查在短时间内失败；立即停止会隐藏后续诊断，降低一次完整检查的反馈价值。
- CI 和发布仍需要严格门禁，打包失败也不能作为普通 warning 忽略。
- 部分前置任务会自行启动 Git、Bun 或 Node 子进程，无界顶层调度仍会放大资源占用。

## 决策
- 采用: `bun run check` 中未显式声明阻断的前置任务失败时，输出 warning、继续领取后续任务，并在没有阻断失败时继续打包；只有 warning 且打包成功时命令退出 `0`。
- 采用: 新增前置任务默认继承非阻断行为；确有永久门禁责任的任务必须在任务定义中显式声明 `blocking: true`，不能通过任务名称或类别推断。
- 采用: `bun run check --strict` 将本次全部前置失败升级为阻断；CI 和其他要求完整门禁的调用方必须显式使用该参数。
- 采用: 阻断失败后停止领取新任务、等待已经启动的任务完成，并跳过 `pack:skills`；`pack:skills` 自身失败始终使完整检查失败。
- 采用: warning 保留原检查的 stdout 和 stderr 诊断，并在单项和最终汇总中显式显示 warning 状态，退出 `0` 不表示全部前置检查通过。
- 采用: 顶层任务默认最多并发两个，并继续允许 `CHECK_CONCURRENCY` 以正整数显式覆盖。
- 不采用: 要求维护者逐项声明哪些新任务属于 warning；非阻断是默认值，阻断才需要额外配置。

# 2026-07-01 - 本地脚本的常见行为优先使用成熟库

## 状态
- 当前状态: active
- 导致状态变化的决策: 无
- 状态说明: 作为当前主仓库脚本依赖选择和实现边界使用。

## 问题
- 为了减少依赖而手写 ZIP 结构、CRC、压缩头等通用底层逻辑，会让本地开发辅助脚本变得难读且难维护。
- 这些脚本不作为运行时产品分发，依赖体积和数量不是主要约束；可读性、正确性和 IDE 类型体验更重要。

## 背景与约束
- 主仓库脚本是本地校验、打包和 CI 的共享工具链，不进入 skill zip。
- 用户明确偏好：常见行为尽量使用库，库可以引入，但要注意选择高质量、高热度的库。
- 仓库仍需要保持脚本入口简单，依赖由 `package.json` 和 `pnpm-lock.yaml` 统一维护。

## 决定
- 采用: 常见格式、协议、解析、压缩和归档能力优先使用成熟库，不因为是辅助脚本就手写底层实现。
- 采用: 选择依赖时优先看维护活跃度、使用热度、API 聚焦程度、类型支持、ESM/Node/Bun 兼容性和依赖复杂度。
- 采用: 文件发现使用 `fast-glob`，Skill frontmatter 使用 `yaml`，ZIP 打包使用 `fflate`。
- 采用: ZIP 打包使用 `fflate`，保留稳定文件排序和固定 `mtime`，避免每次打包产物因当前时间变化。
- 不采用: 继续维护自写 ZIP header、CRC 和 deflate 拼装逻辑；这类代码维护价值低，出错成本高。
- 不采用: 继续用手写递归目录遍历和手写 frontmatter 行解析承接通用格式处理；这些实现容易重复且覆盖不了真实格式边界。
- 触发条件: 后续脚本需要处理成熟生态已有解决方案的通用行为时，先评估库；只有仓库特有规则、owner 判断和校验策略才优先直接实现。

## 影响
- 脚本复杂度应更多集中在仓库规则本身，而不是通用格式细节。
- 新增依赖不再被视为默认负担，但需要能解释选型质量和适用边界。
- 工具链文档需要明确依赖选择标准，避免后续又回到“少依赖优先”的隐性规则。

## 验证
- `scripts/pack-skills.ts` 使用 `fast-glob` 发现文件并用 `fflate` 生成 zip，不再包含自写 ZIP header 和 CRC 实现。
- `scripts/validate.ts` 使用 `fast-glob` 发现文件，并使用 `yaml` 解析 skill frontmatter。
- `docs/tooling.md` 记录常见行为优先使用成熟库的标准。
- `bun run check` 覆盖类型检查、结构校验和 skill 打包。

# Product & Architecture Thinking

Agent 面对工程任务时, 很容易从当前代码、接口或报错开始推理, 然后把已有实现方式当成问题边界。这样可以快速得到技术上自洽的答案, 却可能解决不重要的问题、把局部症状继续补在错误层级, 或为了技术完备性增加缺少产品价值的机制。

`product-architecture-thinking` 为工程判断提供一种思考姿态。技术方案形成前, 判断先向两个方向展开:

1. 向上看产品: 找到真实消费者、使用场景、预期结果和需要长期保持的行为, 判断这件事为什么值得做。
2. 向外看架构: 找到责任 owner、系统边界、事实源、数据流和演进路径, 判断这件事应该由谁、在哪一层负责。

两个方向形成结论后, agent 再向下选择实现。产品结果缺少依据时优先建议取消或简化工作; 产品结果成立但 owner 或边界不合理时把处置提升到系统设计层; 产品与架构都清楚且问题局部时, 直接完成最小实现。具体动作继续遵守用户已经明确的任务目标和修改授权。

这个能力适用于需要作出工程判断的需求分析、方案设计、实现、排障、重构、变更审查和决策。产品或架构视角没有改变方案时保持简短; 确实改变方案时, 只交付会影响当前决策的依据与取舍。目标、owner 和边界已经明确的直接任务继续直接执行。

具体任务继续由各自的行为 owner、验证规则和输出契约承接。该 skill 只改变 agent 的判断起点: 先确认产品结果和架构责任, 再进入技术实现。

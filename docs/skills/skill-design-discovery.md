# Skill Design Discovery

领域专家经常能够稳定完成一项工作，却难以一次说清每个判断的依据、例外和人工介入边界。直接把表面步骤改写成 `SKILL.md`，容易固定偶然实现、遗漏潜藏分支，或者把用户偏好、工具限制和历史 workaround 混成永久规则。

`skill-design-discovery` 用于创建、显著扩展或大幅重构 skill 前的设计发现。它从用户说明、现实案例、既有材料、工具契约和可用运行结果中恢复目标与触发边界、端到端流程、关键决策、约束来源、人类权限和验证义务。

最终交付是一份可实现的设计契约，而不是目标 skill 的目录实现。深度契约包含 Workflow Map、步骤级控制与交接，但不把流程图变成强制落盘或整图确认门禁。设计契约可以留在对话中；只有复杂任务、跨阶段交接或当前环境要求时才保存为文件。

验证设计使用一组通用评估提示，从当前目标即时推导触发、流程、分支、依赖、权限和约束方面的验证义务。评估提示不会被复制成固定验证案例，临时案例也不会因为参与分析就成为 skill 的长期内容。

这项能力与仓库内其他 owner 的关系是：

- `product-architecture-judgment` 判断事情是否值得做、做到什么程度，以及应由谁在哪一层承接。
- `skill-design-discovery` 恢复目标 skill 实际需要表达的行为、判断、约束、权限和验证。
- `skill-maintainer` 负责 skill 与分发单元的组成、实现、集成和交付闭环。
- `decision-records` 只保存已经确认、具有长期影响和回放价值的项目决策，不接收全部发现过程。

这些能力可以协作，但 `skill-design-discovery` 不依赖其他 skill 存在；只安装它时，仍能独立形成设计契约并交给当前环境的实现入口。

当前分发包还包含创建该能力时采用的上游原始参考快照。它供设计尚未稳定期间本地对照和取舍，普通运行不会加载，也不替代当前行为入口；上游法律与署名文件随快照保留。

# Decision Records

`decision-records` 用稳定 Markdown 文件保存重要长期判断的完整目的、背景、决策和按需声明的演进关系，用版本化 JSON 全生命周期索引保存全部记录的状态、秒级创建时间、精简摘要和直接关系。它关注“希望长期达成什么、为何采用当前方向、以后怎样不偏离”，不承接通用知识、任务日志或当前实现状态。

## 核心体验

日常工作继续以原任务为主。当用户正在形成可能持续影响后续行为、owner、边界、兼容性、风险处理或验收方式的决定，或拟议决定可能与既有决定冲突时，agent 会在确定方向或实施依赖该方向的实质改动前恢复相关活动判断。

恢复默认只查看 `active` 记录，`list` 直接返回状态、日期、路径、标题、目的、背景和决策摘要；`--topic <topic-id>` 可以按主题筛选，并与状态筛选和时间显示选项组合。需要完整理由时运行 `show`，需要审计演进时运行 `trace`；历史、迁移、修复或全库生命周期审计再显式选择 `archived` 或 `all`。

索引登记活动和归档记录，并保存秒级 RFC 3339 创建时间及 Markdown 派生的摘要和直接关系。这个时间用于参考先后，尤其区分同日多条记录；索引和列表仍固定按路径排序。`list` 默认只渲染日期，`--full-time` 才显示秒级完整时间。关系不进入列表，而由 `trace` 按方向和深度展开。CLI 或 Node 环境不可用时，索引和 Markdown 仍足以大致理解记录内容和正常命令效果，但严谨维护与校验仍以 CLI 为准。

决策根目录默认是 `docs/decisions`。根目录整体尚不存在表示集合尚未初始化，不需要创建空索引；首条长期判断确认后，写入无日期语义路径 Markdown 并通过 `activate` 初始化索引。已有材料但索引缺失、无效或旧项目需要升级时，agent 只在此类异常入口读取维护恢复手册，优先使用 Git 历史恢复时间和最后有效索引恢复生命周期，再回到严格校验。

发现当前任务与既有判断不一致时，agent 会区分对齐、明确的一次性例外、长期修订和一致性问题。只改变本次任务执行方式的指令属于一次性例外；只有用户已经决定改变后续工作的默认依据时才属于长期修订。

当前指令已经明确长期判断、维护对象和影响范围时，agent 会直接执行而不重复询问；新增记录或改变生命周期前，会明确告知用户将写入或改变的判断及状态变化。由 agent 提炼的新候选或范围不清的调整仍需确认。

原地修改只承接不改变决策语义的编辑性修正。决策语义变化时，agent 会创建新记录并声明真实演进关系。`archive` 只归档指定记录，`activate` 只激活指定记录；关系不再隐藏状态副作用。修订流程先显式归档前序，再写入并激活后续记录。

新记录使用 `<topic-id>/<semantic-slug>.md` 和无日期正文标题。既有带日期路径继续作为稳定兼容身份，普通维护不会批量重命名；既有路径和创建时间的恢复属于另行确认的迁移任务。

索引投影使用 4 至 100 个 Unicode 码点的单行文本，以限制每条列表元数据的长度；完整章节不受该限制，精确字段范围由固定契约承接。

查询带 warning 返回时，索引中仍可恢复的结果会继续输出，但不能据此断言缺失的正文或关系不存在。agent 会说明不完整性及其对结论的影响，写入完成仍以严格 `check` 为准。

## 设计边界

1. 可审核：全生命周期状态、精确时间、索引摘要、Markdown 内容和历史关系都能进入版本审查。
2. 语义分层：完整章节承接完整语义，Markdown 摘要压缩对应章节，JSON 镜像有限长度投影和直接关系。
3. 可演进：语义变化通过显式状态命令和单向直接关系表达；普通退役可以独立归档。
4. 路径稳定：新身份按主题语义命名，既有日期路径保持兼容；生命周期、关系或标题润色都不改变身份。
5. 防偏离：长期决策形成或发生冲突时恢复活动判断，明确区分一次性例外与长期修订。
6. 查询可用：中央索引可解析时，局部问题以 warning 暴露，不阻断其他可恢复结果；warning 同时限定结论完整性。
7. 范围可控：查询、候选、维护和迁移分别按自己的任务出口验收，不把查询自动扩大为修复。

## Owner 分工

1. [`SKILL.md`](../../skills/decision-records/SKILL.md) 承接 agent 的发现、语义恢复、偏离判断、候选确认、CLI 选择和按任务出口交付。
2. [`decision-record-rules.md`](../../skills/decision-records/references/decision-record-rules.md) 是目录、索引 schema、稳定身份、正文格式、关系、生命周期、维护事务和结构校验语义的唯一固定契约。
3. [`maintenance-recovery.md`](../../skills/decision-records/references/maintenance-recovery.md) 只承接冷启动、旧项目升级和故障恢复顺序，不重复定义固定契约。
4. `scripts/decision-records/` 承接 CLI 源码、测试和构建；随 skill 分发的 CLI 提供筛选列表、单条展开、索引关系追溯、派生字段同步和显式生命周期更新。
5. 每条 Markdown 的完整章节承接完整语义，同文件的索引摘要承接作者确认的压缩表达；决策根目录中的 `decision-index.json` 承接全部记录的生命周期元数据和派生检索投影。

## CLI 体验

1. 恢复：`list` 默认查询活动记录，`show` 展开单条正文，`trace` 按方向和深度回放索引关系；局部问题通过 warning 和 `invalid` 标记暴露。
2. 筛选：`list --topic <topic-id>` 按主题收窄结果，`--status active|archived|all` 默认 `active`，两者可以组合；`--full-time` 按需展开秒级完整创建时间。
3. 验证：`check` 严格检查索引、正文、稳定路径、字段长度和关系；查询成功不替代严格验证。
4. 维护：`sync-index --write` 只刷新派生投影和关系，`activate` 与 `archive` 分别改变目标状态，失败时恢复原索引。

## 项目接入

目标项目使用默认决策根目录时无需额外配置；选择其他工作区内目录时，对所有 CLI 调用统一传入 `--decisions-dir`。项目 agent 指令只加入指向索引和本 skill 的短入口，项目专属记录门槛留在自己的行为 owner，不复制固定契约。

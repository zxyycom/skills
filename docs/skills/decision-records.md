# Decision Records

`decision-records` 把重要长期判断保存为公开、可版本化的 Markdown，让项目拥有可审核的决策记忆，而不是依赖模型状态、数据库或工具缓存。

## 核心体验

日常工作继续以原任务为主。只有会持续改变行为、owner、边界、兼容性、风险处理或验收方式的取舍形成时，agent 才在自然阶段给出一份短概览；用户确认后再记录。任务局部选择和一次性例外不会进入长期记忆。

后续维护先从索引读取当前已生效判断，需要审计演进时再沿关系查看历史。`active` 表示决策已经确认并成为后续工作依据，不表示实现任务正在进行。

已有材料不符合固定契约时，skill 先报告差异、可保留语义和修复范围。格式迁移需要独立确认，完成后只维护一种格式。

## 设计边界

1. 可审核：内容、状态和关系都能直接阅读并进入代码审查。
2. 可演进：后续判断通过状态和直接关系修订历史，不覆盖形成原因。
3. 可定位：索引只展示当前已生效决策，历史按需查询。
4. 简单直接：使用固定目录、Markdown、相对链接和 Git 历史，不保存隐藏状态。

## Owner 分工

1. [`SKILL.md`](../../skills/decision-records/SKILL.md) 承接 agent 的发现、确认、读取、写入和交付流程。
2. [`decision-record-rules.md`](../../skills/decision-records/references/decision-record-rules.md) 是目录、状态、正文、关系、索引和维护事务的唯一固定契约。
3. `scripts/decision-records/` 承接 CLI 源码、测试和构建；随 skill 分发的 CLI 提供结构检查、状态列表和活动索引同步。
4. 目标项目的 `docs/decisions/` 只保存索引和实际决策，是长期记忆的数据 owner。

## 后续方向

后续通过真实样例继续校准主动提议时机、候选概览信息量、记录结构重量和非标准格式修复报告，不增加新的记忆入口或并行契约。

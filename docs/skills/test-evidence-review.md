# Test Evidence Review

AI 很容易把 TDD、防回归和覆盖率理解成“尽量多写测试”。结果通常不是更可靠的证据，而是重复等价类、实现细节断言、mock 自证和需要持续维护的测试库存。另一些真实风险受当前架构或环境限制，强行机械测试的成本又可能高于价值。

`test-evidence-review` 是测试价值评估与追踪流程。它先判断测试是否固定了稳定契约和独立、可观察的证明点，再决定不新增、合并、自动化证明或人工审查。测试发现误报使用独立的豁免路径，不作为低价值测试或难测风险的替代处置。

## 适合使用的场景

1. 编写、补充、修改、重构、精简或审查测试，包括 TDD、回归补测和覆盖率补测。
2. 判断真实风险是否值得自动化，或者当前更适合登记为人工审查义务。
3. 实现变更命中已有人工审查范围，需要执行对应检查动作。
4. 测试发现器产生误报，或账本、源码标记、未登记测试和 CI 对账出现问题。
5. 巡检重复、脆弱、实现耦合或维护成本过高的存量测试。

## 工作方式

每次调用都会默认启用测试证据账本。修改类任务会同步当前范围的账本和源码映射；只读审查会给出拟登记或调整结果。

流程包含两个工作面：

1. **语义评估**：判断测试是否值得存在、是否与已有证据重复、是否绑定实现，以及稳定风险应自动验证还是人工审查。
2. **可追溯登记**：用 Markdown 账本保存 automated、review 和 exempt 条目，用标准测试入口清单逐一映射实际入口，再由账本 CLI 检查结构、归属、Git Scope 和未登记结果。

日常恢复优先用账本 CLI 的 `list --query <text>` 按 case ID、行为摘要或代码路径查找既有证据；主题尚不明确时，无查询的 `list` 默认只返回 20 条紧凑摘要，再用 `show <case-id>` 只展开一个条目的原始 Markdown。这些入口不生成或读取测试入口清单。需要完整入口归属、未登记结果和 CI 对账时，再由调用方手动触发采集器，并通过 stdin 把一次性标准清单交给 `check`。自动采集只是一个手动触发的清单入口；即使项目不用它，通用索引仍能降低人工维护账本后的定位、筛选和按需展开成本。`list --triggered` 才显式叠加当前 Git review trigger，正常 trigger 作为匹配 case 的数据返回，入口映射始终留在严格检查和 inspection 中。索引缺失或陈旧时查询明确失败，修改任务通过 `sync-index --write` 从合法 Markdown 账本重建。机器调用可以消费带版本的摘要分页、单条正文、diagnostics、同步结果和 inspection，而不必解析人类错误字符串。只有准备写入、修复结构或需要精确字段语义时，agent 才完整读取账本契约。

账本按稳定契约、共享测试基座和连续行为链路组织，不枚举测试函数。一条没有分支的 happy path 或失败链路也可以在多个顺序阶段形成多个 `Proves:`；证明点数量不与测试入口数量绑定。每个被发现器识别的测试入口必须映射一个角色：`main` 表示自动化规范入口，`derived` 把其他入口归入已有 case，`exempt` 只豁免紧邻的发现误报。case 使用文字 `Contract:` 保存需要固定的行为背景；人工 review 的 `Scope:` 由 CLI 使用 Git 路径校验，并在脏工作区或最近 CR 提交之后发生范围内变化时返回 trigger。精确字段、配置和 CLI 约束由 [账本契约](../../skills/test-evidence-review/references/ledger-contract.md) 定义。

工具分成两个可独立接入的层级：`test-entry-regex.mjs` 在调用方显式运行后，用内置常见规则或项目正则配置自动生成标准 `TestEntryInventory`；它不后台运行，也不写账本。`test-evidence-ledger.mjs` 用清单完成严格对账，并把 Markdown 账本作为领域源接入通用索引。测试侧只定义稳定 case ID、紧凑 state、source revision、`search|status|verification|review-triggered` 查询 key 和动态 Git state；索引文件、新鲜度、同步、筛选、排序、分页、按 ID 获取和全量遍历都由通用层承接。完整 Contract、Proves 或 Review 正文不进入索引；通用源码内联进分发脚本，因此使用者不需要安装其他 skill。复杂项目可以用 AST、框架注册表或自定义收集器替换正则层。配置、清单、报告、索引、查询和单条 show 都以 Valibot Schema 为源码真源，构建时单向生成 JSON Schema 与 TypeScript 数据声明。旧 v2 组合接口或账本配置 v3 的项目按 [升级指南](../../skills/test-evidence-review/references/upgrade-from-v2.md) 一次完成配置、索引、命令和调用方更新。

内置正则按全文匹配，只提供常见入口基线，不猜测注释、字符串、fixture 或生成代码是否可执行。项目只有在默认范围不足时才创建采集配置；要求语法级精度时直接替换采集层。工具仍不执行测试，也不能机械判断自动化证据是否有价值或代替人工审查。测试框架继续负责运行行为，项目测试策略继续拥有测试层级和覆盖要求；这个 skill 负责评估与记录，让后续审计能够恢复测试固定了什么契约、各测试入口归属哪个 case、哪些风险需要重新 CR，以及哪些发现结果经过豁免。

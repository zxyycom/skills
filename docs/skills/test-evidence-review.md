# Test Evidence Review

AI 很容易把 TDD、防回归和覆盖率理解成“尽量多写测试”。结果通常不是更可靠的证据，而是重复等价类、实现细节断言、mock 自证和需要持续维护的测试库存。

`test-evidence-review` 是自动化测试进入或留在代码库前的证据准入流程。新增、修改或保留测试前，先确认规则 owner、独立条件、调用方可观察结果和已有证据，再明确选择待确认、不新增、复用或合并、扩展、新增、删除或降级。无法写清证明目标时，不以 TDD、覆盖率或历史回归名称替代。

调用这个 skill 时，会同时启用两个边界清楚的工作面：

1. 语义准入判断测试是否值得存在、是否重复、是否绑定实现，以及应复用、扩展、新增还是删除测试证据。
2. 可追溯登记默认启用：Markdown 账本保存稳定证明目标，不枚举每个测试函数。账本不存在时，修改类任务直接初始化当前范围，只读任务输出拟登记映射。
3. `@case` 标记主要验证入口，`@supports` 把辅助测试归入已有 case，`@test-exempt` 处理有具体原因的发现豁免。
4. 随 skill 分发的 Node CLI 检查账本状态、主要入口路径、源码标记和未登记测试文件。
5. 跨语言发现先以文件为归属单元，支持 Rust、TypeScript、JavaScript、Python、Go、Java 和 C#，并提供 ignore、warn 和 error 三种接入强度。

CLI 不执行测试，也不能机械判断 `Proves:` 是否真的有价值。测试框架继续负责运行行为，项目测试策略继续拥有测试层级和覆盖要求；这个 skill 负责让 agent 完成测试证据准入，并通过默认账本映射让后续审计能够定位测试为何存在。

实际 skill 位于 [`skills/test-evidence-review/`](../../skills/test-evidence-review/)。

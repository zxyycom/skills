# Test Evidence Review

AI 很容易把 TDD、防回归和覆盖率理解成“尽量多写测试”。结果通常不是更可靠的证据，而是重复等价类、实现细节断言、mock 自证和需要持续维护的测试库存。另一些真实风险受当前架构或环境限制，强行机械测试的成本又可能高于价值。

`test-evidence-review` 是测试价值评估与追踪流程。它先判断测试是否固定了稳定契约和独立、可观察的证明点，再决定不新增、合并、自动化证明或人工审查。测试发现误报使用独立的豁免路径，不作为低价值测试或难测风险的替代处置。

## 适合使用的场景

1. 编写、补充、修改、重构、精简或审查测试，包括 TDD、回归补测和覆盖率补测。
2. 判断真实风险是否值得自动化，或者当前更适合登记为人工审查义务。
3. 实现变更命中已有人工审查范围，需要执行对应检查动作。
4. 测试发现器产生误报，或账本、源码标记、未登记测试和 CI 对账出现问题。
5. 巡检重复、脆弱、实现耦合或维护成本过高的存量测试。

## 工作方式

每次调用都会默认启用测试证据账本。修改类任务会同步当前范围的账本和源码映射；只读审查会给出拟登记或调整结果。

流程包含两个工作面：

1. **语义评估**：判断测试是否值得存在、是否与已有证据重复、是否绑定实现，以及稳定风险应自动验证还是人工审查。
2. **可追溯登记**：用 Markdown 账本保存 automated、review 和 exempt 条目，用测试入口角色逐一映射实际测试入口，再由随附 CLI 检查结构、归属、Git Scope 和未登记结果。

账本按稳定契约、共享测试基座和连续行为链路组织，不枚举测试函数。一条没有分支的 happy path 或失败链路也可以在多个顺序阶段形成多个 `Proves:`；证明点数量不与测试入口数量绑定。每个被发现器识别的测试入口必须映射一个角色：`main` 表示自动化规范入口，`derived` 把其他入口归入已有 case，`exempt` 只豁免紧邻的发现误报。case 使用文字 `Contract:` 保存需要固定的行为背景；人工 review 的 `Scope:` 由 CLI 使用 Git 路径校验，并在脏工作区或最近 CR 提交之后发生范围内变化时返回 trigger。精确字段、配置和 CLI 约束由 [账本契约](../../skills/test-evidence-review/references/ledger-contract.md) 定义。

CLI 不执行测试，也不能机械判断自动化证据是否有价值或代替人工审查。测试框架继续负责运行行为，项目测试策略继续拥有测试层级和覆盖要求；这个 skill 负责评估与记录，让后续审计能够恢复测试固定了什么契约、各测试入口归属哪个 case、哪些风险需要重新 CR，以及哪些发现结果经过豁免。

# Git Commit Organizer

`git-commit-organizer` 的核心是维护提交信息质量。它不是为了自动运行一次 `git commit`，而是为了让提交历史更清楚、更统一、更容易管理。

这个项目的起点，是 agent 在完成改动后很容易把提交当成收尾动作：看到有 diff，就生成一个大致的标题，把文件一次性提交。但提交历史会长期存在，它影响代码审查、问题追踪、回滚判断、发布说明和后来 agent 对项目演进的理解。

## 为什么需要它

好的提交信息应该回答两个问题：这次提交改变了什么，以及为什么这些改动应该放在一起。对长期项目来说，提交风格不统一、范围混乱、标题含糊和正文空泛，都会把管理成本转移给后续阅读者。

`git-commit-organizer` 希望把提交从“保存工作区状态”提升为“整理变更语义”的过程。它关注提交范围、提交类型、标题表达、正文粒度和验证说明，让每个提交都能成为可追踪的项目历史单元。

这对 agent 协作尤其重要。agent 既可能误把用户未提交改动纳入提交，也可能用过于泛化的标题掩盖真实影响。一个专门整理提交的 skill，可以把这些风险变成稳定的检查和表达习惯。

## 希望形成的能力

这个项目希望形成一套统一的提交组织方式：

1. 先理解工作区变化，再决定提交范围。
2. 用稳定的提交类型表达变更性质。
3. 用中文全角冒号和一致正文格式统一提交风格。
4. 让提交正文只描述真实 diff 和实际验证。
5. 把无关改动、未运行验证和范围风险清楚留在最终说明里。

它的目标不是让提交更“自动”，而是让提交更可信。自动化只是手段，真正要降低的是项目维护和协作中的理解成本。

## 发展方向

后续的发展重点会放在更准确的变更归类和更稳定的提交语义上。它需要更好地区分文档、计划、规格、测试、重构、构建和维护类改动，也需要在复杂工作区里保持清楚的提交边界。

长期来看，这个 skill 应该成为 agent 提交代码前的质量门：帮助人类获得一致、可读、可追踪的提交历史，也帮助后续 agent 从历史中更准确地理解项目变化。

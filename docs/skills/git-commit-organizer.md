# Git Commit Organizer

这个仓库维护 `git-commit-organizer` Codex skill。它存在的原因，是把 Git 提交从“把当前改动保存下来”变成一次可审计的工作区整理过程。

这个 skill 面向 agent 创建提交前的关键判断：当前有哪些改动、哪些属于本次请求、是否存在已 staged 内容、是否有用户未提交改动、提交类型如何表达语义影响，以及提交正文能否真实对应 diff。

## 项目意义

AI 代理很容易把提交视为最后一步命令，但提交本身会改变项目历史。一个好的提交应说明为什么这些文件属于同一组变更，也应保护不属于本次任务的用户改动。

`git-commit-organizer` 的长期价值，是让提交过程稳定满足这些要求：

1. 先审计工作区，再决定提交范围。
2. 只 stage 与当前请求相关的文件。
3. 用提交类型表达语义影响，而不是机械套用 `docs` 或 `chore`。
4. 提交正文只描述 diff 中真实发生的改动。
5. 验证结果只陈述实际运行过的命令。

## 发展方向

这个仓库后续应继续强化四类能力：

1. 更精确的范围判断：面对 staged 内容、子仓库指针、生成物和无关改动时，能稳定区分本次提交边界。
2. 更好的提交语义：在文档、规格、计划、测试和自动化改动之间选择更能表达影响的提交类型。
3. 更透明的风险汇报：保留未提交残留、未运行验证和无法归类改动时，能在最终回复中清楚说明。
4. 更适合多仓库协作：支持先提交子仓库内容，再提交主仓库指针的工作方式。

新增规则应优先服务于提交审计、范围选择、提交信息质量或历史安全。共享脚本、CI、发布和多 skill 维护约定不属于本仓库的长期 owner。

## 与主仓库的关系

本仓库只维护 `git-commit-organizer` skill 本体。共享校验、打包、CI、发布和多仓库维护说明由 `skills` 主仓库承接。

这种拆分让提交整理能力可以独立演进，同时仍能作为 `skills` 主仓库统一发布的一部分。

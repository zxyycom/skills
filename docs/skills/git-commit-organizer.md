# Git Commit Organizer

`git-commit-organizer` 的核心是用一套稳定、有明确偏好的项目级风格组织 Git 提交，让提交历史更清楚、更统一、更容易管理。

## 风格定位

这套 skill 面向已经理解其作用、主动安装或显式调用的使用者。选择使用表示希望后续提交默认按它定义的提交粒度、类型、标题和正文格式收敛，而不是在每次执行时重新选择一套表达方式。

这里的“风格”表示一组主动选择的稳定默认，不宣称它是所有 Git 仓库都必须采用的通用规范。为了让 agent 的结果可预期，行为入口仍会用明确规则约束类型、中文全角冒号、中文标题和正文 bullet；用户或项目对当前任务给出更具体要求时，以该要求为准。

## 为什么需要它

Agent 在完成改动后容易把提交当成收尾动作：看到 diff，就生成一个大致标题并一次性提交。但提交历史会长期影响代码审查、问题追踪、回滚判断、发布说明和后来 agent 对项目演进的理解。

好的提交信息应该回答两个问题：这次提交改变了什么，以及为什么这些改动应该放在一起。提交风格不统一、范围混乱、标题含糊和正文空泛，都会把管理成本转移给后续阅读者。

`git-commit-organizer` 希望把提交从“保存工作区状态”提升为“整理变更语义”的过程。它关注提交范围、提交类型、标题表达、正文粒度和验证说明，让每个提交都能成为可追踪的项目历史单元。

## 风格内容

这套风格要求 agent：

1. 先理解工作区变化，再决定提交范围。
2. 用稳定的提交类型表达变更性质。
3. 用中文全角冒号和一致正文格式统一提交风格。
4. 让提交正文只描述真实 diff 和实际验证。
5. 把无关改动、未运行验证和范围风险清楚留在最终说明里。

## 发展方向

后续发展优先提高变更归类、提交边界和结果复核的稳定性。只有真实、重复出现的问题才进入主流程，避免低频 Git 场景稀释日常提交整理。

长期来看，这个 skill 应成为 agent 提交代码前的风格与质量入口：帮助人类获得一致、可读、可追踪的提交历史，也帮助后续 agent 从历史中更准确地理解项目变化。

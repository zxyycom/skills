# Product & Architecture Judgment

Agent 面对已经存在的需求、功能、接口或报错时, 很容易直接进入“怎样把它做好”, 并把当前问题和已有解法当作固定边界。局部实现可能越来越完整, 真实结果却没有相应改善; 也可能为了一个次要功能持续增加例外和维护成本, 而没有重新判断它是否值得做、应做到什么程度, 或是否属于正确的架构层级。

`product-architecture-judgment` 是一项工程决策能力。在深入局部实现前, 它先区分目标结果、当前问题或需求和已有解法, 再从两个方向展开:

1. 产品方向判断真实消费者是谁, 对用户、调用方、运营者和开发者有什么帮助, 结果是否必要, 以及当前投入和完成程度是否合理。
2. 架构方向判断责任 owner、系统边界、事实源、数据流和抽象层是否合理, 当前问题是局部缺口还是系统性症状。

最终结论可以是不做、简化、按真实结果改写需求、重定责任、提炼稳定抽象, 或在正确 owner 内完成局部实现。提炼抽象需要现实复用和演进证据; 放眼大局不意味着默认扩大范围。

这项能力适用于需求分析、方案设计、实现、排障、重构、变更审查和工程决策。它既可以推翻当前问题或解法, 也可以确认原任务成立并直接继续; 判断与输出继续遵守用户授权和当前任务的行为契约。

# Investigation Report

`investigation-report` 用一个稳定主题文件持续追踪同一核心问题。每次形成新认识时追加一份完整报告，让不同时间、背景和证据条件下的结果可以独立阅读、汇报和比较。

## 什么时候使用

仅在用户明确要求记录、沉淀、创建、更新或审阅调查报告时使用。普通调查、排障和问答不会主动建议建报告。

## 每份报告固定保存什么

每份报告使用同一个固定核心：

1. `形成时背景`：当时发生了什么，基于哪些事实、假设、未知和约束意识到需要调查。
2. `调查目的`：本轮具体要回答什么问题、支持什么判断以及预定边界是什么。
3. `调查范围与依据`：实际检查了什么、证据来自哪里、采用什么方法、实际覆盖到哪里，以及没有检查什么。
4. `调查结果与边界`：确认、推断、建议、实际动作和未知分别是什么，结果适用于什么条件，何时需要重新调查。

这四项是调查记录的最小公约数。每项篇幅由本轮实际信息决定；支撑材料需要独立组织才能复核结论时，再在固定核心之后增加时间线、证据、对照、状态模型、恢复或验收等章节。

计量或比较需要补足口径、样本和误差，因果或机制判断需要补足证据链和未闭合环节，候选方案需要区分推荐、采用和验证，实际变更还要记录授权、影响、恢复与验证。这些义务由当前结论所依赖的场景触发，可以同时出现，并先写进对应核心；需要独立展开时再增加章节。

## 怎样保存演进

同一核心问题的新认识追加为完整报告，并重新写清本轮四项核心，不能写“同上”。背景或目的不同但结果相同，仍分别保留，供以后比较适用条件。

最后一份报告只表示最近形成的认识，不自动汇总旧报告或成为累积当前口径；需要当前口径时应显式综合相关报告与当前事实 owner。后来取得的新认识通过追加表达；旧报告没有准确记录当时认识的误写、漏写或格式问题可以原地修正。能够独立调查和结束的新问题另建主题文件。

## 审阅与检查

任取一份报告，读者都应能恢复形成时背景、目的、调查依据和结果边界，不需要补入默认前提。关键主张应有可定位证据，结论强度不能超过证据，推荐、执行和验证状态不能混淆；报告不替代决定、实施计划或当前事实 owner。

主题 Markdown 是唯一事实源，也是索引单位。每个主题文件无论包含多少份报告都只生成一个通用 JSON 索引项；state 保存主题标题、核心问题、状态、最新报告时间、报告数量和按形成顺序排列的报告标题。相对主题路径作为当前集合中的身份，分类、状态、最新报告时间以及主题标题/核心问题/全部报告标题形成查询 key；报告正文不进入主索引。

随 skill 分发的 CLI 用 `sync-index` 显式重建派生索引，用 `list` 在核对 source revision 后按主题路径、category、状态、最新报告时间和文本查询 state，默认命令验证固定核心、时间、主题投影和新鲜度；内容是否真实、充分以及是否遗漏场景义务仍由人工审阅。带 category 或主题路径筛选的检查只验证局部结构，不代表完整索引已经新鲜。

# Investigation Report

`investigation-report` 用于让 agent 稳定形成可快速汇报、降低上手门槛并支持接续探索的调查报告。它以一个稳定主题文件持续追踪同一核心问题，但每次新增的内容都是一份完整、独立的报告，而不是只能与旧内容拼接才能理解的增量记录。

## 什么时候使用

本 skill 采用非主动式入口。用户明确要求记录、沉淀、创建、更新或审阅调查报告时使用；普通调查、排障和问答不会因为结果看起来有参考价值就主动建议建报告。

常见报告对象包括候选库调研、同类产品或竞品研究、机制与性能分析、系统实际执行逻辑，以及并发、时序和状态协作。它们只是帮助识别适用内容，仍以用户明确要求沉淀为前提。

## 报告真正固定的内容

每份完整报告只有三个固定语义章节：

1. `背景`：写理解本次结果所必需的当时上下文，例如项目处境、阶段、对象状态和关键约束。
2. `起因`：写为什么在此时进行调查。重新了解现状时，一句“距上次调查已久，需要了解当前状态”就可以成立。
3. `调查结果`：先直接写实际查到了什么、适用于什么边界；无法判断时明确缺少什么证据。

范围、方法、基线、证据、推断、结论、冲突、未知和下一步都可以按问题增加，但它们是支撑内容，不是每份报告必须填满的固定字段。推断与结论必须回指调查结果和证据，不能把背景假设倒写成结果。

Git hash 也不是固定背景字段。结果确实依赖某个已提交快照时记录完整 hash；未提交改动影响结果时，hash 还必须配合相关 diff、文件内容或工作区状态。调查对象与 Git 提交无关时不写 hash。

## 报告如何演进

一个主题文件可以包含多份按形成时间追加的完整报告。新证据、现状复查、项目阶段变化或解释变化仍属于同一核心问题时，向原文件追加新报告；新报告重新写清当前背景、起因和调查结果，不能只写“同上”或相对上一版的变化。

最新一份报告就是当前认识的报告正文来源，旧报告保留当时的完整快照。文件顶部只保存核心问题、状态和最新报告时间，不再另存一份容易与正文漂移的当前摘要。

纯错别字、格式和失效链接可以原地修正；会改变背景事实、调查结果、推断或适用边界的内容通过追加完整报告表达。能够独立调查、交接和结束的新问题创建兄弟主题文件。

## 写作与审阅重点

报告为没有参与原对话的读者写作。背景必须足以解释项目处境和结果边界，关键事实要保留来源、方法、时间和条件；事实、推断、结论、有限范围内未观察到的现象和未知要明确区分。

审阅时先检查三件事：任取一份报告能否独立理解；最新报告能否代表当前口径；调查结果是否由实际证据支持。之后再判断方法、证据、推断和接续方向是否需要增强。

## 整理与检查

主题文件按语义路径保存，并通过索引提供发现入口。随 skill 分发的 `check-investigations.mjs` 默认检查调查索引和全部主题文件，也可以按主题或文件路径筛选。它机械验证调查信息、每份报告的形成时间、背景、起因、调查结果、追加顺序和索引投影；背景是否真正充分、证据是否可靠以及结果是否成立仍由人工审阅。

实际行为入口位于 [`skills/investigation-report/`](../../skills/investigation-report/)，固定存储契约见其中的 [`investigation-report-contract.md`](../../skills/investigation-report/references/investigation-report-contract.md)。

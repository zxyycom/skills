# Prompt Optimize

`prompt-optimize` 的本质是文档优化。它关注的不是把文字改得更漂亮，而是让文档更适合被 agent 使用，也更适合人类通过文档和 agent 协作。

这个项目的起点，是很多 agent 任务失败并不是因为缺少信息，而是因为文档没有把信息组织成合适的引导：规则堆在一起但没有优先级，背景和动作混在一起，旧方案和当前方案同时出现，坏示例被写成长期提醒，真正的判断标准反而需要临场猜测。

## 为什么需要它

在人类之间，文档可以依靠经验、语境和口头补充被理解。但 agent 会更直接地受文本结构影响：标题、顺序、负向描述、示例、边界和验收标准都会改变它后续的行动路径。

因此文档不只是说明材料，它也是人和 agent 之间的协作接口。`prompt-optimize` 想解决的核心问题，就是让这个接口更清楚、更稳定、更少误导。

它会把“优化 prompt”这件事扩大到更实际的范围：规则、任务、需求、模板、工作流、skill 说明、agent 指令和长期维护文档。只要这些文本会影响 agent 如何理解目标、选择路径或判断完成状态，它们就值得被系统性优化。

## 希望形成的能力

这个项目希望沉淀一套面向 agent 协作的文档判断方法：

1. 先判断文档要引导什么行为，再选择结构。
2. 让目标、范围、边界和验收标准出现在最能影响行动的位置。
3. 把临时反馈、坏示例和历史残留提炼成当前可复用判断。
4. 让人类读者能看懂取舍，也让 agent 能沿着同一套判断执行。

它最终想成为一种文档设计能力：不是事后修补 agent 的错误，而是在文档进入协作流程前，就让文档更适合作为 agent 的工作上下文。

## 发展方向

后续的发展重点会放在更深的文档脉络判断上：区分说明、规则、流程、契约和评估文本；识别内容 owner 和阅读顺序；减少旧方案、坏示例和负向提醒对 agent 的干扰；把模糊要求改成可检查的协作标准。

这个 skill 应该逐步变成一套“面向 agent 的文档优化方法论”，帮助项目把知识、规则和流程写成同时服务人类理解与 agent 执行的协作材料。

# Prompt Optimize

这个仓库维护 `prompt-optimize` Codex skill。它存在的原因，是把 prompt、规则、任务、需求、模板、工作流、skill 说明和 agent 指令的改写，从一次性的文字润色提升为可复用的结构化文本工程能力。

`prompt-optimize` 关注的不是把句子写得更顺，而是让文本的用途、使用者、 owner、边界、验收标准和长期维护位置更清楚。它适合处理规则堆叠、用途错配、负向描述残留、坏示例沉淀、旧版迁移痕迹和复杂结构带来的行为偏移。

## 项目意义

很多 agent 相关文档会同时承担说明、规则、流程、约束和参考资料等角色。如果不先判断文档实际用途，改写很容易变成局部润色，甚至把错误案例、临时提醒或旧方案继续沉淀成长期规则。

这个 skill 的长期价值，是提供一套稳定判断：

1. 先判断文本要让谁做什么，再选择骨架和信息顺序。
2. 先确定内容 owner，再决定保留、移动、降级为引用或删除。
3. 把模糊要求改成目标状态、触发条件、输出格式或验收标准。
4. 把旧方案、坏示例和一次性反馈提炼成当前可复用判断。
5. 让必要边界保留为可执行的正向路径，而不是噪音式提醒。

## 发展方向

这个仓库后续应继续强化三类能力：

1. 更可靠的文档诊断：能识别用途错配、阅读脉络不匹配、规则重复、 owner 冲突和隐蔽防御式说法。
2. 更清晰的改写流程：让入口文件保持可执行，引用资料只承接需要展开的原则、任务和示例。
3. 更可迁移的判断标准：新增规则时优先沉淀可复用判断，而不是记录一次具体修正。

当 `prompt-optimize` 扩展新场景时，应优先判断它是否改变 skill 的触发条件、读取策略、执行流程或验收标准。只影响项目共享工具链、发布方式或仓库维护约定的内容，放到主仓库维护。

## 与主仓库的关系

本仓库只维护 `prompt-optimize` skill 本体和随 skill 分发的资料。共享校验、打包、CI、发布和多 skill 维护文档由 `skills` 主仓库承接。

这种拆分让 `prompt-optimize` 可以保持独立语义历史，同时复用主仓库的统一交付流程。

# Code Review

代码审查很容易退化成两种清单: 一种按文档缺口、实现偏差和代码质量分类, 另一种按理论严重度排序。它们可以描述问题, 却不一定能回答哪些问题真的值得处理, 也容易把同一个系统根因拆成许多局部修补。

`code-review` 从两个问题开始: 这件事产生了什么产品影响或开发影响, 以及它是局部缺口还是系统设计不顺导致的症状。A/B/C 仍用于标记问题来源, 但最终处置由真实场景、证据、可达性或变更频率、恢复成本和维护放大效应共同决定。

产品与开发不是两个独立审查阶段。一个产品问题可能来自开发边界不清, 一个暂时没有用户影响的开发问题也可能持续放大回归风险和相邻功能成本。这个 skill 让两条价值轴在同一组 findings 中完成归因和排序:

1. 产品轴检查用户结果、正确性、数据、安全、兼容承诺、性能和资源边界。
2. 开发轴检查 owner、事实源、非法状态、耦合、修改面、测试性、可观测性和演进成本。
3. 证据不足时, 按需使用相邻小功能、阶段拆分、可达性、变更频率和有限工作等检查验证判断。
4. 每个根问题在补文档、调整系统设计、代码收敛、局部简化、删除约束和延期之间选择主要处置。

输出采用分层方式。正常对话优先呈现阻塞、高优先级和最能改变决策的 findings; 数量较少时直接列全, 数量较多时按根因概括其余内容。完整 finding 清单、证据和跨 finding 归因按需写入系统临时目录中的 Markdown 报告, 再通过路径或链接交付; 它们不会挤占对话, 也不会进入目标仓库或长期文档。

这个 skill 不替代 linter、测试或项目自己的审查规范, 也不默认授权修复。它提供的是一层统一的工程判断框架, 让产品结果、开发质量和系统根因共同决定后续行动。

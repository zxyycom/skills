# Dependency Boundary Design

`dependency-boundary-design` 处理同一责任的工具、库、SDK、存储或基础设施调用已经分散，依赖协议、错误、配置或生命周期处理开始重复和漂移，或者一次同类变化需要跨多个位置修改的情况。它判断是否需要建立明确的依赖边界，并在需要时把后续变化收口到稳定 owner。

边界首先是责任和变化的集中点，不以多实现为前提。一个诚实命名的模块、函数集合或具体类型就可以拥有当前依赖的调用、错误映射和配置；只有多个实现必须履行同一消费者契约，或存在现实选择和切换义务时，才继续建立 provider、adapter 或选择机制。

核心流程是：

1. 恢复依赖面、消费者、现有 owner 和必须保持的行为基线。
2. 按共同变化原因和责任归属组织调用，再用消费者结果与事实源校验范围。
3. 调用仍然局部且责任清楚时保持局部；需要收口时明确 owner、范围、契约、例外和迁移顺序。
4. 迁移后分类剩余直接调用，并用一项代表性的依赖变化验证修改能够留在边界内。
5. 只有现实多实现义务出现后，才扩展可替换契约。

它与相邻能力保持独立：`common-denominator-design` 判断多个现实场景能够共同依赖什么契约，`minimal-implementation` 在目标和责任已明确后比较正确候选的维护面；本 skill 只负责具体依赖调用与治理责任是否需要收口，以及边界应怎样形成。

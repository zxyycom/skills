# Common Denominator Design

`common-denominator-design` 用于处理一种很常见、却容易被“做个抽象”掩盖的判断：多个场景到底能够共同承诺什么，以及这个共同部分是否真的适合存储、理解、使用和演进。

“抽象”通常只提示要寻找共性；“公约数”会迫使设计者继续回答几个更具体的问题：是谁与谁的公约数、在哪个责任层成立、共同约束到底是什么、统一后由谁承担恢复差异的成本。它因此更适合检查那些表面统一、实际把复杂度转移给消费者的方案。

这里的核心概念就是“公约数”：在明确的场景集合和责任层内, 各场景可以按同一含义共同依赖的契约。它有一个直接的消费者检验: 如果消费者只依赖候选契约, 就能正确理解和处理适用集合中的任一场景, 并且除契约公开表达的变体外不需要额外场景知识, 这个候选才构成共同依赖边界。“最小”不进入名称, 因为它会把注意力锚定在削减结构; 数学隐喻中的最小公约数也容易退化为没有设计价值的 `1`。候选公约数先要语义真实、足以承接共同目标并可直接使用, 然后才通过边界收敛移出局部职责和纯便利内容。

一个空对象、万能字符串、任意 map 或全可选 schema 可能覆盖所有输入, 却无法让消费者直接理解和处理。覆盖面广不等于存在公约数; 真正的公约数必须让同一字段、规则或操作在全部适用场景中保持同一语义和约束。

证据不足也不等于已经证明不存在公约数。只有现有场景能够支持否定结论时, 才把候选判为不成立; 缺少的信息会改变分组、共同契约或兼容风险时先恢复或确认该项, 其余未知作为假设和验证义务保留, 并暂时保持局部。

这套判断可以用于多种对象：

1. 格式设计中, 不先规定所有场景必须使用同一组章节, 而是判断哪些信息职责始终相同, 哪些职责在部分场景中需要合并表达或独立表达。
2. 需求合并中, 不因描述相似就压成一句通用需求, 而是恢复消费者、成功标准和约束, 决定共享目标与场景验收应怎样分层。
3. 数据与接口设计中, 检查统一 schema 是否依赖大量空值、标志和调用方猜测, 并判断应使用一个契约、共享核心加显式变体, 还是多个场景契约。
4. 代码复用中, 区分真正共享的责任与仅仅相似的实现形状, 避免为了减少重复而建立会持续增长条件分支的基类或通用服务。

最终结果不预设为一个格式。skill 会在一个公约数、共享核心加变体、多个公约数和保持局部之间选择；存储形式与消费形式承担不同义务时, 还可以分别形成公约数并通过显式映射连接。每个拆分都要能说明合并会造成的误解或处理成本, 每个合并也要能说明拆分只会制造的重复。

它与 `product-architecture-judgment` 的出口不同：后者判断事情是否值得做、应做到什么程度、由谁在哪一层实现；本 skill 在跨场景统一已经成为具体设计任务时, 深入形成共享契约和变体边界。两者不构成安装依赖。

它也不接管 skill 或提示词维护：`skill-design-discovery` 恢复待设计 skill 的真实工作流与验证义务, `skill-maintainer` 判断能力 owner 和分发边界, `prompt-optimize` 优化结构化文本。上述任务内部遇到“多个场景是否真正共享一个契约”时可以采用本方法, 但四项能力保持各自独立, 不形成安装依赖。

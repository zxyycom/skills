# Prompt Optimize 名称与触发对齐调查

## 调查信息
- 核心问题: `prompt-optimize` 的名称应如何表达其真实能力，并同时支持人类显式调用与 agent 主动触发？
- 状态: 暂停
- 最新报告时间: 2026-07-21T18:10:32+08:00

## 调查报告

### 现名未完整表达能力，并暴露双向触发约束
- 形成时间: 2026-07-21T18:10:32+08:00

#### 形成时背景

`prompt-optimize` 是一个独立分发的 skill。它的 `name` 提供稳定身份，`description` 表达能力定位和使用时机，正文承接完整行为。名称调整因此既是发现入口问题，也会影响目录、分发制品、updater 和既有安装的身份兼容，不能只按文案润色处理。

当前 skill 的实际处理对象已经超出狭义 prompt，包括规则、任务、需求、模板、工作流、skill 说明、agent 指令和其他会影响执行、判断、协作或验收的结构化文本。它也不只优化已有文字，而是支持生成前规划、从零设计、审阅、重构和直接改写。现有的人类介绍将其定位为面向 agent 协作的文档优化与设计能力：让文档同时服务人类理解和 agent 执行，并成为两者之间较可靠的协作接口。

实际使用存在两条都必须保留的发现路径：人类显式调用的频率更高，通常从“优化 prompt”“整理规则”“让文档更适合 AI 使用”等直观目标出发；agent 主动触发的频率较低，但仍需在处理具体文档、发现冲突或判断文本不可执行时选择该 skill。agent 不一定会把正在处理的文档理解为“用于引导自己行为的文本”，因此发现入口不能依赖这种自我指涉判断。

#### 调查目的

本次调查源于对 `prompt-optimize` 是否需要改名的讨论。机制导向和结果导向候选分别暴露了人类可发现性、agent 主动触发与能力边界的问题，使用者因此要求先记录当前认识，再继续讨论名称和迁移。

本轮要回答现名是否存在值得继续调查的错位、候选名称应优先表达什么，以及人类显式调用与 agent 主动触发应怎样分工；只形成后续命名调查的工作口径，不决定名称或迁移方案，也不修改 skill 本体。

#### 调查范围与依据

本轮检查了当时的 [`skills/prompt-optimize/SKILL.md`](../../../skills/ai-ready-docs/SKILL.md)、[`docs/skills/prompt-optimize.md`](../../skills/ai-ready-docs.md)、[`skills/skill-maintainer/SKILL.md`](../../../skills/skill-maintainer/SKILL.md) 和 [`docs/repository-model.md`](../../repository-model.md)，并结合使用者提供的显式调用与 agent 主动触发场景比较候选名称。调查依据限于这些仓库材料和使用经验，没有收集系统化正反触发样本，也没有验证候选名称的实际发现率或迁移成本；形成时未记录对应仓库 commit 或工作区 diff，本报告本身不能确定精确材料快照。

#### 调查结果与边界

当前证据支持以下阶段性结果：

1. 现名存在真实但尚未量化的主承诺错位。`prompt` 小于实际处理对象，`optimize` 小于实际行为范围；但现名保留了人类高频使用词，不能只因语义不完整就直接替换。
2. 当前能力可以暂时用一句话概括为：**把人和 agent 共同使用的 prompt、规则、任务、需求、模板与工作流，调整成双方都能准确理解、直接执行、可靠验收并持续维护的结构化工作文档。**
3. 名称应优先帮助人类识别能力和记住稳定身份；agent 的主动触发主要应由 `description` 中的处理对象、问题症状和使用时机保证。agent 不需要先判断“这份文档正在引导我”，只需识别自己正在处理 prompt、规则、任务、需求或工作流，或者文本存在目标不清、规则冲突、owner 错位、历史残留与验收缺失等问题。
4. 机制导向名称容易缩窄能力或要求使用者理解内部引导机制；结果导向名称更接近实际触发语言，其中 `ai-ready-docs` 暂时最有解释力，但仍需验证 `AI`、`ready` 和 `docs` 的边界。
5. 当前证据不足以决定改名。关键缺口是人类显式调用、agent 主动触发和相邻非目标请求的正反样本，而不是更多近义词。

#### 候选名称比较

| 候选名称 | 当前能够表达的部分 | 主要风险 |
| --- | --- | --- |
| `prompt-optimize` | 保留高频人类用语和既有稳定身份 | 对象与动作都小于实际能力，可能漏掉规则、需求和工作流场景 |
| `agent-instruction-design` | 表达从优化扩展到设计 | `instruction` 偏窄且有领域歧义，仍需要额外解释 |
| `agent-guidance-design` | 能覆盖所有影响 agent 行为的文本 | 依赖“文档在引导 agent”的意图视角，不符合部分实际触发过程 |
| `ai-friendly-docs` | 接近日常表达，便于人类显式调用 | `friendly` 模糊，容易退化为格式、措辞或表面可读性优化 |
| `llm-friendly-docs` | 技术对象较明确，名称容易理解 | 可能误触发 LLM 摄取与 RAG 场景，并弱化 agent 工作流边界 |
| `ai-ready-docs` | 以可使用的目标状态概括能力，兼顾人类记忆 | `AI` 和 `docs` 仍偏宽，必须由 description 与触发样本收窄 |

#### 证据

1. [`skills/prompt-optimize/SKILL.md`](../../../skills/ai-ready-docs/SKILL.md) 直接列出 prompt、规则、任务、需求、模板、工作流、技能说明和 agent 指令，并将生成前规划、改写、审阅和方案设计纳入同一执行流程。这证明实际能力不能只按狭义 prompt 优化理解。
2. [`docs/skills/prompt-optimize.md`](../../skills/ai-ready-docs.md) 将该能力解释为面向 agent 协作的文档优化方法，并明确文档同时服务人类理解与 agent 执行。这支持以共享工作文档或协作接口理解最终产物。
3. [`skills/skill-maintainer/SKILL.md`](../../../skills/skill-maintainer/SKILL.md) 区分了 `name` 的稳定身份责任和 `description` 的能力与使用时机责任。这支持将人类记忆与 agent 语义触发分别设计，而不是要求名称独自承担所有触发条件。
4. [`docs/repository-model.md`](../../repository-model.md) 说明每个 skill 独立生成制品并拥有自更新路径。这证明重命名不仅是展示文字变化，还需要考虑分发身份与既有安装的迁移。
5. 本次使用者场景说明提供了当前最关键的行为证据：人类显式调用更频繁，但 agent 主动触发不能放弃；agent 通常从正在处理的对象和发现的问题出发，不会先把文档识别成“引导自己行动的文本”。该证据来自实际使用经验，目前尚未通过系统化样本回归验证。

#### 后续调查

1. 用现实样本验证 `docs`、`AI`、`LLM`、`agent`、`friendly` 和 `ready` 是否会造成新的漏触发或误触发，不再孤立比较词义。
2. 建立覆盖两类发现入口的正反样本：优化 prompt、整理规则、重写任务、修订需求与验收、审阅 `SKILL.md` 或 `AGENTS.md` 应怎样触发；普通润色、文档排版、RAG 预处理和运行时上下文组装为何不触发。
3. 候选名称与 description 通过样本后，把名称选择交给决策 owner；如最终改名，再单独调查目录、release asset、updater、锁文件、文档引用和外部安装的迁移边界。

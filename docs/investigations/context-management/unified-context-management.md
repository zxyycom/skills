## 调查报告

### 现有能力的组合边界与统一方向
- 形成时间: 2026-07-21T16:04:25+08:00

#### 形成时背景

本仓库已经分别维护决策记录、OpenSpec、调查报告、测试证据和子代理编排等 skills，并通过[项目文档导航](../../navigation.md)与[仓库模型](../../repository-model.md)声明本地 owner、按需选择和分发边界。它们在内容上能够互补，但仓库没有因为同仓维护而默认承诺组合安装、统一入口或端到端协作。

本轮仓库证据基于 `main@dcf9f5f1a43a1ca1b6635be8ded2d055b8c38938` 及当时工作区；相关 skills 与仓库模型来自该提交，项目导航含未提交的调查报告路由补充，因此该 commit 不能单独代表全部调查对象。

#### 调查目的

用户观察到这些 skills 组合后已经接近一套项目上下文管理能力，希望先识别当前分工和组合边界，再判断还缺少什么。此时尚未给出“统一系统”的产品定义，也没有作出建设决定。

#### 调查范围与依据

本轮读取了项目导航、仓库模型和六类相关行为入口，比较它们承接什么、明确不承接什么，以及信息怎样可能流向下游。调查只使用仓库内当前 owner 和行为契约，没有创建 OpenSpec change、长期决策或测试账本，也没有尝试从 active change、Git 状态或对话推断全部实时工作。

#### 调查结果与边界

现有 skills 已经覆盖多种互补的持久上下文，但尚未共同承诺一个统一系统。可以先把项目导航理解为持久上下文的项目级路由入口，并保持各类内容由原 owner 承接；是否需要新的组合 skill、公共关系模型或自动化入口，当时仍不能判断。实时并行执行不能由静态 artifact 完整恢复，应与持久上下文管理分开判断。

#### 已确认的能力分工

| 载体 | 当前承接 | 明确边界 |
| --- | --- | --- |
| [项目导航](../../navigation.md) | 按任务定位主要 owner 和必要读取入口 | 不复制各 owner 的完整规则，也不登记单条记录 |
| [decision-records](../../../skills/decision-records/SKILL.md) | 跨任务长期有效的目的、背景、采用方向、状态和关系 | 不承接通用记忆、任务日志、当前实现状态或行为事实 |
| [OpenSpec Explore](../../../skills/openspec-explore/SKILL.md)、[Propose](../../../skills/openspec-propose/SKILL.md)、[Apply](../../../skills/openspec-apply-change/SKILL.md)、[Archive](../../../skills/openspec-archive-change/SKILL.md) | change 前后的探索，以及 change 的目标、设计、局部决策、任务、规格增量、验证和归档 | active change 是持久化变更 artifact；`openspec list` 不能证明当前由谁执行或是否仍在运行 |
| [investigation-report](../../../skills/investigation-report/SKILL.md) | 围绕问题链保存背景、方法、证据、推断、未知和阶段性认识 | 不替代决策、实现计划、当前行为 owner 或稳定测试义务 |
| [test-evidence-review](../../../skills/test-evidence-review/SKILL.md) | 评估测试的契约背景、证明点和维护成本，登记自动化证明、人工审查与发现豁免 | 不拥有产品契约，不执行测试，也不把账本或通过状态当作测试价值证明 |
| [subagent-orchestration](../../../skills/subagent-orchestration/SKILL.md) | 在一次运行时协作中划分目标、历史范围、权限、写入所有权、依赖和结果审计 | 当前契约只定义任务内协调，没有声明跨线程、跨 worktree 或跨进程的持久工作注册表 |

[仓库模型](../../repository-model.md)进一步确认：同仓只表示共享维护和发布基础设施，使用者按需选择 skill；skill 间协作、前置和组合验证只有在行为 owner 或具体项目显式声明后才成立。因此，“这些能力看起来可以组合”目前是观察和候选解释，不是仓库已经兑现的产品承诺。

#### 候选工作模型

以下分层可以解释现有能力怎样互补，但仍只是后续调查用的工作模型：

| 需要恢复的问题 | 候选 owner |
| --- | --- |
| 当前应该遵守什么、实际已经实现什么 | 项目的规范、代码、配置、运行观测和其他行为 owner |
| 为什么长期采用这个方向 | decision-records |
| 某次 change 想改变什么、怎样实施和验收 | OpenSpec change |
| 对某个问题目前知道什么、证据和未知是什么 | investigation-report |
| 稳定契约由哪些自动化证明或人工审查守护 | test-evidence-review 与项目测试策略 |
| 当前哪个执行实例正在处理什么 | 运行时协调系统；不由静态导航、active change 或调查报告推断 |

在这个模型中，项目导航只提供“任务应进入哪个 owner、按什么顺序读取”的路由，不集中保存所有正文。各 owner 之间也不自动复制内容：调查结果可以产生候选决策、change 输入或测试义务；change 内判断只有在项目明确认定其跨 change 继续有效时才可能进入长期决策；测试证据从稳定契约恢复背景，但不成为契约本身。这些交接是否需要统一字段、稳定标识或机械检查，尚无充分证据。

#### 当前缺口与风险

1. **组合承诺缺失**：目前没有 owner 定义这套组合的触发、安装集合、读取顺序、缺失 skill 时的降级路径和端到端验收；仓库模型也明确不默认管理完整安装集合。
2. **现实价值尚未证明**：尚未用多个真实任务比较“独立 skills + 项目导航”和“统一系统”在恢复速度、遗漏、冲突、维护成本上的差异。
3. **交接语义未统一**：各 skill 已声明局部边界，但还没有共同规则说明何时只链接、何时同步 owner、何时产生候选，以及冲突时哪个当前事实优先。
4. **时效与失效条件不一致**：决策有生命周期，调查有形成时间和状态，测试人工审查有 Git Scope，OpenSpec 有 artifact 和 task 状态；这些状态不能直接合并成一个全局“当前”字段。
5. **在途工作不可静态恢复**：active change、任务勾选和单个 worktree 的 Git 状态都不能完整表示并行执行。若以后需要跨线程或跨 worktree 的实时协调，必须单独设计执行身份、作用域、所有权、存活与失效语义。
6. **统一入口可能制造额外负担**：若每个普通任务都查询和维护全部上下文类型，可能增加读取噪音、写入事务和跨 owner 同步成本，违背各 skill 当前的渐进读取与按需触发原则。

#### 仍然未知

1. 现实项目中最常见的上下文丢失发生在任务开始、owner 选择、change 归档、测试维护还是并行协作阶段。
2. 最小有用能力只是项目级导航约定，还是需要一个可执行的只读上下文路由 skill。
3. 是否需要跨 owner 的稳定引用标识；普通 Markdown 链接和语义搜索能否满足大部分接续需求。
4. 统一体系应该只恢复持久上下文，还是还要与某种专门设计的运行时协调系统建立显式接口。
5. 不同项目已经存在自己的调查、决策、规格和测试组织方式时，通用能力应怎样适配而不夺取项目 owner。

#### 后续最有区分力的调查方向

如果未来重新开启本报告，优先选择少量真实任务，逐项记录任务开始时需要恢复的 owner、实际遗漏、冲突和读取成本，再观察任务结束时哪些认识确实需要流向另一个 owner。至少包含一个普通局部修改、一个跨 change 长期判断和一个并行执行场景；并行场景单独评估运行时协调，不用 active change 数量替代执行事实。只有这些案例显示重复且稳定的问题后，再比较以下候选：仅完善项目导航、增加可选交接规则、提供只读上下文投影，或建立新的分发单元。

#### 本轮未形成的结论

本轮没有决定创建统一上下文管理系统，没有确定新 skill 或分发单元，也没有把各 skills 设为互相依赖。报告状态为“暂停”，表示这个方向仍有调查价值，但当前先保留问题、工作模型、风险和接续入口。

### Docnav 的项目级分工样本
- 形成时间: 2026-07-21T16:13:04+08:00

#### 形成时背景

本轮形成时的工作基线是各类上下文可以由不同 owner 承接，但 OpenSpec change 中的决策与长期决策记录如何避免 owner 重叠仍需现实样本。Docnav 是用户提供的实践项目；本轮固定查看 [Docnav 提交 `2df037f`](https://github.com/zxyycom/docnav/tree/2df037fa0a2553a9cabe6dc8d01187a424926393)，观测日期为 2026-07-21。

#### 调查目的

用户指出“OpenSpec 与决策记录存在 owner 重叠”不能只作抽象推断，应查看 Docnav 已采用的解决方案，并以此校准统一上下文管理方向。

#### 调查范围与依据

本轮固定检查 2026-07-21 观测的 [Docnav 提交 `2df037f`](https://github.com/zxyycom/docnav/tree/2df037fa0a2553a9cabe6dc8d01187a424926393)，重点读取其文档导航、长期决策与 OpenSpec governance specification，判断稳定规则、change 内判断、长期理由和当前实现证据怎样分工与交接。范围只包含这一个用户提供的实践项目，没有比较其他项目，也没有测量治理规则的维护成本或实际遗漏率。

#### 调查结果与边界

Docnav 已经通过项目级导航与规范，把稳定规则、change 内判断、跨 change 长期理由和当前实现证据分给不同 owner，并在归档时规定必要的同步条件。这一实例证明相关内容可以在不争夺最终 owner 的前提下协作，也加强了“部分组合治理可以留在具体项目”的认识；单个自有项目仍不能证明相同机制适用于所有项目，也不能证明需要建设统一上下文管理系统。

#### 证据

1. [Docnav 文档导航](https://github.com/zxyycom/docnav/blob/2df037fa0a2553a9cabe6dc8d01187a424926393/docs/navigation.md)规定：`docs/` owner 文档承接稳定行为与 public contract，OpenSpec change 承接当前 change 的目标、设计、决策、任务和验收依据，长期决策记录承接跨 change 仍有效的目的、背景、方向和关系；代码、测试和 release artifact 证明当前实现状态。
2. [分离长期决策、OpenSpec 与规范所有权](https://github.com/zxyycom/docnav/blob/2df037fa0a2553a9cabe6dc8d01187a424926393/docs/decisions/decision-management/separate-decision-spec-ownership.md)保存了这一项目级长期判断，并明确归档 change 不会自动把 change 内决策复制到全局决策集合。
3. [OpenSpec governance specification](https://github.com/zxyycom/docnav/blob/2df037fa0a2553a9cabe6dc8d01187a424926393/openspec/specs/openspec-governance/spec.md)进一步把分工变成归档前可验收规则：只影响当前 change 的判断保留在 `design.md`；跨 change 继续作为默认方向或改变既有活动决策时，归档前同步稳定 owner 和长期决策记录。

这些来源直接证明 Docnav 当前采用了项目级 owner 分层和同步顺序。它们不能证明相同分层适用于所有项目，也不能证明 OpenSpec active change 能代表并行执行中的实时工作。

#### 对既有认识的影响

本轮证据加强了两个方向：第一，稳定规则、change 上下文和长期理由可以在内容上相关而不竞争最终 owner；第二，通用 skills 不必自动互相依赖，具体项目可以用导航、AGENTS 或项目规范声明交接门槛。它同时削弱了“必须新增一个集中存储所有上下文的系统”这一候选，因为现有项目级规则已经能解决部分 owner 重叠。

仍然未知的是，这种项目级治理能否在不同结构的仓库中低成本复用，以及只读上下文路由、跨 owner 引用和组合校验是否还具有独立价值。后续调查仍应使用多个现实项目与任务比较，而不是从 Docnav 单一样本直接形成产品决定。

### 统一系统的最小含义与初步差距
- 形成时间: 2026-07-21T16:21:11+08:00

#### 形成时背景

本仓库已有的决策记录、OpenSpec、调查报告和测试证据分别承接不同持久上下文，项目导航可以把任务导向相应 owner；Docnav 进一步提供了项目级 owner 分层和归档同步样本。此时“统一系统”仍只是用于衡量组合能力的候选概念，不代表集中存储方案或已确认产品。

实时在途工作不能从静态 artifact 简单恢复，需要专门的运行时协调设计；除非以后明确把它纳入目标系统，否则不计入当前持久上下文系统的缺口。

#### 调查目的

用户进一步澄清调查核心：当前已经有很多上下文管理 skills，希望知道它们距离统一上下文管理系统还差多少，并以 Docnav 作为实践参考，而不是在本轮决定是否建设系统。

#### 调查范围与依据

本轮以仓库内决策记录、OpenSpec、调查报告、测试证据等持久上下文 owner，以及项目导航和仓库模型为能力分工基线，并用 Docnav 的项目级分工样本校准“统一系统”的最低含义。调查只比较持久上下文的内容 owner、路由、关系、生命周期、交接和组合校验，没有执行跨项目任务样本或量化差距；实时在途工作因需要独立运行时协调模型而排除在本轮范围外。本轮没有记录仓库材料对应的 commit 或未提交 diff，本报告本身不能确定仓库侧精确快照。

#### 调查结果与边界

这里的“统一系统”可以暂时理解为：从一个任务出发找到最小必要上下文，同时保留各自 owner，并处理跨 owner 的关系、时效、交接和一致性。现有 skills 已经解决了“分别记录什么”，尚未共同解决“怎样作为整体被发现、连接和校验”。Docnav 可以帮助判断其中哪些问题能由项目级治理解决；当前证据不足以给出完成比例、实施规模或建设结论。

#### 初步差距

下表是衡量差距的候选基线，不是系统设计或长期契约。

| 能力 | 已有基础 | 仍需补足或验证 |
| --- | --- | --- |
| 内容与 owner | 决策、change、调查、测试证据和项目事实已有各自 owner；Docnav 也采用稳定文档、OpenSpec、长期决策和实现证据分工 | 主要内容载体已经覆盖，无需先增加通用记忆容器 |
| 任务路由 | 本仓库和 Docnav 都能用项目导航把任务导向 owner | 是否需要跨项目复用的公共入口，以及项目结构不同时怎样适配和降级 |
| 关系与交接 | 各 skill 已声明部分下游边界；Docnav 在 change 归档前同步稳定规范和长期决策 | 何时链接、同步或只产生候选，发生冲突时哪个 owner 优先 |
| 生命周期 | 各载体已有自己的状态、时间或 Git Scope；Docnav 区分 stable owner、active change 和当前实现 | 怎样跨类型判断哪些上下文对当前任务仍然有效，而不把不同状态强行合并 |
| 组合校验 | 各 owner 可以拥有局部检查器；Docnav 已把部分交接要求变成归档验收 | 怎样检查跨 owner 引用、遗漏交接和相互冲突，并形成端到端验收基线 |
| 统一入口 | skills 可以按需选择，具体组合由项目声明 | agent 或人类怎样从一个任务请求获得最小相关上下文，同时不夺取原 owner |
| 实时在途工作 | subagent-orchestration 只覆盖一次任务内协作，Docnav 也没有实时工作注册表 | 属于相邻的运行时协调系统；需要时单独设计，不作为当前缺口计数 |

按这个基线，内容 owner 层已经较完整，项目级路由和局部交接也有实践。主要差距集中在连接层：让上下文能够按任务被发现、跨 owner 关联、判断有效性并接受组合校验。现在还无法判断这些能力应继续由项目规则提供，还是需要通用的只读入口或关系模型。

#### Docnav 的参照价值

Docnav 已经表明，内容相关不等于 owner 重叠：最终事实可以留在稳定 owner，change 内判断留在 OpenSpec，跨 change 继续有效的理由再同步到长期决策。它还表明任务路由和部分交接规则可以留在项目本地。后续需要验证的是这些机制迁移到其他项目的成本，以及它们是否足以覆盖统一入口、有效性判断和组合校验。

#### 后续调查

后续重新开启时，选择少量不同结构的真实项目或任务，按上表记录已有机制、人工补偿、上下文遗漏和交接失败，再比较“只使用项目级导航与治理”和“增加公共只读入口或组合校验”分别解决了什么。只有目标系统明确包含运行时协调接口时，才把并行执行作为独立样本，不用它阻塞持久上下文差距的判断。

本轮没有形成建设决定、系统架构或新 skill 边界；这些问题要等差距在更多实践中被证实后再交给相应 owner。

### 以能力闭环区分系统缺口与证据缺口
- 形成时间: 2026-07-21T17:41:22+08:00

#### 形成时背景

本仓库目前用 [decision-records](../../../skills/decision-records/SKILL.md)、OpenSpec、[investigation-report](../../../skills/investigation-report/SKILL.md)和 [test-evidence-review](../../../skills/test-evidence-review/SKILL.md)分别保存长期判断、变更上下文、阶段性认识和稳定验证义务；[项目导航](../../navigation.md)负责把任务导向 owner，[仓库模型](../../repository-model.md)则明确 skill 间协作、交接和组合验证只有在行为 owner 或具体项目显式声明后才成立。

[Docnav 提交 `2df037f`](https://github.com/zxyycom/docnav/tree/2df037fa0a2553a9cabe6dc8d01187a424926393)提供了一个项目级实践样本：导航负责入口，稳定文档、OpenSpec change、长期决策与实现证据保持不同 owner，归档规则负责把需要跨 change 延续的内容同步到稳定 owner。当前没有统一系统的建设决定；实时并行状态属于需要独立身份、所有权和失效语义的运行时协调问题，不纳入本报告的持久上下文能力缺口。

#### 调查目的

重新审阅报告时发现，“现有 skills 还差多少”无法仅靠 skill 清单回答：载体数量不证明一次任务能否找到、判断、连接并回写正确上下文。报告还需要把尚未由现有组合承诺的能力与尚未取得的验证证据分开，否则容易把“没有证据证明已经解决”误写成“必然需要新系统”。

本轮要建立能够判断统一能力是否闭环的评估口径，并区分已经确认的组合能力缺口、尚待取得的证据以及不属于本主题的运行时协调问题；不在本轮选择实现方案或创建新的系统边界。

#### 调查范围与依据

本轮重新审阅本仓库的 decision-records、OpenSpec、investigation-report、test-evidence-review、项目导航和仓库模型，并继续以 Docnav 提交 `2df037f` 作为项目级机制样本。分析单位是一项任务从路由、内容恢复、关系判断到结果回写和组合校验的闭环；没有取得多个项目的真实任务对照、恢复时间、遗漏率、冲突率或维护成本数据，实时并行状态也不纳入持久上下文闭环。本轮没有记录本仓库材料对应的 commit 或未提交 diff，本报告本身不能确定仓库侧精确快照。

#### 调查结果与边界

以一次任务的上下文闭环衡量，对当前列出的上下文类型，现有组合已经给出较明确的记录 owner，并由本仓库和 Docnav 展示了项目级路由及部分归档交接；尚未共同证明的是“从任务入口恢复最小上下文、跨 owner 判断权威性与时效、把新认识回写给正确 owner，并检查整个交接没有遗漏或冲突”。因此现阶段最准确的距离判断不是百分比或缺少几个 skills，而是：**记录与 owner 层较完整，项目级连接已有样本，跨 owner 闭环及其通用价值尚未得到验证。**

这一区分同时保留两种可能：后续证据可能表明项目导航和项目规则已经足够，也可能暴露需要公共只读入口、关系投影或组合校验的重复问题。现有证据还不能在两者之间作出选择。

#### 统一能力的评估口径

“统一”不要求把所有正文集中到同一载体；最低判断口径是一次任务能否完成以下闭环：

| 环节 | 通过标准 | 当前证据与判断 |
| --- | --- | --- |
| 任务路由 | 从任务描述识别需要读取的最小 owner 集合，并能按项目结构降级 | 本仓库与 Docnav 都有项目导航样本；跨项目公共入口及降级行为未验证 |
| 内容恢复 | 每类必要上下文有明确 owner，能恢复目的、当前规则、变更背景、调查认识和验证义务 | 对本报告列出的上下文类型，现有 skills 与项目事实 owner 已有较完整分工 |
| 关系与有效性 | 能说明上下文之间的来源、影响和交接关系，并在状态或内容冲突时判断当前依据 | 各载体有局部状态和边界，但没有共同的权威性、时效或冲突解释契约 |
| 结果回写 | 任务产生的新判断、规范、调查结果或验证义务能进入正确 owner，不靠复制维持一致 | 各 skill 已声明部分交接边界；Docnav 验证了 change 归档时的部分同步规则 |
| 组合校验 | 能发现遗漏交接、失效引用或 owner 冲突，并证明一次闭环达到项目要求 | 当前主要是各 owner 的局部检查；尚无已声明的端到端验收基线 |

按这一口径，新增“通用记忆容器”不是当前证据指向的首要缺口。真正待验证的是连接层能否在不夺取 owner、不强制安装全部 skills、不过度读取的前提下闭合上述环节。

#### 能力缺口与证据缺口

| 类型 | 当前缺口 | 对后续判断的影响 |
| --- | --- | --- |
| 组合能力缺口 | 当前组合没有共同承诺任务入口、跨 owner 的权威性与时效判断、可检查的交接闭环 | 说明现有 skills 尚未构成一个可统一验收的整体，但不证明这些能力必须由新系统提供 |
| 证据缺口 | 缺少不同项目与真实任务的遗漏、冲突、恢复成本和维护成本对照 | 不能判断能力缺口是否高频，也不能判断项目级治理、公共 skill 或独立系统哪一种更合算 |
| 相邻问题 | 跨线程、跨 worktree 或跨进程的实时在途工作 | 需要单独的运行时协调模型；不应由静态导航或 active change 状态代替，也不计入本轮系统距离 |

#### Docnav 能证明到哪一步

Docnav 的直接参照价值有三点：项目导航可以承担本地任务入口；相关内容可以通过最终 owner 分工避免 OpenSpec 与长期决策互相覆盖；归档阶段可以用明确条件把跨 change 内容同步到稳定规范和决策记录。这些证据支持“统一能力可以由多个 owner 加项目治理组成”，并削弱了“统一必然等于集中存储”的假设。

Docnav 不能单独证明这些规则能低成本迁移到不同项目，也没有证明通用路由、跨 owner 有效性判断和组合校验已经解决。它还没有提供“使用治理前后减少多少遗漏或恢复时间”的对照，因此目前是机制样本，不是系统完整性或建设必要性的证明。

#### 后续最有区分力的调查

后续若重新开启，优先选取两个或以上结构不同的项目，并各取少量真实任务。每个任务记录任务入口、实际需要的 owner、漏读与噪音、冲突处理、结果回写、组合检查和维护成本，再比较两条基线：只使用项目导航与本地治理；增加公共的只读路由或组合校验。调查应先寻找重复失效模式，不以预先设计完整系统为前提。

只有当多组任务反复暴露同一连接缺口，且项目规则难以低成本解决时，才有证据把该缺口交给新 skill、公共关系模型或独立系统设计。运行时并行协调若要继续探索，应另立问题，不用它阻塞持久上下文闭环的判断。

#### 当前结论边界

本轮形成了统一能力的评估口径，并区分了能力缺口与证据缺口；没有形成建设决定、完成比例、系统架构、新 skill 边界或分发组合承诺。报告维持“暂停”，等待真实项目与任务样本提供下一轮证据。

### 当前 skill 组合与统一系统的差距
- 形成时间: 2026-07-21T17:50:15+08:00

#### 形成时背景

本仓库已经用 [decision-records](../../../skills/decision-records/SKILL.md)、[OpenSpec](../../../skills/openspec-explore/SKILL.md)、[investigation-report](../../../skills/investigation-report/SKILL.md)和 [test-evidence-review](../../../skills/test-evidence-review/SKILL.md)分别承接长期判断、change 上下文、阶段性认识和稳定验证义务；规范、代码、配置和测试等项目事实继续由各自行为 owner 承接，[项目导航](../../navigation.md)负责按任务定位这些内容。

[仓库模型](../../repository-model.md)明确说明，同仓维护不自动产生 skill 间的组合、交接、依赖或组合验证承诺。[Docnav 提交 `2df037f`](https://github.com/zxyycom/docnav/tree/2df037fa0a2553a9cabe6dc8d01187a424926393)则提供了多个上下文 owner 在一个真实项目内协作的实践样本。

本报告把“统一上下文管理系统”最低理解为：从一个任务进入，能够找到必要上下文、保留各自 owner、判断跨 owner 关系和有效性，并把新上下文回写到正确位置且完成必要检查。它不预设集中存储。跨线程、跨 worktree 或跨进程的实时在途工作需要独立的运行时协调设计，不计入本报告的持久上下文差距。

#### 调查目的

用户要求报告直接回答：现有上下文管理 skills 组合起来距离统一系统还有多远，并用 Docnav 判断哪些组合机制已经在实践中成立。能力闭环和证据缺口只作为分析工具，不作为调查主轴。

#### 调查范围与依据

本轮综合本仓库中相关 skills 的 owner 契约、项目导航与仓库模型，并复用 Docnav 提交 `2df037f` 中的导航、长期决策分工和归档治理证据，按内容 owner、上下文路由、关系与有效性、交接闭环和组合校验五项能力回答距离问题。材料仍只覆盖本仓库和一个用户自有实践项目，没有多项目任务对照或量化数据；结论不覆盖实时在途工作，也不用于决定具体系统方案。本轮没有记录本仓库材料对应的 commit 或未提交 diff，本报告本身不能确定仓库侧精确快照。

#### 调查结果与边界

已确认的事实是：现有 skills 已为主要持久上下文提供明确 owner，Docnav 已实践项目导航、owner 分离和归档交接。基于这些事实，当前可以形成的阶段性判断是：**现有组合已经越过“是否有内容载体”的阶段，但还没有形成可从任意任务统一进入、跨 owner 协作并整体验收的系统。差距主要在路由与协作层，而不是继续增加记录型 skills。**

这一定性判断不能换算成完成百分比，也不能直接推出是否应建设新系统。

#### 差距判断

| 能力 | 已有基础 | 距离统一系统仍缺什么 |
| --- | --- | --- |
| 内容 owner | 决策、change、调查、测试证据和项目事实已有明确分工 | 通过更多真实任务确认是否仍有重要上下文类型未被承接 |
| 上下文路由 | 本仓库与 Docnav 都用项目导航定位 owner | 从任务识别最小必要上下文，并适配不同项目结构和可选 skill 组合 |
| 关系与有效性 | 各载体有自己的状态和边界；Docnav 已区分稳定规范、change 与长期决策 | 表达跨 owner 来源与影响，并在冲突或失效时判断当前依据 |
| 交接闭环 | 各 skill 已声明部分交接边界；Docnav 已规定归档时的同步条件 | 统一说明何时链接、同步或产生候选，并确认结果进入正确 owner |
| 组合校验 | 各 owner 可以拥有局部检查，Docnav 已有部分归档验收 | 发现漏读、失效引用、遗漏交接和 owner 冲突，形成端到端验收 |

这些是判断统一系统是否成立所需的组合能力，不等于必须新增五个 skills，也不预先指定由项目规则、现有 skill、公共路由能力或独立系统承接。

#### Docnav 的参照作用

Docnav 直接提供了三个实践证据：

1. [项目文档导航](https://github.com/zxyycom/docnav/blob/2df037fa0a2553a9cabe6dc8d01187a424926393/docs/navigation.md)证明一个项目可以把稳定规范、OpenSpec、长期决策和实现证据路由到不同 owner。
2. [长期决策与 OpenSpec 分离](https://github.com/zxyycom/docnav/blob/2df037fa0a2553a9cabe6dc8d01187a424926393/docs/decisions/decision-management/separate-decision-spec-ownership.md)证明 change 内判断与跨 change 长期理由可以相关而不争夺最终 owner。
3. [归档治理规则](https://github.com/zxyycom/docnav/blob/2df037fa0a2553a9cabe6dc8d01187a424926393/openspec/specs/openspec-governance/spec.md)证明归档前可以按条件把跨 change 内容同步到稳定规范和长期决策。

这些证据把 Docnav 定位为项目级组合实践：它证明多个 owner 可以协作，但不能证明相同规则适用于任意项目，也不能证明从任务入口到结果回写的统一系统已经成立。

#### 证据边界与后续调查

当前材料来自本仓库的相关 owner 和一个用户自有实践项目，尚无多个项目中的真实任务对照，也没有上下文恢复时间、漏读、冲突或维护成本数据。因此后续调查应：

1. 在至少两个结构不同的项目中选择少量真实任务，记录实际需要的 owner 和上下文。
2. 观察现有导航是否足以完成定位、有效性判断、结果回写和组合检查，并记录遗漏、噪音、冲突与维护成本。
3. 比较项目导航与本地治理、公共只读路由或组合校验分别改善了什么，再判断重复缺口应由哪个 owner 承接。

只有真实任务反复暴露同一组合缺口，才进一步讨论扩展现有 skill、创建新 skill 或建设独立系统。实时在途工作继续作为独立调查方向。

#### 当前结论边界

本轮确认现有 skills 与统一系统之间的主要差距位于路由与协作层。没有形成建设决定、完成比例、系统架构、新 skill 边界或分发组合承诺；报告维持“暂停”，等待真实项目和任务样本提供下一轮证据。

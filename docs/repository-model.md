# 仓库模型

本文档是本仓库产品定位的 owner，定义集中维护的目的、使用者如何选择 skill，以及通用分发能力的范围。具体脚本、校验、打包、CI 和 updater 行为由 [项目工具链](tooling.md) 承接；单个 skill 的触发、行为、产物和验收由对应 `skills/<skill-name>/` 承接。

## 集中维护

1. 在一个仓库内维护多个可复用 skill，共享项目约定、决策记录、校验、生成、打包和发布工具链，减少每个 skill 独立建仓产生的重复维护。
2. `skills/<skill-name>/` 继续作为单个 skill 的行为和分发内容边界，使各 skill 可以按自身问题独立演进。
3. `tools/` 承接需要构建后随 skill 分发的可维护源码，`scripts/` 承接主仓库自动化，`skills/` 只承接行为本体和实际分发内容；三者不因同仓维护而合并 owner。
4. 根目录只承接跨 skill 的稳定公共能力；skill 专属规则、引用、脚本和素材留在对应 skill 目录。

## 选择与使用

1. 使用者按项目需要选择 skill，并在安装或明确使用前大致了解其用途、主要行为和产物。
2. 同仓表示 skills 共享维护和发布基础设施；使用者只需安装或启用自己选择的部分。
3. 安装或明确调用某个 skill，表示使用者主动选择该能力及其已经说明的工作方式；具体写入、外部状态变更、高风险操作和不可逆动作仍服从当前任务与运行环境的授权边界。
4. skill 间的协作、前置、交接或分发关系由对应行为 owner 或当前项目显式声明，不由仓库位置或主题相近自动产生。

## 轻量分发

1. 主仓库统一发现和校验 `skills/` 下的内容，并为每个 skill 分别生成制品；聚合 release 为这些制品提供同一个获取入口。
2. 随 skill 分发的 updater 由使用者显式运行，只检查并可选替换当前 skill。
3. 通用分发的目标是简化 skill 的获取和更新，不管理使用者的完整安装集合。依赖解析、组合安装和兼容迁移等能力，只有在出现已确认的现实需求并建立独立 owner 后才进入实现范围。
4. 特定 skills 可以声明共同工作流或更强的分发关系；这类局部契约不改变主仓库通用分发能力的范围。

## 验证职责

1. 主仓库验证负责证明源文件结构、共享工具、生成状态和可分发制品符合当前项目契约。
2. 单个 skill 是否改善目标行为、需要哪些现实案例或行为回归证据，由该 skill 的行为与验收 owner 决定。
3. 只有显式声明的共同工作流才产生对应的组合验证责任。

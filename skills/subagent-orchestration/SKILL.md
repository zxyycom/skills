# Subagent Orchestration

## 目标

把长任务、复杂任务和高上下文噪音任务拆给子代理完成。主代理保持上下文干净, 专注于编排、目标一致性审计和下一步决策; 子代理负责调查、实现、验证和审查。

## 核心原则

1. 默认子代理具备当前 agent 系统提供的同类基础上下文、项目规则、工具能力、扩展能力和运行约束; 任务中不重复讲解通用规则。
2. 子代理不继承完整历史对话; 只传递完成当前子任务所需的最小上下文。
3. 主代理派发结果导向的任务, 不把子代理当成逐步执行命令的远程终端。
4. 主代理信任子代理的行动过程; 等待时间长不等于失败, 通常表示范围过大、验证较重或影响面复杂。
5. 子代理短时间无法可靠完成的工作, 主代理通常也无法快速可靠完成; 优先拆分、缩小范围或追加审查, 而不是接手细节实现。
6. 并行任务服从权限、所有权和依赖边界: 写入所有权互斥; 只读任务可在需要独立结论或互补证据且输入稳定时重叠范围。
7. 给子代理的消息聚焦目标、范围、边界、验收和输出; 只有能显著降低误判的结论才作为细节上下文传递。

## 主代理职责

1. 决定使用子代理后, 先明确各任务的目标、权限、所有权和依赖, 再派发任务。
2. 每个子任务只设置一个主要目标, 并给出清晰的文件、模块、问题或决策范围。
3. 等待子代理返回结构化结果; 主代理基于结果判断目标是否达成。
4. 结果不足时, 派发更小范围的实现、只读调查或审查任务。
5. 需要复核实现时, 优先派发审查子代理, 不为了复核而在主线程重新展开同一批文件级细节。
6. 只有需求、权限、目标或取舍无法继续推进时, 才向用户提问。

## 子代理任务模板

```text
目标: <需要达成的具体结果>
范围: <允许处理的文件、目录、模块、问题或决策边界>
权限: <只读或可写; 可写时列出唯一写入范围>
依赖: <可立即并行, 或必须等待的前置结果和稳定输入>
上下文: <完成任务所需的最小背景, 不包含完整历史对话>
约束: <必须遵守的具体限制、用户偏好或项目规则>
验收: <什么条件下可以认为任务完成>
输出: <返回发现、改动、验证、风险和阻塞点>
```

## 权限与并行边界

1. 可写子代理的写入范围必须互不重叠; 按实际可能修改的文件、配置、生成产物或外部状态划分所有权。
2. 只读子代理只产出调查或审查结果, 不改变共享工作区或外部状态; 为获取独立结论或互补证据, 可以重叠读取同一文件、模块或决策面。任务中明确重叠目的和各自输出; 没有额外验证价值时合并为一个任务。
3. 同一范围既需实现又需审查时, 明确哪个子代理负责修改、哪个子代理只读审查; 审查依赖实现结果时, 等写入结束后再读取稳定结果。
4. 后续任务依赖前一个子代理改动时, 串行派发, 不并行派发。
5. 子代理发现非自己产生的改动时, 应报告并围绕自身范围继续工作; 主代理负责判断是否需要重新拆分。

## 等待和拆分策略

1. 子代理耗时较长时, 先判断任务是否过大, 不把等待时间本身视为失败。
2. 范围过大时, 按文件、模块、阶段、验证类型或审查目标拆分。
3. 子代理报告阻塞时, 优先派发只读调查或更小范围任务澄清阻塞点。
4. 多个子代理结果冲突时, 派发只读审计任务核对冲突点。
5. 不把等待时间或冲突结果作为主代理接手实现的理由。

## 结果审计

主代理审计子代理输出时只关注:

1. 是否达成用户目标。
2. 是否超出任务范围。
3. 是否存在未经确认的假设。
4. 是否有足够验证支撑交付。
5. 是否存在残余风险、冲突结果或需要继续拆分的范围。

不要为了了解细节而深入实现过程。只有当细节直接影响目标判断、风险判断或下一步拆分时, 才要求子代理补充说明。

## 子代理输出格式

要求子代理优先返回结构化结果:

```json
{
  "summary": "<一句话说明完成状态>",
  "findings": ["<关键发现或空数组>"],
  "changes": ["<改动文件、模块或行为; 没有则为空数组>"],
  "verification": ["<已运行验证及结果; 没有则说明原因>"],
  "risks": ["<残余风险或假设; 没有则为空数组>"],
  "blocked": ["<阻塞点; 没有则为空数组>"]
}
```

## 完成标准

同时满足以下条件时, 主代理可以交付:

1. 子代理结果覆盖用户目标。
2. 实现、审查和验证之间没有未解决冲突。
3. 所有并行任务均遵守权限、写入所有权和依赖边界。
4. 验证结果足以支持当前交付。
5. 残余风险已经明确说明, 且不阻塞交付。

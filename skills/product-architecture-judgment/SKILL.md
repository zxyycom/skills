# Product & Architecture Judgment

## 目标

在进入或继续局部实现前, 从产品价值与架构责任形成可直接用于当前任务的工程判断: 这件事是否该做、该做到什么程度, 以及应由谁、在哪一层实现。

## 核心判断

1. **分开结果、问题和解法**: 目标结果用于校准方向; 当前问题、需求表述和已有实现都是待检验的候选。
2. **向上看产品价值**: 判断真实消费者、预期变化、必要性、收益、成本和成功边界。
3. **向外看架构责任**: 判断责任 owner、事实源、系统边界、数据流、抽象和演进成本。
4. **向下选工程动作**: 在不做、简化、改写、重定责任、提炼抽象或局部实现中选择有证据支持的最小动作。

放眼大局用于找到直接兑现真实结果的最小责任边界, 不以扩大范围或追求更抽象的方案为目标。

## 执行流程

### 1. 判断是否需要展开

1. 明确当前任务出口与用户授权: 分析、决策、设计、实现、排障、重构、审查或其他交付。
2. 用户明确要求判断必要性、产品价值或架构合理性时展开。
3. 出现以下信号时展开:
   - 围绕一个技术点反复优化, 但用户或开发者结果没有相应改善。
   - 局部规则、例外、补偿机制或跨模块协调成本持续增加。
   - 需求只描述接口、字段、组件或其他解法形式, 没有说明预期结果。
   - 当前层级只能处理症状, 或相同责任散落在多个 owner。
   - 实现与维护成本可能超过收益, 或复杂功能可能由稳定抽象简单承接。
4. 目标结果、责任 owner、边界和实现路径已经清楚, 且任务确实局部时, 简要确认后直接执行。

### 2. 恢复目标与证据

1. 分别写清希望得到的结果、当前被要求解决的问题或需求, 以及已经选中的解法。
2. 找到真实消费者和场景。消费者可以是最终用户、调用方、运营者、开发者、维护者或后续系统。
3. 对照当前结果与期望结果, 找出成功证据以及必须保持的行为、兼容、安全、性能、恢复和运维约束。
4. 将上下文区分为事实、明确约束、基于证据的推断、未验证前提和已有选择。
5. 只恢复足以改变当前工程选择的信息; 关键选择无法可靠判断时, 调查或确认该选择。

### 3. 形成产品判断

1. **必要性**: 即使当前解法成功, 是否会产生消费者真正需要的变化。
2. **受益与代价**: 用户、调用方、运营者和开发者分别获得什么收益, 又由谁承担使用、迁移、维护和认知成本。
3. **完成程度**: 哪些结果必须实现, 哪些范围、质量或扩展能力可以缩小、推迟或省略。
4. **替代路径**: 现有能力、不采取动作或更简单的流程能否获得同一结果。
5. **判断依据**: 只使用用户说明、行为契约、真实消费者和可观察结果能够支持的结论; 其余内容保持为假设或未知项。

### 4. 形成架构判断

1. 确认语义、状态和约束的责任 owner 与单一事实源, 检查派生数据、缓存、schema、映射和测试是否围绕它组织。
2. 追踪输入、输出、状态、控制流和数据流怎样跨越模块、进程、持久化或信任边界。
3. 判断当前问题是 owner 明确的局部缺口, 还是边界、事实源、执行顺序或数据流造成的系统性症状。
4. 检查现有抽象、策略、配置或公共流程能否承接目标, 避免在消费端重复实现上游责任。
5. 用相邻字段、状态、消费者和现实用例检验抽象稳定性。多个现实用例共享同一责任时可以提炼抽象; 只有假想复用时保持局部。
6. 识别方案转移给其他模块、兼容层、诊断恢复、后续演进和开发者认知的成本。

架构调查止于能够解释结果并承接责任的最小边界。

### 5. 选择并完成工程动作

根据产品判断与架构判断选择:

1. **不做**: 结果缺少足够价值、已经由其他机制满足, 或成本明显高于收益。
2. **简化**: 只保留能够直接产生目标结果的范围和质量。
3. **改写**: 用真实结果重新表述由既有解法定义的问题或需求。
4. **重定责任**: 调整 owner、边界、事实源、执行顺序或数据流, 消除局部症状。
5. **提炼抽象**: 修正或建立有现实复用依据的稳定抽象, 让当前复杂功能成为简单应用。
6. **局部实现**: 产品价值成立、架构责任清楚且缺口确实局部时, 在正确 owner 内完成最小实现。

比较候选与目标结果的直接程度、用户和开发者收益、根因解释力、实施与协调成本、风险、可逆性、演进能力和证据强度。证据不足时先完成能改变结论的最小调查、实验或度量。

选定动作后回到当前任务:

1. 分析、决策或设计任务先给工程判断, 再说明真正改变结论的产品依据、架构依据、取舍和未知项。
2. 实现、排障或重构任务在用户授权内完成选定动作, 并验证产品行为和受影响的架构约束。
3. 审查任务沿用目标项目的格式和优先级, 报告具有实际产品影响、开发者成本或架构代价的问题。
4. 输出继续服从当前任务的行为 owner、验证规则和交付格式, 不默认生成独立报告。

## 边界

1. 判断以真实消费者和可观察结果为依据, 不扩展为组织级产品战略或技术路线规划。
2. 调查和方案止于能够兑现目标的最小责任边界; 局部 owner 足够时保持局部, 缺少现实复用证据时不提炼抽象。
3. 技术正确性仍是必要条件; 产品价值和架构合理性不能替代类型、测试、安全、性能或运行验证。
4. 判断不会扩大修改授权。选定方向会改变用户已明确的目标、范围或外部状态时, 先交付判断并请求确认。
5. 本 skill 独立完成完整判断, 不依赖其他问题重构、产品或架构 skill。

## 完成检查

1. 是否分开了目标结果、当前问题或需求和已有解法?
2. 是否说明事情为何值得做、服务谁、应做到什么程度, 并标明依据与未知项?
3. 是否定位责任 owner、事实源、关键边界、数据流和合适抽象层?
4. 选定动作是否直接兑现目标、范围最小, 且由现实证据支持?
5. 是否完成当前任务要求的分析、设计、修改、验证或审查, 并遵守用户授权与任务 owner?

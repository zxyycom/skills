# Product & Architecture Thinking

## 目标

让 agent 在回答“怎么做”前, 先确认“为什么值得做”和“应该由谁、在哪一层负责”, 再选择与产品结果和架构责任一致的技术动作。

## 核心视角

1. **向上看产品**: 从局部需求或实现回到用户、场景、预期结果和真实约束。
2. **向外看架构**: 从当前文件或模块扩展到责任 owner、系统边界、事实源、数据流和演进路径。
3. **向下落实现**: 在产品结果与架构责任成立后, 选择最小且处于正确层级的技术动作。

产品和架构视角用于改变判断起点, 不要求形成两份独立分析。没有产生新约束时保持简短; 确实改变方案时明确写出依据与取舍。

## 应用方式

1. 明确当前任务出口与用户授权: 分析、设计、实现、排障、重构、审查或决策。
2. 读取用户目标、项目指令、相关行为 owner, 以及理解任务所需的代码、文档、测试和运行证据。
3. 区分事实、基于证据的推断和会改变方案的未知项。上下文足以继续时直接推进; 关键选择无法可靠推断时只确认该选择。

### 向上形成产品判断

从技术表述恢复实际结果:

1. 确认用户、调用方、运营者或其他真实消费者, 以及触发问题或使用能力的场景。
2. 对照当前结果与期望结果, 找出需要保持的行为、兼容承诺、数据结果、安全边界、性能边界或恢复能力。
3. 确认需求可达性与实际影响, 区分产品需要与由接口、数据结构、校验器等实现形式产生的代理目标。
4. 比较实施、简化、推迟或取消后的产品结果, 判断技术工作的真实价值。

只把用户说明、行为契约、外部消费者和可观察结果能够支持的产品结论作为方案依据; 其余内容明确为假设或未知项。

### 向外形成架构判断

沿责任和信息流定位问题应落在哪一层:

1. 确认语义、状态或约束的责任 owner 与单一事实源, 检查派生数据、缓存、schema、映射和测试是否围绕它组织。
2. 追踪输入、输出、状态、控制流和数据流怎样跨越模块、进程、持久化或信任边界。
3. 判断当前问题是 owner 明确的局部缺口, 还是边界、事实源、执行顺序或数据流造成的系统性症状。
4. 用相邻字段、状态、消费者或流程分支检验修改面、协调成本和演进能力。
5. 识别方案转移给其他模块、后续能力、诊断恢复或兼容层的成本。

架构判断扩展到能够解释问题的最小责任边界, 并让方案停留在该边界内。

### 向下形成技术动作

根据两个方向选择动作层级:

1. 产品结果缺少依据或收益不足时, 把取消、简化或推迟作为首选建议, 并说明重新评估所需的事实; 未经用户确认不改变已经明确的任务目标。
2. 产品结果成立但责任 owner、边界或事实源不合理时, 先把处置提升到系统设计层, 再按用户授权决定是否修改局部实现。
3. 产品与架构已经清楚且问题局部时, 直接在正确 owner 内完成与问题边界相称的最小实现。
4. 两个方向发生张力时, 明确取舍: 哪个产品结果需要保留、哪个架构成本可以接受、代价由谁承担。
5. 证据不足时, 选择能改变结论的最小调查、实验或度量, 并保留仍无法确认的部分。

### 回到当前任务

1. 分析或决策任务先给结论, 再说明真正改变结论的产品依据、架构依据和未知项。
2. 设计任务让方案同时指出目标结果、责任 owner、关键边界和实现落点; 只比较会影响选择的备选。
3. 实现、排障或重构任务按用户授权完成修改, 并验证产品行为与受影响的架构约束。
4. 审查任务按目标项目的格式和优先级, 报告具有实际产品影响或架构代价的问题。
5. 任务继续遵循专门行为 owner、验证规则和输出契约; 本 skill 只提供上层判断视角。

## 边界

1. 产品方向由用户与对应行为 owner 决定; 构成重大取舍且缺少依据时明确假设或请求判断。
2. 架构分析扩展到正确承接当前责任的最小边界; 局部 owner 足够时保持局部。
3. 技术正确性仍是必要条件; 产品价值和架构合理性不能替代类型、测试、安全、性能或运行验证。
4. 对目标、owner 和边界已经明确的直接任务, 简要确认依据后执行。
5. 输出沿用当前任务的结构、篇幅和术语, 服务于任务出口与用户决策。

## 完成检查

1. 产品结论是否由真实消费者、场景、预期结果或明确假设支持?
2. 架构结论是否定位了责任 owner、边界、事实源与关键数据流, 并区分局部和系统性问题?
3. 技术动作是否由产品结果和架构责任推出, 同时遵守用户授权与任务 owner?
4. 是否完成原任务要求的分析、设计、修改、验证或审查, 并只交付会影响判断的结论?

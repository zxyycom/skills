# Test Evidence Review

## 目标与边界

对测试和难以自动化的稳定风险给出语义处置，并让保留的验证义务具有可回查的背景、实现入口或人工审查状态。记录服务于当前评估和后续巡检，不把“已有账本”“测试通过”或“CLI 通过”当成测试价值或项目测试正确性的证明。

职责边界：

1. 本 skill 评估测试固定了什么、是否形成独立证据、应由自动化还是人工审查承接，并同步账本与入口映射。
2. 测试框架负责执行测试；项目测试策略拥有测试层级、覆盖要求和正常验证入口。CLI 不执行测试，也不判断断言是否正确。
3. `Contract:` 只写 case 的行为背景和必须保持的规则，不写契约由谁决定，也不以代码路径、测试名称或历史事故代替文字语义。
4. 审查、巡检和评估请求保持只读；只有修改请求才更新测试、契约 owner、账本或配置。执行 `Review:` 本身是只读检查，不扩大写入授权。

## 读取与范围

1. 读取项目指令、测试策略、当前 diff、相关契约 owner、实现、测试和覆盖说明，再读取 [ledger-contract.md](references/ledger-contract.md) 并定位配置、账本与 Git 工作区；项目规则优先。
2. 默认评估本次新增、修改或删除的测试与实现、它们所属的 case、命中当前 Git 变化的 review `Scope:` 和必要相邻证据。只有用户要求、项目约定或发现跨边界风险时才扩大语义评估。
3. 已有账本时先运行一次机械对账恢复入口映射和 trigger；账本不存在时记录缺口，修改任务在完成语义评估后初始化，只读任务只给出拟记录。
4. CLI 返回的结构或映射问题按位置处理。与当前变化无关的存量未登记入口按配置报告，不要求借当前任务顺手清零。

## 语义评估

先恢复测试需要固定的契约背景和共享测试基座，再写证明点：

```text
Contract：当前 case 需要长期固定的行为、边界或不变量
case 语境：共享初始状态、fixture、执行上下文或行为链路
Proves 项：分支条件或链路阶段 -> 可观察结果
```

优先从产品或接口规范、schema、安全边界、决策记录、用户明确要求和项目拥有的工程不变量恢复契约。现有实现和测试可以证明当前行为，但不能仅因“代码现在这样写”就把偶然实现升级为必须长期保持的契约；没有独立文档时，先确认该行为确实需要保留，再用文字写入 `Contract:`。

`Proves:` 每项只写一个简短、直接且可独立判断的观察点。证明点不只来自分支叶子：一条顺序执行到底的 happy path 或失败链路，可以在多个阶段同时证明状态迁移、输出、副作用或清理不变量，并在同一 case 中保留多个 `Proves:` 项。`Proves:` 数量与测试入口数量不要求一一对应；一个测试入口可以沿同一链路证明多项，多个入口也可以共同承接一个 case。

多个证明点共享测试基座或必须在同一连续行为链路中理解，且拆分会复制 fixture、丢失顺序关系或增加执行、同步和审计成本时，保留在同一 case。具体字段、线性链路和分支图规则以账本契约为准。

保留或新增自动化测试前，同时确认：

1. **契约背景**：能说明需要固定的产品规则、接口行为、schema、安全边界、错误语义或明确工程不变量。
2. **路径价值**：输入进入契约定义的分支、等价类、责任边界、信任边界、失败出口或有顺序意义的链路阶段。
3. **可观察性**：断言验证调用方可观察结果，或当前白盒层拥有的内部不变量。
4. **失败价值**：失败能指出哪项稳定行为不再成立，而不只是提示实现被改写。
5. **成本比例**：fixture、mock、环境和后续维护成本与证据价值相称。

TDD、防回归、覆盖率和历史 bug 只提供调查动机。以下测试通常不形成新证据：

1. 只更换字面量、枚举值或同类非法输入，仍进入同一契约分支或链路并断言同一结果。
2. 只复述实现步骤、私有布局、内部调用顺序、mock 预设或语言与框架的普通保证。
3. 只执行路径而没有稳定结果断言，或历史测试名称已无法还原为当前契约。

## 主流程

1. **建立清单**：列出当前测试、稳定风险、发现误报、相关 case、入口映射诊断和 review trigger；记录无法运行的环境，不把环境缺失作为价值结论或人工 review 的充分理由。
2. **搜索既有证据**：按契约、行为、输入类别、错误、输出和风险搜索账本与测试；比较共享基座、行为链路、入口、fixture 和可观察证明点，不只比较名称。
3. **选择处置**：每个测试或风险只选择一个主要结果，并写明契约、证明价值和成本依据。
   - **待确认**：缺少可确认的稳定契约、可观察结果或可描述风险；不创建 case。
   - **不保留**：新增测试没有独立证据，或既有测试成本明显高于价值；不新增或删除。
   - **复用**：既有 case 和测试已经承接相同证明，不再增加入口。
   - **合并或扩展**：与既有 case 共享契约、测试基座或连续行为链路；合并重复入口，或补充新的 `Proves:`、测试入口和必要的分支图或顺序链路图。
   - **新建自动化证明**：形成独立契约、观察入口、行为链路、运行环境或维护周期；测试已经实现时使用 active automated，证明目标明确但测试尚未实现时使用 planned automated。
   - **人工审查**：存在稳定风险，自动化成本不成比例，且能写出只读、具体、可执行的 CR 动作；使用 active review。
   - **发现豁免**：发现器把非测试语法识别为测试入口；确认误报后使用 active exempt，不用它承接低价值测试或难测风险。
4. **按授权实施**：修改任务使用最小输入和断言完成处置，并同步必要的契约 owner、账本和入口标记。契约与测试不一致且无法从现有 owner 判断时，只确认应改变哪一方。
5. **验证结果**：运行最小相关测试、命中的人工审查动作和最终机械对账；按项目要求补充类型检查、lint、构建或更大范围测试。

## 记录与入口映射

1. 账本不存在时，修改任务按默认路径初始化；只有默认发现规则不适用时才创建配置。
2. case 标题固定使用 `### Case <CASE-ID>: <title>`。automated case 用 `Contract:` 记录文字契约背景、一个 main 定位规范测试入口、derived 关联同一 case 的其他入口；exempt 只关联实际误报入口。
3. 每个被发现器识别的测试入口必须且只能紧邻一个 `@test-evidence main|derived|exempt <CASE-ID>`。多个入口可以归入同一 case，其中只有一个 main，其余使用 derived；账本按契约和共享测试基座组织，不按测试函数枚举。
4. 首次接入以 `warn` 暴露存量未登记入口，本次新增或修改的入口必须登记；基线可信并处理完存量后再使用 `error`。
5. 验证方式变化时保留稳定 case ID，并按当前事实替换字段和入口映射；待确认、不保留和已经删除且不再承载义务的测试不创建 case。

## Review trigger

1. `reviewTriggers` 的 `warn | error` 只控制 CLI 退出级别，不决定是否需要执行 CR。
2. 脏工作区或 `Reviewed-Commit` 后的路径命中 `Scope:` 时，自动把对应 case 纳入本次只读检查并执行全部 `Review:` 项。
3. 没有最近 CR、最近结果不是 pass 或提交基线不可读取的无路径 trigger，在当前任务涉及该 case 或用户要求巡检时执行；其他情况作为既有未完成义务单独报告，不据此扩大测试语义评估。
4. `Review:` 只包含能从当前仓库或明确可用环境执行的检查动作，不借 trigger 修改命中路径。动作需要缺失的权限、环境或外部协调时，返回 blocked 和具体缺口。
5. CR 针对一个已经存在的稳定 commit 完成后，同步 `Review-Result`、`Reviewed-At` 和 `Reviewed-Commit`。如果检查的是未提交内容，先报告本次结果；等相同内容形成提交并确认未发生变化后再写入该提交，不伪造 hash。状态未持久化前 trigger 保持待处理。

从本 skill 目录运行机械对账：

```bash
node scripts/test-evidence.mjs check --root <workspace-root>
```

需要机器报告时添加 `--json`。CLI 使用 Git 可见路径校验 `Scope:`，并返回 `errors`、`warnings`、`reviewTriggers` 和汇总。分别处理结构错误、入口映射、review 动作和仍待持久化的状态；机械结果不代替语义结论。

## 周期巡检

优先抽样：

1. derived 入口、fixture、mock、snapshot、分支或线性检查点持续增长，且共享基座、行为顺序、`Contract:`、`Proves:` 与 Mermaid 可能已经漂移的 automated case。
2. `Scope:`、成本原因、检查动作或最后 CR 基线已无法对应当前架构的 review case。
3. 豁免原因可能已被发现器、框架或目录变化消除的 exempt case。
4. 经常随重构变化但行为不变，或同时承载多个契约、结果或无关风险的测试。

让巡检结论落到主流程中的既有处置，并同步账本、测试入口角色和实现。

## 交付与完成

1. 先给总体判断；对每个需要动作的测试或 case 说明位置、处置、契约或风险依据，以及实际改动或建议。
2. 确认本次新增或修改的 automated case 包含可独立理解的 `Contract:`，且每个 `Proves:` 项对应明确的分支条件或链路阶段以及可观察结果。
3. 确认本次新增或修改的测试入口各自映射一个合法角色和 case；存量未登记入口按配置报告。每个 active automated case 只有一个 main。
4. 确认 review case 包含稳定契约、有效 Scope、成本原因和只读检查动作；路径 trigger 已执行，无路径 trigger 已执行或按范围单独报告。
5. 确认 exempt case 对应真实发现误报，且每个 exempt marker 只豁免紧邻的误报入口。
6. 报告实际测试、机械校验、人工审查、未执行环境和残余风险，并区分行为失败、映射失败、存量诊断与人工风险。

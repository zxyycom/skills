# Test Evidence Review

## 目标

把本 skill 作为自动化测试进入或留在代码库前的证据准入流程：只新增或保留能对应稳定 owner、独立条件和可观察结果的测试，并让通过准入的证明目标保持可追溯。

调用本 skill 时同时启用两个工作面：

1. **语义准入**判断测试的证明价值、重复证据和实现耦合，并选择最小处置。
2. **可追溯登记**用账本和源码标记保存证明目标与测试实现的映射，再由 CLI 机械校验。

## 读取顺序与边界

1. 先读取目标项目指令、测试策略、被测行为 owner、相关测试和覆盖说明；项目规则优先于本 skill 的默认判断。
2. 每次触发后读取 [ledger-contract.md](references/ledger-contract.md)，再按 `.test-evidence.json` 或默认路径定位账本。账本尚不存在时进入初始化流程，不把缺少账本解释为机械对账未启用。
3. 用户只要求审查、巡检或评估时保持只读；用户同时要求修改时，才改动测试、行为 owner、账本或配置。
4. 默认范围是本次 diff 涉及的测试、对应行为和相邻证据。只有用户要求、项目约定或明确风险支持时才扩大为全量语义巡检。
5. 测试框架负责执行测试，项目测试策略拥有测试层级和覆盖要求，本 skill 负责证据判断；CLI 只负责结构与映射，不能代替语义审查。

## 证据准入标准

先把候选测试写成一句证明陈述：

```text
owner 的稳定承诺 -> 独立条件或边界 -> 被测行为 -> 调用方可观察结果
```

保留或新增测试前，同时确认：

1. **规则来源**：能指出产品规范、接口契约、schema、安全边界、错误语义或明确工程不变量。
2. **独立性**：输入进入新的等价类、责任边界、信任边界、失败出口或观察入口。
3. **可观察性**：断言验证调用方可观察结果，或当前白盒层明确拥有的内部不变量。
4. **失败价值**：测试失败能指出哪项稳定行为不再成立，而不只是提示实现被改写。
5. **成本比例**：fixture、mock、环境和后续维护成本与证据价值相称。

TDD、防回归、覆盖率和历史 bug 是开始调查的动机，不是独立证明目标。以下情况通常不形成新证据：

1. 只替换字面量、枚举值或同类非法输入，仍进入同一 owner 分支并断言同一结果。
2. 断言复制实现步骤、私有数据布局、内部调用顺序或 mock 的预设返回值。
3. 只证明语言、框架或第三方库已经保证的普通行为。
4. 只执行代码路径，没有稳定且有意义的结果断言。
5. 测试名称保留历史回归故事，但当前无法还原为仍有效的契约。

## 证据准入流程

### 1. 建立候选与基线

1. 区分当前任务是新增、修改、审查、精简、语义巡检还是机械对账。
2. 列出本次涉及的测试候选，定位行为 owner、现有测试、实际验证入口和账本 case；账本尚不存在时建立本次范围的拟登记清单。
3. 记录无法读取或运行的测试框架、环境和外部依赖，不把环境缺失解释为测试价值结论。

### 2. 搜索已有证据

1. 按责任域、行为、输入类别、错误或输出搜索账本和现有测试；初次接入时以现有测试和行为 owner 恢复初始 case。
2. 比较 owner、`Proves:`、入口、fixture 和叶子断言，不只比较测试名称。
3. 把多个写法、枚举字面量和同类非法值视为同一候选等价类，除非 owner 定义了不同处理分支。

### 3. 作出处置

每个候选只选择一个主要处置，并写明依据：

1. **待确认**：找不到稳定 owner 或可观察结果；报告缺少的契约，不先写测试固化当前实现。
2. **不新增**：没有独立证据，或维护成本明显高于证明价值。
3. **复用或合并**：已有测试已经证明同一行为；复用现有证据，或把重复测试合并。
4. **扩展**：出现 owner 已承诺的新分支，但仍属于同一稳定证明目标。
5. **新增**：出现新的 owner、责任边界、观察入口、结果类型或需要独立演进的等价类。
6. **删除或降级**：既有测试不再证明稳定行为；删除，或降为已有证据的辅助测试。

### 4. 按授权实施

1. 用户要求修改时，以能证明目标的最小输入集合和断言完成处置。
2. 行为契约与测试不一致时，先确认哪个 owner 应变更，再在同一 patch 更新对应内容。
3. 用户只要求审查时不写文件，只提供可定位、可执行的处置意见。

### 5. 维护账本与映射

1. 账本不存在且任务允许修改时，按默认路径初始化账本并登记本次范围；只有默认发现规则不适用时才创建 `.test-evidence.json`。
2. 账本不存在且任务保持只读时，不写文件；把缺失账本作为审查结论，并给出本次范围的拟 case、主要入口和辅助归属。
3. 更新 case、`Code:` 和源码标记，让账本保存稳定证明目标，不为每个测试函数创建 case；主要入口使用 `@case`，辅助证据使用 `@supports`。
4. 首次接入以 `warn` 暴露存量未登记测试，但本次新增或修改的测试必须登记；基线可信后再切换为 `error`。
5. `@test-exempt` 只处理发现误差或不承接产品证明的特殊测试，不用它隐藏低价值或漏登记测试。
6. 从本 skill 目录运行：

```bash
node scripts/test-evidence.mjs check --root <workspace-root>
```

需要机器报告时添加 `--json`。CLI 失败时修正结构或映射；CLI 通过后仍保留本 skill 的语义结论。

### 6. 验证与交付

1. 运行能覆盖处置结果的最小测试和机械对账。
2. 按项目要求补充类型检查、lint、构建或更大范围测试。
3. 区分行为失败、机械映射失败和环境未执行；只读任务因账本尚未建立而无法运行 CLI 时，明确报告拟登记结果和未执行原因。

## 周期巡检

用户要求周期或存量巡检时，优先抽样：

1. `@supports`、fixture、入口或分支持续增长的证明目标。
2. 经常因重构修改、但产品行为没有变化的测试。
3. 大量依赖 mock、snapshot、生成材料或复杂环境的测试。
4. 长期未触达、而对应实现频繁变化的证据。
5. 一个证明目标或 case 同时承载多个 owner 或可观察结果的区域。

巡检结论必须落到待确认、不新增、复用或合并、扩展、新增、删除或降级之一，不只输出“测试不足”或“建议增加覆盖率”。

## 交付格式

先给总体判断。每个需要动作的候选说明：

1. 测试位置或 case。
2. 处置。
3. 证明陈述，或与现有证据的重叠点。
4. 所需的测试、owner、账本或标记改动。

最后列出实际运行的测试和机械检查，以及未执行环境、待确认契约和残余风险。

## 完成检查

1. 每个保留或新增的测试是否能指出 owner、独立条件和可观察结果。
2. 是否搜索过已有证据，并处理证明目标相同的重复测试。
3. 处置是否验证稳定契约，而不是固化实现步骤或 mock 配置。
4. case、主要入口、辅助归属、豁免和发现结果是否一致。
5. 是否报告实际验证、未执行项和仍待确认的行为契约。

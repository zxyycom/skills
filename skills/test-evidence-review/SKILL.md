# Test Evidence Review

## 目标

把本 skill 作为测试与相关风险进入或留在代码库前的准入流程。先判断候选是否形成需要长期维护的验证义务；形成义务时选择自动化证明、人工审查或发现豁免，不形成义务时选择待确认、不新增、合并或删除，且不创建新的账本 case。

调用本 skill 时同时启用两个工作面：

1. **语义准入**判断自动化测试的证明价值、重复证据和实现耦合，并判断风险应自动验证还是人工审查。
2. **可追溯登记**默认用账本保存验证义务，用源码标记保存测试文件的角色，再由 CLI 机械校验。

## 读取顺序与边界

1. 先读取目标项目指令、测试策略、被测行为 owner、相关测试和覆盖说明；项目规则优先于本 skill 的默认判断。
2. 每次触发后读取 [ledger-contract.md](references/ledger-contract.md)，再按 `.test-evidence.json` 或默认路径定位账本。账本尚不存在时进入初始化流程。
3. 用户只要求审查、巡检或评估时保持只读；用户同时要求修改时，才改动测试、行为 owner、账本或配置。
4. 默认范围是本次 diff 涉及的测试、对应行为、命中的 review `Scope:` 和相邻证据。只有用户要求、项目约定或明确风险支持时才扩大为全量巡检。
5. 测试框架负责执行测试，项目测试策略拥有测试层级和覆盖要求，本 skill 负责验证义务判断；CLI 只检查结构、角色和映射，不能代替语义审查或人工 CR。

## 验证义务分类

每个进入账本的 case 只使用一种验证方式：

1. **automated**：存在稳定、可观察且成本相称的机械证明。active case 有唯一主入口，也可以有归属于主 case 的衍生测试源码。
2. **review**：风险真实且需要持续关注，但当前架构、环境或故障注入成本使机械测试不成比例。登记作用范围、风险、无法自动化的原因和人工检查动作，不创建虚假测试入口。
3. **exempt**：测试发现器因框架、语法、fixture 或生成材料把文件识别为测试，但该文件并不是项目测试入口，发现结果属于误报。豁免必须登记到账本并从源码引用。

发现稳定目标但自动化测试尚未实现时，使用 planned automated case。已经存在的人工审查义务和发现豁免直接保持 active。

待确认和不新增的候选不创建账本 case。真实但低价值的测试使用不新增、合并或删除处置；exempt 只承接测试发现结果本身需要豁免的源码文件。

## 自动化证据准入标准

先把候选测试写成一句证明陈述：

```text
owner 的稳定承诺 -> 独立条件或边界 -> 被测行为 -> 调用方可观察结果
```

保留或新增自动化测试前，同时确认：

1. **规则来源**：能指出产品规范、接口契约、schema、安全边界、错误语义或明确工程不变量。
2. **独立性**：输入进入新的等价类、责任边界、信任边界、失败出口或观察入口。
3. **可观察性**：断言验证调用方可观察结果，或当前白盒层明确拥有的内部不变量。
4. **失败价值**：测试失败能指出哪项稳定行为不再成立，而不只是提示实现被改写。
5. **成本比例**：fixture、mock、环境和后续维护成本与证据价值相称。

TDD、防回归、覆盖率和历史 bug 是开始调查的动机，不是独立证明目标。以下情况通常不形成新证据：

1. 只替换字面量、枚举值或同类非法输入，仍进入同一 owner 分支并断言同一结果。
2. 断言复制实现步骤、私有数据布局、内部调用顺序或 mock 的预设返回值。
3. 只证明语言、框架或第三方库已经保证的普通行为。
4. 只执行代码路径，没有稳定且有意义的结果断言。
5. 测试名称保留历史回归故事，但当前无法还原为仍有效的契约。

## 准入流程

### 1. 建立候选与基线

1. 区分当前任务是新增、修改、审查、精简、不可测试性评估、语义巡检还是机械对账。
2. 列出本次涉及的候选，定位行为 owner、现有测试、实际验证入口、人工审查风险和账本 case；对实现变更同时检查是否命中已有 review `Scope:`。账本尚不存在时建立本次范围的拟登记清单。
3. 记录无法读取或运行的测试框架、环境和外部依赖，不把环境缺失解释为测试价值结论或 review 的充分理由。

### 2. 搜索已有义务

1. 按责任域、行为、输入类别、错误、输出和风险范围搜索账本、现有测试与行为 owner。
2. 比较 owner、证明目标、风险、入口、fixture 和叶子断言，不只比较测试名称。
3. 把多个写法、枚举字面量和同类非法值视为同一候选等价类，除非 owner 定义了不同处理分支。

### 3. 作出处置

每个候选只选择一个主要处置，并写明依据：

1. **待确认**：找不到稳定 owner、可观察结果或可描述风险；报告缺少的契约。
2. **不新增**：没有独立证据或风险，或维护成本明显高于价值。
3. **复用或合并**：已有 case 已经承接同一目标；复用、合并或归入现有主 case。
4. **扩展**：出现 owner 已承诺的新分支，但仍属于同一稳定目标。
5. **新增自动化**：出现新的 owner、责任边界、观察入口、结果类型或需要独立演进的等价类。
6. **登记人工审查**：存在稳定风险，但已明确说明当前机械测试为何不成比例，并能给出可执行的 CR 检查动作。
7. **登记发现豁免**：发现器把非项目测试入口的文件识别为测试，确认发现结果属于误报后登记。
8. **删除或转换**：删除失去价值的测试，或在 automated、review、exempt 之间按当前事实转换同一义务。

### 4. 按授权实施

1. 用户要求修改时，以能证明目标的最小输入集合和断言完成自动化处置。
2. 行为契约与测试不一致时，先确认哪个 owner 应变更，再在同一 patch 更新对应内容。
3. 使用 review 时登记实际作用范围、风险、成本原因和检查动作；使用 exempt 时登记误报原因和对应源码。
4. 用户只要求审查时不写文件，只提供可定位、可执行的处置意见。

### 5. 维护账本与映射

1. 账本不存在且任务允许修改时，按默认路径初始化账本并登记本次范围；只有默认发现规则不适用时才创建 `.test-evidence.json`。
2. 账本不存在且任务保持只读时，不写文件；给出拟 case、验证方式、源码角色或人工审查范围。
3. 让账本保存稳定验证义务，不为每个测试函数创建 case。自动化主入口使用 `@test-evidence main`，归属于已有主 case 的测试源码使用 `derived`，发现豁免使用 `exempt`。
4. review 和 planned case 不添加源码测试标记；任务触及 review 的 `Scope` 时执行并报告对应 `Review` 动作。
5. 首次接入以 `warn` 暴露存量未登记测试，但本次新增或修改的测试源码必须登记；基线可信后再切换为 `error`。
6. 从本 skill 目录运行：

```bash
node scripts/test-evidence.mjs check --root <workspace-root>
```

需要机器报告时添加 `--json`。CLI 失败时修正结构或映射；CLI 通过后仍保留语义结论和人工检查结果。

### 6. 验证与交付

1. 运行能覆盖处置结果的最小测试和机械对账。
2. 执行本次作用范围命中的人工审查动作，并单独报告结论。
3. 按项目要求补充类型检查、lint、构建或更大范围测试。
4. 区分行为失败、机械映射失败、人工审查风险和环境未执行。

## 周期巡检

用户要求周期或存量巡检时，优先抽样：

1. derived 源码、fixture、入口或分支持续增长的自动化 case。
2. review 的作用范围、成本原因或检查动作已经无法对应当前架构的 case。
3. exempt 原因可能已被发现器、框架或目录结构变化消除的 case。
4. 经常因重构修改、但产品行为没有变化的测试。
5. 大量依赖 mock、snapshot、生成材料或复杂环境的测试。
6. 一个 case 同时承载多个 owner、可观察结果或无关风险的区域。

巡检结论必须落到既有处置之一，并同步账本类型、源码角色和实际实现。

## 交付格式

先给总体判断。每个需要动作的候选说明：

1. 测试位置、风险范围或 case。
2. 处置和验证方式。
3. 证明陈述、风险说明，或与现有义务的重叠点。
4. 所需的测试、owner、账本、源码标记或人工检查改动。

最后列出实际运行的测试、机械检查、人工审查结论、未执行环境和残余风险。

## 完成检查

1. 每个 automated case 是否能指出 owner、独立条件和可观察结果。
2. 每个 review case 是否包含稳定风险、明确范围、不成比例的自动化成本和可执行检查动作。
3. 每个 exempt case 是否对应真实发现误报，并由源码标记指向账本原因。
4. 是否搜索过已有义务，并处理证明目标相同的重复测试。
5. main、derived、exempt、review、planned 和发现结果是否一致。
6. 是否报告实际测试、机械校验、人工审查和仍待确认的行为契约。

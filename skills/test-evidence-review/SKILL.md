# Test Evidence Review

## 目标与边界

把测试与相关风险作为候选验证义务。先判断候选保护的稳定承诺或风险，再决定是否值得长期维护以及如何承接；账本条目、测试通过或 CLI 通过都不能代替价值判断。

每次调用：

1. 读取项目指令、测试策略、行为 owner、相关测试和覆盖说明，再读取 [ledger-contract.md](references/ledger-contract.md) 并定位配置与账本；项目规则优先。
2. 先执行语义准入，再同步账本和源码映射。默认范围是本次测试与实现 diff、命中的 review `Scope:` 和相邻证据；只有用户要求、项目约定或明确风险支持时才扩大。
3. 审查、巡检或评估任务保持只读并给出拟调整项；修改任务才更新测试、行为 owner、账本或配置。
4. 让测试框架负责执行测试，让项目测试策略拥有测试层级和覆盖要求；CLI 只检查账本结构、源码角色和映射。

## 自动化证据准入

先定位候选测试的行为 owner 和共享测试基座，再写证明点：

```text
case 语境：owner 明确语义 + 共享初始状态、fixture、执行上下文或行为链路
Proves 项：分支条件 -> 可观察结果
```

`Proves:` 每项只写一个简短、直接的叶子结果，背景由 case 标题、行为 owner、`Code:` 和 Mermaid 承接。多个分支共享测试基座或连续行为链路，且拆分会增加 fixture、执行、同步或审计成本时，保留在同一 case；具体分组与拆分规则以账本契约为准。

保留或新增自动化测试前，同时确认：

1. **规则来源**：能指向产品规范、接口契约、schema、安全边界、错误语义或明确工程不变量。
2. **分支价值**：输入进入 owner 明确定义的分支、等价类、责任边界、信任边界、失败出口或观察入口。
3. **可观察性**：断言验证调用方可观察结果，或当前白盒层拥有的内部不变量。
4. **失败价值**：失败能指出哪项稳定行为不再成立，而不只是提示实现被改写。
5. **成本比例**：fixture、mock、环境和后续维护成本与证据价值相称。

TDD、防回归、覆盖率和历史 bug 只提供调查动机。以下候选通常不形成新证据：

1. 只更换字面量、枚举值或同类非法输入，仍进入同一 owner 分支并断言同一结果。
2. 只复述实现步骤、私有布局、内部调用顺序、mock 预设或语言与框架的普通保证。
3. 只执行路径而没有稳定结果断言，或历史测试名称已无法还原为当前契约。

## 主流程

1. **建立范围**：列出测试、稳定风险、发现误报和受机械诊断影响的 case。实现变更先匹配 review `Scope:`；记录无法运行的环境，不把环境缺失作为价值结论或 review 的充分理由。
2. **搜索既有义务**：按 owner、行为、输入类别、错误、输出和风险搜索账本与测试；比较共享基座、行为链路、入口、fixture 和叶子断言，不只比较名称。
3. **作出处置**：每个候选只选择一个主要结果并写明依据。
   - **待确认**：缺少稳定 owner、可观察结果或可描述风险。
   - **不新增或删除**：没有独立证据或风险，或成本明显高于价值。
   - **复用或合并**：既有 case 已承接同一目标。
   - **扩展**：出现 owner 已承诺的新分支，且与既有 case 共享初始状态、fixture、执行上下文或连续行为链路；补充简短 `Proves:` 项和多分支 Mermaid。
   - **自动化证明**：出现独立 owner、边界、观察入口或行为链路，或者共享基座已经稳定集中且拆分确实降低总维护与审计成本；已有测试使用 active automated，目标明确但尚未实现时使用 planned automated。
   - **人工审查**：存在稳定风险，自动化成本不成比例，且能给出可执行 CR 动作；使用 active review。
   - **发现豁免**：测试发现器把非测试入口识别为测试；确认误报后使用 active exempt，不用它承接低价值测试或难测风险。
4. **按授权实施**：修改任务使用最小输入和断言完成处置。行为契约与测试不一致时先确认应变更的 owner；只读任务只报告可定位、可执行的建议。
5. **同步追踪**：
   - 账本不存在时，修改任务按默认路径初始化，只读任务给出拟 case；只有默认发现规则不适用时才创建配置。
   - 账本按稳定义务和共享测试基座组织，不枚举测试函数。active automated 使用一个 main 定位主入口，其他归属源码使用 derived，发现误报使用 exempt；review 和 planned 不添加源码标记。
   - 首次接入以 `warn` 暴露存量未登记测试，本次新增或修改的测试源码必须登记；基线可信后再使用 `error`。
6. **验证结果**：运行最小测试、命中的人工审查动作和机械对账；按项目要求补充类型检查、lint、构建或更大范围测试。

从本 skill 目录运行机械对账：

```bash
node scripts/test-evidence.mjs check --root <workspace-root>
```

需要机器报告时添加 `--json`。CLI 失败时修正结构或映射；CLI 通过后仍保留语义结论。验证方式变化时保留稳定 case ID，并按当前事实同步字段与源码映射；待确认、不新增和已删除且不再承载义务的候选不创建 case。

## 周期巡检

优先抽样：

1. derived 源码、fixture、mock、snapshot 或分支持续增长，且共享基座、`Proves:` 与 Mermaid 可能已经漂移的 automated case。
2. `Scope:`、成本原因或检查动作已无法对应当前架构的 review case。
3. 豁免原因可能已被发现器、框架或目录变化消除的 exempt case。
4. 经常随重构变化但行为不变，或同时承载多个 owner、结果或无关风险的测试。

让巡检结论落到主流程中的既有处置，并同步账本、源码角色和实现。

## 交付与完成

1. 先给总体判断；对每个需要动作的候选说明位置或 case、处置、证明或风险依据，以及所需改动。
2. 确认 automated case 能指出 owner、共享测试基座或行为链路，并让每个 `Proves:` 项对应明确的分支条件和可观察叶子结果。
3. 确认 review case 包含稳定风险、明确范围、成本原因和可执行检查动作。
4. 确认 exempt case 对应真实发现误报，并由源码标记指向账本原因。
5. 报告实际测试、机械校验、人工审查、未执行环境和残余风险，并区分行为失败、映射失败与人工风险。

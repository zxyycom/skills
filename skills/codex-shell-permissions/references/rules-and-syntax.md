# Shell Rules And Syntax

本文件是 shell 失败时的规则速查。只保留 agent 需要执行的判断：什么写法容易被 runner 或权限规则卡住，什么情况要提权，什么时候拆成简单命令，什么时候可以提权执行复杂命令。

## 复杂命令的处理选择

这些写法容易被外层 runner 分段处理，权限规则对每段分别生效：

1. `cmd1 | cmd2`
2. `cmd1 && cmd2`
3. `cmd1 || cmd2`
4. `cmd1; cmd2`
5. `(cmd1)`
6. `$(cmd1)`
7. `cmd > file`、`cmd >> file`、`cmd < file`

按场景选择处理方式：

1. 排查失败原因、命令意图不清、每段风险不同、或涉及未确认的删除/移动/覆盖时，拆成多次简单 `shell_command`。
2. 命令意图明确、目标范围清楚、一次执行更可靠或更省步骤，且失败来自 runner、sandbox 或权限匹配时，可以按当前环境规则用 `require_escalated` 执行整条复杂命令。
3. 提权执行复杂命令时，说明整条命令为什么是完成当前任务所必需；不要在普通失败处理中配置持久权限规则。

## PowerShell 中慎用的写法

这些写法在真实 PowerShell 中可能合法，但不适合作为排查权限问题时的首选：

1. 脚本块：`{ ... }`
2. 数组字面量：`@(...)`
3. 哈希表字面量：`@{...}`
4. 子表达式：`$(...)`
5. 复杂引号嵌套。
6. 逗号参数列表。
7. 运行前临时设置多个变量再执行命令。

处理方式：

1. 能拆成普通命令参数就拆。
2. 复杂逻辑本身更清楚、且目标范围和风险已经说明时，可以提权执行原命令。
3. 需要长期复用或逻辑较长时，优先修改文件或脚本后运行脚本入口。
4. 需要确认 PowerShell 语法本身时，再用 `codex sandbox powershell -NoProfile -Command "<command>"` 做对照。

## 不要用来绕权限的写法

遇到审批或 sandbox 失败时，不要改成这些形式来绕开：

1. `cmd /c <destructive command>`
2. `powershell -Command "<destructive command>"`
3. 先 `Get-ChildItem` 枚举路径，再管道传给删除或移动。
4. 用通配符删除或移动：`Remove-Item *`、`rm -r *`。
5. 通过重定向写长期源文件。
6. 用临时下载脚本绕过网络限制。

正确做法是申请审批、缩小命令范围，或用专门工具编辑文件。

提权执行复杂命令是正常路径，不是绕权限；前提是命令目的、影响范围和必要性要说清楚。

## 适合直接运行的命令

通常可以直接尝试：

1. 已允许的只读命令，例如 `git status --short`、`git diff`、`rg`、`Get-Content`、`Get-ChildItem`。
2. 不写文件、不联网、不启动服务、不修改配置的检查命令。
3. 用户当前任务所需、且已在环境批准前缀内的命令。

如果这些命令仍出现 `CreateProcessAsUserW failed: 5`，不要继续改语法；按提权规则处理。

## 必须提权或确认的命令

按当前环境规则申请提权：

1. 网络：下载、安装、registry、DNS、host resolution、API 请求。
2. 写工作区外路径：用户全局配置、工具缓存、系统目录。
3. 启动服务或 GUI：dev server 可见窗口、浏览器、系统应用。
4. 登录、认证、修改配置。
5. 破坏性操作：删除、递归删除、移动、覆盖、`git reset`。
6. 已知重要命令被 sandbox 创建进程失败拦住。
7. 复杂 shell 组合被 runner 或权限匹配拦住，但整条命令比拆成多步更适合完成当前任务。

破坏性命令提权前要确认目标路径；不要为破坏性命令配置持久权限规则。

## 转入 rules 维护的边界

如果用户明确要求添加、修改或审查 Codex 权限 rules，再转入 [permission-rules-maintenance.md](permission-rules-maintenance.md)。

普通 shell 失败仍按本文件处理：按场景选择拆命令、换成普通参数、申请必要提权、确认破坏性范围，或回到目标程序错误。不要把一次命令失败直接当成 rules 维护任务。

# Investigation Report

## 目标

围绕一个稳定核心问题维护主题文件。每轮调查都留下形成于特定时点、能够独立阅读的认识快照，使未来读者可以比较不同背景、目的和证据条件下的结果，不必补入默认前提或拼接原对话。

报告可以停在部分结果、暂定解释或明确未知，不以形成决定、方案或强结论为完成前提。

## 使用条件

1. 用户明确要求记录、沉淀、创建、更新或审阅调查报告时使用。
2. 普通调查、排障和问答没有沉淀要求时，不主动建议或创建报告。

## 内容 owner

1. 本文件承接报告导向的调查流程、场景判断、内容质量以及何时追加、拆分或原地修正。
2. [固定契约](references/investigation-report-contract.md) 唯一承接目录、派生索引、精确标题与顺序、时间、状态和 CLI 语义；创建、更新或结构审阅前完整读取。
3. `scripts/check-investigations.mjs` 通过 `list` 查询已经核对新鲜度的主题级 JSON 索引，默认命令只读检查固定结构、时间和索引；只有显式 `sync-index` 命令写入可删除重建的派生索引，不写主题文件。
4. [索引 Schema](references/investigation-index.schema.json) 提供机器可读的当前 JSON 结构；主题 Markdown 继续是标题、调查信息和报告正文的唯一事实源。
5. 代码、规范、配置、决策和测试材料继续承接当前事实与长期选择；调查报告保存某一时点的调查输入、证据基础、认识和边界，不取代这些 owner。

## 工作流程

### 1. 确定主题与本轮调查

1. 先确定任务出口：只读审阅不写文件；形成新认识时创建或追加完整报告；只改变是否继续调查时只更新状态；旧内容没有准确记录形成时认识或违反固定格式时才原地修正。
2. 读取目标工作区指令，通过 `list` 从已核对新鲜度的主题索引发现可能相关的主题文件，再读取主题原文和当前任务需要的事实 owner，判断应追加、创建还是拆分。索引缺失或失效时不使用旧投影猜测成员；需要维护时从当前主题文件同步重建。
3. 一个主题文件只追踪一个能够持续调查的核心问题。相同问题的复查或新证据追加报告；能够独立调查和结束的问题另建文件。
4. 一次报告对应一份能够独立汇报的认识，不按命令、来源或每次试错拆分。
5. 把未来读者视为没有参与当前对话的人；本轮报告不能依赖“同上”、相邻报告或未写出的默认前提。

### 2. 固定形成时输入

1. 调查开始前能够取得信息时，先在工作记录或草稿中固定本轮背景与目的；整理已经完成的调查时，从当时材料和原对话恢复，不能用后来认识替换形成时输入。
2. `形成时背景` 保存当时发生了什么、基于哪些已知事实、假设、未知和约束意识到需要调查，以及影响结果解释的版本、环境、样本或工作区状态。
3. `调查目的` 写明本轮具体要回答的问题、准备支持的判断和预定边界。这里记录准备回答什么，后续的 `调查范围与依据` 记录实际检查到了哪里。
4. 形成时信息无法可靠恢复时，将缺口标为未知并说明现有依据；缺口会改变主题身份、调查目的或结果解释时，再向用户确认。

### 3. 调查并形成结果

1. 优先使用项目事实 owner、实际运行结果、可复现样本和一手来源；关键主张保留可定位证据和必要观测条件，易变外部事实、实验和日志同时记录各自的观测时间。
2. 用 `调查范围与依据` 记录实际检查的对象、来源、时点或版本、方法、实际覆盖范围，以及没有检查或不能覆盖的内容。依据只有一次直接观察或单一材料时如实写明，不虚构方法。
3. 结果依赖已提交 Git 快照时记录完整 commit hash；未提交改动影响结果时还要记录相关 diff、内容或工作区状态。与 Git 无关的对象不要求 hash。
4. 用 `调查结果与边界` 区分已确认事实、基于证据的推断、建议、实际动作和当前未知，并说明适用条件、不能外推的内容以及需要重新调查的条件。
5. 结论依赖计量或比较时，补足时间窗口、样本、指标、单位、纳入排除规则和误差；依赖因果或机制时，补足候选解释、直接证据、反证和未闭合环节。
6. 调查涉及候选方案时，区分推荐、已采用和已验证；涉及实际变更时，记录授权、影响范围、变更前状态、恢复来源、执行后验证和残余风险。
7. 调查默认只读。实验或其他副作用只在当前任务已经授权时执行；敏感材料只保留复核主张所需的最小信息。

### 4. 写入并演进主题文件

1. 按固定契约写入四项核心，并按本轮实际情况完整重述。即使内容与上一轮相同也重新写清；背景或目的不同但结果相同，也分别保留。
2. 先在四项核心中写清直接相关的信息；支撑材料需要独立组织才能复核结论时，再在其后增加时间线、证据、对照、状态模型、恢复或验收等章节。报告按形成的认识组织，原始搜索和命令过程只提炼为必要证据。
3. 本轮形成新的可独立汇报认识时，在原主题文件末尾追加完整报告。最新报告只是最近一次形成的认识记录，不自动汇总旧报告或充当主题的累积当前口径；旧报告继续保留形成时快照。
4. 新事实改变旧报告当时真实采用的认识时，通过追加表达演进；旧报告没有准确记录当时认识的误写、漏写、格式或链接问题可以原地修正。
5. 只改变是否继续调查时可以只更新状态。子问题的核心问题、当前 owner、授权边界、验证周期或复核条件能够独立变化时，拆成新的主题文件。

### 5. 审阅与检查

1. 只读审阅不修改报告。逐份检查是否能够恢复形成时背景、目的、调查依据以及结果边界，并确认时间顺序足以比较认识演进；需要当前口径时显式综合相关报告与当前事实 owner。
2. 按结论实际依赖检查场景内容是否完整，并只保留有助于判断和复核的章节。
3. 创建、更新、删除或移动主题文件后，先同步派生索引，再从本 skill 目录运行默认全量检查；也可以使用 CLI 的实际安装路径：

   ```text
   node <investigation-report-skill>/scripts/check-investigations.mjs sync-index --root <workspace-root>
   node <investigation-report-skill>/scripts/check-investigations.mjs --root <workspace-root>
   node <investigation-report-skill>/scripts/check-investigations.mjs list --root <workspace-root>
   ```

4. `list` 可以按 category、主题路径、状态、主题标题/核心问题/报告标题文本和最新报告时间筛选，并默认按最新报告时间倒序返回主题 state；单个分类或主题路径可以用 `--category` 或 `--path` 收窄只读结构检查。筛选检查不证明全局索引新鲜度；默认全量检查只证明固定结构、时间、派生 state 与索引一致性；通过后仍按完成标准人工审阅内容。

## 完成标准

1. 任取一份报告，都能独立恢复形成时背景、调查目的、调查范围与依据以及调查结果与边界。
2. 背景没有隐藏默认前提，事实、假设与未知能够区分；必要基线足以解释结果差异。
3. 证据范围与方法足以判断主张强度，结论没有超过证据，推荐、执行和验证状态没有混淆。
4. 当前场景真正需要的计量、因果、状态、方案或变更信息已经补足，但没有扩张为固定模板或原始过程日志。
5. 每份报告都可以作为其形成时认识独立汇报，报告序列保留可比较的历史输入；需要当前口径时没有把“最新”等同于“累积完整”，报告也没有取代决定、实施计划或当前事实 owner。

## 交接

调查形成长期采用方向、实施任务或稳定验证义务时，分别交给决策、规划或测试 owner。没有下游决定或任务，也不影响调查报告成立。

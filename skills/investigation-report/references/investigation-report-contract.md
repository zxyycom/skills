# 调查报告固定契约

本文件是 `investigation-report` 的唯一固定存储契约，承接调查根目录、索引、主题文件、完整报告、状态、时间、维护结果的落盘方式，以及检查器的精确语义。固定结构只保留稳定身份和每份报告不可缺少的语义；调查方法、证据质量、正文判断、维护方式选择和人工审阅由 [SKILL.md](../SKILL.md) 承接。

## 内容 owner

1. 每个主题文件的 `调查信息` 承接稳定核心问题、调查状态和最新报告时间；不重复维护当前认识摘要。
2. `调查报告` 中的每个三级标题承接一份形成于特定时点的完整独立报告。其固定语义核心只有背景、起因和调查结果；范围、基线、方法、证据、推断、结论、未知和下一步按需要补充。
3. 同一主题文件中最新形成的完整报告是当前认识的报告正文 owner；较早报告保留各自形成时的完整快照。
4. `<investigation-root>/investigation-index.md` 只投影主题文件的标题、核心问题、状态和最新报告时间，用于发现，不承接调查结果或当前事实。
5. `scripts/check-investigations.mjs` 只读检查路径、固定 Markdown 结构、字段、核心章节、形成时间和索引对应关系，不写入或重排文件。
6. 代码、配置、规范、指南、决策和测试材料继续承接各自当前事实与行为；报告通过概括和链接引用这些 owner，不复制其长期契约。

## 调查根目录与稳定身份

默认调查根目录是工作区内的 `docs/investigations`：

```text
<investigation-root>/
├── investigation-index.md
└── <topic-id>/
    └── <semantic-slug>.md
```

1. `topic-id` 和 `semantic-slug` 使用英文小写、数字与连字符组成的 kebab-case 名词或名词短语。
2. 一个相对路径只承接一个稳定核心问题链。文件名表达调查主题，不使用日期前缀、阶段、版本或工单号作为身份。
3. 主题文件创建后，相对调查根目录的路径保持稳定；标题澄清、追加报告、状态变化和重新开启不改变路径。
4. 同时涉及多个主题时选择主要主题。默认不为单个主题文件再建立目录；独立材料确实需要共同分发时由目标项目另行约定。
5. 调查根目录可以通过检查器的 `--investigations-dir <workspace-relative-path>` 改为工作区内其他位置；同一集合始终使用同一根目录。
6. 调查根目录在首次创建主题文件时建立并同时创建索引；除索引外的 Markdown 文件都必须是主题目录中的调查主题文件。

## 索引结构

索引首个非空行固定为：

```text
# 调查索引
```

每个主题使用二级标题，每个主题文件使用一个顶层列表项和三个嵌套字段：

```markdown
# 调查索引

## codex

- [项目 Shell MCP 注册调查](codex/project-shell-mcp-registration.md)
  - 核心问题: 为什么项目 Shell MCP 没有注册到可用工具列表？
  - 状态: 调查中
  - 最新报告时间: 2026-07-21T14:05:30+08:00
```

索引规则：

1. 主题标题逐字等于主题文件路径第一段；索引链接使用相对调查根目录的 POSIX 路径。
2. 链接文字逐字投影主题文件 H1 的纯文本；三个字段分别逐字投影 `调查信息` 中的 `核心问题`、`状态` 和 `最新报告时间`。
3. 每个主题文件有且只有一个索引条目，每个索引条目存在对应主题文件。
4. 一个条目只包含链接和三个非空字段；字段名称、顺序和合法值按示例固定。
5. 索引可以在 H1 与首个主题之间保留简短用途说明；主题内部不插入会打断条目结构的说明或额外层级。

## 主题文件结构

主题文件首个非空行是唯一一级标题，前两个二级标题依次固定为 `调查信息` 和 `调查报告`：

```markdown
# <调查主题>

## 调查信息
- 核心问题: <本文件持续追踪的一个问题链>
- 状态: 调查中 | 暂停 | 已结束
- 最新报告时间: 2026-07-21T14:05:30+08:00

## 调查报告

### <本次报告的语义标题>
- 形成时间: 2026-07-21T14:05:30+08:00

#### 背景
<理解本次结果所需的完整当时背景>

#### 起因
<为什么在此时进行并形成这次调查>

#### 调查结果
<本次实际查到什么、适用边界或明确未知>

#### 证据
<按需增加的支撑内容；本节只是示例，不是固定字段>
```

调查信息规则：

1. 三个字段按示例顺序各出现一次，每个值使用一个非空单行列表项；值内可以使用普通 Markdown 行内语法。
2. `核心问题` 定义主题文件身份。它可以包含共同服务于同一认识链的子问题，但不能只是宽泛主题名称。
3. `状态` 只表达是否继续调查，不表达问题是否解决、结果是否完整或是否已经形成决定。
4. `最新报告时间` 必须逐字等于最后一份完整报告的 `形成时间`。状态或纯编辑变化不单独改变该时间。
5. 调查信息不保存起因、背景、范围或当前认识。它们属于具体报告；当前口径由最新一份完整报告承担，避免顶部摘要成为第二份可能漂移的事实来源。

完整报告规则：

1. `调查报告` 只出现一次且至少包含一个三级标题。三级标题概括本次形成的认识、调查对象或复查目的，不使用单纯的“第一次”“继续调查”等无语义阶段名。
2. 每份报告的首个非空内容固定为 `- 形成时间: <timestamp>`。形成时间表示背景、调查结果和必要推断被整理成这份完整报告的时刻，不代替证据自身的观测或访问时间。
3. 形成时间之后，前 3 个四级标题依次固定为 `背景`、`起因`、`调查结果`。三者各出现一次且包含实际内容。
4. `背景` 写理解本次结果所需的当时上下文，例如项目处境和阶段、对象责任、现有状态、关键约束及显著影响结果的条件。它不能只写“同上”、差异摘要或一个不加概括的链接。
5. `起因` 写本次调查为何发生，不把尚未证明的根因写成动机。重新了解现状时可以简单写“距上次调查已久，需要了解当前状态”。
6. `调查结果` 先写本次实际查到的事实性结果及适用边界；暂时不能判断时，写明不能判断什么和缺少什么证据。推断或建议不能替代调查结果。
7. 三个必需章节之后可以增加任意有语义名称的四级章节，承接范围与方法、对象基线、证据、推断、结论、冲突、未知或下一步。可选章节不能插入三个必需章节之间。
8. 每份报告必须能够脱离主题文件中的其他报告独立阅读。可以比较旧报告，但不得依赖旧报告补足当前背景、起因或结果。
9. 报告是阶段性认识，不是原始命令日志。多条动作共同回答一个局部问题时合并；新背景、证据或解释形成可独立汇报的认识时再追加下一份完整报告。
10. `调查报告` 之后可以按需增加附录、术语或其他二级章节；这些补充不替代每份报告承担的完整语义。

## 背景与对象基线

1. 背景是固定语义，具体基线不是固定字段。只有版本、环境、样本、时间范围、配置或工作区状态显著影响调查结果时，才在背景或可选章节中记录相应基线。
2. 调查结果依赖已提交 Git 快照时，使用完整 commit hash 作为可复核定位，并说明它定位的仓库或组件；hash 不自动代表依赖、运行环境或外部状态。
3. 未提交改动影响调查时，commit hash 不能单独代表调查对象。报告同时记录相关 diff、文件内容、工作区状态或其他可复核定位，并明确结果基于未提交状态。
4. 非 Git 对象、无法由 commit 定位的状态，或与提交快照无关的结果不要求 hash。不得为了填充固定格式而虚构或滥用版本基线。
5. 易变外部事实、实验、日志和运行观测在对应证据附近保留各自时间与条件，不能只用报告形成时间代替。

## 时间、状态与演进

1. `最新报告时间` 和每份报告的 `形成时间` 都使用带显式时区且不含小数秒的 RFC 3339 时间戳，例如 `2026-07-21T14:05:30+08:00` 或 `2026-07-21T06:05:30Z`。
2. 完整报告按形成时间非递减顺序追加；最后一份报告的形成时间与 `最新报告时间` 使用完全相同的文本。
3. `调查中` 表示当前仍在推进或仍值得继续；`暂停` 表示仍有价值但当前缺少条件或暂不推进；`已结束` 表示当前不再计划继续。状态都不承诺问题已经解决或获得完整答案。
4. 行为流程选择追加时，新报告写在原主题文件的 `调查报告` 容器末尾，并同步调查信息和索引；最新报告成为当前口径，旧报告保持不变。
5. 行为流程选择原地修正时，不新增报告，也不改变 `最新报告时间`；需要同步投影的字段同时更新索引。
6. 行为流程选择只更新状态时，同时更新主题文件和索引中的状态，不新增报告或改变 `最新报告时间`。
7. 行为流程选择拆分时，在同一主题目录创建兄弟主题文件；文件间联系使用普通相对链接表达，索引不维护正式关系图。
8. 调查主题文件不进入物理归档目录；状态、完整报告序列和 Git 历史共同表达调查演进。

## 检查器

从 skill 目录运行，或使用脚本绝对路径：

```text
node scripts/check-investigations.mjs --root <workspace-root>
```

默认检查所选调查根目录中的完整索引和全部主题文件。可选参数：

```text
--investigations-dir <path>  使用工作区内其他调查根目录
--topic <topic-id>           只检查一个主题，可重复
--report <relative-path>     只检查一个主题文件，可重复
```

`--report` 名称为兼容现有命令行保留，其值始终是相对调查根目录的主题文件路径，不选择文件中的单份完整报告。

筛选规则：

1. 同一参数重复时取并集；同时使用 `--topic` 和 `--report` 时取交集。
2. `--report` 相对调查根目录解析，命令行中的反斜杠会归一化为 POSIX 分隔符。
3. 筛选检查只校验选中主题文件及对应索引条目；无关文件和其他主题的结构问题不使本次筛选失败。
4. 调查根目录或索引无法读取仍然阻断筛选检查；筛选没有命中任何文件或索引目标时失败，避免拼写错误静默通过。

检查器机械验证：

1. 调查根目录、索引文件、主题目录、路径层级和 kebab-case 语义文件名；除索引外的每个 Markdown 文件都按主题文件检查，文件名是否实际表达日期、阶段或工单语义由人工判断。
2. 索引 H1、主题、条目结构、三个字段、重复项和目标文件。
3. 主题文件唯一 H1、前两个固定 H2、三个调查信息字段、状态，以及至少一份三级标题报告。
4. 每份报告的形成时间，`背景`、`起因`、`调查结果` 三个非空四级章节及其顺序，报告形成时间的非递减顺序，以及最新报告时间与最后一份报告形成时间的一致性。
5. 主题文件与索引之间的标题、核心问题、状态、最新报告时间和路径一一对应。

检查器不判断背景是否足以解释项目处境和阶段、报告能否真正独立阅读、调查结果是否充分、来源是否可靠、推断是否成立、状态是否恰当、语义文件名是否合适或是否应该拆分；这些语义由 `SKILL.md` 的形成与审阅流程承接。

退出状态：`0` 表示所选范围结构通过，`1` 表示结构或对应关系失败，`2` 表示命令参数错误。

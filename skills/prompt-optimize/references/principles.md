# 提示词设计原则

本文件解释“为什么这样改”。具体执行路径由入口文件承接。每节按“原理或思考 -> 行为”组织。

## 解空间锚定原理 -> 先建立目标框架, 再保留必要边界

原理: LLM 即使准确理解“不使用 X”, X 及其常见用法、修正路径和反例仍已进入当前上下文。生成替代方案时, 模型可能继续围绕 X 的正确做法或反面搜索, 把解空间限制在由 X 建立的局部框架; 更优方案有时需要从目标、约束和结果重新建立解题路径, 而不是继续解释 X。

行为:

1. 先从目标状态、必要约束和预期结果建立正向框架, 再处理“不要 X”。
2. 判断 X 是迁移残留、低频极端方案、默认倾向冲突还是硬性边界。
3. 当前推荐路径明确时, 写“使用 Y 完成目标”, 不围绕 X 继续展开替代方案。
4. X 是默认容易回落的方案、明确契约或重要边界时, 保留一次必要边界并补正向做法。
5. 推荐路径、接口契约或合法值集合已经自然排除 X 时, 不再把 X 写成额外特例。

## 文档行为引导原理 -> 先定任务出口

原理: 文档不只传递信息, 还会引导读者或 agent 的行为。优化文档时, 要检查当前文本会让使用者做什么, 以及这种行为是否符合文档真实用途。

行为:

1. 先确定用途、使用者、目标状态和风险等级。
2. 使用者判断服务于内容取舍、结构和表达判断, 不默认写进正文。
3. 再按任务出口选择执行、流程、规则、说明、参考、契约、检查、评估或协作骨架。

## 文档主承诺原则 -> 正文重心要兑现主承诺

原理: 标题、副标题、序言、开头段、一级标题、触发条件、前置条件、资源名、字段集合、适用范围和输入输出描述会先于细节建立读者或 agent 的预期。后续篇幅、层级、结论、验收或校验方式如果集中到另一个主题, 文档实际用途会偏离表面承诺。owner 判断只能解决内容完整解释的位置, 不能单独判断整篇文档的重心是否偏移。

行为:

1. 从标题、副标题、序言、开头段、一级标题、触发条件、前置条件、资源名、字段集合、适用范围或输入输出描述推断文档主承诺。
2. 对照正文反复解释、篇幅最多、层级最高、结论落点、验收标准和校验方式, 检查实际重心。
3. 正文偏离主承诺时, 移动、降级或删减偏离内容。
4. 标题、开头、适用范围或契约入口过宽时, 收窄入口承诺。
5. 偏离内容有独立维护价值时, 移到对应 owner; 当前文档只留摘要、触发条件或引用。

## 内容所有权原则 -> 一个判断只在一个 owner 位置完整解释

原理: 同一规则在多个位置完整展开, 会带来重复、漂移和局部修补。

行为:

1. 入口文件写触发条件、主动引用策略、主执行流程、冲突处理和完成检查。
2. 原则文件解释为什么; agent 任务文件写协作边界和输出结构。
3. 非 owner 位置只保留摘要、触发条件或引用。

## 用途决定阅读脉络 -> 只在理解型文本里强化自然展开

原理: 阅读脉络不是通用审美要求。说明型文本需要帮助读者理解判断如何展开; 流程型、契约型和机器消费型文本更依赖稳定结构。

行为:

1. 说明型、介绍型、原理型和设计理由型文本, 检查问题引入、概念递进、转折和结论。
2. 教程、上手指南、迁移指南和决策说明, 同时保留必要上下文、操作路径和验收标准。
3. SOP、runbook、API 文档、字段说明、规则索引和机器消费格式, 优先保证可执行、可检索、可对照、可验证。

## 显式关系原理 -> 使用稳定默认格式和受控图示

原理: 表格列对齐、目录树缩进和普通思维导图的空间关系对模型只是线性 token, 关系需要在内容中显式存在。Mermaid flowchart 用代码声明节点、边、方向和分组, 能在保留逻辑关系的同时改善层级、分支和跨节点关系的可读性。

行为:

1. 目标已有明确格式契约时遵循契约; 没有契约时使用稳定默认格式。
2. 步骤、规则和优先级优先使用有序列表; 复杂嵌套、结构化输入输出和 agent 返回结果优先使用 JSON。
3. 简单对照默认使用不超过 3 列、每行可独立理解的表格。
4. 层级、分支或跨节点关系在图示能显著改善可读性且环境支持 Mermaid 时, 使用 `flowchart LR` 或 `flowchart TD`, 并用 `subgraph` 分组。
5. 图中用节点、边标签和方向显式表达关系; 不适合图示时改成扁平路径、字段对象或 nodes/edges。

## 可验证性原则 -> 把形容词改成检查点

原理: “更好”“高级”“合理”“适当”会把判断留给使用者临场猜测, 让输出难以复现和验收。

行为:

1. “更好”改成目标状态和验收标准。
2. “高级”改成受众、复杂度、示例深度和输出格式。
3. “合理”改成决策优先级; “适当”改成触发条件、范围和上限。

## 长期沉淀原则 -> 错误样本先当诊断材料

原理: 坏示例、旧版描述、迁移残留、临时提醒和单次问题通常说明当前规则哪里没覆盖, 但它们本身不是长期规则。

行为:

1. 能提炼成当前目标、推荐做法、边界或验收标准时, 改写后保留。
2. 只是已有规则的重复例子时, 合并到已有规则或删除。
3. 只描述一次性错误、历史过程或旧版本做法时, 不写入长期文档。

## 风险分级原则 -> 冲突先按后果处理

原理: 用户目标、项目约定、工具默认路径和安全边界可能互相冲突。不同冲突的后果不同, 不能用同一种策略处理。

行为:

1. 涉及安全、权限、合规、事实真实性、不可逆操作、更高优先级指令或明确契约时, 先说明冲突并确认可行路径。
2. 项目约定、依赖或相邻模块做法冲突但风险可控时, 可以按用户目标处理, 并在交付中汇报取舍。
3. 无法判断冲突风险时, 问一个具体问题确认。

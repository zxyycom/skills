# 提示词设计原则

## 核心原理

LLM 按 token 序列处理 prompt。负向描述如果包含具体技术、工具、库、框架、API 或实现方案, 相关 token 仍会进入上下文, 在某些任务里可能提高模型复用该路径的概率。

因此, 负向描述不要机械保留或机械删除。先判断它是迁移残留、低频极端方案、高频误用还是硬性边界; 能写成推荐路径时优先写推荐路径, 必要边界保留并配套正向做法。

用户提供的坏示例、旧版描述、迁移残留、临时提醒和单次问题默认是诊断材料, 不是长期规则。面向执行的文档应沉淀当前可复用判断, 不沉淀错误清单。

对人类读者而言, 正向目标、显式结构和可验证标准能降低理解成本, 减少解释空间, 提高执行一致性。对长期维护而言, 文档应沉淀稳定判断, 不沉淀临时反馈和错误清单。

## 判断规则

1. 迁移残留或低频极端方案:
   - 例子: 从 Redux 迁到 Zustand 后, 文档仍写“不要用 Redux”; 在 Vue 项目里写“不要用 WinForms”。
   - 风险: 旧方案 token 或无关方案进入上下文, 增加噪音。
   - 做法: 当前推荐路径已明确时只写推荐路径; 无判断价值的提醒删除。

2. 高频误用或明确排除的方案:
   - 例子: “不要用 Redux”“别写原始 SQL”“不要用 class component”。
   - 风险: 这些方案可能是模型或团队容易回到的默认路径。
   - 做法: 能明确替代方案时写推荐路径; 该方案容易反复出现或属于明确契约时, 保留边界并补推荐路径。

3. 禁止抽象行为:
   - 例子: “不要猜测”“不要过度设计”“不要泛泛总结”。
   - 风险: 较低, 因为没有激活具体技术路径。
   - 做法: 可以保留否定式; 更稳妥时补充可验证标准, 例如“缺少来源时标注未知”。

4. 安全、权限和真实性边界:
   - 例子: “不要泄露密钥”“不要越权访问”“不要编造数据”。
   - 风险: 这些是必要边界。
   - 做法: 保留边界, 并写清正向路径, 例如“从环境变量读取密钥并对日志脱敏”。

## 正向改写模板

```text
原意: 排除某个具体方案
交付: 使用 <推荐方案> 完成 <目标>

原意: 文档迁移后残留旧方案提醒
交付: 删除旧提醒, 或只写 <当前推荐方案>

原意: 高频误用或硬性排除的具体方案
交付: 使用 <推荐方案> 完成 <目标>; 边界: <仍需排除的方案>

原意: 避开某类历史错误
交付: <目标状态>; 验收时检查 <可验证标准>

原意: 约束安全、权限或真实性边界
交付: 保留必要边界, 并写清 <安全路径>
```

## 格式原则

AI 不依赖视觉布局理解信息。表格列对齐、目录树缩进和思维导图空间关系对人类直观, 但对模型只是线性 token。结构必须在文本中显式存在。

优先级:

1. 有序列表: 适合步骤、规则、优先级, token 开销低。
2. JSON: 适合复杂嵌套、结构化输入输出和 agent 返回结果。
3. 简单表格: 只适合不超过 3 列、每行独立的内容。
4. 缩进列表: 只适合不超过 2 层的固定层级。

需要改写的结构:

```text
原结构: 多列 feature 对比表, 含合并单元格和跨列依赖
改写: 按功能点拆成有序列表; 每个条目内写清适用场景、限制和推荐做法

原结构: 深层目录树, 仅靠缩进表达父子关系
改写: 扁平化为“路径: 说明”列表, 每行完整写出从根到叶的路径

原结构: 依赖图示空间关系描述模块关系
改写: 使用 JSON 的 nodes/edges 或层级编号列表描述关系
```

## 最小检查清单

1. “不要 X”里的 X 是迁移残留或低频极端方案吗? 是则删除或改成当前推荐路径。
2. “不要 X”里的 X 是高频误用、明确契约或重要边界吗? 是则保留边界并补正向做法。
3. 指令是否包含主观形容词或模糊程度词? 是则改成检查点或完成条件。
4. 信息结构是否依赖复杂表格、深层缩进或图示布局? 是则改成有序列表、JSON 或扁平路径说明。
5. 最终文档是否沉淀了坏示例、旧版描述、迁移残留、临时提醒或单次错误? 是则提炼成当前可复用判断, 否则删除。

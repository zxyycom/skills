# 改写规则

## 负向描述处理

先判断负向描述的来源、频率和风险, 再决定删除、改成推荐路径或保留为边界。

1. 迁移残留: 文档从 A 迁到 B 后留下“不要 A”等旧提醒, 且当前推荐路径已明确时, 删除残留或改成“使用 B 完成目标”。
2. 低频极端方案: 目标读者通常不会选择, 且保留只会增加干扰时, 删除。
3. 默认倾向冲突: 用户目标、项目约束或目标文档要求偏离模型、团队或生态常规做法时, 保留必要边界, 并补推荐路径。
4. 硬性契约: 安全、权限、合规、事实真实性、不可逆操作或项目明确契约, 保留边界; 有正确做法时必须补正向做法。

```text
原意: 排除 Redux
改写: 使用 Zustand 管理客户端状态
```

```text
原意: 限制原始 SQL
改写: 通过项目现有 ORM 访问数据库; 只有 ORM 无法表达查询时, 才在参数化查询中使用 SQL
```

```text
原意: 不要用原始 HTML
改写: 主要使用 Vue 组件和项目现有组件库实现页面结构
```

```text
原意: 不要使用前端框架
改写: 主要使用 HTML、CSS 和 JavaScript 实现; 边界: 不使用 Vue、React、Angular 等前端框架
```

这个例子不是因为前端框架本身错误, 而是因为“使用框架”常是前端任务的默认倾向; 当用户目标是原生实现时, 需要把边界写清。

判断方法:

1. 如果 X 是迁移前旧方案且当前方案 Y 已明确, 写 Y 或删除旧提醒。
2. 如果 X 是低频极端方案, 且不是安全、权限、合规、事实真实性、不可逆操作或明确契约, 删除。
3. 如果 X 是默认倾向冲突或硬性边界, 保留边界并补推荐路径。
4. 如果没有可靠的 Y, 先根据用户目标、项目约定、依赖、相邻模块和行业常见方案推断。
5. 仍无法推断时, 明确缺口并问一个具体问题。

## 必须保留的边界

以下边界可以保留否定式; 有正确做法时必须补正向做法:

1. 事实真实性: 不编造数据、来源、实验结果。
2. 安全边界: 不泄露密钥、不执行危险代码。
3. 权限边界: 不越权访问、不绕过审批。
4. 不可逆操作: 不删除用户数据、不重写历史记录。
5. 合规要求: 不处理未授权隐私数据。

输出格式:

```text
边界: 密钥不进入代码或日志
做法: 从环境变量或密钥管理服务读取密钥, 并在日志输出前做脱敏
```

## 抽象行为约束

抽象行为可以保留否定式, 但如果不可验证, 要补具体标准。

```text
原意: 不要泛泛总结
改写: 先列出具体发现、文件位置或证据, 再给简短总结
```

```text
原意: 不要过度设计
改写: 只在重复逻辑、跨模块契约或长期维护成本明确存在时新增抽象
```

## 模糊表达

把主观形容词改成可检查标准。

```text
原意: 提高代码质量
改写: 提取重复逻辑, 保持函数单一职责, 并为新增分支添加测试
```

常见处理:

1. “更好”: 写成目标状态和验收标准。
2. “高级”: 写成受众、复杂度、示例深度和输出格式。
3. “合理”: 写成决策优先级。
4. “适当”: 写成触发条件和上限。

## 信息结构

优先使用这些格式:

1. 有序列表: 适合步骤、规则、优先级。
2. JSON: 适合复杂嵌套、结构化输入输出、agent 返回结果。
3. 简单表格: 仅在不超过 3 列、每行可独立理解时使用。
4. 缩进列表: 仅用于深度不超过 2 层的固定层级。

需要改写的结构:

```text
原结构: 用 6 列表格描述 API 参数, 且多列互相依赖
改写: JSON 数组; 每个参数一个对象, 字段名显式标注
```

```text
原结构: 4 层目录树仅靠缩进表达关系
改写: 扁平列表, 每行使用“路径: 说明”
```

## 沉淀判断

用户提供的坏示例、旧版描述、临时提醒和单次问题默认是诊断材料, 不是长期规则。

文档迁移时, AI 常把旧方案 A 写成“不要 A”等提醒。若新方案 B 已经在当前文档中成立, 这类提醒通常是迁移残留, 只有它代表默认倾向冲突、硬性契约或验收边界时才保留。

写入目标文档前先判断:

1. 能提炼成当前目标、推荐做法、边界或验收标准时, 改写后保留。
2. 只是已有规则的重复例子时, 合并到已有规则或删除。
3. 只描述一次性错误、历史过程或旧版本做法时, 不写入长期文档。
4. 只有引入新的判断标准、边界条件或输出要求时, 才新增规则。

最终文档应沉淀可复用判断, 不沉淀错误清单。

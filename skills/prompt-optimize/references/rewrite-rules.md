# 改写规则

## 用途错配处理

用途一致性审计发现当前信息、行为引导和实际用途不一致时, 按错配类型改写。实际使用者用于判断内容取舍和信息顺序, 不等于默认要在正文声明使用者范围。

1. 使用者差异影响执行:
   - 如果不说明使用者也不会影响理解或执行, 不把使用者声明写进正文。
   - 如果多类使用者会导致误用、权限或前置条件不同, 改写标题、开头、适用范围或分流说明。
   - 删除只服务于非实际使用者的解释, 或移动到参考资料。

2. 信息不足:
   - 补充使用者完成目标所需的输入、判断标准、步骤、边界或示例。
   - 把需要使用者自行推断的内容改成显式条件。

3. 信息过载:
   - 删除与实际用途无关的背景、历史过程和低价值细节。
   - 将背景信息降级为参考, 保留直接支撑判断或执行的内容。

4. 行为引导偏移:
   - 如果文档希望使用者执行操作, 改成步骤、输入、输出和验收标准。
   - 如果文档希望使用者做决策, 改成判断条件、取舍依据和推荐路径。
   - 如果文档只是长期参考, 避免写成任务指令或强行动作。

5. 用途和结构不匹配:
   - 执行型文档优先写目标、步骤、边界、验收。
   - 规则型文档优先写触发条件、规则、例外、优先级。
   - 参考型文档优先写概念、背景、示例和索引。
   - 评估型文档优先写检查项、通过标准和失败表达。

## 负向描述处理

先判断负向描述的来源、频率和风险, 再决定删除、改成推荐路径或保留为边界。

1. 迁移残留: 文档从 A 迁到 B 后留下“不要 A”等旧提醒, 且当前推荐路径已明确时, 删除残留或改成“使用 B 完成目标”。
2. 低频极端方案: 实际使用者通常不会选择, 且保留只会增加干扰时, 删除。
3. 默认倾向冲突: 用户目标、项目约束或目标文档要求偏离模型、团队或生态常规做法时, 保留必要边界, 并补推荐路径。
4. 硬性契约: 安全、权限、合规、事实真实性、不可逆操作或项目明确契约, 保留边界; 有正确做法时必须补正向做法。

```text
原意: 排除 Redux
改写: 使用 Zustand 管理客户端状态
```

```text
原意: 从旧枚举 A 迁到新枚举 B 后, 不要再接受 A
容易误写: 使用 B; 边界: 单独排除 A
改写: 使用 B 的合法值集合校验输入
```

这里 A 已经不是 B 的合法值, 正常的合法值校验会自然排除 A。继续点名 A 会暗示 A 需要特殊处理, 属于隐蔽防御式说法。

```text
原意: 限制原始 SQL
改写: 通过项目现有 ORM 访问数据库; 只有 ORM 无法表达查询时, 才在参数化查询中使用 SQL
```

```text
原意: 不要用原始 HTML
改写: 主要使用 Vue 组件和项目现有组件库实现页面结构
```

```text
原意: 不要使用前端框架
改写: 主要使用 HTML、CSS 和 JavaScript 实现; 边界: 不使用 Vue、React、Angular 等前端框架
```

这个例子不是因为前端框架本身错误, 而是因为“使用框架”常是前端任务的默认倾向; 当用户目标是原生实现时, 需要把边界写清。

判断方法:

1. 如果 X 是迁移前旧方案且当前方案 Y 已明确, 写 Y 或删除旧提醒。
2. 如果 X 是低频极端方案, 且不是安全、权限、合规、事实真实性、不可逆操作或明确契约, 删除。
3. 如果 X 是默认倾向冲突或硬性边界, 保留边界并补推荐路径。
4. 如果推荐路径、接口契约或合法值集合已经使 X 非法, 不把 X 写成额外特例; 写当前合法值、校验方式或验收标准。
5. 如果没有可靠的 Y, 先根据用户目标、项目约定、依赖、相邻模块和行业常见方案推断。
6. 仍无法推断时, 明确缺口并问一个具体问题。

## 必须保留的边界

以下边界可以保留否定式; 有正确做法时必须补正向做法:

1. 事实真实性: 不编造数据、来源、实验结果。
2. 安全边界: 不泄露密钥、不执行危险代码。
3. 权限边界: 不越权访问、不绕过审批。
4. 不可逆操作: 不删除用户数据、不重写历史记录。
5. 合规要求: 不处理未授权隐私数据。

输出格式:

```text
边界: 密钥不进入代码或日志
做法: 从环境变量或密钥管理服务读取密钥, 并在日志输出前做脱敏
```

## 抽象行为约束

抽象行为可以保留否定式, 但如果不可验证, 要补具体标准。

```text
原意: 不要泛泛总结
改写: 先列出具体发现、文件位置或证据, 再给简短总结
```

```text
原意: 不要过度设计
改写: 只在重复逻辑、跨模块契约或长期维护成本明确存在时新增抽象
```

## 模糊表达

把主观形容词改成可检查标准。

```text
原意: 提高代码质量
改写: 提取重复逻辑, 保持函数单一职责, 并为新增分支添加测试
```

常见处理:

1. “更好”: 写成目标状态和验收标准。
2. “高级”: 写成受众、复杂度、示例深度和输出格式。
3. “合理”: 写成决策优先级。
4. “适当”: 写成触发条件和上限。

## 信息结构

优先使用这些格式:

1. 有序列表: 适合步骤、规则、优先级。
2. JSON: 适合复杂嵌套、结构化输入输出、agent 返回结果。
3. 简单表格: 仅在不超过 3 列、每行可独立理解时使用。
4. 缩进列表: 仅用于深度不超过 2 层的固定层级。

需要改写的结构:

```text
原结构: 用 6 列表格描述 API 参数, 且多列互相依赖
改写: JSON 数组; 每个参数一个对象, 字段名显式标注
```

```text
原结构: 4 层目录树仅靠缩进表达关系
改写: 扁平列表, 每行使用“路径: 说明”
```

## 沉淀判断

用户提供的坏示例、旧版描述、临时提醒和单次问题默认是诊断材料, 不是长期规则。

文档迁移时, AI 常把旧方案 A 写成“不要 A”等提醒。若新方案 B 已经在当前文档中成立, 这类提醒通常是迁移残留, 只有它代表默认倾向冲突、硬性契约或验收边界时才保留。

改写后还要检查是否出现隐蔽防御式说法: 表面上写了推荐路径 B, 但又继续把旧方案 A 写成额外边界、特例处理或专项排除。若 A 已被 B 的合法值集合、接口契约或验收标准自然排除, 删除 A 的点名, 只保留正常校验或验收规则。

写入目标文档前先判断:

1. 能提炼成当前目标、推荐做法、边界或验收标准时, 改写后保留。
2. 只是已有规则的重复例子时, 合并到已有规则或删除。
3. 只描述一次性错误、历史过程或旧版本做法时, 不写入长期文档。
4. 只有引入新的判断标准、边界条件或输出要求时, 才新增规则。

最终文档应沉淀可复用判断, 不沉淀错误清单。

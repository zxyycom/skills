# 工作流

本文件负责“按什么顺序做”。原则解释归 [principles.md](principles.md), 具体改写判断归 [rewrite-rules.md](rewrite-rules.md)。

按任务选择一个或多个工作流执行。只加载当前判断和改写需要的部分。

## 生成前规划

适用于从零生成 prompt、规则、需求、任务或模板。

1. 确定用途: 执行、流程、协作、审阅、生成、说明、参考、契约、检查、分类、检索或评估。
2. 确定使用者: 人类、主 agent、子 agent、工具调用 agent 或长期维护者。
3. 确定目标: 写成可交付的目标状态, 不写成愿望。
4. 确定范围: 写清负责对象、排除对象和输入来源。
5. 确定边界: 写清安全、权限、真实性、不可逆操作和依赖条件。
6. 确定输出: 指定格式、字段、排序、粒度和失败时的表达方式。
7. 确定验收: 写成可检查的完成条件。

## 改写已有文本

适用于优化已有 prompt、规则、需求、模板或工作流。

1. 先定任务出口: 用途、使用者、目标状态和风险等级。
2. 再选文档骨架: 执行、流程、规则、说明、参考、契约、检查、评估或协作。
3. 做行为审计: 检查使用者错配、信息不足、信息过载和行为偏移。
4. 给内容找 owner: 完整解释只保留在 owner 位置, 其他位置改成摘要、触发条件或引用。
5. 收敛规则关系: 删除重复, 标出冲突, 确定优先级, 控制例外和示例。
6. 处理负向描述: 区分迁移残留、低频极端方案、默认倾向冲突和硬性契约。
7. 落地表达和格式: 把模糊词、抽象否定和复杂结构改成可验证表达。
8. 判断是否值得沉淀: 只保留当前可复用判断, 不沉淀错误清单。

## 审阅模式

适用于用户只要求检查、评估或 review, 且没有要求直接编辑文件。

输出顺序:

1. 问题: 指出当前信息、行为引导和实际用途的错配, 以及冲突、重复、模糊、过载、沉淀风险或触发风险。
2. 建议调整: 给出可直接替换的表达, 或明确结构、顺序、删减、补充方案。
3. 理由: 说明改写如何降低误触发、歧义或维护成本。
4. 残余风险: 标注仍需要用户选择或项目上下文确认的点。

## 直接编辑模式

适用于用户要求优化文件、更新规则或改造结构。

1. 先读目标文件和相邻引用文件。
2. 先确定新增或移动内容的 owner, 再改结构。
3. 保持原意, 优先做结构收敛和按需拆分。
4. 只创建直接支撑 skill 工作流、引用资料或资产的文件。
5. 新增辅助文档前, 先确认它会被 skill 触发、执行或交付流程直接使用。
6. 完成后检查入口文件是否能清楚导航到每个引用文件。

## 交付格式

根据用户请求选择:

1. 已编辑文件: 简述改动、列出新增或修改文件、说明验证。
2. 只审阅: 使用“问题 -> 建议调整 -> 理由”。
3. 只给方案: 给出结构设计、拆分边界和迁移步骤。
4. 信息不足: 说明缺口, 再问一个具体问题。

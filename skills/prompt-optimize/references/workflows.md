# 工作流

按任务选择一个或多个工作流执行。只加载当前判断和改写需要的部分。

## 生成前规划

适用于从零生成 prompt、规则、需求、任务或模板。

1. 确定用途: 执行、协作、审阅、生成、分类、检索或评估。
2. 确定使用者: 人类、主 agent、子 agent、工具调用 agent 或长期维护者。
3. 确定目标: 写成可交付的目标状态, 不写成愿望。
4. 确定范围: 写清负责对象、排除对象和输入来源。
5. 确定边界: 写清安全、权限、真实性、不可逆操作和依赖条件。
6. 确定输出: 指定格式、字段、排序、粒度和失败时的表达方式。
7. 确定验收: 写成可检查的完成条件。

## 改写已有文本

适用于优化已有 prompt、规则、需求、模板或工作流。

1. 先判断文本角色:
   - 执行型: 让 agent 完成任务。
   - 规则型: 约束长期行为。
   - 参考型: 提供背景、领域知识或示例。
   - 评估型: 检查输出是否合格。
2. 做用途一致性审计: 根据上下文推断实际使用者, 再识别当前信息、文档可能引导的行为和实际希望用途。
3. 为跨文档或跨结构节点的内容确定 owner; 当前节点不是 owner 时, 移动内容、降级为摘要、触发条件或引用, 或删除无判断价值的信息。
4. 删除或合并重复规则。
5. 标出冲突规则并确定优先级。
6. 按风险和用途处理负向描述, 写清保留、删除或改成推荐路径的原因。
7. 检查改写结果是否把旧方案包装成特例边界; 如果推荐路径、接口契约或合法值集合已经自然排除旧方案, 删除旧方案点名, 只保留当前可验证规则。
8. 把模糊表达改成检查点、完成条件或输出字段。
9. 把迁移残留、历史问题、坏示例和旧版本描述提炼成当前判断。
10. 保留必要边界, 并补正向做法。

## 审阅模式

适用于用户只要求检查、评估或 review, 且没有要求直接编辑文件。

输出顺序:

1. 问题: 指出当前信息、行为引导和实际用途的错配, 以及冲突、重复、模糊、过载、沉淀风险或触发风险。
2. 建议调整: 给出可直接替换的表达, 或明确结构、顺序、删减、补充方案。
3. 理由: 说明改写如何降低误触发、歧义或维护成本。
4. 残余风险: 标注仍需要用户选择或项目上下文确认的点。

## 直接编辑模式

适用于用户要求优化文件、更新规则或改造结构。

1. 先读目标文件和相邻引用文件。
2. 保持原意, 优先做结构收敛和按需拆分。
3. 只创建直接支撑 skill 工作流、引用资料或资产的文件。
4. 新增辅助文档前, 先确认它会被 skill 触发、执行或交付流程直接使用。
5. 完成后检查入口文件是否能清楚导航到每个引用文件。

## 交付格式

根据用户请求选择:

1. 已编辑文件: 简述改动、列出新增或修改文件、说明验证。
2. 只审阅: 使用“问题 -> 建议调整 -> 理由”。
3. 只给方案: 给出结构设计、拆分边界和迁移步骤。
4. 信息不足: 说明缺口, 再问一个具体问题。

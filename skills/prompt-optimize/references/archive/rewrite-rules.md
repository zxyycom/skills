# 改写规则

迁移保留说明: 本文件是将默认改写管线合并回入口文件后的保留副本。保留原因是迁移范围较大, 需要在稳定期内保留原拆分内容, 方便回溯旧结构和必要时恢复。原作用是承接任务出口、骨架选择、行为审计、内容归位、规则收敛、负向描述、表达落地和沉淀判断。当前不再作为主动读取入口, 新维护以入口文件为准。

合并前, 本文件负责“怎么判断、怎么改”。不要把它当成主题词清单逐条套用; 按下面的顺序推进, 每一步的产物决定下一步怎么处理。

以下为迁移前内容, 仅用于回溯旧结构。

## 1. 先定任务出口

改写前先回答: 这份文本最终要让谁在什么条件下完成、判断、查找或校验什么。

输出四个判断:

1. 用途: 执行、流程、规则、说明、参考、契约、检查、评估或协作。
2. 使用者: 人类、主 agent、子 agent、工具调用 agent 或长期维护者。
3. 目标状态: 改写后使用者应该能做什么, 不写成愿望。
4. 风险等级: 是否涉及安全、权限、合规、事实真实性、不可逆操作、明确契约或更高优先级指令。

使用者判断只服务于内容取舍、结构和表达判断; 不说明也不影响执行时, 不写进正文。

## 2. 再选文档骨架

先选骨架, 再改句子。文本同时承担多种用途时, 以最容易造成误用的用途为主骨架, 其他用途降级为附录、参考或局部说明。

1. 执行型: 目标、范围、输入、步骤、边界、验收、输出。
2. 流程型: 前置条件、状态转移、步骤、失败处理、恢复路径、验收。
3. 规则型: 触发条件、规则、例外、优先级、检查方式。
4. 说明型: 问题引入、概念、推理递进、取舍、结论。
5. 参考型: 索引、概念、条目、示例、相关链接。
6. 契约型: 字段、合法值、约束、兼容性、校验方式。
7. 检查型或评估型: 检查项、通过标准、失败表达、残余风险。
8. 协作型: 目标、范围、所有权、协作边界、输出字段、阻塞表达。

阅读脉络只在理解型文本里强化。说明、介绍、原理和设计理由文本检查承接、递进、转折和结论; SOP、runbook、API、字段、规则索引和机器消费格式优先稳定结构。

## 3. 做行为审计

用当前骨架检查原文会把使用者引向哪里。按错配类型改:

1. 使用者错配:
   - 多类使用者会导致误用、权限或前置条件不同, 改标题、开头、适用范围或分流说明。
   - 只服务于非实际使用者的解释, 删除或移动到参考资料。
2. 信息不足:
   - 补输入、判断标准、步骤、边界、输出或示例。
   - 把需要临场推断的条件写成显式规则。
3. 信息过载:
   - 删除与目标无关的背景、历史过程和低价值细节。
   - 仍有判断价值但不服务主流程的信息, 降级为参考。
4. 行为偏移:
   - 想让使用者执行时, 改成步骤、输入、输出和验收。
   - 想让使用者决策时, 改成判断条件、取舍依据和推荐路径。
   - 只是长期参考时, 避免写成强行动作。

## 4. 给内容找 Owner

内容归位先于局部润色。出现跨文档、跨标题、跨步骤、跨字段或跨规则节点的内容时, 先判断哪里最适合完整承接。

判断 owner:

1. 哪个位置是使用者最可能查找这条规则的地方。
2. 哪个位置拥有这条规则的触发条件、上下文和验收标准。
3. 哪个位置变更时最需要同步修改这条规则。

处理方式:

1. 当前节点是 owner: 保留完整解释, 写成目标、判断、步骤、边界或验收标准。
2. 当前节点不是 owner: 移动到 owner; 当前节点只保留摘要、触发条件或引用。
3. 找不到 owner 且内容只是历史过程、低价值背景或非实际使用者需要的信息: 删除或降级为参考。
4. 涉及安全、权限、合规、事实真实性、不可逆操作或明确契约: 不直接删除, 移到边界、例外或验收位置。

## 5. 收敛规则关系

内容归位后, 再处理规则之间的关系。

1. 重复规则: 保留最接近 owner 的一处完整版本, 其他位置改成引用或删除。
2. 冲突规则: 按风险和来源确定优先级; 无法判断时问一个具体问题。
3. 例外规则: 只有高频、明确契约或重要边界才写成例外; 低频极端情况删除或降级参考。
4. 示例: 只保留能说明判断标准的代表性示例; 不保留错误清单。

优先级参考: 更高优先级指令和硬性契约 > 安全、权限、合规、真实性、不可逆操作 > 用户当前目标 > 项目约定 > 已安装依赖和相邻模块做法 > 行业常见方案。

## 6. 处理负向描述

负向描述放在规则关系之后处理, 因为它经常是迁移残留、例外规则或硬性边界的表现。

判断顺序:

1. X 是迁移前旧方案, 且当前方案 Y 已明确: 写 Y 或删除旧提醒。
2. X 是低频极端方案, 且不是硬性契约: 删除或降级参考。
3. X 是默认容易回落的方案: 写推荐路径; 必要时保留边界。
4. X 是安全、权限、合规、事实真实性、不可逆操作或明确契约: 保留边界, 有正确做法时补正向路径。
5. 推荐路径、接口契约或合法值集合已经使 X 非法: 不把 X 写成额外特例, 只写当前合法值、校验方式或验收标准。
6. 没有可靠的 Y: 根据用户目标、项目约定、依赖、相邻模块和行业常见方案推断; 仍无法推断时问一个具体问题。

示例:

```text
原意: 排除 Redux
改写: 使用 Zustand 管理客户端状态

原意: 从旧枚举 A 迁到新枚举 B 后, 不要再接受 A
改写: 使用 B 的合法值集合校验输入

原意: 限制原始 SQL
改写: 通过项目现有 ORM 访问数据库; 只有 ORM 无法表达查询时, 才在参数化查询中使用 SQL

原意: 不要使用前端框架
改写: 主要使用 HTML、CSS 和 JavaScript 实现; 边界: 不使用 Vue、React、Angular 等前端框架
```

## 7. 落地表达和格式

到这一步才做句子级和格式级处理。

1. 模糊词:
   - “更好”改成目标状态和验收标准。
   - “高级”改成受众、复杂度、示例深度和输出格式。
   - “合理”改成决策优先级。
   - “适当”改成触发条件、范围和上限。
2. 抽象否定:
   - 可以保留, 但要补可验证标准。
   - 例: “不要泛泛总结”改成“先列出具体发现、文件位置或证据, 再给简短总结”。
3. 结构格式:
   - 步骤、规则和优先级用有序列表。
   - 复杂嵌套、结构化输入输出和 agent 返回结果用 JSON。
   - 简单表格只用于不超过 3 列、每行可独立理解的内容。
   - 深层目录树、跨列依赖表格和图示关系改成扁平路径、字段对象或 nodes/edges。
4. 语言:
   - 保持原文主要语言。
   - 只有用户要求翻译或目标文档需要统一语言时才切换语言。

## 8. 最后判断是否值得沉淀

坏示例、旧版描述、临时提醒和单次问题先当诊断材料, 不直接写入长期规则。

写入前检查:

1. 能提炼成当前目标、推荐做法、边界或验收标准时, 改写后保留。
2. 只是已有规则的重复例子时, 合并到已有规则或删除。
3. 只描述一次性错误、历史过程或旧版本做法时, 不写入长期文档。
4. 只有引入新的判断标准、边界条件或输出要求时, 才新增规则。
5. 改写后仍把已被推荐路径、接口契约或合法值集合自然排除的旧方案写成额外边界、特例处理或专项排除时, 删除旧方案点名, 只保留正常校验或验收规则。

最终结果应沉淀可复用判断, 不沉淀错误清单。

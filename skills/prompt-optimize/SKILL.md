# Prompt Optimize

## 目标与边界

把 prompt、规则、任务、需求、模板、工作流和 agent 指令等结构化文本, 改成目标清晰、范围明确、正向可执行、结构显式、边界和验收标准清楚、便于复用维护的文本。

核心判断:

1. 负向描述按用途处理: 迁移残留和低频极端方案删除或改成当前推荐路径; 高频误用、硬性契约和重要边界保留。
2. 安全、合规、权限、事实真实性和不可逆操作边界可以保留否定式; 有正确做法时必须补正向做法。
3. 模糊要求改成可验证的检查点、完成条件或输出格式。
4. 复杂表格、深层树状结构和思维导图式描述改成有序列表、JSON 或扁平路径说明。
5. 坏示例、旧版描述、临时提醒和单次问题先当作诊断材料, 只沉淀可复用判断。

## 读取策略

先读本文件。按当前任务需要加载一个或多个引用; 一个任务可以同时触发多个子流程:

1. 改写、审阅或生成 prompt、规则、需求、模板时, 读取 [workflows.md](references/workflows.md)。
2. 需要处理负向描述、模糊表达、结构格式或沉淀判断时, 读取 [rewrite-rules.md](references/rewrite-rules.md)。
3. 需要写 worker、explorer、并行 agent 或子 agent 任务时, 读取 [agent-tasks.md](references/agent-tasks.md)。
4. 用户要求解释原理、争议取舍或长期维护理由时, 读取 [principles.md](references/principles.md)。
5. 需要整体审阅 skill 或判断引用一致性时, 可以读取入口文件和所有直接引用文件。

## 最小流程

1. 定位目标:
   - 用户指定文件或目录时, 只处理指定范围。
   - 用户只给出项目或工作区时, 查找 `AGENTS.md`、`CLAUDE.md`、`SKILL.md`、规则文件、prompt 模板和 agent 任务模板。
   - 存在多个高风险目标且优先级不清时, 用一个简短问题确认范围。

2. 判断用途:
   - 执行型文本: 优先给目标、范围、步骤、边界、验收标准。
   - 规则型文本: 先设计结构, 再写规则。
   - 审阅型任务: 先列问题, 再给建议改写和理由。
   - agent 任务: 先界定所有权、协作边界和输出字段。
   - 文档迁移任务: 识别从旧方案 A 迁到新方案 B 后残留的“不要 A”等提醒, 只保留仍有判断价值的边界。

3. 按需加载引用:
   - 加载当前任务需要的引用文件。
   - 引用文件之间按当前任务需要继续读取; 本文件列出的引用就是主要入口。

4. 处理文本:
   - 删除重复规则, 合并冲突要求, 写清优先级。
   - 按风险和用途处理负向描述, 区分迁移残留、低频极端方案、高频误用和硬性边界。
   - 把主观形容词改成检查点。
   - 把坏示例和旧说法提炼成当前可复用判断。
   - 保持原文主要语言; 只有用户要求翻译或目标文档需要统一语言时才切换语言。

5. 交付:
   - 直接编辑目标文件时, 保持改动范围最小。
   - 只做审阅时, 使用“问题 -> 建议改写 -> 理由”。
   - 无法可靠推断替代方案时, 说明缺口并问一个具体问题。

## 推荐路径与冲突处理

选择推荐路径时默认以用户当前目标和显式偏好为主, 因为用户可能掌握项目文档外的信息。执行前后按风险处理冲突:

1. 涉及安全、权限、合规、事实真实性、不可逆操作、更高优先级指令或明确契约时, 先说明冲突并确认可行路径。
2. 与项目约定、已安装依赖或相邻模块做法冲突但风险可控时, 可以按用户目标处理, 并在交付中汇报取舍。
3. 无法判断冲突风险时, 问一个具体问题确认。

选择具体方案时参考:

1. 用户当前目标和显式偏好。
2. 项目已有约定和文档契约。
3. 已安装依赖。
4. 相邻模块的做法。
5. 行业常见方案。

## 完成检查

完成前检查:

1. 是否按用途处理负向描述: 迁移残留和低频极端方案删除或改写, 高频误用和重要边界保留。
2. 是否保留了必要安全边界并配套正向做法。
3. 是否删除或合并了重复、冲突和优先级不清的规则。
4. 是否把模糊表达改成可验证标准。
5. 是否把复杂格式改成列表、JSON 或扁平路径说明。
6. 是否保持原文主要语言, 或说明了切换语言的原因。
7. 是否清除了坏示例、旧版本描述、迁移残留和临时提醒, 只沉淀可复用判断。

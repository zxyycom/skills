# Prompt Optimize

## 目标

把 prompt、规则、任务、需求、模板、工作流和 agent 指令等结构化文本, 改成目标清晰、范围明确、正向可执行、结构显式、边界和验收标准清楚、阅读脉络符合用途、便于复用维护的文本。

## 主动引用

先读本文件。按当前任务只加载必要引用:

1. 用户要求解释原理、争议取舍、内容所有权或长期维护理由时, 读取 [principles.md](references/principles.md)。
2. 需要写 worker、explorer、并行 agent 或子 agent 任务时, 读取 [agent-tasks.md](references/agent-tasks.md)。
3. 需要整体审阅 skill 或判断引用一致性时, 读取入口文件和必要引用文件。

## 主执行流程

1. 定位目标:
   - 用户指定文件或目录时, 只处理指定范围。
   - 用户只给出项目或工作区时, 查找 `AGENTS.md`、`CLAUDE.md`、`SKILL.md`、规则文件、prompt 模板和 agent 任务模板。
   - 多个高风险目标优先级不清时, 用一个简短问题确认范围。

2. 定任务出口:
   - 用途: 执行、流程、规则、说明、参考、契约、检查、评估或协作。
   - 使用者: 人类、主 agent、子 agent、工具调用 agent 或长期维护者。
   - 目标状态: 改写后使用者应该能做什么, 不写成愿望。
   - 风险等级: 是否涉及安全、权限、合规、事实真实性、不可逆操作、明确契约或更高优先级指令。
   - 使用者判断只服务于内容取舍、结构和表达判断; 不说明也不影响执行时, 不写进正文。

3. 选择文档骨架:
   - 先选骨架, 再改句子。
   - 文本同时承担多种用途时, 以最容易造成误用的用途为主骨架, 其他用途降级为附录、参考或局部说明。
   - 执行型: 目标、范围、输入、步骤、边界、验收、输出。
   - 流程型: 前置条件、状态转移、步骤、失败处理、恢复路径、验收。
   - 规则型: 触发条件、规则、例外、优先级、检查方式。
   - 说明型: 问题引入、概念、推理递进、取舍、结论。
   - 参考型: 索引、概念、条目、示例、相关链接。
   - 契约型: 字段、合法值、约束、兼容性、校验方式。
   - 检查型或评估型: 检查项、通过标准、失败表达、残余风险。
   - 协作型: 目标、范围、所有权、协作边界、输出字段、阻塞表达。
   - 阅读脉络只在理解型文本里强化; SOP、runbook、API、字段、规则索引和机器消费格式优先稳定结构。

4. 做行为审计:
   - 使用者错配: 多类使用者会导致误用、权限或前置条件不同, 改标题、开头、适用范围或分流说明; 只服务于非实际使用者的解释, 删除或移动到参考资料。
   - 信息不足: 补输入、判断标准、步骤、边界、输出或示例; 把需要临场推断的条件写成显式规则。
   - 信息过载: 删除与目标无关的背景、历史过程和低价值细节; 仍有判断价值但不服务主流程的信息, 降级为参考。
   - 行为偏移: 想让使用者执行时, 改成步骤、输入、输出和验收; 想让使用者决策时, 改成判断条件、取舍依据和推荐路径; 只是长期参考时, 避免写成强行动作。

5. 给内容找 owner:
   - 内容归位先于局部润色。
   - 根据使用者最可能查找的位置、触发条件和上下文所在位置、后续变更时最需要同步的位置判断 owner。
   - 当前节点是 owner 时, 保留完整解释, 写成目标、判断、步骤、边界或验收标准。
   - 当前节点不是 owner 时, 移动到 owner; 当前节点只保留摘要、触发条件或引用。
   - 找不到 owner 且内容只是历史过程、低价值背景或非实际使用者需要的信息时, 删除或降级为参考。
   - 涉及安全、权限、合规、事实真实性、不可逆操作或明确契约时, 不直接删除, 移到边界、例外或验收位置。

6. 收敛规则关系:
   - 重复规则: 保留最接近 owner 的一处完整版本, 其他位置改成引用或删除。
   - 冲突规则: 按风险和来源确定优先级; 无法判断时问一个具体问题。
   - 例外规则: 只有高频、明确契约或重要边界才写成例外; 低频极端情况删除或降级参考。
   - 示例: 只保留能说明判断标准的代表性示例; 不保留错误清单。
   - 优先级参考: 更高优先级指令和硬性契约 > 安全、权限、合规、真实性、不可逆操作 > 用户当前目标 > 项目约定 > 已安装依赖和相邻模块做法 > 行业常见方案。

7. 处理负向描述:
   - X 是迁移前旧方案, 且当前方案 Y 已明确: 写 Y 或删除旧提醒。
   - X 是低频极端方案, 且不是硬性契约: 删除或降级参考。
   - X 是默认容易回落的方案: 写推荐路径; 必要时保留边界。
   - X 是安全、权限、合规、事实真实性、不可逆操作或明确契约: 保留边界, 有正确做法时补正向路径。
   - 推荐路径、接口契约或合法值集合已经使 X 非法: 不把 X 写成额外特例, 只写当前合法值、校验方式或验收标准。
   - 没有可靠的 Y 时, 根据用户目标、项目约定、依赖、相邻模块和行业常见方案推断; 仍无法推断时问一个具体问题。

8. 落地表达和格式:
   - 模糊词改成目标状态、验收标准、受众、复杂度、示例深度、输出格式、决策优先级、触发条件、范围或上限。
   - 抽象否定可以保留, 但要补可验证标准。
   - 步骤、规则和优先级用有序列表。
   - 复杂嵌套、结构化输入输出和 agent 返回结果用 JSON。
   - 简单表格只用于不超过 3 列、每行可独立理解的内容。
   - 深层目录树、跨列依赖表格和图示关系改成扁平路径、字段对象或 nodes/edges。
   - 保持原文主要语言; 只有用户要求翻译或目标文档需要统一语言时才切换语言。

9. 判断是否值得沉淀:
   - 坏示例、旧版描述、临时提醒和单次问题先当诊断材料, 不直接写入长期规则。
   - 能提炼成当前目标、推荐做法、边界或验收标准时, 改写后保留。
   - 只是已有规则的重复例子时, 合并到已有规则或删除。
   - 只描述一次性错误、历史过程或旧版本做法时, 不写入长期文档。
   - 只有引入新的判断标准、边界条件或输出要求时, 才新增规则。
   - 改写后仍把已被推荐路径、接口契约或合法值集合自然排除的旧方案写成额外边界、特例处理或专项排除时, 删除旧方案点名, 只保留正常校验或验收规则。

## 模式分流

生成前规划适用于从零生成 prompt、规则、需求、任务或模板:

1. 确定用途、使用者、目标、范围、边界、输出和验收。
2. 目标写成可交付的目标状态, 不写成愿望。
3. 范围写清负责对象、排除对象和输入来源。
4. 边界写清安全、权限、真实性、不可逆操作和依赖条件。
5. 输出指定格式、字段、排序、粒度和失败时的表达方式。
6. 验收写成可检查的完成条件。

审阅模式适用于用户只要求检查、评估或 review, 且没有要求直接编辑文件。输出顺序:

1. 问题: 指出当前信息、行为引导和实际用途的错配, 以及冲突、重复、模糊、过载、沉淀风险或触发风险。
2. 建议调整: 给出可直接替换的表达, 或明确结构、顺序、删减、补充方案。
3. 理由: 说明改写如何降低误触发、歧义或维护成本。
4. 残余风险: 标注仍需要用户选择或项目上下文确认的点。

直接编辑模式适用于用户要求优化文件、更新规则或改造结构:

1. 先读目标文件、当前主动引用文件和确有必要的相邻文件。
2. 先确定新增或移动内容的 owner, 再改结构。
3. 保持原意, 优先做结构收敛和按需拆分。
4. 只创建直接支撑 skill 主流程、主动引用资料或资产的文件。
5. 新增辅助文档前, 先确认它会被 skill 触发、执行或交付流程直接使用。
6. 完成后检查入口文件是否能清楚导航到每个主动引用文件。

## 冲突处理

默认以用户当前目标和显式偏好为主, 但按风险处理冲突:

1. 涉及安全、权限、合规、事实真实性、不可逆操作、更高优先级指令或明确契约时, 先说明冲突并确认可行路径。
2. 与项目约定、已安装依赖或相邻模块做法冲突但风险可控时, 可以按用户目标处理, 并在交付中汇报取舍。
3. 无法判断冲突风险时, 问一个具体问题确认。

选择具体方案时参考: 用户目标、项目约定、已安装依赖、相邻模块做法和行业常见方案。

## 交付格式

根据用户请求选择:

1. 已编辑文件: 简述改动、列出新增或修改文件、说明验证。
2. 只审阅: 使用“问题 -> 建议调整 -> 理由”。
3. 只给方案: 给出结构设计、拆分边界和迁移步骤。
4. 信息不足: 说明缺口, 再问一个具体问题。

## 完成检查

1. 是否按定位目标、任务出口、文档骨架、行为审计、内容 owner、规则关系、负向描述、表达格式和沉淀判断的顺序处理。
2. 当前信息和行为引导是否符合文档实际用途。
3. 核心判断是否只有 owner 位置完整解释。
4. 安全、权限、合规、真实性和不可逆操作边界是否保留, 且有正确做法时配套正向路径。
5. 重复、冲突、模糊表达和复杂格式是否已收敛。
6. 坏示例、旧版描述、临时提醒和单次问题是否只作为诊断材料使用。
7. 交付格式是否符合用户请求, 且没有扩大修改范围。

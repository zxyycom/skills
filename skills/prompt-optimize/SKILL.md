# Prompt Optimize

## 目标

把 prompt、规则、任务、需求、模板、工作流和 agent 指令等结构化文本, 改成目标清晰、范围明确、正向可执行、结构显式、边界和验收标准清楚、阅读脉络符合用途、便于复用维护的文本。

## 主动引用

先读本文件。按当前任务只加载必要引用:

1. 用户要求解释原理、争议取舍、内容所有权或长期维护理由时, 读取 [principles.md](references/principles.md)。
2. 需要整体审阅 skill 或判断引用一致性时, 读取入口文件和必要引用文件。

## 主执行流程

按顺序处理, 但每步只展开当前任务需要的判断:

1. 定位目标和范围:
   - 用户指定文件或目录时, 只处理指定范围。
   - 用户只给出项目或工作区时, 查找 `AGENTS.md`、`CLAUDE.md`、`SKILL.md`、规则文件、prompt 模板和 agent 任务模板。
   - 多个高风险目标优先级不清时, 用一个简短问题确认范围。

2. 定任务出口:
   - 用途: 执行、流程、规则、说明、参考、契约、检查、评估或协作。
   - 使用者: 人类、主 agent、子 agent、工具调用 agent 或长期维护者。
   - 目标状态: 改写后使用者应该能做什么, 不写成愿望。
   - 风险等级: 是否涉及安全、权限、合规、事实真实性、不可逆操作、明确契约或更高优先级指令。
   - 使用者判断只服务于内容取舍、结构和表达判断; 不说明也不影响执行时, 不写进正文。

3. 建立主承诺:
   - 从标题、序言、触发条件、前置条件、资源名、字段集合、适用范围或输入输出描述推断文档承诺。
   - 对照正文篇幅、层级、结论、验收或校验方式, 判断正文是否兑现承诺。
   - 不兑现时, 判断是正文偏离、入口承诺过宽, 还是内容应移动到其他 owner。

4. 选择骨架:
   - 先选骨架, 再改句子。
   - 文本同时承担多种用途时, 以最容易造成误用的用途为主骨架, 其他用途降级为附录、参考或局部说明。
   - 执行型写目标、范围、输入、步骤、边界、验收、输出。
   - 流程型写前置条件、状态转移、步骤、失败处理、恢复路径、验收。
   - 规则型写触发条件、规则、例外、优先级、检查方式。
   - 说明型写问题引入、概念、推理递进、取舍、结论。
   - 参考型写索引、概念、条目、示例、相关链接。
   - 契约型写字段、合法值、约束、兼容性、校验方式。
   - 检查型或评估型写检查项、通过标准、失败表达、残余风险。
   - 协作型写目标、范围、所有权、协作边界、输出字段、阻塞表达。

5. 审计内容和行为:
   - 使用者错配时, 改标题、开头、适用范围或分流说明。
   - 信息不足时, 补输入、判断标准、步骤、边界、输出或示例。
   - 信息过载时, 删除无关背景、历史过程和低价值细节; 有判断价值但不服务主流程的信息降级为参考。
   - 行为偏移时, 按真实用途改成执行步骤、决策条件或长期参考。

6. 归位和收敛规则:
   - owner 指最适合完整承接某项判断的位置: 使用者最可能查找, 拥有触发上下文和验收标准, 且后续变更时最需要同步。
   - 内容归位先于局部润色; 完整解释只留在最稳定的 owner。
   - 非 owner 位置只保留摘要、触发条件或引用。
   - 重复规则保留 owner 处完整版本; 冲突规则按风险和来源定优先级。
   - 例外和示例只保留高频、明确契约、重要边界或能说明判断标准的内容。

7. 处理负向描述和沉淀:
   - 迁移残留和低频极端方案删除或降级参考。
   - 默认容易回落、重要边界或明确契约写成推荐路径和必要边界。
   - 推荐路径、接口契约或合法值集合已经排除的旧方案, 不额外点名。
   - 坏示例、旧版描述、临时提醒和单次问题只作为诊断材料; 只有提炼成可复用判断时才沉淀。

8. 落地表达和格式:
   - 模糊词改成目标状态、验收标准、触发条件、范围、上限或输出格式。
   - 抽象否定可以保留, 但要补可验证标准。
   - 步骤、规则和优先级用有序列表; 复杂结构化输入输出用 JSON。
   - 简单表格只用于不超过 3 列、每行可独立理解的内容。
   - 深层目录树、跨列依赖表格和图示关系改成扁平路径、字段对象或 nodes/edges。
   - 保持原文主要语言; 只有用户要求翻译或目标文档需要统一语言时才切换语言。

## 交付边界

同一套主执行流程适用于生成、审阅、改写、直接编辑和方案设计; 不按这些请求类型拆成独立模式。按用户请求决定是否改文件和如何交付:

1. 用户要求编辑文件时, 先读目标文件、当前主动引用文件和确有必要的相邻文件; 保持原意, 优先做结构收敛和按需拆分。
2. 用户只要求检查、评估、review、方案或结构设计时, 不直接编辑文件。
3. 从零生成 prompt、规则、需求、任务或模板时, 仍按任务出口确定用途、使用者、目标、范围、边界、输出和验收。
4. 审阅类输出先给问题, 再给建议调整和理由; 有未确认的项目上下文时列出残余风险。
5. 已编辑文件时, 简述改动、列出新增或修改文件、说明验证。
6. 信息不足时说明缺口, 再问一个具体问题。

## 冲突处理

默认以用户当前目标和显式偏好为主, 但按风险处理冲突:

1. 涉及安全、权限、合规、事实真实性、不可逆操作、更高优先级指令或明确契约时, 先说明冲突并确认可行路径。
2. 与项目约定、已安装依赖或相邻模块做法冲突但风险可控时, 可以按用户目标处理, 并在交付中汇报取舍。
3. 无法判断冲突风险时, 问一个具体问题确认。

选择具体方案时参考: 用户目标、项目约定、已安装依赖、相邻模块做法和行业常见方案。

## 完成检查

1. 是否已定位目标、任务出口、主承诺、文档骨架、内容 owner 和交付边界。
2. 主承诺是否被正文层级、篇幅、结论、验收或校验方式兑现。
3. 当前信息和行为引导是否符合文档实际用途。
4. 核心判断是否只有 owner 位置完整解释。
5. 安全、权限、合规、真实性和不可逆操作边界是否保留, 且有正确做法时配套正向路径。
6. 重复、冲突、模糊表达、复杂格式和旧方案点名是否已收敛。
7. 坏示例、旧版描述、临时提醒和单次问题是否只作为诊断材料使用。
8. 交付是否符合用户请求, 且没有扩大修改范围。

# Decision Records

## 目标

用显式、可版本化的 Markdown 保存长期决策的目的、形成背景、采用方向和必要关系，用 JSON 全生命周期索引保存全部记录的状态、秒级创建时间、精简摘要和直接关系。让人类和 agent 在形成可能产生长期影响的决定前快速恢复希望达成的结果、当时约束和当前方向，在需要时低成本回放理由，并避免后续维护静默偏离。

决策记录不承接通用记忆、任务日志或当前实现状态；代码、配置、规范和项目文档继续承接当前事实与行为。本文中的行为 owner 指权威承接这些事实或规则的文件。skill 只在重要长期取舍形成时介入，不反复打断原任务。

## 内容 owner

1. 本文件承接发现、语义恢复、偏离判断、候选确认、CLI 选择和按任务出口交付。
2. [decision-record-rules.md](references/decision-record-rules.md) 是唯一固定契约，承接决策根目录、索引 schema、稳定身份、正文格式、关系、生命周期、维护事务和 CLI 精确语义。
3. [maintenance-recovery.md](references/maintenance-recovery.md) 是仅在冷启动、升级或故障时读取的恢复手册，承接错误分类和恢复顺序，不重复定义格式契约。
4. `scripts/decision-records.mjs` 是活动决策查询、单条展开、关系追溯、生命周期变更、派生字段同步和结构检查的确定性接口。
5. 目标项目的决策根目录承接全生命周期索引和 Markdown；项目专属的更高记录门槛写入 `AGENTS.md` 或相关行为 owner。

## 主动读取

1. 先读目标工作区的指令文件和相关行为 owner。
2. 在形成可能产生长期影响的决定或处理与既有决定的冲突前，确定决策根目录并定位索引。根目录整体尚不存在时，将集合解释为尚未初始化；已有决策材料但索引缺失或无法解析时，进入维护恢复手册。
3. 日常恢复运行 `list`，默认查看全部主题的 `active` 记录；`--topic <topic-id>` 是可与状态筛选和时间显示选项组合的主题筛选条件。从状态、日期、路径、标题、目的、背景和决策摘要定位相关判断。
4. 列表摘要足以判断时停止；需要单条完整正文时使用 `show`，需要审计演进时使用 `trace`。只有明确需要历史清单、迁移、修复或全库生命周期审计时，才给 `list` 传入 `archived` 或 `all` 状态筛选。
5. 恢复当前判断、归类偏离和提出未确认候选时不读取固定契约；创建、修订、归档、归并、显式修复、迁移或结构审阅前完整读取固定契约。
6. 只有现有集合冷启动或升级、CLI/运行时不可用、索引缺失或损坏、写入中断或严格检查失败且普通诊断不足时，才完整读取维护恢复手册；正常查询和维护不读取。

## 执行流程

### 1. 确定任务出口

1. 恢复、解释、检查或审阅已有决策时，交付查询或审阅结果；当前指令没有要求维护时，不扩大改动范围。
2. 当前指令已经明确长期判断、维护对象和影响范围时，将其视为已经确认并直接维护，不重复询问；新增记录或改变生命周期前，先告知将写入或改变的判断及状态变化。
3. 普通工作中形成新的重要取舍时，先恢复相关活动决策，在自然更新或交付阶段提出候选；只有必须先确定该取舍才能继续时才即时提出。
4. 修复非标准格式或迁移旧索引、路径、时间时，先清点并报告拟转换范围；当前指令只授权已经说明并确认的范围。

写入内容、长期含义、维护对象或影响范围仍不清楚时，先给出最小概览并只问一个确认问题。

### 2. 恢复当前判断

1. 从 `list` 输出的 `path`、`title`、`purpose`、`background` 和 `decision` 定位相关活动记录。
2. 将 `status: active` 解释为已经确认并仍作为后续工作默认依据的决策；`archived` 记录只在历史恢复或审计时读取。
3. 从 `目的` 恢复希望达成或维护的长期结果，从 `背景` 理解当时为何需要作出选择，从 `决策` 恢复采用的方向，再对照当前事实和行为 owner 判断当前任务是否仍应沿用；活动记录是默认依据，不是脱离目的和背景的永久规则。
4. 将当前任务与相关决策归为以下一种情况：
   - 对齐：按当前判断继续。
   - 明确的一次性例外：按当前指令完成本次任务，说明例外范围，并明确索引状态和历史未改变。
   - 长期修订：形成候选概览；当前指令已经明确要求长期记录或修订时复用该确认，否则等待确认后再写入。
   - 一致性问题：活动记录互相冲突、记录与行为 owner 冲突，或偏离性质不清且会影响实质改动时，报告冲突并请求判断。
5. 当前指令只改变本次任务的执行方式时，归为一次性例外；只有用户已经决定改变后续工作的默认依据时，才归为长期修订。
6. 当前事实与记录背景已经明显不同时，说明变化、受影响判断和需要确认的新取舍，不机械沿用旧背景。
7. 没有当前指令或既有记录提供确认依据时，不自行选择冲突方向，也不静默改变长期行为。
8. `list`、`show` 或 `trace` 成功只表示返回了索引仍可恢复的结果。warning 可能影响相关主题时，不根据缺失正文、关系或结果断言它们不存在；说明不完整性，并按需检查索引、Markdown 或运行严格 `check`。

### 3. 提出候选决策

候选判断需要同时具备长期影响和回放价值。依次确认：

1. 它是否会持续作为后续工作依据，或改变长期行为、owner、边界、兼容性、风险处理或验收方式？
2. 缺少记录时，未来是否难以恢复为何采用当前方案？
3. 它是否包含可复用判断或取舍，而不是事实、状态、任务或执行结果？
4. 它是否是一项能够独立修订或替代的判断？需要分别演进的判断分别处理。

当前指令已经明确表达这项长期判断时，复用该确认并进入维护。否则只给出形成判断所需的最小概览：

1. 目的。
2. 背景。
3. 采用的决策。
4. 与既有决策的直接关系；没有关系时省略。

让用户确认、修改或放弃。任务局部选择、一次性例外、建议和开放问题继续留在原 owner。

### 4. 写入或修复

1. 完整读取固定契约，并据此确定决策目录和维护事务。
2. 按固定契约区分编辑性修正与决策演进，并选择新增、修订、替代、判定无效、归并或独立归档事务。
3. 准备 Markdown、真实关系和需要同步的行为 owner，再通过 CLI 登记记录、改变显式状态和同步 JSON 派生字段，不直接维护生成字段。
4. 新增独立判断时写入无日期语义路径，再运行 `activate` 登记并激活；首条记录同样由 `activate` 初始化索引。
5. 修订、替代、判定无效或归并时，先准备新结论，在文件落盘前显式 `archive` 直接前序，再写入新 Markdown 并显式 `activate`；关系不替代状态命令，`archive` 不隐式激活后续记录。
6. 目录不符合契约时，先报告差异、可保留语义、修复范围和不确定项；得到修复范围确认后再执行转换。
7. CLI 或 Node 环境不可用时，按维护恢复手册直接读取索引和 Markdown，说明正常命令会产生的大致效果；不把无法机械校验的人工操作描述为等价事务。
8. 修复或迁移中的不确定项会改变决策含义、生命周期、关系、创建时间或 owner 时，请用户判断。

### 5. 验证与交付

1. 确认记录表达了已经确认的目的、背景和采用方向，并检查当前事实、行为 owner、相关决策和本次生命周期变化之间没有未解释的冲突或不必要的敏感信息。
2. 确认标题、目的、背景和决策索引投影均为 4 至 100 个 Unicode 码点的单行文本；完整正文不受该长度限制。
3. 完成写入或迁移后运行 `sync-index --write` 刷新全部派生投影与关系，再运行严格 `check`，修复到退出码为 `0`。
4. 等待确认时，交付候选概览、未确认项和“尚未写入”的状态。
5. 已写入时，列出新增、修订、替代、判定无效、归并、归档或迁移，说明活动与归档状态变化、偏离分类、行为 owner 检查以及同步和校验结果。
6. 只恢复或审阅时，交付相关活动判断、偏离分类、查询 warning 及其对结论完整性的影响。

## CLI

从 skill 目录运行，或使用脚本的绝对路径：

```text
node scripts/decision-records.mjs list [--topic <topic-id>] [--status <status>] [--full-time] --root <workspace-root>
node scripts/decision-records.mjs show <topic/file.md> --root <workspace-root>
node scripts/decision-records.mjs trace <topic/file.md> [--direction <direction>] [--depth <n>] --root <workspace-root>
node scripts/decision-records.mjs check --root <workspace-root>
node scripts/decision-records.mjs sync-index --write --root <workspace-root>
node scripts/decision-records.mjs activate <topic/file.md> --root <workspace-root>
node scripts/decision-records.mjs archive <topic/file.md...> --root <workspace-root>
```

需要在现有 ESM 进程中复用时，从本 skill 的实际安装路径直接导入同一个 `scripts/decision-records.mjs`。模块导入不会执行 CLI，导出 `scanDecisionRecords`、`validateDecisionRecords` 和返回退出码的 `runDecisionRecordsCli(argv)`；相邻的 `decision-records.d.mts` 提供 TypeScript 声明。

`list` 默认只列全部主题的活动记录；`--topic <topic-id>` 可以按主题筛选，并与状态筛选和时间显示选项组合。只有明确需要历史清单时才使用 `--status archived` 或 `--status all`。默认只显示创建日期，需要秒级完整时间时增加 `--full-time`。`show` 展开索引独有元数据和原始正文，`trace` 恢复关系，`check` 严格检查结构；读取固定契约后再使用写命令。`--decisions-dir <path>` 可以选择工作区内的非默认决策根目录。命令参数、合法值、初始化、warning、状态变化和失败恢复的精确语义以固定契约与 `--help` 为准；冷启动、升级或错误恢复顺序见维护恢复手册。

## 完成标准

### 恢复或审阅

1. 已从活动索引投影和必要的 Markdown 恢复相关判断，并结合目的、背景、当前事实和行为 owner 判断是否沿用。
2. 当前任务的偏离已经归类；查询 warning 已被披露，且结论没有超出可恢复结果能够证明的范围。
3. 交付与当前查询或审阅目标一致，没有扩大为未要求的维护。

### 候选决策

1. 候选同时满足长期影响和回放价值，粒度能够独立演进。
2. 确认状态清楚；未确认候选没有进入索引。

### 维护或迁移

1. 只有明确确认的长期取舍进入索引；新增记录和生命周期变化前已经告知用户，非标准格式转换没有超出确认范围。
2. 精确维护或结构审阅前已经读取固定契约，历史语义通过正确事务、显式状态和真实关系保留。
3. 活动记录、归档记录、创建时间、决策内容、行为 owner 和按需关系没有未解释的冲突。
4. 严格 `check` 返回 `0`，交付说明实际语义检查、生命周期变化和校验结果。

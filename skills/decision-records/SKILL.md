# Decision Records

## 目标

用自包含 Markdown 保存已经生效的长期决策、生命周期、对齐状态、形成背景、完整采用方向和历史关系，并从这些文件确定性生成领域 state、派生查询 key 和 source revision 组成的通用 JSON 索引。让人类和 agent 能区分未激活候选、已经生效的活动决策，以及决策目标是否已经核对并建立为当前基线：`active` 决策立即生效，`alignment` 表达完整目标是否已通过实际行为 owner 和事实核对并建立为必须保持的单向基线，并由索引原样投影。需要约束后续工作的限制、允许范围或例外直接写入决策正文；当前差距由完整决策与实际行为 owner 和事实比较得出。

决策记录不承接通用记忆、任务日志、进度清单或当前实现事实；代码、配置、规范和项目文档继续承接当前事实与行为。本文中的行为 owner 指权威承接这些事实或规则的文件。

## 内容 owner

1. 本文件承接发现、语义恢复、对齐判断、偏离分类、候选确认、调用方式选择和按任务出口交付。
2. [decision-record-rules.md](references/decision-record-rules.md) 是唯一固定契约，承接目录、Markdown、frontmatter、索引 schema、状态语义、关系、维护事务和 CLI 精确行为。
3. [maintenance-recovery.md](references/maintenance-recovery.md) 只在 CLI/Node 不可用、索引缺失或损坏、写入中断或严格检查失败且普通诊断不足时读取。
4. [decision-index.schema.json](references/decision-index.schema.json) 是随包分发的通用索引外壳与决策 state、keys 的当前 JSON Schema。
5. `scripts/decision-records.mjs` 是查询、展开、追溯、维护和检查决策记录的确定性接口。

## 主动读取

1. 先读目标工作区指令和相关行为 owner。
2. 在形成长期决定或处理潜在冲突前，按 `--root` 和可选的 `--decisions-dir` 确定决策根目录并定位索引。决策根目录必须位于能够读取 `HEAD` 路径的 Git 仓库；根目录整体不存在时解释为尚未初始化，已有材料但索引不可用时进入恢复手册。
3. 日常恢复先运行 `list`；主题明确时使用 `--topic <topic-id>`，只需要已对齐基线或未落实方向时再使用 `--alignment`。
4. 列表摘要足以判断时停止；需要完整理由时使用 `show`，需要审计演进时使用 `trace`。只有任务明确需要历史时才扩大生命周期范围。
5. 相关记录为 `active + unaligned` 时必须运行 `show`，完整读取目标决策并检查相关行为 owner 和当前事实；从两者比较中确定实际差距和本次动作，不能从对齐字段推断额外许可。
6. 恢复判断和提出候选时不读取固定契约；任何写入、状态变化或结构审阅前完整读取固定契约。
7. 只有索引或工具故障、写入中断或严格检查无法普通修复时才完整读取恢复手册。

## 执行流程

### 1. 确定任务出口

1. 恢复、解释、检查或审阅已有决策时，只交付查询或审阅结果；没有维护授权时不扩大为写入。
2. 当前指令已经明确长期判断、维护对象和影响范围时，视为已经确认并直接维护；新增记录或改变状态前，先告知将写入的判断和状态变化。
3. 普通工作中形成重要取舍时，在自然更新或交付阶段提出候选；只有继续任务必须先确定该取舍时才即时提出。
4. 写入内容、长期含义、维护对象或影响范围仍不清楚时，给出最小概览并只确认这一项。

### 2. 恢复当前判断

1. 从 `list` 的路径、生命周期、对齐状态、标题、目的、背景和决策摘要定位相关记录。
2. 活动记录在 `activate` 成功后即生效；`HEAD` 只决定其路径是否临时显示为 `pending`，不决定确认、生命周期或生效状态。
3. `active + aligned` 表示完整决策已经通过实际行为 owner 和事实核对，并建立为必须持续遵守的当前基线。
4. `active + unaligned` 表示决策已经生效，但完整目标尚未核对并建立为当前基线；实际差距仍需比较完整决策、行为 owner 和事实，它不改变决策正文，也不自动增加或取消任何约束。
5. 从 `目的` 恢复希望长期达成的结果，从 `背景` 理解为何需要选择，从 `决策` 恢复采用方向和必须遵守的限制；未对齐时将完整决策与实际行为 owner 和事实比较，确定当前差距和本次动作。
6. 将当前任务归为以下一种情况：
   - 已对齐继续：遵守活动已对齐基线。
   - 向目标推进：活动未对齐，当前工作按决策正文中的方向和明确约束推进。
   - 明确的一次性例外：按当前指令完成本次任务，说明例外范围，状态和历史不变。
   - 长期修订：形成新的完整候选和真实直接关系。
   - 一致性问题：活动记录互相冲突、已对齐记录与行为 owner 偏离，或当前动作违反活动决策正文中的明确约束。
7. 已对齐记录发生事实偏离时不改回 `unaligned`；报告一致性问题。新的未来目标通过新决策表达。
8. 查询出现 warning 时，不根据未登记文件或结果断言它们不存在；说明不完整性并按需运行严格 `check`。查询因 source revision 失配而失败时，不使用陈旧 state 推断当前状态，先审阅原文变化并同步索引。

### 3. 提出候选决策

候选判断需要同时具备长期影响和回放价值。依次确认：

1. 它是否持续影响后续行为、owner、边界、兼容性、风险处理或验收？
2. 缺少记录时，未来是否难以恢复为何采用当前方案？
3. 它是否是判断或取舍，而不是事实、任务、进度或执行结果？
4. 它是否能够独立修订或替代？需要分别演进的判断分别记录。

当前指令已明确表达判断时复用该确认。否则只给出目的、背景、采用方向、必须长期遵守的限制和必要的直接关系，等待确认后再写入。当前实现差距属于行为 owner 和事实，不作为决策正文中的专门结构保存。

### 4. 写入和改变状态

1. 完整读取固定契约，确认稳定主题、路径、目标对齐状态和维护事务。
2. 按固定契约写自包含 Markdown；无论选择何种对齐状态都使用同一正文结构，必须长期遵守的限制、允许范围或例外直接写在 `决策` 中。已经同时确认多条决策时，可以先写好多条 `createdAt: null` 的未激活候选文件。
3. 按目标对齐状态选择 `activate` 参数：完整决策已经核对满足并应建立为当前基线时使用 `aligned`，否则使用 `unaligned`。每次命令只激活显式目标，命令成功后该活动记录立即生效；其他候选仍未生效并由 CLI 逐条 warning。路径进入 `HEAD` 前仍可在原地收敛或使用 `discard` 放弃。
4. 将完整决策与实际实现、行为 owner 和事实逐项比较，确认已经满足并应建立为当前基线后运行 `mark-aligned <path>`；状态写入不代替实施与核对。
5. 独立退役使用 `archive`；决策演进先归档直接前序，再创建能够独立表达当前结论的新活动记录和真实关系。
6. 编辑性修正后运行 `sync-index --write`；索引缺失或损坏时同一命令从全部已建立的当前格式 Markdown 完整重建领域 state、派生 keys 和 source revision。未激活候选保持在索引外并显示 warning。
7. 所有状态命令都修改 Markdown 并重建索引；关系不隐式改变生命周期或对齐状态。
8. 不确定项会改变决策含义、状态、关系、创建时间或 owner 时，请用户判断。结束维护前必须激活或丢弃全部未激活候选，不能把作用域命令的 warning 当作最终通过。

### 5. 验证与交付

1. 确认记录表达了已经确认的目的、背景、采用方向，以及任何必须长期遵守的限制、允许范围或例外。
2. 检查相关行为 owner：标记 aligned 前必须已有事实证据；保持 unaligned 时按决策正文约束当前动作，并从实际 owner 恢复差距。
3. 写入后运行 `sync-index --write` 所需的同步事务和严格 `check`；状态命令已经自动重建索引，仍以最终 `check` 为验收。逐条激活期间 warning 是允许的中间状态，交付时不得残留未激活候选。
4. 已写入时说明维护类型、生命周期与对齐变化、行为 owner 核对和校验结果；只恢复或审阅时说明相关状态、偏离分类和 warning 边界。

## CLI

从 skill 目录运行，或使用脚本绝对路径：

```text
node scripts/decision-records.mjs <command> [options] --root <resolution-root>
```

日常恢复使用 `list`，单条展开使用 `show`，关系追溯使用 `trace`，结构验证使用 `check`。`sync-index`、`activate`、`mark-aligned`、`archive` 和 `discard` 是写命令，完整读取固定契约后再使用。查询只在通用索引与已登记 Markdown 的 source revision 一致时返回 state；查询和作用域写命令仍可能在退出码为 `0` 时 warning 未激活候选。所有参数、失败语义和输出以固定契约与 `--help` 为准。

需要在现有 ESM 进程中复用时，从已安装 skill 的实际路径导入 `scripts/decision-records.mjs`；公开导出和类型见相邻 `decision-records.d.mts`。

## 完成标准

### 恢复或审阅

1. 已恢复相关活动判断、生命周期和对齐状态；未对齐记录已与相关行为 owner 和当前事实比较。
2. 当前任务已归为遵守基线、向目标推进、一次性例外、长期修订或一致性问题。
3. 结论没有违反决策正文中的明确约束，也没有越过查询 warning 能够证明的范围。

### 候选决策

1. 候选具有长期影响和回放价值，粒度能够独立演进。
2. 面向未来的候选清楚表达采用方向和必须长期遵守的约束，不把当前差距写成决策内容 owner。
3. 未确认候选没有进入索引。

### 维护

1. 只有明确确认的长期取舍进入记录；已建立 Markdown 自包含全部权威状态，索引完全由其生成，未激活候选保持在索引外且不生效。
2. 对齐状态只由 frontmatter 表达并由索引投影，标记 aligned 已将完整决策与行为 owner 和事实核对并建立为当前基线。
3. 路径尚未进入 `HEAD` 的已激活记录已原地收敛或丢弃；所有未激活候选也已激活或丢弃，没有形成虚假历史或遗漏文件。
4. 内容、状态、行为 owner、索引和关系没有未解释冲突。
5. 严格 `check` 返回 `0`；无法运行时按恢复手册完成当前格式恢复，并明确剩余校验。

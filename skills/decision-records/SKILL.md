# Decision Records

## 目标

用普通 Markdown 和版本历史保存已经确认的决策依据，使人类和 agent 能定位当前有效判断，并在需要时回放形成原因。

让 `docs/decisions/` 承接决策的原因、取舍和演进关系。让代码、配置、规范和项目文档继续承接当前事实与行为，不用决策记录替代它们。

## Owner 分工

1. 本文件承接使用流程和语义判断：读取什么、何时记录、怎样修订以及如何交付。
2. [decision-record-rules.md](references/decision-record-rules.md) 承接唯一固定契约：目录、命名、状态、关系、正文模板、索引、格式修复和维护事务。
3. `scripts/decision-records.mjs` 承接确定性的结构检查、状态列表和活动索引同步。
4. 目标工作区的 `docs/decisions/` 承接实际记录；skill 包不保存用户的动态决策内容。

## 读取顺序

1. 先读取目标工作区的 `AGENTS.md`、项目说明和与当前任务相关的行为 owner。
2. 只需恢复上下文时，读取 `decision-record-index.md` 和相关 `active` 记录；当前记录足以说明现行判断时停止。
3. 初始化、修复、创建、修订、状态变更或结构审阅前，完整读取 [decision-record-rules.md](references/decision-record-rules.md)。
4. 需要审计演进原因时，再沿直接关系逐步读取历史；读到归并决策时，将其作为默认追溯边界。

## 执行流程

### 1. 定位并检查体系

1. 优先使用用户指定的决策目录，未指定时使用 `<workspace>/docs/decisions/`。
2. 用户只要求解释、检查或审阅时保持只读。
3. 用户要求建立或记录且目录不存在时，按 reference 创建最小体系。
4. 目录已存在时，对照固定契约检查根 owner、影响面、文件名、状态、关系、正文结构和索引。
5. 已有材料不符合契约时，将当前固定格式作为唯一目标完成修复；保留能够确认的决策语义，完成承接后收敛旧入口，不维护并行格式。

### 2. 判断是否记录

只把已经确认、以后仍会影响工作的取舍写成决策。依次判断：

1. 缺少这条记录时，未来是否难以解释为何采用当前方案而不是其他合理方案？
2. 这项判断是否会改变后续行为、owner、边界、兼容性、风险处理或验收方式？
3. 它是否包含可复用的取舍，而不是事实、状态快照、任务或执行结果？

把用户明确选择、批准或要求长期沿用的方案视为已确认。将建议、推测和开放问题留在讨论或任务 owner，确认后再写入 `active`。

### 3. 读取当前判断

1. 从活动索引按稳定影响面定位相关 `active` 记录，不按日期遍历全部文件。
2. 对照当前行为 owner，区分“现在怎样做”和“为什么形成当前做法”。
3. 把 `active` 记录作为完整的当前判断；关系链接只用于审计历史。
4. 当前指令改变既有判断时，以当前指令为准，并在同一轮更新决策历史。

### 4. 新增、修订或修复

1. 每个文件只记录一个需要回放的判断，并放入未来最可能查找它的稳定影响面。
2. 按 reference 选择最小或完整正文，写清采用方案、关键备选、适用条件、影响和验证方式。
3. 修复已有材料时，根据项目 owner、版本历史和相互引用恢复能够确认的日期、状态与关系；只有继续推断会改变决策含义时才请求用户判断。
4. 新判断改变旧判断时，让新 `active` 记录完整表达变化后的当前结论，只链接本次直接前序；同时更新旧记录状态、文件名和指向新记录的状态来源。
5. 当前结论已经分散或关系链妨碍日常理解时，创建归并决策作为新的日常读取起点，更早历史继续用于审计。
6. 同步当前行为 owner 和活动索引，使现行规则、决策原因与导航一致。

### 5. 审阅与验证

同时完成结构检查和语义检查：

1. 确认内容达到记录门槛，并有明确的确认依据。
2. 确认一份记录只承接一个判断，问题、采用方案、关键取舍、适用条件、影响和验证方式具体可读。
3. 确认 `active` 记录独立表达当前结论，状态与文件名一致，关系只指向真实存在的直接前序。
4. 确认当前行为 owner 已同步，记录没有复制完整现行规范。
5. 确认内容不包含密钥、敏感个人信息、私人路径内容或无助于未来判断的环境标识。
6. 同步活动索引并运行 `check`，逐项修复到退出码为 `0`。

### 6. 交付

1. 列出实际读取、新增、修订、归并、迁移或改变状态的记录。
2. 说明当前任务沿用了哪些有效判断，以及同步了哪些行为 owner。
3. 报告实际运行的校验及结果。
4. 没有写入时，说明是记录门槛不足、尚未确认，还是当前任务要求只读。

## CLI

从 skill 目录运行，或使用脚本的绝对路径：

```text
node scripts/decision-records.mjs check --root <workspace-root>
node scripts/decision-records.mjs list --root <workspace-root>
node scripts/decision-records.mjs list --all --root <workspace-root>
node scripts/decision-records.mjs list --status amended,superseded --root <workspace-root>
node scripts/decision-records.mjs sync-index --write --root <workspace-root>
```

不写命令时默认执行只读 `check`。使用 `--decisions-dir <path>` 指定非默认目录；`sync-index` 只有带 `--write` 时更新索引末尾的 `## 活动决策`。

## 边界

1. 使用普通 Markdown、相对链接和 Git 历史作为公开事实来源，不建立隐藏的决策状态。
2. 状态同时写入文件名和正文，使文件树、索引和正文都能表达有效性。
3. 每个目标目录只维护当前固定契约；既有材料修复完成后，由版本历史保留旧形态。
4. 将完整对话、操作日志、提交摘要和普通实现过程留在各自 owner，只把已确认的长期取舍写入决策记录。

## 完成标准

1. 活动索引能够按影响面定位全部 `active` 记录。
2. 每条 `active` 记录无需拼接前序即可表达当前判断；关系只链接直接前序，归并记录提供明确读取边界。
3. 文件名、正文状态、关系、索引和当前行为 owner 保持一致。
4. 目标目录只保留当前固定契约，`invalidated` 位于专用归档目录。
5. CLI 检查通过，交付说明包含实际语义检查和校验结果。

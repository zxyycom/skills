# Decision Records

## 目标

用显式、可版本化的 Markdown 保存长期取舍和形成原因，用 JSON 索引保存当前仍作为后续工作依据的决策集合。让人类和 agent 在形成可能产生长期影响的决定前快速恢复当前方向，在需要时低成本回放原因，并避免后续维护静默偏离。

决策记录不承接通用记忆、任务日志或当前实现状态；代码、配置、规范和项目文档继续承接当前事实与行为。只在重要长期取舍形成时介入，不反复打断原任务。

## Owner 与读取

1. 本文件承接发现、确认、读取、写入和交付流程。
2. [decision-record-rules.md](references/decision-record-rules.md) 是唯一固定契约，承接目录、JSON 索引、稳定文件名、正文、关系和维护事务。
3. `scripts/decision-records.mjs` 承接当前成员变更、派生字段同步、结构检查和查询。
4. 目标项目的 `docs/decisions/decision-index.json` 承接当前成员；各主题目录中的 Markdown 承接显式索引摘要、完整背景、采用的决策和历史关系。
5. 项目专属的更高记录门槛写入 `AGENTS.md` 或相关行为 owner。

按任务逐步读取：

1. 先读目标工作区的指令文件和相关行为 owner。
2. 用户正在形成可能持续影响后续行为、owner、边界、兼容性、风险处理或验收方式的决定，或拟议决定可能与既有决定冲突时，在确定方向或实施依赖该方向的实质改动前读取 JSON 当前索引，并按路径分类、标题、精简背景和决策定位相关记录。
3. 索引信息足以判断相关性时不展开全部文件；需要完整回放或关系审计时，再读取命中的 Markdown。
4. 创建、修订、逻辑归档、归并、显式修复或结构审阅前，完整读取固定契约。
5. 只有审计演进原因时才沿直接关系读取历史；归并记录是默认追溯边界。

## 执行流程

### 1. 区分任务意图

1. 恢复、解释、检查或审阅保持只读。
2. 当前指令已经表达决策维护意图时，直接执行而不重复询问；新增、调整或退出当前集合前，先告知将写入或改变的判断及集合变化。
3. 用户要求修复非标准格式时，只将已经说明的修复范围视为授权。
4. 普通工作中需要形成可能产生长期影响的决定时，先恢复相关决策；形成新的重要取舍后，在自然更新或交付阶段提出候选，只有必须先确定该取舍才能继续时才即时提出。

写入内容、长期范围或确认意图仍不清楚时，先给出概览并只问一个确认问题。

### 2. 恢复当前判断

1. 从 `decision-index.json` 的 `path`、`title`、`background` 和 `decision` 定位并简要恢复相关当前记录，不按日期遍历全部 Markdown。
2. 将索引成员解释为已经确认并成为后续工作依据的决策；不在索引中的记录只用于历史回放。
3. 从 `背景` 理解当时为什么选择该方向，从 `决定` 恢复采用的做法，再对照当前事实和行为 owner 判断两者是否仍一致；当前成员是后续工作的默认依据，不是脱离背景的永久规则。
4. 将当前任务与相关决策归为以下一种情况：
   - 对齐：按当前判断继续。
   - 明确的一次性例外：按当前指令完成本次任务，说明例外范围，并明确当前索引和历史未改变。
   - 长期修订：形成候选概览；当前指令已经明确要求长期记录或修订时复用该确认，否则等待确认后再写入。
   - 一致性问题：当前记录互相冲突、记录与行为 owner 冲突，或偏离性质不清且会影响实质改动时，报告冲突并请求判断。
5. 当前指令只改变本次任务的执行方式时，归为一次性例外；只有用户已经决定改变后续工作的默认依据时，才归为长期修订。
6. 当前事实与记录背景已经明显不同时，说明变化、受影响判断和需要确认的新取舍，不机械沿用旧背景。
7. 没有当前指令或既有记录提供确认依据时，不自行选择冲突方向，也不静默改变长期行为。

### 3. 提出候选决策

候选判断需要同时具备长期影响和回放价值。依次确认：

1. 它是否会持续作为后续工作依据，或改变长期行为、owner、边界、兼容性、风险处理或验收方式？
2. 缺少记录时，未来是否难以恢复为何采用当前方案？
3. 它是否包含可复用判断或取舍，而不是事实、状态、任务或执行结果？

候选尚未确认时，只给出形成判断所需的最小概览：

1. 背景。
2. 采用的决策。
3. 与既有决策的直接关系；没有关系时省略。

用户确认后，将最终确认的背景和采用决策直接写入 `## 索引摘要` 的对应字段；正文沿用该方向展开原因、约束和细节。

让用户确认、修改或放弃。任务局部选择、一次性例外、建议和开放问题继续留在原 owner。

### 4. 写入或修复

1. 完整读取固定契约，并使用用户指定的决策目录；未指定时使用 `<workspace>/docs/decisions/`。
2. 已确认写入且目录不存在时，创建 JSON 空索引和当前需要的主题分类。
3. 新增独立判断时，写入包含显式索引摘要的稳定路径 Markdown，再运行 `activate` 加入当前集合。
4. 一项当前判断只需退出后续工作依据且没有后续判断时，运行 `archive` 将其移出当前集合，不强制建立关系。
5. 修订、替代、判定无效或归并时，写入完整的新判断，只从新记录链接直接前序，再用 `archive --by` 校验直接关系并更新当前集合；已有文件不重命名或移动。
6. 目录不符合契约时，先报告差异、可保留语义、修复范围和不确定项；得到显式修复确认前保持只读。
7. 修复中的不确定项会改变决策含义、当前成员、关系或 owner 时，请用户判断。

### 5. 验证与交付

1. 按固定契约完成语义检查和结构检查，复核 `索引摘要` 与完整背景、采用决定保持一致，并确认没有未解释的当前决策冲突或不必要的敏感信息。
2. 同步 JSON 派生字段并运行 `check`，修复到退出码为 `0`。
3. 等待确认时，交付候选概览并明确尚未写入。
4. 已写入时，列出新增、修订、归并、逻辑归档或迁移，说明当前集合变化、偏离分类和同步的行为 owner。
5. 没有写入时，说明当前任务属于对齐、一次性例外、门槛不足、尚未确认、只读要求或等待修复授权中的哪一种。

## CLI

从 skill 目录运行，或使用脚本的绝对路径：

```text
node scripts/decision-records.mjs check --root <workspace-root>
node scripts/decision-records.mjs list --root <workspace-root>
node scripts/decision-records.mjs list --archived --root <workspace-root>
node scripts/decision-records.mjs list --all --root <workspace-root>
node scripts/decision-records.mjs trace <topic/file.md> --root <workspace-root>
node scripts/decision-records.mjs trace <topic/file.md> --direction predecessors --depth 1 --root <workspace-root>
node scripts/decision-records.mjs sync-index --write --root <workspace-root>
node scripts/decision-records.mjs activate <topic/file.md> --root <workspace-root>
node scripts/decision-records.mjs archive <old.md...> --root <workspace-root>
node scripts/decision-records.mjs archive <old.md...> --by <new.md> --root <workspace-root>
```

不写命令时默认执行只读 `check`。使用 `--decisions-dir <path>` 指定非默认目录。`trace` 默认双向查询完整关系链，使用 `--direction predecessors|successors|both` 选择方向，使用 `--depth <n>` 限制关系跳数。`sync-index --write` 只刷新标题、精简背景、决策和排序，不改变当前成员。

## 完成标准

1. 只有明确确认的长期取舍进入当前索引；新增、调整或退出当前集合前已经告知用户将写入或改变的判断及集合变化。
2. 形成可能产生长期影响的决定前，或拟议决定与既有决定冲突时，已读取相关当前记录，并结合背景、当前事实和行为 owner 判断是否沿用。
3. JSON 索引包含每条当前判断的 `path`、`title`、精简 `background` 和 `decision`；摘要由 Markdown 显式提供，不重复路径中的分类，也不包含逻辑归档记录。
4. 当前任务的偏离已经归类，没有未说明的一次性例外或静默长期修订。
5. 当前集合、Markdown 内容、按需声明的单向关系和行为 owner 保持一致；关系不存在环，`archive --by` 的后续记录直接引用全部前序，CLI 能够反查关系链，已有决策路径未因生命周期变化而漂移。
6. 非标准格式在未获授权时只形成修复报告，获授权后收敛到唯一固定契约。
7. CLI 检查通过，交付包含实际语义检查和校验结果。

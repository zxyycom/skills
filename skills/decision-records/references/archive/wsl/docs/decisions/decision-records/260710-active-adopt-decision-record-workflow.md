# 2026-07-10 - 采用可回放的决策记录流程

## 状态
- 当前状态: active
- 导致状态变化的决策: 无
- 状态说明: 作为本仓库当前的决策记录组织、状态追踪和校验方式使用。

## 问题
- WSL 设置会同时涉及 Windows 宿主、Linux 发行版、权限、生效条件和恢复方式；只保留最终配置或提交差异，后续难以回放重要方案为什么成立。
- 如果把每次配置值变化都写成决策，真正影响长期维护方式的判断又会被普通变更淹没。

## 背景与约束
- 本仓库既要保存期望配置，也要让后续维护者和 agent 能安全判断配置属于哪一侧、如何应用以及如何验证。
- 决策记录需要服务当前仓库，而不是充当操作日志、系统快照或提交历史的副本。
- 项目脚本统一通过 Bun 运行；决策结构校验不应为这项基础能力引入第三方运行时依赖。

## 决定
- 采用: `docs/decisions/` 按稳定影响面组织决策，使用独立的规则 owner 和索引 owner，并让每条决策只承接一个需要回放的长期判断。
- 采用: 文件名同时包含短日期、状态和英文短标题；正文记录完整日期、状态关系、问题、决定、影响和验证，复杂决策按需补充背景与决策过程。
- 采用: 决策只记录会改变配置 owner、宿主与发行版边界、兼容性、应用与恢复方式、自动化契约或安全边界的长期判断。
- 采用: 使用 TypeScript 编写决策校验器，通过 `bun run validate:decisions` 执行，并只使用 Bun 兼容的内置 API。
- 不采用: 使用单个按时间追加的决策日志。原因是影响面和状态会比时间顺序更适合后续查找当前有效依据。
- 不采用: 为每次配置值变化新增决策。原因是普通调整应由配置差异和提交说明承接。
- 触发条件: 后续出现需要解释“为什么长期采用当前 WSL 管理方式而不是另一种合理方案”的判断时，沿用本流程记录。

## 影响
- agent 在改变长期 WSL 管理契约前需要先查索引和相关影响面，再判断是否新增或更新决策。
- 历史判断可以保留回放价值，同时通过 `active`、`amended`、`superseded` 和 `invalidated` 区分当前有效性。
- 普通设置调整继续保持轻量，不把决策目录变成变更日志。

## 验证
- `docs/decisions/decision-record-rules.md` 定义记录门槛、结构、状态和更新流程。
- `docs/decisions/decision-record-index.md` 列出影响面与本决策。
- `scripts/validate-decisions.ts` 可以通过 `bun run validate:decisions` 校验当前决策目录。

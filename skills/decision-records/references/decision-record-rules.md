# 决策记录方案

本文件定义可在不同项目中复用的固定决策记录契约。项目可以增加更严格的语义门槛和领域说明；目录、文件名、状态、结构字段、索引关系和校验语义保持一致。

## 设计原则

1. 显式：使用普通 Markdown、目录、文件名和相对链接表达全部决策状态，不依赖隐藏存储。
2. 可审核：让人类和 agent 能直接阅读问题、取舍、适用条件和验证方式。
3. 可优化：保留所有已形成的决策，通过状态、关系和归档位置修订旧判断。
4. 简单：一条记录承接一个判断，固定最小结构，只在确有价值时增加背景与过程。
5. 可移植：不同项目采用同一结构契约，只让领域内容和记录门槛随项目变化。

## 目录与 owner

```text
docs/decisions/
├── decision-record-index.md
├── decision-record-rules.md
├── <impact-area-id>/
│   └── YYMMDD-<status>-short-title.md
└── archive/
    └── <impact-area-id>/
        └── YYMMDD-invalidated-short-title.md
```

1. `decision-record-index.md` 承接阅读方式、状态速查、影响面说明和当前 `active` 决策链接。
2. `decision-record-rules.md` 承接当前项目采用的记录门槛、固定格式和维护流程。
3. 每个影响面目录承接该稳定责任边界下的决策文件。
4. 当前配置、规范和行为继续由各自 owner 承接；决策文件解释形成原因和适用条件。
5. `archive/<impact-area-id>/` 只保存 `invalidated` 决策；没有失效记录时不创建归档目录。
6. 只创建当前需要的影响面，不预建空目录。

## 记录门槛

满足以下条件之一，且判断已经被用户确认时，通常值得记录：

1. 未来如果看不到原因，会难以解释为什么采用当前方案而不是另一个合理方案。
2. 判断会改变后续行为、owner、数据边界、兼容性、安全边界或验收方式。
3. 判断沉淀了以后会复用的偏好、取舍标准或长期约束。
4. 新判断会修订、替代或判定既有决策无效。

以下内容通常留在其他 owner：

1. 普通事实、状态快照、任务清单、提交摘要和执行日志。
2. 未确认建议、开放问题、头脑风暴和 agent 推断出的偏好。
3. 一次性实现细节、普通文字修正、格式调整和已有规则的重复说明。

## 影响面

影响面表示稳定责任边界。

1. 使用 kebab-case 名词或名词短语。
2. 不包含日期、阶段、版本、工单号或一次性动作。
3. 同时影响多个边界时，选择未来最可能查找该判断的主要影响面，并在正文中说明其他影响。
4. 新增影响面时，同步在索引中解释该 ID 的职责。

## 状态

1. `active`：当前仍完整生效。
2. `amended`：仍有回放价值，但部分内容已被后续决策修订。
3. `superseded`：已被后续决策完整替代，不再作为当前规则执行。
4. `invalidated`：因冲突、前提错误或无效结论不再作为依据。

状态同时写入文件名和正文。

1. 新决策保持 `active`。
2. 非 `active` 记录必须在 `导致状态变化的决策` 字段使用内联 Markdown 相对链接，指向导致变化的后续决策。
3. `active` 记录的状态来源固定写 `无`。
4. `superseded` 表示正常演进后的完整替代；只有旧结论不应再作为依据时使用 `invalidated`。
5. `active`、`amended` 和 `superseded` 保留在原影响面；`invalidated` 移入 `archive/<impact-area-id>/`。

## 文件名与标题

```text
YYMMDD-<status>-short-title.md
```

1. 日期使用决策首次形成时的本地日期，并按 `20YY-MM-DD` 写入正文标题。
2. `<status>` 使用四个固定状态之一。
3. `short-title` 使用简短英文 kebab-case，表达判断而不是任务。
4. 正文第一行固定为 `# YYYY-MM-DD - <标题>`。
5. 状态变化时重命名文件，并同步所有索引和状态来源链接。

## 最小正文

固定结构字段使用下列中文标签；标题和正文内容可以使用项目主要语言。

```markdown
# YYYY-MM-DD - <标题>

## 状态
- 当前状态: <active | amended | superseded | invalidated>
- 导致状态变化的决策: <active 写“无”；其他状态使用后续决策的内联 Markdown 相对链接>
- 状态说明: <当前怎样使用这条记录>

## 问题
- <为什么这项判断值得长期保留>

## 决定
- 采用: <最终规则、结构或做法>
- 不采用: <关键备选和原因；没有明显备选时省略>
- 触发条件: <以后何时沿用或重新评估>

## 影响
- <未来行为或维护方式怎样变化>

## 验证
- <如何确认决定已落地>
```

1. 必需小节为 `状态`、`问题`、`决定`、`影响` 和 `验证`。
2. `状态` 必须包含三个固定字段。
3. `决定` 必须包含 `采用` 和 `触发条件`；`不采用` 在没有关键备选时可以省略。
4. 每个已出现的小节必须包含实际内容。

## 完整正文

需要回放关键上下文或多轮收敛时，在 `问题` 和 `决定` 之间按顺序增加：

```markdown
## 背景与约束
- <当时已经成立且会影响判断的边界>

## 决策过程
1. <初始问题或备选>
2. <新增约束、反例或用户确认>
3. <最终收敛>
```

1. 两个扩展小节均可独立省略。
2. 省略时不保留空标题。
3. `决策过程` 只保留会帮助未来重新判断的收敛过程，不写普通执行流水。

## 索引

`decision-record-index.md` 至少包含：

1. 阅读方式：先按影响面定位，再按状态判断有效性。
2. 状态速查：解释四个固定状态。
3. 影响面：用内联代码列出每个已存在影响面及其责任边界。
4. 活动决策：在文档最后使用 `## 活动决策`，按影响面列出当前 `active` 记录的内联 Markdown 相对链接。

示例：

```markdown
`workflow-policy`：

1. [active: 2026-07-10 - 采用显式审批门禁](workflow-policy/260710-active-use-explicit-approval-gate.md)
```

1. `## 活动决策` 必须是索引最后一个二级小节，由 CLI 生成和同步。
2. 默认索引只覆盖 `active`；使用 `list --all` 或 `list --status` 查看保留的历史状态。
3. 新增、重命名或改变状态后运行 `sync-index --write`，不手工维护生成区。

## 维护事务

### 新增

1. 选择或创建稳定影响面。
2. 写入新的 `active` 决策。
3. 在索引中说明新影响面。
4. 同步当前行为 owner。
5. 运行 `sync-index --write` 和 `check`。

### 修订、替代或判定无效

1. 写入导致变化的新 `active` 决策。
2. 将旧文件重命名为 `amended`、`superseded` 或 `invalidated`。
3. 更新旧正文状态、状态说明和来源链接。
4. 如果状态是 `invalidated`，将旧文件移入 `archive/<impact-area-id>/`。
5. 搜索和修复重命名或移动造成的所有相对链接变化。
6. 同步当前行为 owner。
7. 运行 `sync-index --write` 和 `check`。

所有已形成的决策默认保留。后续判断改变其有效性时更新状态；完全不应再作为依据时标为 `invalidated` 并归档，不通过删除抹去历史。

## 校验契约

`scripts/decision-records.mjs` 提供三个命令：

1. `check`：只读检查目录、文件、正文、链接、归档位置和活动索引。
2. `list`：默认列出 `active`，通过 `--all` 或 `--status` 显式查看历史。
3. `sync-index`：默认检查索引漂移；`--write` 只更新索引最后的活动决策区。

`check` 检查以下确定性约束：

1. 决策目录、两个根 owner 文件和影响面目录结构。
2. 影响面 kebab-case、非空和无嵌套目录。
3. 决策文件名、日期、状态和正文标题一致。
4. 必需小节存在、顺序正确且非空。
5. 状态与决定小节包含固定字段，正文状态与文件名一致。
6. 索引说明每个影响面，活动决策区与全部 `active` 文件一致且不包含其他状态。
7. 非 `active` 状态来源使用内联相对链接，目标位于决策目录且真实存在。
8. `invalidated` 只位于 `archive/<impact-area-id>/`，归档目录不包含其他状态。

校验器不检查：

1. 判断是否值得记录、是否已经被用户确认或是否合理。
2. 正文事实是否真实、取舍是否充分或结论是否正确。
3. 当前 owner 是否已完成语义同步。
4. 内容是否包含隐私信息。
5. 何时应把判断改为 `invalidated`。

这些语义检查由使用 skill 的 agent 和用户负责。

## 内容边界

1. 只写完成未来判断所需的最小上下文。
2. 不写密钥、令牌、敏感个人信息、私人路径内容或不必要的本机标识。
3. 当前指令与历史决策冲突时，以当前指令为准，并通过新决策显式更新历史。
4. 让决策目录保持可直接阅读；不把必要状态藏入 frontmatter、数据库、缓存或脚本输出。
5. CLI 不维护隐藏状态；唯一写操作是用户显式调用 `sync-index --write` 后更新公开索引。

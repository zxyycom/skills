# 决策记录契约

本文件是 `decision-records` 的唯一固定契约，只承接精确存储和维护约束。触发、语义恢复、偏离判断、候选确认、CLI 调用时机和交付流程由 `SKILL.md` 承接。目标项目不复制本契约，目录、索引、文件名、正文、关系和校验语义保持一致。

## Owner 与内容边界

1. `docs/decisions/decision-index.json` 是“哪些决策当前仍作为后续工作依据”的唯一 owner。
2. 各分类目录中 Markdown 的 `目的`、`背景` 和 `决策` 完整章节承接完整语义，`索引摘要` 承接作者确认的精简投影，`关系` 承接历史演进。
3. 代码、配置、规范和项目文档承接当前事实与行为；决策记录承接希望长期达成的结果、形成判断的背景和最终采用的方向。
4. CLI 承接 JSON 索引的成员变更、派生字段同步、确定性检查和查询，不保存缓存或隐藏状态。
5. 只保留未来判断所需的最小上下文；排除任务清单、执行日志、完整对话、提交摘要、密钥、令牌、敏感个人信息、私人路径内容和无助于判断的本机标识。

## 目录与稳定身份

```text
docs/decisions/
├── decision-index.json
└── <topic-id>/
    └── YYMMDD-short-title.md
```

1. 根目录只保留 `decision-index.json` 和主题分类目录，不建立物理归档目录。
2. 主题目录是便于查找的稳定分类，使用 kebab-case 名词或名词短语，不表示当前实现 owner。
3. 分类排除日期、阶段、版本、工单号和一次性动作；同时涉及多个主题时选择主要主题。
4. 文件名使用决策首次形成时的本地短日期和简短英文 kebab-case 标题，例如 `260713-use-json-current-index.md`。
5. 文件路径是决策的稳定身份。创建后不因退出当前集合、关系变化、标题润色或 owner 调整而重命名或移动。
6. 决策退出索引后仍保留在原主题分类目录；“归档”只表示不再属于 JSON 当前集合。

## JSON 当前索引

`decision-index.json` 固定使用：

```json
{
  "schemaVersion": 2,
  "current": [
    {
      "path": "workflow-policy/260710-use-explicit-approval-gate.md",
      "title": "采用显式审批门禁",
      "purpose": "让高风险操作在执行前经过一致、可审计的确认。",
      "background": "审批边界分散在多个入口，后续维护容易产生不一致。",
      "decision": "使用统一的显式审批门禁。"
    }
  ]
}
```

1. `current` 只列出已经确认并成为后续工作依据的当前决策；不在数组中的 Markdown 记录视为逻辑归档。
2. `path` 成员关系是当前集合的唯一来源；Markdown 正文和文件名不重复保存状态。
3. `path` 的第一段已经表达主题分类，索引不再重复保存 `area`。
4. `title` 从正文标题生成，`purpose`、`background` 和 `decision` 分别从 `## 索引摘要` 的同名字段生成。
5. 通过 CLI 改变成员关系或刷新派生字段；不直接维护生成字段。
6. JSON 使用 UTF-8、两空格缩进、稳定排序和文件末尾换行。
7. 删除或损坏索引会丢失当前成员判断，因此 CLI 必须校验 schema、路径、重复项和派生字段一致性；Git 历史承接恢复依据。

## 正文结构

正文第一行固定为：

```text
# YYYY-MM-DD - <标题>
```

最小正文：

```markdown
# YYYY-MM-DD - <标题>

## 索引摘要
- 目的: <足以判断期望结果的精简目的>
- 背景: <足以判断是否相关的精简背景>
- 决策: <完整表达采用方向的精简决策>

## 目的
- <希望长期达成或维护的结果；必要时展开对象、范围和优先级>

## 背景
- <当时需要作出选择的背景，以及足以理解原因的关键事实>

## 决策
- 采用: <最终做法，以及支持它的核心理由>
```

正文规则：

1. 必需小节为 `索引摘要`、`目的`、`背景` 和 `决策`，且均包含实际内容。
2. `索引摘要` 包含且仅包含一个 `目的`、一个 `背景` 和一个 `决策` 字段；三者由作者明确概括，CLI 不从正文位置或顺序猜测摘要。
3. 三个完整章节分别承接完整语义：`目的` 写希望长期达成或维护的结果，`背景` 写促成选择的事实、问题和关键约束，`决策` 写最终采用的方向和核心理由。
4. `索引摘要` 是完整章节的压缩投影：摘要目的足以判断期望结果，摘要背景足以判断记录是否相关，摘要决策完整表达采用方向。摘要不得引入对应完整章节没有明确表达的独立语义。
5. 创建或修改记录时，同步更新完整章节与对应摘要；完整章节可以展开对象、范围、原因、约束和细节，影响相关性或采用方向的限定同时写入摘要。
6. `目的` 简单时可以只有一句话，但不能省略或只由背景隐含；需要说明对象、范围或优先级时在完整章节展开。
7. `决策` 至少包含一个 `采用`；关键备选确实有助于回放时可以增加 `不采用`。
8. `背景` 只保留足以理解选择原因的事实和约束；`采用` 写清最终做法和核心理由。
9. 不单设适用场景或重评条件，也不预测未来变化；决策本身已有的对象、范围和边界写入目的、背景或决策。未来任务结合当时目的、背景、当前事实和行为 owner 判断是否继续沿用。
10. `关系` 按需使用，省略时不保留空标题。

## 关系与逻辑归档

新判断改变已有判断时，在正文最后增加 `## 关系`：

```markdown
## 关系
- 修订: [<直接被部分修订的决策>](relative/path.md)
- 替代: [<直接被完整替代的决策>](relative/path.md)
- 判定无效: [<直接被判定无效的决策>](relative/path.md)
- 归并: [<被归并为当前结论的直接前序>](relative/path.md)
```

1. 所有关系从新记录指向本次直接作用的前序：
   - `修订`：保留前序的主体方向，同时改变其中一部分。
   - `替代`：用新的完整判断取代前序，前序不再构成当前结论。
   - `判定无效`：前序所依据的前提、证据或推理不成立，不再作为后续依据。
   - `归并`：把分散的直接前序整合为一条能够独立使用的当前判断。
2. 关系只由新记录单向保存；CLI 通过反向扫描恢复指向本记录的后续决策，并沿两个方向查询关系链。
3. 前序关系的目标必须已经退出 JSON 当前集合；没有后续判断的逻辑归档记录可以不声明关系。
4. 关系不指向自身，不形成环，同一关系类型不重复声明同一前序。
5. 当前记录独立表达完整的当前判断；部分修订时不要求读者与前序自行拼接。
6. 当前结论已经分散或日常理解必须反复追溯时，创建归并决策完整重述当前结论，并将直接承接的记录退出索引。

## 维护事务

所有写入事务都同步当前行为 owner、关系和 JSON 索引，最后运行 `check`。

### 初始化或新增

1. 创建 `decision-index.json`，使用空 `current` 数组。
2. 确认待记录判断已经得到明确确认。
3. 选择稳定主题分类，写入显式索引摘要和完整目的、背景与决策。
4. 运行 `activate <path>` 将新记录加入当前集合。

### 独立逻辑归档

1. 确认一项当前判断不再作为后续工作依据，且没有需要记录的后续判断。
2. 同步当前行为 owner，再运行 `archive <path...>` 将记录移出当前集合。
3. 原文件保留在稳定路径；没有真实演进关系时不新增关系或占位记录。

### 修订、替代、判定无效或归并

1. 写入完整表达新当前结论的新记录，并链接直接前序。
2. 运行 `archive <old-path...> --by <new-path>`；CLI 确认后续记录直接引用全部前序，再在一次校验式索引更新中移除前序并激活后续。
3. 不重命名或移动任何已有记录。

### 显式修复非标准格式

1. 保持只读，清点已有决策、索引、相互引用和当前行为 owner，区分长期判断与任务、日志、规范或普通说明。
2. 报告不符合项、可保留语义、稳定路径、关系、当前成员映射和仍需判断的不确定项。
3. 只有用户明确确认修复范围后才执行转换；普通记录、解释、检查或维护请求不自动授权迁移。
4. 按本契约完成正文转换、稳定命名、关系恢复和 JSON 当前集合迁移，并收敛旧索引、状态字段和物理归档目录。
5. 不确定项会改变决策含义、当前成员、关系或 owner 时请求用户判断。

## CLI

`scripts/decision-records.mjs` 提供：

1. `check`：只读检查目录、稳定文件名、正文、关系和 JSON 当前索引。
2. `list`：默认只列当前记录；`--archived` 只列逻辑归档，`--all` 同时列出两者。
3. `trace <path>`：从指定记录出发查询关系链；默认沿直接前序和 CLI 推导的后续关系双向遍历，使用 `--direction predecessors|successors|both` 选择方向，使用 `--depth <n>` 限制关系跳数。
4. `sync-index`：默认检查 JSON 派生字段漂移；`--write` 只刷新 `title`、`purpose`、`background`、`decision` 和排序，不改变成员关系。
5. `activate <path>`：把已有 Markdown 记录加入当前集合。
6. `archive <path...> [--by <path>]`：把一条或多条记录移出当前集合；指定 `--by` 时，后续记录必须直接引用全部前序，并在同一次校验式更新中进入当前集合。

CLI 先生成完整候选索引，写入后重新运行完整校验，校验失败时恢复原索引。这属于校验式更新，不提供进程或系统中断级的原子保证；Git 历史承接异常恢复。

`check` 确定性检查：

1. 目录根文件、主题分类目录、稳定文件名和正文标题日期。
2. `索引摘要`、唯一的摘要目的、背景与决策、完整 `目的`、`背景`、`决策`、`采用`、小节顺序和非空内容。
3. JSON schema、路径唯一性、文件存在性、当前成员，以及标题、精简目的、背景和决策派生字段。
4. 关系目标的生命周期以及关系是否形成环。
5. 关系链接位于决策目录内、使用相对 POSIX 路径且目标存在。

索引摘要与对应完整章节的语义一致性、`SKILL.md` 定义的记录门槛、理由质量、当前 owner 同步、隐私、关系是否确为直接前序和归并时机由 agent 与用户完成语义检查。

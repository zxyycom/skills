# 决策记录契约

本文件是 `decision-records` 的唯一固定契约。目标项目不复制或改写本契约；项目可以在 `AGENTS.md` 或相关行为 owner 中增加更严格的记录门槛，但目录、文件名、状态、关系、正文结构、索引和校验语义保持一致。

## Owner 与内容边界

1. 本文件承接全部固定契约，不在目标项目建立第二份格式规则或契约导航。
2. 目标项目的 `decision-record-index.md` 承接阅读方式、状态速查、影响面说明和当前 `active` 决策链接。
3. 每个影响面目录承接该稳定责任边界下的决策文件。
4. 代码、配置、规范和项目文档承接当前事实与行为；决策文件只承接形成原因、关键取舍和演进关系。
5. 项目专属的更高记录门槛由 `AGENTS.md` 或相关行为 owner 承接，不复制本文件的固定结构。
6. 使用普通 Markdown、相对链接、文件名和 Git 历史表达公开状态，不建立数据库、缓存或隐藏状态。
7. 只保留未来判断所需的最小上下文；排除任务清单、执行日志、完整对话、提交摘要、密钥、令牌、敏感个人信息、私人路径内容和无助于判断的本机标识。

## 记录门槛

只记录已经明确确认将作为后续工作依据的判断。确认后满足以下任一条件时，通常值得记录：

1. 缺少记录会让未来难以解释为何采用当前方案而不是其他合理方案。
2. 判断会改变后续行为、owner、数据边界、兼容性、安全边界、风险处理或验收方式。
3. 判断沉淀了可复用的偏好、取舍标准或长期约束。
4. 新判断会修订、替代、判定既有决策无效或归并分散的当前结论。

以下内容留在原 owner：

1. 普通事实、状态快照、任务和执行结果。
2. 尚未确认的建议、开放问题、头脑风暴和 agent 推断。
3. 一次性实现细节、任务局部选择、文字修正、格式调整和已有规则的重复说明。

一条记录只承接一个能够独立修订或替代的判断。共享同一确认主体、适用条件和重评条件的内容可以留在同一条记录；否则拆分。

## 目录与影响面

```text
docs/decisions/
├── decision-record-index.md
├── <impact-area-id>/
│   └── YYMMDD-<status>-short-title.md
└── archive/
    └── <impact-area-id>/
        └── YYMMDD-invalidated-short-title.md
```

目录规则：

1. 根目录只保留 `decision-record-index.md`、影响面目录和可选的 `archive/`。
2. 只创建当前需要的影响面，不预建空目录。
3. `active`、`amended` 和 `superseded` 位于原影响面。
4. `invalidated` 位于 `archive/<impact-area-id>/`；没有失效记录时省略 `archive/`。

影响面表示未来最可能查找该判断的稳定责任边界：

1. 使用 kebab-case 名词或名词短语。
2. 排除日期、阶段、版本、工单号和一次性动作。
3. 同时影响多个边界时，选择主要影响面，并在正文说明其他影响。
4. 新增影响面时，在索引中同步说明该 ID 的职责。

## 状态与关系

状态同时写入文件名和正文：

1. `active`：已经确认并成为后续工作依据的已生效决策；不是进行中的任务状态。
2. `amended`：部分内容已被后续决策修订，继续保留历史价值。
3. `superseded`：已被后续决策完整替代，不再作为当前规则执行。
4. `invalidated`：因冲突、前提错误或无效结论不再作为依据。

状态规则：

1. 新决策使用 `active`；实现进度由任务或行为 owner 承接。
2. `active` 的 `导致状态变化的决策` 固定写 `无`。
3. 非 `active` 记录在 `导致状态变化的决策` 中使用内联 Markdown 相对链接，指向造成变化的后续决策；多个后续决定使用多个链接。
4. `superseded` 表达正常演进后的完整替代；`invalidated` 表达旧结论不再具备依据。
5. 已形成的记录默认保留，通过状态、关系和归档位置表达后续有效性。

新 `active` 记录改变已有判断时，在 `## 状态` 的三个固定字段之后增加 `### 关系`：

```markdown
### 关系
- 修订: [<直接被部分修订的决策>](relative/path.md)
- 替代: [<直接被完整替代的决策>](relative/path.md)
- 判定无效: [<直接被判定无效的决策>](relative/path.md)
- 归并: [<被归并为当前结论的直接前序>](relative/path.md)
```

关系规则：

1. 只保留实际发生的关系类型，每条关系使用相对当前文件的内联 Markdown 链接。
2. 只链接本次直接作用的前序；更早祖先继续由前序记录承接。
3. 新记录的关系与旧记录的状态及状态来源保持一致，形成双向导航。
4. 每条 `active` 记录独立表达完整的当前判断；日常读取不需要打开前序。
5. 部分修订时，新记录写出修订后的完整结论，不要求读者与 `amended` 记录自行拼接。
6. 当前结论已经分散、继续修订需要累计祖先链接或日常理解必须反复追溯时，创建归并决策，将直接承接的记录改为 `superseded`。
7. 归并决策成为新的日常读取起点，只链接直接前序；更早历史仅用于审计。
8. 没有改变已有判断的新决策省略 `### 关系`。

## 文件名与正文

文件名固定为：

```text
YYMMDD-<status>-short-title.md
```

1. 日期使用决策首次形成时的本地日期；`260710` 表示 `2026-07-10`。
2. `<status>` 使用四个固定状态之一。
3. `short-title` 使用简短英文 kebab-case，表达判断而不是任务。
4. 正文第一行固定为 `# YYYY-MM-DD - <标题>`。
5. 状态变化时重命名文件，并同步索引、状态来源和其他相对链接。

固定字段使用下列中文标签；标题和正文内容使用项目主要语言。

最小正文：

```markdown
# YYYY-MM-DD - <标题>

## 状态
- 当前状态: <active | amended | superseded | invalidated>
- 导致状态变化的决策: <active 写“无”；其他状态链接后续决策>
- 状态说明: <当前怎样使用这条记录>

## 问题
- <触发问题、确认背景和为什么值得长期保留>

## 决定
- 采用: <最终做法，以及支持它的核心理由>
- 不采用: <关键备选和原因；没有明显备选时省略>
- 触发条件: <适用范围，以及哪些变化需要重新评估>

## 影响
- <未来行为或维护方式怎样变化>

## 验证
- <如何确认决定已落地>
```

需要回放关键上下文或多轮收敛时，在 `问题` 和 `决定` 之间按顺序增加：

```markdown
## 背景与约束
- <当时已经成立且会影响判断的边界>

## 决策过程
1. <初始问题或备选>
2. <新增约束、反例或用户确认>
3. <最终收敛>
```

正文规则：

1. 必需小节为 `状态`、`问题`、`决定`、`影响` 和 `验证`，且均包含实际内容。
2. `状态` 包含三个固定字段；`决定` 包含 `采用` 和 `触发条件`。
3. `采用` 同时写清最终做法和核心理由。
4. `不采用` 只记录会帮助未来重新判断的关键备选。
5. `触发条件` 同时写清适用范围和重新评估时机。
6. `问题` 写触发原因和确认背景，`影响` 写未来变化，`验证` 写可执行的确认方式。
7. `背景与约束` 与 `决策过程` 按需使用，省略时不保留空标题。

## 索引

`decision-record-index.md` 至少包含：

1. 阅读方式：先按影响面定位，再按状态判断有效性。
2. 状态速查：解释四个固定状态。
3. 影响面：使用内联代码列出每个现有影响面及其职责。
4. 活动决策：在文档最后使用 `## 活动决策`，按影响面列出全部 `active` 记录。

活动决策示例：

```markdown
`workflow-policy`：

1. [active: 2026-07-10 - 采用显式审批门禁](workflow-policy/260710-active-use-explicit-approval-gate.md)
```

1. `## 活动决策` 是索引最后一个二级小节，由 CLI 生成和同步。
2. 默认索引只覆盖 `active`；使用 `list --all` 或 `list --status` 查看其他状态。
3. 新增、重命名或改变状态后运行 `sync-index --write`。

## 维护事务

所有写入事务都同步当前行为 owner、受影响链接和活动索引，最后运行 `check`。

### 初始化或新增

1. 创建或确认索引 owner，不在目标项目复制固定契约。
2. 确认待记录判断已经得到明确确认。
3. 选择稳定影响面，写入新的 `active` 决策。
4. 在索引中说明新增影响面。

### 修订、替代或判定无效

1. 写入完整表达新当前结论的 `active` 决策，并链接直接前序。
2. 将旧文件重命名为对应状态，更新正文状态、状态说明和指向新记录的状态来源。
3. 将 `invalidated` 移入 `archive/<impact-area-id>/`。

### 归并

1. 完整重述当前结论，创建新的 `active` 归并决策。
2. 只链接直接承接的记录，并将这些记录改为 `superseded`。
3. 将归并记录作为新的日常读取起点，更早历史继续用于审计。

### 显式修复非标准格式

1. 保持只读，清点已有决策、索引、相互引用和当前行为 owner，区分长期判断与任务、日志、规范或普通说明。
2. 报告不符合项、可保留语义、重命名或移动范围、关系与索引影响，以及仍需判断的不确定项。
3. 只有用户明确确认修复范围后才执行转换；普通记录、解释、检查或维护请求不自动授权迁移。
4. 将达到门槛的判断映射为一条记录一个判断，保留能够确认的首次形成日期、问题、决定、取舍、状态和验证依据。
5. 使用项目 owner、版本历史和相互引用恢复有效性与直接关系；不确定项会改变决策含义、当前有效性、关系或 owner 时请求用户判断。
6. 按本契约完成归类、正文转换、重命名、关系恢复、链接修复和索引同步。
7. 完整承接内容后收敛旧目录、旧索引和重复记录，不维护并行格式。

## 语义检查

1. 判断已经明确确认并达到记录门槛。
2. 一份记录只承接一个能够独立修订或替代的判断。
3. 核心理由、关键备选和触发条件足以支持未来重新判断。
4. `active` 记录独立表达当前已生效结论，状态、关系和当前行为 owner 语义一致。
5. 关系只链接真实存在的直接前序，归并时机能够限制日常读取深度。
6. `验证` 给出可执行检查，内容符合隐私和内容边界。

## CLI 校验

`scripts/decision-records.mjs` 提供：

1. `check`：只读检查目录、文件、正文、状态来源链接、归档位置和活动索引。
2. `list`：默认列出 `active`，通过 `--all` 或 `--status` 查看历史。
3. `sync-index`：默认检查索引漂移；`--write` 更新索引最后的活动决策区。

`check` 确定性检查：

1. 决策目录、索引 owner 和影响面目录结构。
2. 影响面 kebab-case、非空且无嵌套目录。
3. 文件名、日期、状态和正文标题一致。
4. 必需小节存在、顺序正确且非空，固定字段齐全。
5. 活动索引与全部 `active` 文件一致。
6. 非 `active` 状态来源链接真实存在且位于决策目录。
7. `invalidated` 只位于归档，归档只包含 `invalidated`。

记录门槛、理由质量、当前 owner 同步、隐私、关系直接性和归并时机由 agent 与用户完成语义检查。

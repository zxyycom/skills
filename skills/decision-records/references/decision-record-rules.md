# 决策记录契约

本文件是 `decision-records` 的唯一固定契约，只承接精确存储和维护约束。触发、语义恢复、偏离判断、候选确认、CLI 调用时机和交付流程由 `SKILL.md` 承接。本文将保存索引和 Markdown 的目录称为“决策根目录”；它默认是工作区内的 `docs/decisions`，可以通过 `--decisions-dir <path>` 选择工作区内的其他位置。目标项目不复制本契约，所选根目录内的索引、文件名、正文、关系和校验语义保持一致。

## Owner 与内容边界

1. `<decision-root>/decision-index.json` 是全部决策的生命周期状态、秒级创建时间、检索投影和直接关系的集中 owner。
2. 各分类目录中 Markdown 的 `目的`、`背景` 和 `决策` 完整章节承接完整语义，`索引摘要` 承接作者确认的精简投影，`关系` 承接历史演进。
3. 代码、配置、规范和项目文档承接当前事实与行为；决策记录承接希望长期达成的结果、形成判断的背景和最终采用的方向。
4. CLI 承接索引登记、生命周期变化、派生字段与关系同步、确定性检查和带诊断的查询，不保存缓存或隐藏状态。
5. 只保留未来判断所需的最小上下文；排除任务清单、执行日志、完整对话、提交摘要、密钥、令牌、敏感个人信息、私人路径内容和无助于判断的本机标识。

## 目录与稳定身份

```text
<decision-root>/
├── decision-index.json
└── <topic-id>/
    └── <semantic-slug>.md
```

1. 同一决策集合的查询和维护始终指向同一个决策根目录；未指定 `--decisions-dir` 时使用默认目录。
2. 决策根目录只保留 `decision-index.json` 和主题分类目录，不建立物理归档目录。
3. 主题目录是便于查找的稳定分类，使用 kebab-case 名词或名词短语，不表示当前实现 owner。
4. 分类排除日期、阶段、版本、工单号和一次性动作；同时涉及多个主题时选择主要主题。
5. 新记录文件名使用简短英文 kebab-case 语义名称，例如 `use-lifecycle-index.md`；文件名和正文标题都不保存日期。
6. 已经存在的 `YYMMDD-short-title.md` 及对应 `# YYYY-MM-DD - <标题>` 作为兼容身份继续有效，普通维护不重命名；是否迁移及如何恢复时间属于显式迁移任务。
7. 在同一决策集合内，相对决策根目录的文件路径是决策的稳定身份，根目录本身不进入身份。创建后不因归档、关系变化、标题润色或 owner 调整而改变相对路径。
8. 归档记录仍保留在原主题分类目录；生命周期只由索引中的 `status` 表达。
9. 决策根目录整体尚不存在时，集合视为尚未初始化，查询不创建任何文件。已有决策材料但索引缺失或无法解析时，Markdown 不能替代生命周期、创建时间和集中关系来源。

## JSON 全生命周期索引

`decision-index.json` 固定使用：

```json
{
  "schemaVersion": 3,
  "records": [
    {
      "path": "workflow-policy/use-explicit-approval-gate.md",
      "status": "active",
      "createdAt": "2026-07-20T11:47:38Z",
      "title": "采用显式审批门禁",
      "purpose": "让高风险操作在执行前经过一致、可审计的确认。",
      "background": "审批边界分散在多个入口，后续维护容易产生不一致。",
      "decision": "使用统一的显式审批门禁。",
      "relations": []
    }
  ]
}
```

1. `records` 登记集合内全部活动和归档记录；每个 Markdown 必须有且只有一个索引条目，每个索引条目必须存在对应 Markdown。
2. `status` 只能是 `active` 或 `archived`。活动记录是后续工作的默认依据，归档记录只用于历史恢复和审计。
3. `createdAt` 是索引拥有的不可变秒级元数据，使用带显式时区且不含小数秒的 RFC 3339 时间戳。它为记录提供参考时间和同日多条记录的可比较先后，不承担判断形成瞬间的取证语义。CLI 首次登记新记录时使用激活时刻；既有记录的时间恢复由显式迁移处理。
4. `title` 从正文标题生成，`purpose`、`background` 和 `decision` 分别从 `## 索引摘要` 的同名字段生成，`relations` 从 `## 关系` 生成。
5. `title`、`purpose`、`background` 和 `decision` 去除首尾空白后必须分别包含 4 至 100 个 Unicode 码点，且为单行文本；CLI 报告实际长度，不截断、补齐或改写内容。
6. 关系项保存 `type` 和直接前序的 `target` 路径；后续关系由 CLI 反向推导，不重复写入。
7. `records` 固定按 POSIX `path` 升序排列，不按状态或时间重排；JSON 使用 UTF-8、两空格缩进和文件末尾换行。
8. 通过 CLI 改变状态、登记记录或刷新派生字段；不直接维护生成字段和关系投影。
9. 索引足以在 CLI 或 Node 环境不可用时直接恢复全部记录的大致状态、时间、摘要和直接关系；正常维护和严谨校验仍以 CLI 为准，冷启动、升级和错误恢复顺序见 `maintenance-recovery.md`。
10. 删除或损坏索引会丢失生命周期、创建时间和关系投影，因此 CLI 必须校验 schema、路径、重复项、状态、时间和派生字段一致性；Git 历史承接异常恢复依据。

## 正文结构

新记录第一行固定为：

```text
# <标题>
```

最小正文：

```markdown
# <标题>

## 索引摘要
- 目的: <足以判断期望结果的精简目的>
- 背景: <足以判断是否相关的精简背景>
- 决策: <完整表达采用方向的精简决策>

## 目的
- <希望长期达成或维护的结果；必要时展开对象、范围和优先级>

## 背景
- <当时需要作出选择的背景，以及足以理解原因的关键事实>

## 决策
- 采用: <最终做法，以及支持它的核心理由>
```

正文规则：

1. 必需小节为 `索引摘要`、`目的`、`背景` 和 `决策`，且均包含实际内容。
2. `索引摘要` 包含且仅包含一个 `目的`、一个 `背景` 和一个 `决策` 字段；三者由作者明确概括，CLI 不从正文位置或顺序猜测摘要。
3. 正文标题与三个摘要字段都遵循索引投影的 4 至 100 个 Unicode 码点单行限制；完整目的、背景和决策章节不受该长度限制。
4. 三个完整章节分别承接完整语义：`目的` 写希望长期达成或维护的结果，`背景` 写促成选择的事实、问题和关键约束，`决策` 写最终采用的方向和核心理由。
5. `索引摘要` 是完整章节的压缩投影：摘要目的足以判断期望结果，摘要背景足以判断记录是否相关，摘要决策完整表达采用方向。摘要不得引入对应完整章节没有明确表达的独立语义。
6. 原地修改只用于错别字、格式、链接或不改变决策语义的表述澄清；修改后同步更新完整章节与对应摘要。
7. 目的、适用范围、关键背景前提、采用方向、核心理由或关系语义发生变化时，创建新的活动记录并按真实演进关系处理，不原地改写既有判断；无法确认是否改变语义时按决策演进处理。
8. 完整章节可以展开对象、范围、原因、约束和细节，影响相关性或采用方向的限定同时写入摘要。
9. `目的` 简单时可以只有一句话，但不能省略或只由背景隐含；需要说明对象、范围或优先级时在完整章节展开。
10. `决策` 至少包含一个 `采用`；关键备选确实有助于回放时可以增加 `不采用`。
11. `背景` 只保留足以理解选择原因的事实和约束；`采用` 写清最终做法和核心理由。
12. 不单设适用场景或重评条件，也不预测未来变化；决策本身已有的对象、范围和边界写入目的、背景或决策。未来任务结合当时目的、背景、当前事实和行为 owner 判断是否继续沿用。
13. `关系` 按需使用，省略时不保留空标题。

## 关系与生命周期

新判断改变已有判断时，在正文最后增加 `## 关系`：

```markdown
## 关系
- 修订: [<直接被部分修订的决策>](relative/path.md)
- 替代: [<直接被完整替代的决策>](relative/path.md)
- 判定无效: [<直接被判定无效的决策>](relative/path.md)
- 归并: [<被归并为当前结论的直接前序>](relative/path.md)
```

1. 所有关系从新记录指向本次直接作用的前序：
   - `修订`：保留前序的主体方向，同时改变其中一部分。
   - `替代`：用新的完整判断取代前序，前序不再构成当前结论。
   - `判定无效`：前序所依据的前提、证据或推理不成立，不再作为后续依据。
   - `归并`：把分散的直接前序整合为一条能够独立使用的活动判断。
2. 关系只由新记录单向保存；CLI 通过索引关系反向恢复指向本记录的后续决策，并沿两个方向查询关系链。
3. 前序关系的目标必须是 `archived`；没有后续判断的归档记录可以不声明关系。
4. 关系不指向自身，不形成环，同一关系类型不重复声明同一前序。
5. 活动记录独立表达完整的当前判断；部分修订时不要求读者与前序自行拼接。
6. 当前结论已经分散或日常理解必须反复追溯时，创建归并决策完整重述当前结论，并归档直接承接的记录。
7. 关系本身不改变任何记录的 `status`；`activate` 和 `archive` 分别显式表达各自状态变化。

## 维护事务

所有写入事务都检查当前行为 owner、关系和 JSON 索引，先生成完整候选索引，写入后重新运行完整校验，失败时恢复原索引。

### 首次初始化或新增

1. 确认待记录判断已经得到明确确认。
2. 选择稳定主题分类和无日期语义文件名，写入显式索引摘要和完整目的、背景与决策。
3. 运行 `activate <path>`。目标尚未登记时，CLI 以当前秒级时刻建立 `createdAt`、登记完整索引条目并设为 `active`；目标已经登记为归档时只改变状态并保留 `createdAt`。
4. 索引不存在时，只在决策目录中恰好存在这一条有效 Markdown、目标就是该记录且没有其他结构错误时初始化 schema v3 索引；已存在但无法解析或不符合 schema 的索引不被覆盖。
5. 已有索引时，未登记 Markdown 是严格检查错误；`activate` 只允许其明确目标作为唯一未登记记录进入同一校验式登记事务，其他未登记记录或结构错误仍会阻断。
6. 首次写入后的完整校验失败时删除新建索引；已有索引的更新失败时恢复原内容。

### 编辑性修正

1. 确认改动不改变目的、适用范围、关键背景前提、采用方向、核心理由或关系语义。
2. 原地修正 Markdown，并同步完整章节和索引摘要。
3. 运行 `sync-index --write` 刷新全部记录的 `title`、`purpose`、`background`、`decision`、`relations` 和排序；`status` 与 `createdAt` 保持不变。

### 独立归档

1. 确认一项活动判断不再作为后续工作依据，且没有需要记录的后续判断。
2. 检查并按需同步当前行为 owner，再运行 `archive <path...>` 把指定记录设为 `archived`。
3. 原文件、创建时间和索引条目全部保留；没有真实演进关系时不新增关系或占位记录。

### 修订、替代、判定无效或归并

1. 在写入文件前准备能够完整表达新活动结论的内容和指向直接前序的真实关系。
2. 先运行 `archive <old-path...>` 显式归档全部直接前序。
3. 再写入新的无日期语义路径 Markdown，并运行 `activate <new-path>` 完成登记和激活；CLI 验证新记录的关系目标已经归档。
4. 不由 `archive` 隐式激活后续记录，也不因关系存在而改变状态。若未来需要原子组合事务，应由名称和参数直接表达全部状态变化的独立命令承接。
5. 不重命名或移动任何已有记录。

### 显式修复或迁移

1. 先清点已有决策、索引、相互引用和当前行为 owner，区分长期判断与任务、日志、规范或普通说明。
2. 报告不符合项、可保留语义、稳定路径、关系、生命周期映射、创建时间来源和仍需判断的不确定项。
3. 只有用户明确确认修复或迁移范围后才执行转换；普通记录、解释、检查或维护请求不自动授权批量重命名或时间恢复。
4. 恢复既有 `createdAt` 时优先使用最早且可追溯的 Git 作者时间，并沿文件重命名历史查找；Git 证据不存在时再参考操作系统时间或其他可靠来源。所有来源统一到秒级；来源冲突或只能推断日期时披露不确定性。
5. 按本契约完成正文、路径、关系和全生命周期索引迁移，并收敛旧 schema、状态字段和物理归档目录。
6. 不确定项会改变决策含义、生命周期、关系、时间或 owner 时请求用户判断。

## CLI

`scripts/decision-records.mjs` 提供：

1. `check`：只读检查目录、路径、Markdown、全生命周期索引和关系。
2. `list [--topic <topic-id>] [--status active|archived|all] [--full-time]`：默认只列全部主题的活动记录；`--topic` 按路径第一段筛选 kebab-case 主题，可与状态筛选和时间显示选项组合。输出状态、创建日期、路径、标题、目的、背景和决策，不输出关系。默认时间只显示 `createdAt` 的日期部分，`--full-time` 显示秒级完整时间戳。
3. `show <path>`：输出路径、状态和完整创建时间等索引独有元数据，再输出原始 Markdown；元数据区不重复正文已有的标题、摘要和关系。
4. `trace <path>`：使用索引中的直接关系查询关系链；默认双向遍历，使用 `--direction predecessors|successors|both` 选择方向，使用 `--depth <n>` 限制关系跳数。
5. `sync-index`：默认检查派生字段和关系漂移；`--write` 刷新全部记录的派生投影、关系和排序，不改变 `status` 或 `createdAt`。
6. `activate <path>`：把已登记记录设为 `active`，或登记唯一目标并激活；满足首次初始化条件时创建 schema v3 索引。
7. `archive <path...>`：只把指定活动记录设为 `archived`，不接受后续关系参数，也不激活其他记录。

同一 MJS 可以从已安装 skill 的实际路径直接导入；导入不会执行命令。模块导出 `scanDecisionRecords`、`validateDecisionRecords` 和 `runDecisionRecordsCli(argv)`，其中前两者返回结构化结果，后者复用本节 CLI 输出语义并返回退出码；相邻的 `decision-records.d.mts` 是这些 exports 的 TypeScript 声明。

CLI 写命令属于校验式更新，不提供进程或系统中断级的原子保证；Git 历史承接异常恢复。`check`、`sync-index`、`activate` 和 `archive` 保持严格校验并在结构或索引错误时失败。

`list`、`show` 和 `trace` 是非阻断查询：中央索引缺失或无法解析、查询目标未登记时失败；其他校验问题通过 stderr warning 暴露，能够返回查询结果时退出码为 `0`。查询使用索引投影和索引关系，因此 Markdown 缺失、无效或与索引漂移时仍返回可恢复部分并标记或警告；成功退出不证明结果完整或目录符合契约。

`check` 确定性检查：

1. 决策根目录、根文件、主题分类目录、语义文件名和兼容的既有日期路径。
2. 正文标题、`索引摘要`、唯一的摘要目的、背景与决策、完整 `目的`、`背景`、`决策`、`采用`、小节顺序和非空内容。
3. 标题与三个摘要投影的单行、4 至 100 个 Unicode 码点边界，并报告实际长度。
4. JSON schema、全量路径唯一性、Markdown 与索引一一对应、生命周期状态、秒级 RFC 3339 创建时间、路径排序及标题、目的、背景、决策和关系投影一致性。
5. 关系目标已经归档、关系不形成环，链接位于决策目录内、使用相对 POSIX 路径且目标存在。

索引摘要与对应完整章节的语义一致性、`SKILL.md` 定义的记录门槛、理由质量、当前 owner 同步、隐私、创建时间来源可靠性、关系是否确为直接前序和归并时机由 agent 与用户完成语义检查。

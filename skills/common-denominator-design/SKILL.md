# Common Denominator Design

## 目标

为明确的场景集合和责任层选择可共同依赖的契约边界, 使消费者无需未声明的场景知识即可正确使用, 并由现实差异决定公约数的数量、变体和分层。

## 核心判据

1. **共同依赖**: 公约数是在明确的场景集合和责任层内可共同依赖的契约; 消费者只依赖它就能正确处理任一成员, 除公开表达的变体外不需要额外场景知识。
2. **语义而非同形**: 名称、字段或实现相似不构成共性; 含义、约束、生命周期、失败语义和使用方式一致才构成共享职责。
3. **充分后收敛**: 共同契约先完整承接共同目标, 再把局部职责和纯便利内容移出共享层。

## 执行流程

### 1. 界定场景与义务

1. 固定要统一的对象、场景集合、责任层、消费者、共同目标和主要操作。
2. 记录各场景会改变设计的语义、合法状态、操作、失败、事实源、生命周期、责任归属和变化原因; 非简单比较使用“场景 × 义务”矩阵。
3. 按含义归一化后再比较, 并区分契约、稳定规则、当前实现、偏好和假设。证据不足不等于没有公约数: 缺失信息会改变分组、共同契约或兼容风险时只确认该项; 其余未知作为假设或验证义务并暂时保持局部, 不为假想复用增加契约。

### 2. 形成并检验候选

1. 提取含义、约束、责任归属、生命周期和失败语义一致的职责; 部分场景才需要的义务留在可稳定识别的变体或局部适配层。
2. 至少比较全局共享、按稳定差异拆分和保持局部三类候选。
3. 使用正常场景、边界场景和表面相近但语义不同的场景检验:
   - **语义与使用**: 共享项含义一致, 消费者无需猜测、隐藏约定或重复分支。
   - **合法状态**: 契约直接表达合法组合, 不把校验责任散给消费者。
   - **操作与存储**: 常用操作、查询、更新、迁移和必要往返可直接完成, 事实源明确。
   - **变化隔离**: 单个场景变化不会迫使无关场景修改共享契约。
   - **分组信号**: 选择分组或变体的信号稳定、可观察且可校验。
   - **边界收敛**: 共享层不遗漏共同职责, 也不吸收局部义务。

可选字段、标志和分支可以属于公约数, 前提是它们的含义及合法组合明确, 消费者无需重建隐藏类型。

### 3. 选择结构与分层

1. 全部场景共享同一语义和操作义务, 差异只在展示、命名或局部便利时, 选择一个公约数并在边缘适配。
2. 场景共享稳定身份、生命周期或核心不变量, 但部分义务合法地不同, 选择共享核心加显式变体。
3. 场景能够形成内部一致、彼此差异稳定的分组, 且全局共享核心会丢失语义或增加处理成本时, 选择多个场景公约数。
4. 只有表面相似、没有共同责任, 或统一成本高于重复时, 保持局部。

存储、传输、领域和消费义务不同时, 在各责任层分别选择结构, 并明确事实源、映射方向、信息损失和兼容责任。用合并与拆分的反事实解释最终边界: 每个独立公约数都应消除真实差异带来的成本。

### 4. 交付结论

先给最终结构及是否分层, 随后只提供当前任务需要的内容:

1. 适用场景、分组和稳定区分信号。
2. 每个公约数的共享语义、不变量、责任归属以及变体或局部内容。
3. 存储、传输和消费表示之间的映射与事实源。
4. 被否决候选的失败场景, 以及证据不足处、验证义务和必要的迁移或兼容事项。

## 边界

本 skill 只处理已经进入跨场景统一或复用判断的对象, 不替代产品价值、项目优先级、系统责任或整体架构方向判断。

## 完成检查

1. 场景集合、消费者、责任层、目标操作和证据边界清楚; “没有公约数”与“证据不足”没有混淆。
2. 最终结构通过候选检验, 数量与分层由现实差异支持, 并能解释合并和拆分的反事实成本。
3. 交付内容足以继续当前任务, 且未知项和验证义务明确。

# 审查证据与验证方法

本文件补充主审查流程的证据检查, 不构成独立审查方法。仅当现有证据不足以证明影响、问题层级或处置时, 选择与当前判断相关的检查, 不机械套用全部内容。

## 证明产品影响

1. 指出用户或外部消费者、触发路径和可观察结果。
2. 区分任务失败、错误结果、数据或安全风险、契约破坏、资源失控和诊断展示变化。
3. 兼容性判断列出实际消费者: 仓库外调用者、持久数据、用户自动化或明确的机器协议。
4. 根据消费者区分稳定产品契约、有限稳定管理接口和内部诊断材料。

## 证明开发影响

1. 指出重复发生的工程动作, 例如新增相邻字段、修改流程、测试失败路径或排查线上故障。
2. 列出实际修改位置、平行事实源、回归入口或不可构造的状态。
3. 把维护问题落到变更频率、修改面、测试难度、诊断成本、恢复成本或对后续能力的阻塞。
4. 对性能或算法复杂度候选确认真实数据规模、hot path 或可证明的无界工作。

## 证明局部或系统性

局部缺口通常满足:

- owner 和边界明确。
- 修复集中在一个责任域。
- 可以通过纯函数、构造约束或边界映射完成。
- 不新增平行事实源、循环依赖或隐藏 handoff。

系统性症状通常表现为:

- 同一事实在多个 owner、表、schema 或测试中重复定义。
- 简单相邻功能需要修改多个平行入口和映射层。
- owner 在加载顺序、初始化顺序或验证责任上互相依赖。
- 中间类型、双重读取或第二套流程主要用于协调职责。
- 上游阶段必须读取下游私有数据才能完成工作。

## 证明契约与信任边界

1. 把校验放在用户输入、持久配置、进程或模块外输入、adapter 返回值和机器协议等真实边界。
2. 对进程内值使用解析结果、构造器或领域类型承接已证明的不变量。
3. 让文档固定用户可观察结果和必要工程约束, 例如 bounded、确定性、可继续分页、失败分类和输出通道。
4. 只有算法本身属于产品或工程契约时才固定实现细节。

## 选择最小验证实验

1. 相邻功能试验: 假设增加一个简单字段、选项或状态, 列出必须修改的位置, 检查 canonical declaration 是否是真正事实源。
2. 阶段拆分试验: 列出各阶段输入输出, 找出重复上游语义、隐藏状态或读取下游私有数据的位置。
3. 可达性与频率试验: 沿产品入口、错误路径和常见开发动作证明问题何时发生。
4. 有限工作试验: 区分限制输出与限制读取、解析、分配和重复扫描。
5. 度量试验: 为性能提供数据规模和基线, 为维护性提供变更频率、修改位置和回归证据。

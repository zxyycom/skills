# 详细审查报告

## 用途与位置

把完整 finding 清单、证据链和根因关系写入临时文件, 让对话只承接结论和决策入口。

1. 使用 Markdown 生成详细报告。
2. 写入操作系统或当前环境提供的临时目录, 使用 `code-review-<scope>-<timestamp-or-random>.md` 一类避免冲突的名称。
3. 保持目标仓库、长期文档、源码目录和待提交内容不变。
4. 在最终对话中提供绝对路径或可点击链接, 并说明文件是临时交付物。

## Markdown 结构

```markdown
# Code Review Detailed Report

## 审查范围
- 目标:
- 基线或 diff:
- 已读取证据:
- 未覆盖范围:

## 综合判断
- 总体结论:
- 产品影响摘要:
- 开发影响摘要:
- 根问题:

## Findings

### [优先级] 标题
- 位置:
- 来源: A | B | C
- 价值视角: 产品 | 开发 | 两者
- 证据:
- 产品影响:
- 开发影响:
- 局部或系统性:
- 根因:
- 最终处置:
- 优先级依据:
- 验收标准:

#### 证据补充
- 兼容性消费者:
- 信任边界:
- 相邻功能修改面:
- 最小验证实验:

## 根问题归并
- 根问题:
  - 关联 findings:
  - 共同机制:
  - 建议下一步:

## 待确认项与残余风险
- 待确认项:
- 未覆盖测试或环境:
- 延期项的重新评估条件:
```

每个 finding 保留核心字段。`产品影响` 或 `开发影响` 没有直接证据时写“无直接证据”; `证据补充` 只保留会改变判断的项目。

## 填写规则

1. 把事实、推断和待确认项分别放入对应字段。
2. 证据定位到文件、行、符号、命令结果、测试或明确契约。
3. 多个 findings 共享机制时, 在单项中保持简短, 在根问题归并中完整表达关系。
4. 优先级依据说明适用视角、触发概率或频率、影响范围、恢复成本和维护放大效应。
5. 删除重复背景、完整执行日志和不影响结论的调查过程。

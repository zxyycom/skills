# Code Review

## 目标

形成可行动、可追踪且便于阅读的审查结论。让每个 finding 至少在产品或开发视角上证明实际影响, 再完成根因归并、处置选择和优先级判断。

## 主动引用

先读本文件, 再按问题加载引用:

1. finding 的成立依赖兼容性消费者、信任边界、性能规模、维护放大效应或系统性根因的进一步证明时, 读取 [review-evidence.md](references/review-evidence.md), 只选择与当前判断相关的证据检查。
2. findings 超过 5 项、存在跨 finding 根因、需要保留长证据链, 或用户要求详细、可追踪报告时, 读取 [detailed-report.md](references/detailed-report.md) 并生成临时 Markdown 报告。

## 审查契约

1. 默认只读审查目标范围。只有用户同时要求修复时才修改项目或外部状态; 临时详细报告写入系统临时目录, 不进入目标仓库。
2. 让 finding 指向具体文件、行、符号、运行结果、测试或明确契约。区分事实、基于证据的推断和待确认项。
3. 审查变更时区分本次引入、由本次暴露和既有的问题, 只报告与当前范围有可操作关系的内容。
4. 优先采用目标项目已有的优先级和审查格式; 没有项目规则时使用本文件的判断与输出约定。
5. 没有达到 finding 门槛的问题不进入问题清单。没有有效 finding 时明确结论并说明未覆盖范围。

## 主执行流程

### 1. 确定范围和基线

1. 读取项目指令、用户指定范围、目标 diff 或实现, 以及理解行为所必需的文档和测试。
2. 明确产品基线: 用户或外部消费者、核心场景、预期结果和已承诺契约。
3. 明确开发基线: 事实 owner、主要数据流、构造约束、测试边界、诊断路径和常见变更动作。
4. 标记未覆盖模块、运行环境和会改变结论的上下文缺口。

### 2. 从双视角发现候选问题

- 产品视角检查用户结果、正确性、数据、安全、隐私、兼容性、性能、资源边界和故障恢复。
- 开发视角检查 owner、单一事实源、非法状态、耦合、修改面、测试性、可观测性、构建发布、认知成本和演进成本。

任一视角都可以独立形成 finding。产品问题可以来自清晰实现中的行为错误; 开发问题即使尚无直接用户影响, 也可以因高频变更、回归风险或系统阻塞而成立。

### 3. 验证候选问题

对每个候选问题确认:

1. 场景: 哪个产品路径或工程动作会触发?
2. 证据: 哪些代码、契约、调用关系、测试或运行结果能证明?
3. 影响: 谁会受到什么可观察代价?
4. 发生条件: 产品路径是否可达, 工程动作是否重复, 影响范围和恢复成本如何?
5. 依据: 建议依赖的能力、稳定性或工程约束是否已经成立?

至少一条视角同时具备场景、证据和实际影响时保留 finding。信息不足但可能改变结论时保留为待确认项; 只有审美差异而无实际代价时结束调查。

### 4. 归因并确定问题层级

1. 标记来源:
   - A: 文档或设计契约缺口。
   - B: 实现没有达到已确认的文档、契约或工程约束。
   - C: 内部设计或代码质量已经产生实际代价。
2. 判断层级:
   - 局部缺口: owner 与边界清楚, 修复集中在一个责任域, 不增加平行事实源或隐藏协调。
   - 系统性症状: owner、事实源、执行顺序或数据流使多个位置重复承担同一责任。
3. 把共享机制的 symptoms 归并为最少的根问题, 同时保留各 finding 的独立影响和位置。
4. 现有证据不足以确定层级或处置时读取 [review-evidence.md](references/review-evidence.md), 选择能改变结论的最小检查。

### 5. 选择处置和优先级

为每个根问题选择一个主要处置:

- 补文档: 当前行为合理, 但产品结果、工程约束、owner 或外部契约缺失。
- 调整系统设计: owner、事实源、执行顺序或数据流导致多处症状。
- 代码收敛: 设计清楚, 实现没有达到既有契约或约束。
- 局部简化: 删除已经产生维护成本的无收益抽象、重复工作或冗余数据流。
- 删除约束或 validator: 当前约束没有消费者或信任边界依据。
- 延期: 当前证据或价值不足, 同时写明重新评估的产品或工程触发条件。

按适用视角形成优先级依据:

- 产品依据: 影响强度、可达性、发生概率、影响范围、可恢复性和外部承诺。
- 开发依据: 变更频率、维护放大系数、回归概率、检测与恢复成本、认知负担和系统阻塞。

任一视角有高影响证据时都可以进入高优先级。统一排序时保留具体依据, 不机械平均两条视角, 也不让 A/B/C 决定优先级。

### 6. 分层交付

#### 正常对话

1. 先给总体结论, 再写 findings, 最后按需补充根因、待确认项和残余风险。
2. findings 不超过 5 项时全部列出。超过 5 项时列出全部阻塞或高优先级项; 若不足 5 项, 再按优先级补足到最多 5 项, 其余按根因和数量概括。
3. 每项使用不超过两句话表达优先级、标题、位置、价值视角、关键影响、问题层级和主要处置。
4. 完整分析卡、长证据链和全部低优先级项进入临时报告。对话提供报告的绝对路径或可点击链接。
5. 用户明确要求内联完整报告时按其要求输出。临时文件不可用时说明限制, 保留紧凑摘要, 并让用户决定是否展开。

#### 临时详细报告

1. 按 [detailed-report.md](references/detailed-report.md) 写入所有 findings、证据、根因关系和验收标准。
2. 使用 Markdown, 并保持标题和核心字段稳定、便于人类阅读与后续引用。
3. 写入操作系统或当前环境提供的临时目录, 使用避免冲突的文件名, 保持目标仓库和待提交内容不变。
4. 报告承接细节, 对话承接结论和决策入口; 阻塞或高优先级 finding 始终出现在对话中。

## 完成检查

1. 是否明确审查范围、产品基线、开发基线和未覆盖范围?
2. 每个 finding 是否有具体位置、证据和至少一条成立的影响视角?
3. 事实、推断和待确认项是否清楚分开?
4. A/B/C 是否只表示来源, 局部或系统性是否由 owner、事实源和数据流证明?
5. 处置是否对应根因, 优先级是否说明产品或开发依据?
6. 对话是否遵守信息上限并保留关键结论, 详细报告是否承接全部必要细节?
7. 是否保持只读审查边界, 并排除没有实际代价的风格偏好?

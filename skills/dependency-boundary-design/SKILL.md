# Dependency Boundary Design

## 目标

把散落的底层调用收口到明确边界，让后续变化集中处理。

边界首先是责任和变化的集中点，不以多实现为前提。默认路线是：
**先收口，再演进。**

## 核心规则

1. **按责任收口**：先按共同变化原因和责任归属划定边界，再用消费者结果与
   数据或状态的事实源校验范围；不按可执行文件、包或 SDK 名称归并全部调用。
2. **具体边界即可成立**：模块、函数集合或具体类型都可以集中调用、协议、
   错误和配置，不必等待通用接口或第二实现。
3. **契约服务消费者**：只承诺消费者需要且能够验证的行为；具体依赖边界
   可以保留依赖特有概念，只有消费者契约确实需要时才向外暴露，并明确暴露
   范围与变化影响。
4. **收口与演进分开**：先集中责任并避免计划外语义变化；语义改进和多实现
   适配只在现实义务出现后继续。

## 执行流程

### 1. 恢复依赖面

1. 明确任务是分析、设计、实现还是审查，以及已有修改授权。
2. 定位依赖、消费者、现有 owner、直接调用和重复的协议、错误、配置、
   生命周期或性能处理，并说明希望集中处理的变化。
3. 先以用户明确要求和已有契约确定迁移行为基线，用测试与当前实现补充证据；
   将行为分为消费者必须保持、本次明确改变和尚未确认三类。关键未知会改变
   owner、范围或契约时先调查或确认；其余作为假设和验证义务。

### 2. 判断并划定范围

1. 先按共同变化原因和责任归属组织调用，再用消费者结果与数据或状态的事实源
   校验范围；同一工具服务不同责任时可以拥有不同 owner。
2. 同一责任的调用分散、依赖处理重复或漂移、消费者反复接触无关底层细节，
   或已确认同类变化需要跨多个位置处理时，建立收口边界。
3. 调用仍然局部、责任清楚，且新增层只会转发参数和返回值时保持局部。记录
   owner、理由和重新评估信号后直接交付，不继续设计边界或迁移。
4. 需要收口时明确边界 owner、纳入的责任与消费者、允许直接调用底层依赖的
   位置和预期吸收的变化；边界外例外必须写明 owner、原因、范围和重新评估
   信号。

### 3. 定义边界与契约

1. 稳定消费者语义尚未形成时，先建立诚实命名的具体依赖边界；语义稳定且
   底层细节不属于消费者责任时，可在其上增加消费者语义门面。
2. 多个消费者共享依赖治理但语义不同，使用一个依赖边界加多个局部门面，
   不压成巨型扁平接口。
3. 从消费者操作和明确约束定义输入输出、失败、顺序、能力限制及必要的运行
   义务。依赖特有概念只在消费者契约确实需要时暴露，并记录暴露范围以及
   哪些依赖变化可能越过边界。

### 4. 规划或完成收口

1. 分析或设计任务形成 owner、范围、契约、例外、迁移顺序和验证方案；实现
   任务才在授权范围内修改代码。
2. 按已确定的责任边界迁移底层调用及其依赖处理，避免顺带统一不同责任或
   改变未授权语义，并为范围内新增调用提供明确入口。
3. 迁移后核对剩余直接调用，逐一归为边界实现、已说明例外或尚未完成的授权
   范围。
4. 选择一项代表性的契约内依赖变化，例如协议、错误、配置或参数调整。分析
   或设计任务说明它应由何处吸收；实现任务通过静态影响核对或测试确认修改
   能够留在边界内。不能满足时重新检查 owner、范围、契约或例外。

### 5. 按现实义务继续演进

1. 只有多个实现必须履行同一消费者契约，或存在实际选择、切换义务时，才定义
   provider、adapter 或选择机制。
2. 接口、构造函数或工厂可以只服务封装、注入和生命周期管理；代码形态本身
   不构成可替换承诺。
3. 声明可替换能力时，写清共同操作、失败、能力差异和选择方式，并由实际
   多实现或替换场景验证；只有未来设想时保留具体边界和重新评估信号。

## 交付

1. 保持局部时，给出 owner、理由、允许范围和重新评估信号。
2. 分析或设计任务给出收口依据、owner、范围、例外、边界结构、契约、迁移、
   验证义务、假设和未知项。
3. 实现任务说明实际收口范围、消费者变化、剩余直接调用和验证证据。
4. 审查任务报告责任错误归并、仍在扩散的调用或依赖处理、无依据例外、
   代表性变化仍需跨边界处理、计划外语义变化、契约夸大和无证据的替换承诺。

## 完成标准

1. 收口义务和要集中处理的变化明确；保持局部时已经形成完整结论。
2. owner、范围、例外和契约按责任组织，当前实现没有被自动升级为规则。
3. 实现任务中，授权范围内的调用及依赖处理已进入边界，剩余调用均已分类。
4. 收口没有产生计划外语义变化；正常、失败及必要的生命周期和性能场景符合
   选定行为基线。
5. 剩余调用核对以及代表性变化的影响分析或测试表明，已声明的契约内变化
   能够在边界内处理；可替换承诺仅在实际声明时由多实现或替换场景证明。

## 边界

1. 本 skill 处理具体工程任务中的依赖收口，不替代产品价值、整体架构方向或
   跨领域共享契约判断。
2. 同一工具的调用可以属于不同 owner；收口不以减少文件、接口或直接调用总数
   为成功标准。
3. 明确边界不自动保证底层可替换；替换范围由消费者契约和验证证据决定。
4. 安全、权限、不可逆操作和外部状态变更继续服从当前任务与运行环境授权。

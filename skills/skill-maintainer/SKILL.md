# Skill Maintainer

## 目标

为一项可复用的 agent 能力确定应由新 skill、既有 skill 还是其他 owner 承接，并让选定的 skill 成为触发清楚、边界独立、可验证且可交付的行为单元。维护位置决定修改与交付入口；skill 的适用范围由其行为契约决定。

## Skill 模型

Skill 是在特定使用场景下为 agent 提供可复用行为的能力单元。发现入口让 agent 判断何时使用，行为正文决定触发后如何执行，按需资源为执行提供知识、确定性操作或输出素材；三者共同兑现同一项能力。

### 组成

必需入口是一个 skill 目录及其中的 `SKILL.md`。Frontmatter 的 `name` 提供稳定身份，`description` 表达能力和使用时机；正文承接目标、判断、步骤、边界与完成标准。

可选资源包括：`references/` 保存按需读取的知识与契约，`scripts/` 保存适合确定性执行的重复操作，`assets/` 保存直接用于输出的素材。每项资源都由正文说明使用条件。

Metadata、人类文档、索引、打包与发布配置属于环境集成层；当前环境使用这些能力时再接入。

### 按主要作用分类

以下分类用于维护时选择正文骨架、资源和验证重点：

1. **知识与规则型**：提供领域知识、约束和判断标准，重点验证准确性、覆盖范围与适用边界。
2. **流程与决策型**：组织步骤、分支和状态变化，重点验证路径完整性与完成条件。
3. **工具与自动化型**：调用工具或脚本完成可重复操作，重点验证输入、输出与失败处理。
4. **产出与模板型**：生成文档、代码、图像或其他交付物，重点验证输出结构与质量。
5. **协调与入口型**：选择 owner、组合能力或安排交接，重点验证触发、路由与责任边界。

一个 skill 可以兼具多种特征；以主承诺占比最高的类型决定正文重心、资源选择和验证方式。

## 结构验证器

本 skill 随包提供 `scripts/validate-skill.mjs`。它机械检查可移植基线：skill 目录、`SKILL.md`、YAML frontmatter、`name` 与 `description`、名称与目录一致性、正文、资源目录和内部 Markdown 链接。

从本 skill 目录运行，或使用脚本的绝对路径：

```text
node scripts/validate-skill.mjs <skill-directory>
```

该脚本验证通用结构；当前环境定义的 metadata、索引、打包或其他契约由对应验证入口补充。

## 获取上下文

1. 根据用户指定的路径或当前上下文定位维护对象；需要时区分源码位置、安装位置和 skill 实际服务的环境。
2. 更新或审阅已有 skill 时，完整读取它的 `SKILL.md` 和当前任务需要的引用；创建前检查能够访问到的相邻 skill。
3. 读取当前环境中会影响结果的指令、运行时约定和验证入口；其余部分使用本 skill 的最小基线。

## 主流程

### 1. 确定 owner

1. 将当前任务归为设计、创建、更新、重构或审阅，并判断它是否形成稳定、可复用的执行能力。
2. 一次性工作和背景知识保留在当前任务；已有专职契约交给对应 owner。
3. 对可复用能力，以用户意图、行为责任和验收入口为判断依据：三者与既有 skill 相同时扩展该 skill，三者共同形成可独立识别的边界时创建新 skill，并明确交接条件。

### 2. 定义行为契约

1. 选择主要类型，用一句话定义触发后的目标结果，再明确该 skill 拥有的判断、步骤、验收以及相邻 owner。
2. 默认名称使用唯一、简洁且与目录一致的 kebab-case；当前环境有命名约定时采用该约定。
3. `description` 只表达能力定位和使用时机；正文给出必要输入、判断顺序、执行步骤、边界、完成条件和交付内容。

### 3. 形成交付

1. 按 Skill 模型组织入口、正文和资源，让最小交付能够独立兑现行为契约。
2. 当前环境使用 metadata、人类说明、索引、共享工具或打包配置时，按其既有入口完成集成。
3. 将安装、启停、发布、提交和全局配置变更作为用户明确授权的交付项。

### 4. 验证并交付

1. 运行 `validate-skill.mjs` 检查通用结构，再运行当前环境提供的验证入口。
2. 实际运行新增或修改的其他脚本，并覆盖会改变结果的代表性分支。
3. 按主要类型检查核心风险，再用现实请求确认预期场景由该 skill 承接，并确认相邻请求能够交给正确 owner。
4. 交付时说明实际改动、验证结果、未执行项和残余风险。

## 边界

1. 当前环境的明确契约决定已定义部分的实现方式，本 skill 的最小基线补足其余部分。
2. 当维护对象、交付位置或相邻 owner 无法从现有事实确定，且选择会改变结果时，只确认这一项判断。
3. 安全、权限、不可逆操作和外部状态变更遵循当前环境的授权边界。

## 完成标准

1. 新增、扩展或交给其他 owner 的选择能够由触发、责任和验收解释。
2. 主要类型、名称、description、正文和资源共同兑现同一主承诺，并与相邻 owner 清楚衔接。
3. 结构验证器与当前环境已有的集成验证均已通过，最小交付能够独立使用。
4. 实际验证证据、未执行项和残余风险已经报告。

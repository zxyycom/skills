# Skill Maintainer

## 目标

为一项可复用的 agent 能力确定应由新 skill、既有 skill 还是其他 owner 承接，让每个 skill 形成清楚、可验证的行为单元，并让包含一个或多个 skill 的分发单元拥有明确依赖、完整验收和独立交付边界。

## Skill 模型

Skill 是在特定使用场景下为 agent 提供可复用行为的能力单元。发现入口让 agent 判断何时使用，行为正文决定触发后如何执行，按需资源为执行提供知识、确定性操作或输出素材；三者共同兑现同一项能力。

分发单元是安装、更新、兼容性和交付的边界，可以只包含一个 skill，也可以包含一组共同完成工作流且不承诺单独使用的 skill。同一分发单元内的成员可以互相引用、依赖和指定交接；跨单元依赖通过显式前置条件和缺失路径表达。

### 组成

必需入口是一个 skill 目录及其中的 `SKILL.md`。Frontmatter 的 `name` 提供稳定身份，`description` 表达能力和使用时机；正文承接目标、判断、步骤、边界与完成标准。

可选资源包括：`references/` 保存按需读取的知识与契约，`scripts/` 保存适合确定性执行的重复操作，`assets/` 保存直接用于输出的素材。每项资源都由正文说明使用条件。

一个分发单元包含成员 skill 和运行所需资源。Metadata、索引、打包、安装和发布配置属于环境集成层，用于声明、构建或验证单元，不要求进入分发内容。成员目录不自动等于分发单元；单元边界由用户意图和当前环境的安装、更新或交付契约决定。

### 按主要作用分类

以下分类用于维护时选择正文骨架、资源和验证重点：

1. **知识与规则型**：提供领域知识、约束和判断标准，重点验证准确性、覆盖范围与适用边界。
2. **流程与决策型**：组织步骤、分支和状态变化，重点验证路径完整性与完成条件。
3. **工具与自动化型**：调用工具或脚本完成可重复操作，重点验证输入、输出与失败处理。
4. **产出与模板型**：生成文档、代码、图像或其他交付物，重点验证输出结构与质量。
5. **协调与入口型**：选择 owner、组合能力或安排交接，重点验证触发、路由与责任边界。

一个 skill 可以兼具多种特征；以主承诺占比最高的类型决定正文重心、资源选择和验证方式。

## 结构验证器

本 skill 随包提供 `scripts/validate-skill.mjs`。它机械检查单个 skill 的可移植结构基线：skill 目录、`SKILL.md`、YAML frontmatter、`name` 与 `description`、名称与目录一致性、正文、资源目录和内部 Markdown 链接。

从本 skill 目录运行，或使用脚本的绝对路径：

```text
node scripts/validate-skill.mjs <skill-directory>
```

该脚本不推断分发单元，也不验证跨 skill 依赖。分发单元的成员完整性、内部依赖、metadata、索引、打包和安装契约由当前环境的对应入口补充验证；没有机械入口时逐项人工检查，并在交付中说明证据和残余风险。

## 获取上下文

1. 根据用户指定的路径或当前上下文定位维护对象；区分 skill 行为入口、源码位置、安装位置和分发单元。
2. 更新或审阅已有 skill 时，完整读取它的 `SKILL.md` 和当前任务需要的引用；多个 skills 属于同一分发单元时，同时读取会影响依赖和交接的成员入口。
3. 从用户说明和当前环境的指令、metadata、打包、安装、更新或发布入口恢复分发单元边界；边界缺失且不同选择会改变依赖或交付时，只确认这一项。
4. 创建前在当前维护环境允许时检查能够访问到的同类能力；其余部分使用本 skill 的最小基线。

## 主流程

### 1. 确定 owner 和分发单元

1. 将当前任务归为设计、创建、更新、重构或审阅，并判断它是否形成稳定、可复用的执行能力。
2. 一次性工作和背景知识保留在当前任务；已有专职契约交给对应 owner。
3. 对可复用能力，以用户意图、行为责任和验收入口为判断依据：三者与既有 skill 相同时扩展该 skill，三者共同形成可独立识别的边界时创建新 skill，并明确交接条件。
4. 根据安装、更新、兼容性和完整工作流判断分发单元：不承诺单独使用且共同完成同一交付的 skill 可以组成一个单元；只是在同一仓库、主题相近或偶尔协作不足以自动合并。

### 2. 定义行为和依赖契约

1. 为每个 skill 选择主要类型，用一句话定义触发后的目标结果，再写清输入、前置条件、判断、步骤、输出和验收。
2. 同一分发单元内的成员可以显式点名、调用或依赖其他成员；正文写清触发条件、交接输入和完成状态。
3. 跨分发单元的能力不能作为未声明前提；确有依赖时写成显式前置条件，并说明缺失时的失败或降级路径。
4. 默认名称使用唯一、简洁且与目录一致的 kebab-case；当前环境有命名约定时采用该约定。
5. `description` 只表达能力定位和使用时机；正文承接完整行为与依赖契约。

### 3. 形成交付

1. 按 Skill 模型组织每个成员的入口、正文和资源，并让分发单元包含兑现整体契约所需的全部成员和共享资源。
2. 当前环境使用 metadata、人类说明、索引、共享工具或打包配置时，按其既有入口完成集成。
3. 将安装、启停、发布、提交和全局配置变更作为用户明确授权的交付项。

### 4. 验证并交付

1. 对每个成员运行 `validate-skill.mjs` 检查通用结构，再运行当前环境已有的分发单元验证入口；没有机械入口时逐项人工检查并记录未自动化的证明。
2. 检查成员清单、内部引用、交接输入、共享资源和版本或兼容性约束；跨单元依赖要写明前置条件和缺失路径。
3. 实际运行新增或修改的脚本，并覆盖会改变结果的代表性分支。
4. 用现实请求验证单个成员的触发和单元内完整工作流，同时用缺失外部分发单元的场景验证失败或降级路径。
5. 交付时说明实际改动、验证结果、未执行项和残余风险。

## 边界

1. 当前环境的明确契约决定 skill 和分发单元的实现方式，本 skill 的最小基线补足其余部分。
2. 当维护对象、分发单元边界或交付位置无法从现有事实确定，且选择会改变结果时，只确认这一项判断。
3. 安全、权限、不可逆操作和外部状态变更遵循当前环境的授权边界。

## 完成标准

1. 新增、扩展或交给其他 owner 的选择能够由触发、责任和验收解释。
2. 每个 skill 的主要类型、名称、description、正文和资源共同兑现自身行为承诺。
3. 分发单元包含全部必需成员和资源；单元内依赖明确，跨单元没有未声明前提。
4. 每个成员的结构验证与当前环境已有的单元集成验证均已通过；没有机械入口时已报告人工证据和残余风险。
5. 实际验证证据、未执行项和残余风险已经报告。

# Skill Design Discovery

## 目标

在实现前恢复 skill 真正需要承载的结果、行为、判断和边界，形成足以指导实现与验收的设计契约。契约默认在对话中交付；只有复杂交接或当前环境要求时才写入文件。

## 主动读取

先读本文件。深度发现、复杂判断或需要稳定交接时，再读取 [设计契约](references/design-contract.md)；轻量任务不加载引用。

维护本 skill、审计早期取舍或回收原方案机制时，按需读取 [上游原始参考快照](references/org/README.md)。快照随当前分发单元提供，但普通设计发现仍以本文件和设计契约为行为 owner，不加载原版流程。

## 主流程

### 1. 选择发现深度

1. 目标、现实案例、主要分支、约束来源和验收已经清楚时走轻量路径，只补会改变设计的缺口。
2. 依赖专家经验、多个工具或人类介入，既有 skill 经常跳步或误判，证据互相冲突，或者错误代价明显时走深度路径。
3. 当前只是结构、打包或其他局部维护时，直接回到对应任务，不制造额外访谈和规划产物。

### 2. 恢复真实行为

1. 先读取用户材料、目标 skill、项目规则、相关资源和可用运行结果；能从现有证据获得的信息不转问用户。
2. 区分已确认目标、规范性约束、观察行为、当前实现、用户经验、推断和未知项。当前文字或实现只证明现状，不自动成为应保留的规则。
3. 用正常案例、失败案例或表面相近但处理不同的对照案例，恢复消费者、目标结果、触发与 near miss、端到端流程、分支、前置条件和完成标准。

### 3. 提取潜藏判断

1. 对会改变路径或结果的决策，写清信号、默认方向、例外、依据和错误影响。
2. 区分硬契约、稳定规则、启发式默认值、用户或项目偏好、环境限制、历史 workaround 和待验证假设；重要约束保留来源、范围和 owner。
3. 明确 agent 可以自主执行、执行后说明、只能建议、必须确认或禁止执行的权限。
4. 用反事实检查候选规则：删除后是否改变行为，什么证据会改变选择，何时存在合法例外。无法说明作用或来源的内容不升级为硬规则。

### 4. 形成可实现设计

1. 只在未决选择会改变主承诺、责任边界、关键路径、权限或验收时询问用户；其余未知项写成显式假设或验证义务。
2. 深度路径先形成端到端 Workflow Map，表达起点、完成态、顺序、分支、并行或可选路径、人类介入和步骤交接；它是设计结构，不自动成为落盘或整图确认门禁。
3. 为关键步骤选择与风险相称的控制方式，写清必需资源、完成证据、交接条件和失败处理；优先采用足以约束失败的最轻机制。
4. 提出实现归位建议：所有路径都需要的内容进入 `SKILL.md`，分支知识进入 references，确定性重复操作进入 scripts，输出素材进入 assets，外部事实留在原 owner；最终 skill 与分发 owner 由实现或维护入口确认。
5. 使用 [设计契约](references/design-contract.md) 中的评估提示，从当前目标即时推导验证义务；只交付会约束实现和验收的结论。

### 5. 交付与交接

1. 轻量路径交付目标与边界、关键流程、主要判断和验证义务；深度路径按 [设计契约](references/design-contract.md) 组织。
2. 标明设计是可实现、带显式假设可实现，还是仍有会改变设计的待决项；不要把计划中的验证写成已通过。
3. 当前请求包含实现时，将契约作为实现输入继续执行；只要求设计时，不擅自创建、安装或发布目标 skill。

## 边界

1. 不强制用户确认整张流程图，也不要求轻量任务生成文件；中间沟通只展示会改变下一步的证据、矛盾或选择。
2. 不拥有 skill 的结构校验、分发、metadata、打包、安装、更新或发布。
3. 发现到的判断只有在当前环境认定需要长期回放且已经确认时，才进入对应长期 owner。
4. 当前环境限制不得无证据泛化为通用规则；安全、权限、隐私和不可逆操作继续遵循当前授权。

## 完成标准

1. 目标结果、触发边界、端到端流程和非目标足以约束实现。
2. 关键决策、重要约束、人类权限及剩余假设清楚且有依据。
3. 关键步骤的控制方式、必需资源、完成证据、交接条件和失败处理已经说明。
4. `SKILL.md`、references、scripts、assets 与外部 owner 的归位建议没有重复事实源。
5. 验证义务覆盖会改变结果的触发、分支、依赖、权限和约束，并能追溯到当前目标与证据。
6. 设计状态、未运行验证和后续实现输入已经说明。

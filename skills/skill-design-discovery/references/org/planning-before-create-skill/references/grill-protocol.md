# Grill 协议

设计 Skill 前先读本文件。目标不是做普通需求访谈，而是榨出用户脑中的隐性 workflow。

## 原则

用户通常已经有一套隐性的行为模式：

- 先做什么
- 看什么资料
- 凭经验怎么判断
- 哪些地方必须人来介入
- 进入下一步前检查什么
- 过去模型最容易犯什么错

不要一次性问完所有问题。分轮追问。每轮结束后，简短总结你学到的内容，并让用户纠正。

## 第 1 轮：任务与工作流全景

问到足够理解整体任务：

- 这个 Skill 要让 agent 能稳定完成什么？
- 如果你手动做，会从头到尾怎么做？
- 从开始到结束有哪些主要步骤？
- 哪些步骤必须严格按顺序？哪些可以并行？哪些是可选分支？
- 最痛苦或最容易失败的部分是什么？
- 在实际工作里，什么叫“完成”？

完成标准：你能列出主要步骤，且不是靠自己猜。

## 第 2 轮：逐步深挖

针对每一步榨出：

- 目的
- 输入
- 输出文件
- 必需 references
- 必需工具或脚本
- 人类判断
- completion criterion
- 常见失败
- 下游依赖

必要时使用表格：

| 步骤 | 目的 | 输入 | 输出 | Refs | Scripts/Tools | 人类输入 | 完成标准 | 失败风险 |
|---|---|---|---|---|---|---|---|---|

完成标准：每一步至少有输入、输出和完成条件。

## 第 3 轮：可复用资源

追问：

- 哪些知识是非显然的，应该成为 reference？
- 哪些重复动作应该成为 script？
- 哪些模板、示例、图片、字体或项目骨架应该成为 asset？
- 是否涉及外部平台、API、dashboard、浏览器页面或数据集？
- 哪些资源每次都需要？哪些只在某个分支需要？

完成标准：得到 `references/`、`scripts/`、`assets/` 的候选清单。

## 第 4 轮：人机介入点

追问：

- 哪些节点 agent 必须停下来问用户？
- 哪些节点需要用户选择方向？
- 哪些节点需要用户 review 中间产物？
- 每个介入点用户应该看到什么？
- 哪些信息对用户是噪音，应该隐藏？
- 反馈应该用短回答、选项、表格，还是 HTML review 页面收集？

完成标准：每个人类介入点都有时机、展示格式和反馈去向。

## 第 5 轮：触发和边界

追问：

- 用户需要这个 Skill 时会怎么说？
- 有哪些非正式说法也应该触发？
- 哪些相邻任务看起来像，但不应该触发？
- 可能和哪些已有 Skill 重叠？
- 应该 model-invoked，还是 user-invoked？
- 用哪 1-3 个 leading words 锚定行为？

完成标准：能起草 description 和 near-miss 列表。

## Grill 阶段输出

结束 grill 时输出：

- 5-10 行 Skill 意图总结
- 步骤列表
- 资源列表
- 人机介入点列表
- 未解决问题

未解决问题被回答或明确延后之前，不要进入实现。

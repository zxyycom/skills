# Skill 设计契约

深度发现或跨阶段交接时使用本文件。设计契约可以是对话输出，不因采用本结构而强制落盘。

## 判断模型

1. **证据**：区分已确认目标或契约、现实行为、当前实现、用户经验和推断。当前实现只说明现状；推断必须暴露依据和未知项。
2. **决策**：只记录会改变路径、输出或风险的选择，包含信号、默认方向、例外、依据、权限和错误影响。
3. **约束**：区分硬契约、稳定规则、启发式默认值、偏好、环境限制、历史 workaround 和待验证假设，并记录来源、范围和 owner。
4. **权限**：使用自主执行、执行后说明、只能建议、必须确认或禁止执行，说明人类决定怎样进入下游。
5. **控制**：根据失败风险选择清晰指令、必需产物或交接契约、确定性脚本或检查、人类介入、隔离执行或评估、固定编排等机制；采用足以约束失败的最轻方式。

证据冲突时先判断它们描述的是目标、当前事实、局部范围还是不同环境。只有不同选择会改变主承诺、owner、关键路径、权限或验收时才请求用户决定。

## 评估提示

以下提示用于从多个视角检查当前设计。它们不是预设验证案例，也不提供领域答案；agent 应结合当前证据即时推导验证义务，只保留会约束实现和验收的结论。

1. **触发边界**：哪些可观察信号应触发，哪些表面相近的请求不应触发？两者的行为差异是什么？
2. **流程完整性**：agent 最可能在哪一步跳过、抢跑或提前宣布完成？什么产物、检查或交接能证明该步完成？
3. **判断与约束**：哪些信号区分默认方向和合法例外？哪些规则来自硬契约，哪些只是偏好、环境限制或待验证假设？
4. **控制与依赖**：哪些步骤只需清晰指令，哪些需要确定性检查、人类决定或固定顺序？依赖缺失时应停止、降级还是请求补充？
5. **人类权限**：agent 必须在哪些节点停下？用户需要看到和决定什么？反馈如何改变下一步？
6. **归位与行为变化**：哪些事实必须留在外部 owner？有无该 skill 时，可观察行为应发生什么变化？
7. **跨次反馈**：只有当目标承诺依赖持续使用经验时，反馈如何收集、升级、失效和删除；否则不设计长期记忆机制。

## 准备度检查

1. 目标是消费者可观察结果，触发与 near miss 能解释边界差异。
2. Workflow Map 能说明起点、完成态、顺序、分支、回流、人类节点和关键交接。
3. 关键步骤有完成证据和失败处理；重要决策能由信号区分默认与例外，控制方式与风险相称。
4. 硬规则、默认值、偏好、环境限制、workaround 和假设没有混写。
5. 人类介入说明何时停、展示什么、决定什么，以及怎样传给下游。
6. 每项资源候选直接支持行为或验收，外部事实没有复制成新的 owner。
7. 评估提示已经转化为当前目标的验证义务，没有被复制成固定案例；验证义务与实际证明保持分开。
8. 剩余待决项不会被实现细节掩盖。
